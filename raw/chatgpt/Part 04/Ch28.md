Chapter 28 — Naming Conventions and When They Matter

Purpose

Names are the first interface most engineers encounter. Before understanding an implementation, they read its identifiers.

This chapter distinguishes names that communicate architectural or domain information from naming conventions that are primarily stylistic. Good naming reduces the need for explanation; stylistic consistency reduces friction. These are related goals, but they are not equally important.

Prerequisites

This chapter assumes familiarity with:

Part I, Chapter 4: Abstraction and information hiding

Part II, Chapter 13: Bounded contexts and ubiquitous language

Part IV, Chapter 27: File and module structure


Chapter 30 expands on the relationship between naming and comments.


---

Key Concepts

Names are part of the API of a codebase.

A good name communicates intent rather than implementation.

Consistency is valuable, but only after clarity.

Names should reflect the problem domain before they reflect programming language mechanics.


---

Decision: Name Things by Their Intent

What it is

An identifier should describe what a thing represents or accomplishes, not how it happens to be implemented.

A reader should be able to understand the purpose of a function, variable, or type without reading its implementation.


---

Why it exists

Reading code is substantially more common than writing it.

Names are therefore an optimization for comprehension rather than typing speed.

A well-chosen name eliminates the need to repeatedly rediscover intent.


---

Options

Domain-oriented names

Names come from the language of the problem being solved.

Examples:

Order

Invoice

Settlement

RetryPolicy



---

Implementation-oriented names

Names describe mechanisms instead of concepts.

Examples:

DataProcessor

Manager

Handler

Util



---

Historical or accidental names

Names persist long after the implementation has changed.

Examples include classes named XmlParser that now parse JSON or methods named LoadCache() that actually perform remote network requests.


---

Trade-offs

Domain-oriented names

Advantages

Align with business concepts.

Easier communication between developers and domain experts.

More resilient to implementation changes.


Disadvantages

Require understanding the domain.

Domain terminology may evolve over time.



---

Implementation-oriented names

Advantages

Easy to invent.

Often describe current implementation accurately.


Disadvantages

Become misleading as implementations change.

Encourage thinking in terms of mechanisms rather than behavior.

Frequently result in vague "manager" or "helper" classes.



---

Historical names

Advantages

Avoid immediate churn.


Disadvantages

Gradually become deceptive.

Increase cognitive load because names and behavior diverge.



---

When to choose each option

Prefer domain terminology whenever the identifier represents a domain concept.

Reserve implementation-oriented names for infrastructure whose purpose is genuinely technical rather than business-oriented.

Rename identifiers when their meaning changes significantly rather than preserving obsolete terminology.


---

Common failure modes

Classes named Manager, Helper, Processor, or Engine gradually accumulate unrelated responsibilities because the names impose almost no semantic constraints.

Variables named after implementation details survive multiple refactorings until the code no longer matches the name.

Teams invent internal terminology instead of using the vocabulary already established by the business, creating unnecessary translation between conversations and code.


---

Example

The emphasis on intention-revealing names originates in the traditions of Extreme Programming and later became one of the central themes of the Clean Code movement.

A function named calculateOutstandingBalance() communicates substantially more than one named processData(), even if both contain similar amounts of code.


---

Decision: Encode Meaning, Not Types

What it is

Names should communicate semantics, not information the language already knows.

Modern type systems already record types. Identifiers should provide information the compiler cannot.


---

Why it exists

Every piece of duplicated information eventually becomes inconsistent.

If both the compiler and the identifier describe a value's type, one of them will eventually be wrong.


---

Options

Intention-revealing names

customer
order
timeout


---

Type-prefixed names

strName
iCount
pBuffer
lpData


---

Mixed conventions

Some codebases retain legacy prefixes while newer code omits them.


---

Trade-offs

Intention-revealing names

Advantages

Stable across refactoring.

Shorter.

Communicate purpose.


Disadvantages

Depend on the language exposing type information elsewhere.



---

Type prefixes

Advantages

Useful in languages where type information is otherwise difficult to discover.

Historically valuable in older C code and some assembly environments.


Disadvantages

Redundant in modern statically typed languages.

Become incorrect after refactoring.

Increase visual noise.



---

Mixed conventions

Advantages

Easier migration of legacy systems.


Disadvantages

Inconsistent expectations.

Different parts of the codebase communicate differently.



---

When to choose each option

In modern languages such as C#, Java, Rust, Go, Kotlin, Swift, TypeScript, and similar environments, avoid encoding types into names.

In traditional C codebases without editor support or strong type visibility, historical conventions were more defensible because they compensated for genuine limitations of the tooling and language.

The historical context explains the convention. It does not justify carrying it forward unchanged.


---

Common failure modes

A variable named strName becomes a richer object but keeps its original prefix.

A pointer becomes a value.

An integer becomes a 64-bit integer.

The prefixes remain while the code evolves, making the names progressively less truthful.


---

Example

Hungarian notation emerged in an era when C offered little assistance in exposing type information. Editors were primitive, IDEs did not exist, and programmers often navigated large source files with minimal tooling.

Those constraints largely disappeared. In modern languages, the compiler and development environment already expose type information, leaving the prefixes to duplicate information that is available elsewhere.


---

Decision: Use Naming Conventions That Convey Meaning

What it is

Some naming conventions encode useful semantic information.

Others merely express stylistic preference.

The two should not be treated as equally important.


---

Why it exists

Conventions are worthwhile when they reduce ambiguity.

They are much less valuable when they merely standardize aesthetics.


---

Options

Semantic conventions

Examples include:

isReady

hasPermission

canRetry

shouldPersist



---

Stylistic conventions

Examples include:

camelCase

snake_case

PascalCase

kebab-case



---

Project-specific conventions

Organizations sometimes establish additional conventions appropriate to their ecosystem.


---

Trade-offs

Semantic conventions

Advantages

Clarify expected behavior.

Reduce ambiguity.

Improve readability in conditional logic.


Disadvantages

Require consistent application.



---

Stylistic conventions

Advantages

Improve consistency.

Reduce visual friction.


Disadvantages

Carry almost no semantic meaning.

Can become sources of unnecessary debate.



---

Custom conventions

Advantages

May integrate well with existing tooling.


Disadvantages

Increase onboarding cost.

Can become isolated from ecosystem norms.



---

When to choose each option

Use semantic conventions consistently because they communicate information.

Follow the idiomatic style of the language for capitalization and formatting.

Do not spend architectural energy debating camelCase versus snake_case unless interoperability or ecosystem expectations require one.

Consistency matters far more than the particular choice.


---

Common failure modes

Boolean variables named enabled or permission leave readers wondering whether the value is a flag or an object.

Names such as valid or ready become confusing when read in conditions:

if (ready)

compared to

if (isReady)

The latter communicates both type and intended interpretation.


---

Example

Across languages, predicates commonly adopt prefixes such as is, has, can, or should.

Unlike capitalization conventions, these prefixes convey additional semantic information rather than merely following style.


---

Decision: Match Name Length to Scope

What it is

Identifier length should correspond to how much context the reader already possesses.

Short-lived local variables can often be brief.

Objects with broader visibility should generally have more descriptive names.


---

Why it exists

Names exist to distinguish concepts.

Inside a five-line loop, very little distinction is required.

Across an entire package, substantially more context must be encoded.


---

Options

Uniform verbosity

Every identifier is fully descriptive regardless of scope.


---

Scope-dependent naming

Short names in small scopes.

Longer names as visibility increases.


---

Aggressively abbreviated naming

Everything is shortened to minimize typing.


---

Trade-offs

Uniform verbosity

Advantages

Predictable.

Self-descriptive.


Disadvantages

Local code becomes unnecessarily noisy.



---

Scope-dependent naming

Advantages

Balances readability with brevity.

Reflects available context.

Produces less visual clutter.


Disadvantages

Requires judgment rather than fixed rules.



---

Aggressive abbreviation

Advantages

Less typing.


Disadvantages

Poor readability.

Difficult onboarding.

Ambiguous meaning.



---

When to choose each option

Use very short names only where the surrounding context makes the meaning immediately obvious.

As visibility expands—from loop variable to function parameter to exported API—the identifier should become increasingly descriptive.


---

Common failure modes

Developers apply one rule everywhere.

Either every variable becomes excessively verbose, or every identifier becomes cryptic.

Neither extreme optimizes comprehension.


---

Example

Idiomatic Go deliberately uses short names such as i and err in limited scopes because their meaning is obvious from nearby context.

At package scope, exported types and functions generally receive much more descriptive names.

Many Java codebases instead favor descriptive identifiers even for relatively local variables. Neither approach is inherently more correct; they optimize for different balances between explicitness and brevity.


---

Decision: Prefer Better Names Over Explanatory Comments

What it is

When a short block of code requires a comment explaining its purpose, first ask whether the code should instead become a well-named function.

This is an introduction to a topic explored more fully in Chapter 30.


---

Why it exists

Comments describe behavior.

Names become behavior.

A reader calling recalculateCreditLimit() rarely needs an accompanying comment explaining what the function does.


---

Options

Explain inline with comments

Leave the code intact and document it.


---

Extract into a named function

Move the behavior behind an intention-revealing name.


---

Leave complex code unchanged

Accept that understanding requires careful reading.


---

Trade-offs

Comments

Advantages

Quick to write.

Helpful for exceptional cases.


Disadvantages

Can become stale.

Duplicate information.



---

Named functions

Advantages

Improve reuse.

Improve readability.

Strengthen abstraction boundaries.


Disadvantages

May introduce excessive indirection if every few lines become another function.



---

No explanation

Advantages

Minimal structure.


Disadvantages

Readers repeatedly rediscover intent.



---

When to choose each option

If the comment merely answers "what does this block do?", consider extracting it into a well-named function.

If the comment explains why an unusual decision exists, the comment is often still valuable. Chapter 30 discusses this distinction in detail.


---

Common failure modes

Large functions accumulate section headers such as:

// Validate input

// Build request

// Retry failed operations

// Persist results

Each heading describes a logical operation that could frequently become a separate function with a meaningful name.


---

Example

Many modern refactoring tools include "Extract Method" as a primary operation because replacing explanatory comments with well-named functions has repeatedly proven to improve readability in long-lived codebases.


---

Why Smart Engineers Disagree

Engineers rarely disagree about the value of intention-revealing names. They disagree about the amount of information a name should contain.

One camp prefers highly descriptive identifiers everywhere, arguing that explicitness reduces ambiguity.

Another prefers concise names when surrounding context already provides the missing information, arguing that excessive verbosity obscures the structure of the code.

The Go community generally favors shorter local names than many Java communities. Neither philosophy rejects clarity; they simply optimize for different assumptions about how much context the reader already has.

Both approaches converge on the same underlying principle: names should communicate enough information to make the surrounding code easy to understand, but no more.


---

Summary

Naming is the least expensive form of information hiding. A well-chosen identifier communicates intent without introducing additional abstractions or documentation.

The most valuable names come from the domain rather than the implementation, matching the ubiquitous language established with domain experts. Modern code should describe what something means, not what type it happens to be. Historical conventions such as Hungarian notation solved real problems in early C environments but largely duplicate information already provided by contemporary languages and tools.

Only a small number of naming conventions carry semantic weight. Predicate prefixes like is, has, and can reduce ambiguity, while choices such as camelCase versus snake_case are matters of consistency rather than correctness.

Finally, names and decomposition reinforce one another. When code requires comments simply to explain what it does, the better solution is often not a better comment but a better name attached to a better abstraction—a topic developed further in Chapter 30.
