Chapter 32 — Error Handling: Typed Errors vs. Exceptions vs. Result Types

Purpose

Every non-trivial program must communicate failure internally before that failure ever reaches a user, another process, or a network boundary.

This chapter examines the language-level mechanisms used to represent and propagate errors within a single codebase. The goal is not to determine whether an operation can fail—that is unavoidable—but how that possibility should be expressed to callers.

Prerequisites

This chapter assumes familiarity with:

Part I, Chapter 7: Fail-fast and failure taxonomy

Part III, Chapter 21: API error contracts


Retries and distributed error handling are covered later. This chapter concerns only in-process communication of failure.


---

Key Concepts

Failure is part of a function's contract.

The mechanism used to represent failure affects readability, correctness, and maintenance.

Visible control flow is often more verbose.

Invisible control flow is often more convenient.

Different categories of failure justify different mechanisms.


---

Decision: Represent Failure with Exceptions or Values

What it is

Modern languages generally fall into three broad approaches:

Unchecked exceptions

Checked exceptions

Error-as-value


Each represents failure differently and asks different things of callers.


---

Why it exists

A caller needs to know:

Can this operation fail?

How is failure communicated?

Am I expected to recover?


Different language designs answer these questions in different ways.


---

Options

Unchecked exceptions

Functions may throw exceptions without declaring them.

Examples include Python, JavaScript, and Java's RuntimeException.


---

Checked exceptions

Functions explicitly declare recoverable exceptions.

The compiler requires callers to acknowledge them.

Historically associated with Java.


---

Error-as-value

Failure is returned as an ordinary value.

Examples include:

Go's (value, error)

Rust's Result<T, E>

Traditional C return codes



---

Trade-offs

Unchecked exceptions

Advantages

Happy path remains uncluttered.

Minimal boilerplate.

Straightforward code for successful execution.


Disadvantages

Control flow becomes invisible.

Callers may not realize failures are possible.

Exceptions can propagate unexpectedly.



---

Checked exceptions

Advantages

Failure is visible in the interface.

Compiler enforces handling.


Disadvantages

Verbose.

Frequently encourages meaningless catch-and-rethrow code.

Difficult to evolve public APIs.



---

Error-as-value

Advantages

Failure is explicit.

Ordinary control flow.

Easy to reason about propagation.


Disadvantages

Boilerplate at call sites.

Happy path becomes visually interrupted.



---

When to choose each option

Choose unchecked exceptions when failures represent programming mistakes or genuinely exceptional situations.

Choose error-as-value when failure is expected as part of normal operation.

Avoid checked exceptions for most new designs unless working within ecosystems where they remain an accepted convention.


---

Common failure modes

An unchecked exception unexpectedly escapes multiple abstraction layers because no intermediate caller realized it could occur.

Conversely, an error-as-value API may accumulate repetitive error forwarding that obscures the primary logic.

Checked-exception systems often accumulate empty catch blocks written only to satisfy the compiler.


---

Example

Traditional C communicates failure through return values combined with errno.

Go adopts the same fundamental philosophy while replacing global state with explicit error return values.

Rust evolves the same approach further through richer types and language support.


---

Decision: Make Expected Failures Explicit

What it is

Routine operational failures should generally appear in the function's normal interface.

Missing files, invalid input, timeouts, or failed validation are ordinary outcomes that callers frequently know how to handle.


---

Why it exists

Expected failures are not exceptional.

Treating them as unusual obscures the normal behavior of the program.

A caller should immediately recognize that handling failure is part of using the function correctly.


---

Options

Return an error value

Represent failure directly.


---

Throw an exception

Interrupt normal execution.


---

Hide failure

Convert failures into default values or silent behavior.


---

Trade-offs

Error values

Advantages

Explicit.

Encourages deliberate handling.

Easier to compose.


Disadvantages

Additional code at every call site.



---

Exceptions

Advantages

Concise success path.


Disadvantages

Failure paths become less visible.

Recovery locations may be distant.



---

Hidden failure

Advantages

Simpler interfaces.


Disadvantages

Silent bugs.

Incorrect behavior.

Difficult debugging.



---

When to choose each option

If the immediate caller is realistically expected to recover, use an explicit mechanism that makes recovery obvious.

If opening a file may legitimately fail because the file does not exist, that possibility belongs in the function's contract.

If parsing user input may fail because users make mistakes, callers should expect to handle that outcome.


---

Common failure modes

Functions silently return empty collections, zero values, or null after failures that callers expected to distinguish.

The program continues running while operating on incorrect assumptions.


---

Example

Rust's Result<T, E> makes the possibility of failure visible in the type signature, requiring callers to consciously decide whether to recover, propagate, or transform the error.


---

Decision: Reserve Exceptions for Exceptional Conditions

What it is

Exceptions are most appropriate when normal local recovery is impossible or undesirable.

They should represent situations where continuing ordinary execution is itself questionable.


---

Why it exists

Exception handling deliberately bypasses ordinary control flow.

That mechanism is justified only when ordinary control flow is not the right model.


---

Options

Exceptions

Represent unexpected or unrecoverable conditions.


---

Error values

Represent expected operational outcomes.


---

Program termination

Abort immediately when execution cannot safely continue.


---

Trade-offs

Exceptions

Advantages

Separate error path.

Minimal impact on successful execution.


Disadvantages

Hidden propagation.

Harder reasoning about control flow.



---

Error values

Advantages

Explicit recovery.

Predictable execution.


Disadvantages

More verbose.



---

Immediate termination

Advantages

Prevents corrupted execution.


Disadvantages

No opportunity for local recovery.



---

When to choose each option

Programming bugs, violated invariants, impossible states, or catastrophic runtime failures naturally align with exceptions or fail-fast termination.

Routine operational failures generally do not.

This aligns with the broader failure philosophy introduced in Chapter 7 without replacing it.


---

Common failure modes

Applications begin throwing exceptions for ordinary validation failures.

Soon nearly every operation requires exception handling despite failures being expected parts of everyday execution.

The distinction between exceptional and routine failure disappears.


---

Example

Attempting to access an element beyond an array's bounds typically indicates a programming error.

Failing to open a user-selected file because it does not exist generally does not.

These failures deserve different mechanisms.


---

Decision: Balance Explicitness Against Boilerplate

What it is

Every error-handling mechanism chooses a different point on the spectrum between visibility and convenience.

No approach eliminates this trade-off.


---

Why it exists

Making failure explicit inevitably requires additional syntax.

Hiding failure paths simplifies successful execution but obscures possible outcomes.

Language designers choose different balances.


---

Options

Prioritize concise success paths

Unchecked exceptions.


---

Prioritize explicit failure handling

Error-as-value.


---

Reduce boilerplate through language features

Typed results combined with dedicated syntax.


---

Trade-offs

Concise success paths

Advantages

Readable primary logic.

Less repetitive code.


Disadvantages

Hidden control flow.



---

Explicit handling

Advantages

Visible contracts.

Easier reasoning.


Disadvantages

Repetitive checks.



---

Language-assisted propagation

Advantages

Preserves explicit contracts.

Reduces repetitive forwarding.


Disadvantages

Requires language support.



---

When to choose each option

If the language already provides ergonomic support for explicit propagation, prefer using it rather than inventing custom abstractions.

Otherwise, accept that some verbosity is the cost of explicit correctness.


---

Common failure modes

Developers create elaborate helper frameworks solely to eliminate a few repeated error checks.

The abstraction becomes harder to understand than the original boilerplate.


---

Example

Go's repeated:

if err != nil {
    return err
}

is perhaps the best-known criticism of error-as-value.

Rust addresses the same concern with the ? operator, allowing explicit propagation while preserving Result<T, E> in the function's type signature.

The underlying philosophy remains the same even though the syntax differs.


---

Decision: Avoid Checked Exceptions for General Error Propagation

What it is

Checked exceptions attempt to make failures visible through the type system while preserving exception-based control flow.

In practice, many ecosystems have moved away from them.


---

Why it exists

The goal was admirable: prevent ignored failures.

The practical result was often excessive ceremony and brittle APIs.


---

Options

Checked exceptions

Compiler-enforced handling.


---

Unchecked exceptions

Optional handling.


---

Error-as-value

Explicit return types.


---

Trade-offs

Checked exceptions

Advantages

Visibility.

Compiler enforcement.


Disadvantages

Verbose signatures.

Frequent catch-and-rethrow patterns.

Difficult API evolution.



---

Unchecked exceptions

Advantages

Simpler interfaces.


Disadvantages

Hidden failure paths.



---

Error-as-value

Advantages

Explicit contracts without exception hierarchies.

Straightforward composition.


Disadvantages

Boilerplate.



---

When to choose each option

Modern languages increasingly favor either unchecked exceptions or typed result values.

Checked exceptions remain appropriate primarily when working within existing ecosystems that already depend upon them.

Designing new systems around checked exceptions is rarely considered best practice today.


---

Common failure modes

Developers catch checked exceptions solely to wrap them in another checked exception or immediately rethrow them.

The compiler's requirement has been satisfied, but no additional understanding or recovery has been gained.


---

Example

Java's checked exceptions were one of the language's most ambitious experiments in reliability.

Over time, many major Java frameworks—including Spring—shifted toward unchecked exceptions because the practical costs of checked exceptions frequently outweighed their intended benefits.


---

Why Smart Engineers Disagree

Most experienced engineers agree that routine failures should not disappear silently.

The disagreement concerns whether making those failures explicit is worth the additional syntax.

Supporters of exceptions argue that successful execution should dominate the appearance of the code and that exceptional paths should remain visually separate.

Supporters of error-as-value argue that failures are part of the contract and therefore belong in the ordinary type system, even if that requires more explicit handling.

Rust demonstrates that language design can soften this trade-off through features like the ? operator. Go deliberately accepts more verbosity in exchange for maximal explicitness. Different languages place the balance at different points, but none eliminate the underlying tension.


---

Summary

Language-level error handling determines how failures move through a codebase before they ever reach an API boundary.

Unchecked exceptions keep successful execution concise but make failure paths implicit. Error-as-value approaches make failures visible in function signatures and ordinary control flow at the cost of additional boilerplate. Checked exceptions attempted to combine compiler enforcement with exceptions, but their practical drawbacks have led many modern ecosystems to move away from them.

The choice of mechanism should reflect the nature of the failure. Routine operational outcomes—missing files, invalid input, network timeouts—are usually best represented explicitly because callers are expected to respond. Programming errors, violated invariants, and genuinely exceptional conditions align naturally with exceptions or fail-fast behavior.

No mechanism is universally superior. Each balances explicitness, readability, and convenience differently. The objective is not to eliminate error handling from the code, but to make the contract between caller and callee accurately reflect the kinds of failures that are realistically expected.
