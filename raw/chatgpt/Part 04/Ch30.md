Chapter 30 — Comments: What to Comment, What Not To

Purpose

Comments exist to communicate information that the code itself cannot.

Unlike code, comments are not executed, type-checked, or validated by the compiler. Every comment therefore becomes a maintenance obligation. This chapter explains when that cost is justified and when it is better to improve the code instead.

Prerequisites

This chapter assumes familiarity with:

Part I, Chapter 4: Information hiding

Part III, Chapter 15: API contracts

Part IV, Chapter 28: Naming conventions


Repository-level design rationale is covered later in Part VI through Architecture Decision Records. API documentation is covered in detail in Part VIII.


---

Key Concepts

Comments are secondary sources of truth.

Every comment must justify its maintenance cost.

Code explains what it does.

Comments should explain what the code cannot.

Public interface documentation serves a different audience and purpose than implementation comments.


---

Decision: Comment the Why, Not the What

What it is

The primary purpose of an implementation comment is to record information that cannot be recovered by simply reading the code.

A comment that merely translates code into English adds maintenance cost without adding knowledge.


---

Why it exists

Source code already describes behavior.

Comments should provide context, constraints, assumptions, or rationale that are invisible from the implementation itself.

Without this distinction, comments become redundant documentation that inevitably diverges from reality.


---

Options

Explain why

Document hidden constraints, design rationale, assumptions, or surprising decisions.


---

Explain what

Describe the operations already visible in the code.


---

No comment

Allow the implementation to speak for itself.


---

Trade-offs

Why-oriented comments

Advantages

Preserve knowledge unavailable elsewhere.

Help future maintainers understand unusual decisions.

Remain valuable even as implementations evolve.


Disadvantages

Still require maintenance.

Can become obsolete if circumstances change.



---

What-oriented comments

Advantages

Easy to write.

May briefly help inexperienced readers.


Disadvantages

Duplicate information.

Become stale quickly.

Increase noise.



---

No comment

Advantages

No maintenance burden.

Forces code to remain understandable.


Disadvantages

Hidden assumptions may disappear with the original author.



---

When to choose each option

Write a comment when answering questions such as:

Why is this implemented this way?

What external constraint requires this?

What invariant must remain true?

What surprising behavior would a careful reader miss?


Do not write comments that merely describe the next few lines of code.


---

Common failure modes

// Increment index
index++;

The comment contributes no information.

By contrast:

// Must update the secondary index first.
// Replication depends on this ordering.

records knowledge unavailable from the statements themselves.


---

Example

The Linux kernel reserves many implementation comments for genuinely non-obvious requirements, such as lock-ordering rules, processor memory-ordering assumptions, or hardware-specific behavior. The comments explain constraints that are not evident from the code alone rather than narrating the implementation.


---

Decision: Treat Comments as Maintenance Liabilities

What it is

Every comment introduces another artifact that must remain synchronized with the implementation.

Unlike code, comments receive no automatic correctness checks.


---

Why it exists

Software evolves.

Comments frequently do not.

A stale explanation is often more dangerous than no explanation because readers naturally assume comments are accurate.


---

Options

Keep comments minimal

Comment only when the benefit clearly exceeds the maintenance cost.


---

Comment extensively

Attempt to document every significant section of code.


---

Remove obsolete comments aggressively

Delete comments that no longer communicate useful information.


---

Trade-offs

Minimal commenting

Advantages

Less maintenance.

Higher average quality.

Less visual clutter.


Disadvantages

Important rationale may occasionally be omitted.



---

Extensive commenting

Advantages

Appears well documented.

Can assist initial onboarding.


Disadvantages

High maintenance burden.

Significant risk of comment rot.

Readers must determine which comments remain trustworthy.



---

Aggressive removal

Advantages

Eliminates misleading documentation.

Keeps surviving comments meaningful.


Disadvantages

Requires discipline during refactoring.



---

When to choose each option

Assume every comment will eventually become outdated unless someone actively maintains it.

If a comment no longer provides unique information, removing it is usually preferable to updating it.


---

Common failure modes

The implementation changes.

The accompanying comment does not.

Future developers trust the comment, introducing new defects because they acted on obsolete information.

Eventually engineers stop believing comments entirely because too many have become unreliable.


---

Example

Many production outages have been prolonged because documentation embedded in code described behavior that had changed years earlier. The misleading comment delayed discovery of the actual implementation.


---

Decision: Distinguish Implementation Comments from Interface Documentation

What it is

Not all comments serve the same audience.

Implementation comments explain code to future maintainers.

Documentation comments describe an interface to its consumers.

These are fundamentally different responsibilities.


---

Why it exists

A maintainer needs to understand internal reasoning.

A caller needs to understand how to use the interface correctly.

Mixing these purposes produces documentation that satisfies neither audience well.


---

Options

Inline implementation comments

Located inside functions or beside implementation details.


---

Documentation comments

Associated with public types, functions, modules, or APIs.


---

No documentation

Rely entirely on names and signatures.


---

Trade-offs

Implementation comments

Advantages

Capture local reasoning.

Explain unusual implementation constraints.


Disadvantages

Easily become stale during refactoring.



---

Documentation comments

Advantages

Define interface expectations.

Improve discoverability.

Become part of the public contract.


Disadvantages

Require stronger maintenance discipline.

Incorrect API documentation misleads every consumer.



---

No documentation

Advantages

Zero maintenance.


Disadvantages

Public interfaces become difficult to discover and use correctly.



---

When to choose each option

Use implementation comments for local reasoning.

Use documentation comments for public interfaces whose consumers should not need to inspect the implementation.

Treat documentation comments as part of the API itself, not as optional implementation notes.


---

Common failure modes

Implementation details leak into public documentation.

Alternatively, API documentation simply repeats the function name without describing behavior, preconditions, side effects, or guarantees.

Neither helps the intended audience.


---

Example

Python's PEP 257 deliberately standardizes docstrings as interface documentation rather than ordinary implementation comments.

A docstring describes what callers should know, not how the function achieves its result.


---

Decision: Improve the Code Before Adding a Comment

What it is

When a comment exists only to explain a block of code, first ask whether the code can be reorganized so the explanation becomes unnecessary.

Frequently the answer is to extract a well-named function.


---

Why it exists

Names become part of the executable program.

Comments remain external explanations.

Whenever the intent can be expressed directly in the structure of the code, that expression is generally more reliable than explanatory prose.


---

Options

Add a comment

Explain the existing implementation.


---

Refactor

Extract functions or reorganize code to communicate intent directly.


---

Leave the code unchanged

Accept that readers must infer the behavior themselves.


---

Trade-offs

Comments

Advantages

Fast.

Low immediate effort.


Disadvantages

Permanent maintenance cost.



---

Refactoring

Advantages

Improves readability.

Eliminates redundant explanation.

Reduces future maintenance.


Disadvantages

Requires more immediate effort.

Can introduce excessive indirection if overused.



---

No action

Advantages

No structural changes.


Disadvantages

Future readers repeatedly rediscover intent.



---

When to choose each option

If the comment merely answers "What does this block do?", prefer improving the code.

If the comment answers "Why must this be done this way?", the comment is often justified because the rationale cannot usually be expressed through structure alone.


---

Common failure modes

Large methods become divided into sections such as:

// Validate request

// Calculate totals

// Persist results

Each section heading often identifies a natural function that should exist independently.


---

Example

Modern IDEs make "Extract Method" one of the most common refactorings precisely because replacing explanatory comments with intention-revealing functions generally improves readability while reducing maintenance burden.


---

Decision: Record Hidden Constraints Explicitly

What it is

Some information cannot reasonably be encoded into names, types, or control flow.

When the implementation depends on invisible constraints, a comment becomes the appropriate place to preserve that knowledge.


---

Why it exists

Certain assumptions originate outside the code itself.

Examples include operating system behavior, hardware quirks, compiler bugs, protocol requirements, or dependencies on third-party systems.

Without documentation, future maintainers may "simplify" code that only appears unnecessary.


---

Options

Document the hidden constraint

Explain the external dependency or invariant.


---

Leave the constraint implicit

Assume readers will discover it independently.


---

Encode the constraint in code

Where practical, enforce the rule programmatically.


---

Trade-offs

Explicit comments

Advantages

Preserve institutional knowledge.

Prevent accidental regressions.


Disadvantages

Must be maintained if circumstances change.



---

Implicit assumptions

Advantages

Less documentation.


Disadvantages

Easy to violate unintentionally.



---

Programmatic enforcement

Advantages

Compiler or runtime enforces correctness.

Reduces reliance on documentation.


Disadvantages

Not every external constraint can be encoded.



---

When to choose each option

Prefer expressing constraints in code whenever possible.

When that is impossible, document the assumption clearly and specifically.

Avoid vague comments such as "Don't change this."

Instead explain what breaks if it changes.


---

Common failure modes

Comments reference long-fixed bugs, obsolete compiler limitations, or removed hardware behavior.

Maintainers cannot determine whether the workaround is still required and become reluctant to modify the surrounding code.


---

Example

Implementation comments explaining that a particular ordering exists because of a processor erratum or an external protocol requirement communicate information that cannot be reconstructed simply by reading the implementation.


---

Why Smart Engineers Disagree

Most experienced engineers agree that comments should not narrate obvious code.

The disagreement concerns how aggressively code should be refactored before comments become acceptable.

Some engineers argue that almost every explanatory comment indicates code that should be reorganized.

Others point out that complex algorithms, hardware interactions, concurrency protocols, or mathematically sophisticated implementations may remain difficult even after careful refactoring.

The practical distinction is not between "comment" and "no comment." It is between comments that preserve knowledge unavailable elsewhere and comments that merely compensate for avoidable complexity.


---

Summary

Comments are expensive because they create a second description of the system that must evolve alongside the code. Unlike the implementation itself, nothing verifies that they remain correct.

The most valuable implementation comments explain why a decision exists, document hidden constraints, preserve non-obvious invariants, or identify external dependencies. Comments that merely restate visible behavior add maintenance cost without increasing understanding.

Implementation comments and documentation comments serve different audiences. Inline comments explain local reasoning to maintainers, while documentation comments describe public contracts to consumers and should be treated as part of the interface itself.

Finally, comments should not become substitutes for better structure. If a block of code needs prose simply to explain what it does, the more durable solution is often to reorganize the code so that its intent is evident from its names and decomposition. Comments should preserve knowledge the code cannot express—not repeat what it already says.
