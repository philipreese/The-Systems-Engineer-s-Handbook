Chapter 27 — File and Module Structure

Purpose

Architecture is only useful if it survives contact with the codebase. Previous chapters established principles such as coupling, cohesion, information hiding, ports and adapters, and dependency direction. This chapter explains how those architectural decisions become physical structure: packages, modules, directories, and import relationships.

A good module structure makes the intended architecture the easiest architecture to maintain. A poor one forces developers to constantly fight the organization of the code.

Prerequisites

This chapter assumes familiarity with:

Part I, Chapter 2: Complexity

Part I, Chapter 3: Coupling and cohesion

Part I, Chapter 4: Abstraction and information hiding

Part II, Chapter 11: Architectural styles (especially hexagonal architecture)

Part II, Chapter 12: Dependency inversion and the dependency rule


Naming within packages is covered in Chapter 28. Deciding when to split a single concept across multiple files is covered in Chapter 29.


---

Key Concepts

A file tree is not merely a storage mechanism. It communicates architectural intent.

A package boundary is an architectural boundary.

Import relationships express dependency direction.

Folder structure should reinforce the architecture rather than merely categorize source files.

Most architectural violations begin as organizational ones before becoming runtime problems.


---

Decision: Organize by Technical Layer or by Feature

What it is

The primary organizational decision is whether packages are organized by technical role or by business capability.

Package-by-layer groups similar technical responsibilities together.

controllers/
services/
repositories/
models/

Package-by-feature groups everything needed for one capability together.

orders/
    controller
    service
    repository

billing/
    controller
    service
    repository

inventory/
    controller
    service
    repository

Both are common. They optimize for different concerns.


---

Why it exists

Every source file belongs somewhere.

The question is whether the primary organizing principle is:

"What kind of code is this?"

or

"What problem does this code solve?"


Those answers produce very different systems as projects grow.


---

Options

Package-by-layer

Organize around technical responsibilities.

Typical examples:

controllers

services

repositories

DTOs

validators


This style became popular through early enterprise frameworks because it mirrored application execution.


---

Package-by-feature

Organize around cohesive business capabilities.

Each feature owns its API, business logic, persistence, tests, and supporting code.

Infrastructure shared across features lives outside the feature packages.


---

Hybrid

Some projects organize application code by feature while keeping genuinely generic infrastructure separate.

Example:

orders/
billing/
inventory/

database/
logging/
metrics/
config/

This is the dominant structure in many mature systems.


---

Trade-offs

Package-by-layer

Advantages

Familiar to many developers.

Easy to locate similar technical components.

Works well when most code performs generic framework responsibilities.


Disadvantages

Business logic becomes fragmented.

Simple changes require editing many unrelated directories.

Technical layers become tightly coupled across the application.



---

Package-by-feature

Advantages

High cohesion.

Changes remain localized.

Easier ownership by teams.

Encourages bounded contexts.

Reduces unnecessary dependencies.


Disadvantages

Some infrastructure code is duplicated.

Shared abstractions require discipline.

Initial organization may appear less uniform.



---

Hybrid

Advantages

Keeps infrastructure centralized.

Preserves feature cohesion.

Avoids unnecessary duplication.


Disadvantages

Requires judgment about what is truly shared.

Easy to accidentally recreate layer-based coupling if everything migrates into shared packages.



---

When to choose each option

Choose package-by-feature when:

Building business applications.

Features evolve independently.

Multiple teams own different domains.

Domain concepts dominate technical concerns.


Choose package-by-layer when:

Building frameworks.

Building reusable libraries.

Building infrastructure components.

The technical role itself is the primary abstraction.


Choose a hybrid when:

Business features are the dominant organization.

Infrastructure components are genuinely generic.


For most application code, package-by-feature has become the prevailing industry practice because it minimizes coordination across unrelated parts of the codebase.


---

Common failure modes

Developers implementing a small feature modify:

controllers

services

repositories

models

DTOs

validators

tests


The feature itself has no home.

Instead, every technical layer accumulates unrelated changes.

Another common symptom is that every service imports every repository because organizational boundaries no longer correspond to conceptual ones.


---

Example

Idiomatic Go projects strongly discourage package-by-layer for application code.

Structures such as

controllers/
models/
utils/

are widely regarded within the Go community as early-project organization that scales poorly.

Instead, Go projects typically organize packages around cohesive capabilities, with infrastructure packages remaining explicitly separated.


---

Decision: Place Ports and Adapters in Separate Packages

What it is

Ports belong with the domain.

Adapters belong with infrastructure.

A package layout implementing hexagonal architecture commonly resembles:

core/
    orders/
        service.go
        order.go
        repository.go   <- port

adapters/
    postgres/
    sqlite/
    redis/
    http/

The dependency direction follows the package structure.


---

Why it exists

Part II established that dependencies point inward.

The physical module structure should make violating this rule difficult.

Interfaces define what the domain needs.

Infrastructure satisfies those interfaces.

The interface therefore belongs with the consumer, not the provider.


---

Options

Ports inside the core

The domain owns its abstractions.

Adapters depend inward.


---

Ports inside infrastructure

Infrastructure defines the interfaces.

Business logic depends outward.

This effectively reverses dependency inversion.


---

Shared interface package

Interfaces live in a common package.

Both domain and infrastructure depend on it.

This is appropriate only when the package truly represents stable contracts rather than miscellaneous abstractions.


---

Trade-offs

Core-owned ports

Advantages

Enforces dependency inversion.

Domain remains technology-independent.

Infrastructure becomes replaceable.


Disadvantages

Infrastructure developers must understand domain interfaces.

Initial design requires more thought.



---

Infrastructure-owned ports

Advantages

Simpler for very small applications.

Easy initial implementation.


Disadvantages

Domain becomes coupled to infrastructure.

Database choices leak into business logic.

Technology changes become expensive.



---

Shared contracts

Advantages

Useful for stable APIs.

Reduces duplication.


Disadvantages

Shared packages often become dumping grounds.

Ownership becomes ambiguous.



---

When to choose each option

Choose core-owned ports for nearly all business systems.

Choose shared contract packages only when the interfaces genuinely represent stable public contracts.

Avoid infrastructure-owned interfaces except in deliberately infrastructure-centric libraries.


---

Common failure modes

Database types appear inside domain packages.

HTTP request objects appear inside business logic.

Infrastructure-specific exceptions propagate into core logic.

Eventually changing databases requires changing domain code because the dependency rule has been violated.


---

Example

Many production systems implementing hexagonal architecture place repository interfaces alongside the domain services that consume them while database implementations live in separate infrastructure packages.

This makes dependency direction visible simply by inspecting imports.


---

Decision: Use Package Boundaries to Enforce Dependency Direction

What it is

Imports should reflect architecture.

Allowed imports should be intentionally constrained.

A developer should be able to determine whether an architectural dependency is legal simply by reading package relationships.


---

Why it exists

Architecture diagrams are descriptive.

Import graphs are executable.

If the compiler permits forbidden dependencies, architectural drift becomes inevitable.

Package organization provides a mechanical enforcement mechanism.


---

Options

Free imports

Any package may import any other.


---

Directed dependency graph

Imports follow explicitly defined dependency direction.


---

Tool-enforced dependency rules

Static analysis rejects forbidden imports.

Many large organizations augment package structure with automated dependency checking.


---

Trade-offs

Free imports

Advantages

Minimal friction.

Fast early development.


Disadvantages

Architectural erosion.

Increasing coupling.

Difficult refactoring.



---

Directed dependencies

Advantages

Architecture becomes visible.

Easier reasoning.

Safer evolution.


Disadvantages

Requires occasional restructuring.

Some convenience imports become impossible.



---

Tool enforcement

Advantages

Prevents accidental violations.

Scales across large teams.


Disadvantages

Additional maintenance.

Rules must evolve with architecture.



---

When to choose each option

Small experiments may tolerate unrestricted imports.

Long-lived production systems should establish dependency direction early.

Large organizations should automate enforcement whenever practical.


---

Common failure modes

Developers bypass intended APIs because importing an internal package is easier.

Over time the architecture diagram and the import graph describe different systems.

Refactoring becomes increasingly risky because dependencies are no longer obvious.


---

Example

Rust's module system encourages explicit visibility through pub and pub use.

Instead of exposing entire internal module trees, libraries commonly re-export selected public APIs while hiding implementation modules.

Information hiding therefore exists not only in types and functions but also in module organization.


---

Decision: Prevent Circular Package Dependencies

What it is

Two packages should not depend on each other.

If package A imports package B while package B imports package A, the architecture almost always contains an incorrect boundary.


---

Why it exists

Circular dependencies eliminate independent reasoning.

Neither package can be understood, tested, or evolved without understanding the other.

This is the package-level equivalent of tightly coupled classes discussed in Part I.


---

Options

Allow cycles

Some languages permit circular package references.


---

Prohibit cycles

Languages or tooling reject them.


---

Refactor shared concepts

Extract the shared responsibility into a separate package or relocate ownership entirely.


---

Trade-offs

Allowing cycles

Advantages

Fast short-term development.


Disadvantages

Increasing architectural entanglement.

Difficult builds.

Difficult testing.

Refactoring becomes progressively harder.



---

Eliminating cycles

Advantages

Clear ownership.

Independent evolution.

Simpler dependency graph.


Disadvantages

Occasionally requires larger restructuring.



---

When to choose each option

Production systems should treat package cycles as design defects rather than inconveniences.

When a cycle appears, first question whether the concept has been split incorrectly before introducing additional abstractions.


---

Common failure modes

Developers solve one cycle by introducing a new shared package.

Eventually the shared package imports everything and everything imports it.

The cycle disappears, but only because the system now revolves around a single enormous dependency.


---

Example

Languages such as Go reject cyclic package imports entirely.

This forces architectural problems to be addressed immediately rather than accumulated over time.


---

Decision: Avoid the "God Package"

What it is

Many projects eventually accumulate packages named:

common

shared

helpers

utilities

misc


These packages contain code unrelated except for lacking an obvious owner.


---

Why it exists

Creating another package requires making an ownership decision.

Adding another utility function does not.

Over time convenience replaces architecture.


---

Options

Central utility package

Everything reusable goes into one location.


---

Domain ownership

Reusable code belongs with the concept that owns it.

Only genuinely generic infrastructure receives dedicated shared packages.


---

Trade-offs

Utility package

Advantages

Easy to discover initially.

Convenient for small projects.


Disadvantages

Ownership disappears.

Dependencies spread everywhere.

Package size grows indefinitely.



---

Domain ownership

Advantages

Strong cohesion.

Clear maintenance responsibility.

Stable architectural boundaries.


Disadvantages

Occasional duplication.

Requires discipline when deciding ownership.



---

When to choose each option

Shared packages should represent stable technical capabilities such as logging, metrics, configuration, or serialization.

Avoid creating generic dumping grounds whose only organizing principle is "used in multiple places."


---

Common failure modes

A package named utils eventually contains:

string helpers

date formatting

SQL generation

HTTP helpers

authentication

configuration

random business rules


No developer understands what belongs there because the package no longer represents a concept.

Eventually nearly every other package imports it, making it the highest-coupled component in the system.


---

Example

Large legacy systems frequently develop a common package that becomes one of the largest directories in the repository.

Attempts to modify it become high-risk because seemingly unrelated systems depend upon it.


---

Why Smart Engineers Disagree

The primary disagreement is not whether package-by-feature is generally superior for application development. Industry practice has increasingly converged on that conclusion.

The disagreement concerns what qualifies as a "feature."

One engineer may organize around business domains such as orders and billing.

Another may organize around bounded contexts defined by organizational ownership.

A framework engineer may reasonably organize around parsers, schedulers, allocators, or protocol implementations because those technical components are the primary domain.

The correct organizing principle follows the primary axis of change. If features evolve independently, organize by feature. If reusable technical components evolve independently, organize by technical role. Problems arise when a project chooses a structure optimized for one axis while most changes occur along another.


---

Summary

File organization is architecture made concrete.

Package-by-feature should be the default organization for most application code because it aligns physical structure with business capabilities and reduces cross-feature coupling.

Package-by-layer remains appropriate where technical roles are themselves the primary abstraction, particularly in frameworks, runtimes, and infrastructure libraries.

Ports belong with the domain. Adapters belong with infrastructure. Dependency direction should be visible in import relationships, not merely documented in diagrams.

Circular package dependencies usually indicate an incorrectly drawn architectural boundary, while ever-growing common or utils packages signal that ownership has been abandoned in favor of convenience.

A well-organized module tree does not merely store code—it constrains future changes so that the intended architecture remains the path of least resistance.
