Chapter 33 — When to Write Unsafe or Low-Level Code

Purpose

Most software should be written within the safety guarantees of its language. Memory safety, type safety, bounds checking, lifetime tracking, and managed runtimes exist because they eliminate entire classes of defects.

This chapter explains the narrow circumstances in which stepping outside those guarantees is justified, and the engineering discipline required when it is. The question is not whether unsafe code is powerful—it is whether the benefits outweigh the loss of compiler-enforced correctness.

Key Concepts

Unsafe code transfers responsibility from the compiler to the programmer.

The unsafe boundary should be as small and auditable as possible.

Unsafe implementations should expose safe interfaces.

Unsafe code requires stronger evidence before it is introduced and stronger verification after it is written.


---

Decision: Stay Within the Language's Safety Guarantees or Bypass Them

What it is

Languages differ in how much protection they provide by default.

Writing unsafe or low-level code means intentionally bypassing some of those protections.

The exact mechanism varies by ecosystem.


---

Why it exists

Safety checks impose constraints.

Occasionally those constraints prevent direct interaction with hardware, foreign code, or highly optimized implementations.

Unsafe facilities exist to enable these exceptional situations without forcing an entire language to abandon safety.


---

Options

Fully safe code

Remain entirely within the language's guarantees.


---

Localized unsafe code

Confine unsafe operations to narrowly scoped regions.


---

Unsafe by default

Work in environments where the programmer bears responsibility everywhere.


---

Trade-offs

Fully safe code

Advantages

Compiler verifies more invariants.

Easier review.

Fewer memory safety bugs.


Disadvantages

Some low-level operations are impossible.

Certain optimizations may remain inaccessible.



---

Localized unsafe code

Advantages

Enables specialized capabilities.

Limits the loss of compiler guarantees.

Easier auditing.


Disadvantages

Requires careful reasoning.

Mistakes have broader consequences.



---

Unsafe by default

Advantages

Maximum control.

Direct hardware access.

Minimal runtime restrictions.


Disadvantages

Programmer assumes responsibility continuously.

Greater risk of memory corruption and undefined behavior.



---

When to choose each option

Remain in safe code unless a concrete requirement cannot reasonably be satisfied otherwise.

If unsafe behavior is necessary, isolate it to the smallest possible region.

Avoid allowing unsafe operations to spread through otherwise safe codebases.


---

Common failure modes

An initially small unsafe implementation gradually expands because each nearby function now "already has access."

Eventually the unsafe region ceases to be identifiable.


---

Example

Rust requires unsafe operations to appear inside explicit unsafe blocks.

The surrounding file remains fully checked.

This deliberate design makes the loss of compiler guarantees visible, local, and auditable.

By contrast, C and C++ operate largely without memory safety guarantees throughout the program, making the entire codebase responsible for maintaining those invariants.


---

Decision: Enter Unsafe Code Only for Demonstrable Need

What it is

Unsafe code should solve a proven problem rather than an anticipated one.

The burden of justification is substantially higher than for ordinary implementation choices.


---

Why it exists

Unsafe code increases maintenance cost and defect risk.

Those costs should purchase something concrete.


---

Options

Stay with the safe implementation

Accept the language's guarantees.


---

Introduce unsafe code after measurement

Optimize only after evidence identifies the safe abstraction as the bottleneck.


---

Introduce unsafe code preemptively

Assume lower-level code will eventually be necessary.


---

Trade-offs

Safe implementation

Advantages

Simpler maintenance.

Strong compiler assistance.

Lower defect rates.


Disadvantages

May leave some performance unrealized.



---

Measurement-driven unsafe code

Advantages

Solves verified problems.

Justifies increased complexity.

Limits unnecessary risk.


Disadvantages

Requires investigation before implementation.



---

Speculative unsafe code

Advantages

May appear highly optimized.


Disadvantages

Often solves imaginary problems.

Permanently increases maintenance burden.



---

When to choose each option

Choose unsafe code only when one of the following applies:

Profiling demonstrates that a safety mechanism is the actual bottleneck.

An FFI boundary requires interaction with native code.

A foundational low-level component must itself perform operations unavailable in safe code.


Mechanical sympathy should justify unsafe code only after measurement confirms the suspected cost.


---

Common failure modes

Developers assume bounds checking, garbage collection, or runtime safety must be responsible for poor performance.

After profiling, the real bottleneck proves to be disk access, networking, or algorithmic complexity.

The unsafe optimization delivered additional risk but negligible benefit.


---

Example

A developer measures a hot loop and demonstrates that repeated bounds checks dominate execution time.

Only after confirming this through profiling does the implementation replace the checked operation with a carefully verified unsafe equivalent.

The unsafe code exists because the cost was measured, not guessed.


---

Decision: Build Safe Abstractions on Top of Unsafe Implementations

What it is

Unsafe code should rarely become part of an application's everyday programming model.

Instead, it should support higher-level interfaces that remain entirely safe for ordinary callers.


---

Why it exists

Most developers should never need to reason about pointer validity, memory ownership, or unchecked indexing.

The complexity belongs inside a carefully designed implementation.


---

Options

Expose unsafe operations directly

Require callers to manage invariants themselves.


---

Hide unsafe code behind a safe API

Unsafe implementation, safe public contract.


---

Eliminate unsafe code entirely

Remain completely within safe language facilities.


---

Trade-offs

Direct exposure

Advantages

Maximum flexibility.


Disadvantages

Every caller must reason about correctness.

Defects propagate outward.



---

Safe wrapper

Advantages

Centralizes verification.

Reduces misuse.

Limits unsafe reasoning to one location.


Disadvantages

Requires careful API design.



---

Fully safe implementation

Advantages

Strong compiler guarantees.


Disadvantages

Some capabilities remain unavailable.



---

When to choose each option

Whenever unsafe code cannot be avoided, encapsulate it immediately behind a safe interface.

Only a small number of engineers should ever interact with the unsafe implementation directly.


---

Common failure modes

Unsafe pointers, raw memory, or unchecked operations leak into higher layers.

Soon every consumer becomes responsible for maintaining invariants originally confined to one implementation.


---

Example

Most standard libraries implement collections, allocators, or runtime primitives using carefully reviewed unsafe code internally while presenting interfaces that ordinary programmers use without ever writing unsafe code themselves.

This mirrors the "thin binding, thick wrapper" approach discussed for FFI boundaries in Chapter 26.


---

Decision: Document Invariants the Compiler Can No Longer Verify

What it is

Unsafe code frequently depends on assumptions that the compiler cannot check.

Those assumptions become part of the implementation's correctness.


---

Why it exists

Inside safe code, the compiler enforces many invariants automatically.

Unsafe operations intentionally disable some of those guarantees.

Future maintainers therefore need explicit documentation describing what must remain true.


---

Options

Document required invariants

State exactly what assumptions make the code safe.


---

Rely on implementation details

Assume readers will infer the requirements.


---

Encode invariants in surrounding safe code

Enforce as many assumptions as possible before entering unsafe regions.


---

Trade-offs

Explicit documentation

Advantages

Improves review.

Preserves reasoning.

Prevents accidental violations.


Disadvantages

Requires maintenance.



---

Implicit assumptions

Advantages

Less documentation.


Disadvantages

Easy to violate unknowingly.



---

Safe enforcement

Advantages

Compiler assists whenever possible.

Reduces reliance on documentation.


Disadvantages

Not every invariant can be expressed mechanically.



---

When to choose each option

Every unsafe block should have a clearly understood answer to the question:

"What assumptions make this code safe?"

If the answer cannot be stated clearly, the implementation is unlikely to be ready.


---

Common failure modes

Months later, another engineer modifies surrounding code without realizing an undocumented pointer validity assumption has been broken.

The resulting defect appears far from the actual change.


---

Example

A concise comment documenting that a pointer remains valid because ownership has not yet been transferred preserves information the compiler can no longer verify automatically.


---

Decision: Hold Unsafe Code to a Higher Standard

What it is

Unsafe code deserves more scrutiny than ordinary application code.

The loss of compiler guarantees should be compensated for through engineering discipline.


---

Why it exists

Mistakes in unsafe code frequently produce consequences far beyond ordinary defects.

Memory corruption, undefined behavior, and security vulnerabilities often originate in small regions of low-level code.


---

Options

Treat unsafe code like ordinary code

Normal review and testing.


---

Apply elevated standards

Additional review, testing, and verification.


---

Avoid unsafe code entirely

Refuse any implementation requiring it.


---

Trade-offs

Ordinary standards

Advantages

Lower development cost.


Disadvantages

Greater risk for high-consequence defects.



---

Elevated standards

Advantages

Improved confidence.

Better long-term maintainability.

Easier auditing.


Disadvantages

Additional engineering effort.



---

Complete avoidance

Advantages

Maximum safety.


Disadvantages

Some legitimate capabilities become unavailable.



---

When to choose each option

Unsafe implementations should receive:

more careful code review,

stronger testing,

more explicit documentation,

narrower ownership,


than ordinary application code.

The greater the reduction in compiler guarantees, the greater the obligation to compensate through process.


---

Common failure modes

Unsafe helper functions proliferate throughout a repository until reviewers stop treating them as exceptional.

Eventually unsafe code becomes ordinary, defeating the purpose of making it explicit in the first place.


---

Example

Rust's explicit unsafe blocks naturally support focused code review because reviewers can immediately identify precisely where additional reasoning is required rather than auditing an entire module.


---

Why Smart Engineers Disagree

Most experienced engineers agree that unnecessary unsafe code should be avoided.

The disagreement concerns where the boundary lies.

Some engineers argue that carefully written unsafe implementations are justified whenever they produce measurable performance improvements.

Others place greater value on long-term maintainability and require much stronger evidence before relinquishing compiler guarantees.

Both positions generally converge on one principle: unsafe code should remain exceptional. The debate concerns how compelling the justification must be before accepting the additional responsibility.


---

Summary

Unsafe programming is not an alternative style of software development. It is an exception to the normal safety model of a language.

Legitimate reasons for crossing that boundary are limited: measured performance bottlenecks attributable to safe abstractions, unavoidable interaction with foreign code, and the implementation of foundational low-level components that expose safe interfaces to everyone else.

When unsafe code is necessary, it should be confined to the smallest practical region, immediately wrapped by a safe public API, and accompanied by clear documentation describing the invariants the compiler can no longer verify. That mirrors the same architectural pattern used at FFI boundaries: concentrate unavoidable unsafety into a small, auditable core while allowing the rest of the system to remain safe.

Ultimately, unsafe code transfers responsibility from the compiler to the programmer. Because that transfer carries real cost and risk, it should occur only when the benefits have been demonstrated and only with correspondingly higher standards of review, testing, and documentation.
