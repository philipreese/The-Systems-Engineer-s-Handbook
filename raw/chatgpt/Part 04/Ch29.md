Chapter 29 — When to Split Files vs. Keep Together

Purpose

Once packages and modules have been organized, a more localized question remains: how much should live in a single file?

This is not an architectural decision in the sense of Chapter 27. The package boundary already exists. The question here is whether a single concept should be spread across multiple files or kept together.

The governing principle is the same one introduced earlier for abstraction: split when evidence shows multiple concerns exist, not because a style guide says every declaration deserves its own file.

Prerequisites

This chapter assumes familiarity with:

Part I, Chapter 3: Coupling and cohesion

Part I, Chapter 4: Information hiding and the Rule of Three

Part IV, Chapter 27: File and module structure

Part IV, Chapter 28: Naming conventions


Test organization is covered in greater depth in Part V.


---

Key Concepts

A file is a unit of comprehension.

Files should maximize cohesion rather than satisfy arbitrary size limits.

Splitting files is an organizational decision, not automatically an abstraction decision.

The Rule of Three applies here: avoid preemptive decomposition until the need is demonstrated.


---

Decision: One Class Per File or One Concept Per File?

What it is

Different ecosystems disagree on what a source file represents.

Some treat a file as the home of exactly one primary type.

Others treat it as a container for one cohesive concept, even if that concept consists of multiple closely related types and functions.


---

Why it exists

Files exist to help humans navigate a codebase.

The question is whether navigation is improved by maximizing consistency or by maximizing locality.


---

Options

One class per file

Each significant type receives its own source file.

Supporting types are minimized or extracted separately.


---

One cohesive concept per file

A file contains whatever declarations are needed to understand one idea.

This may include several small types, helper functions, constants, or internal state.


---

Hybrid

Large public types receive dedicated files.

Small supporting types remain nearby until they demonstrate independent value.


---

Trade-offs

One class per file

Advantages

Predictable navigation.

Easy file discovery.

Simple tooling.

Works well for large object-oriented systems.


Disadvantages

Related concepts become physically separated.

Reading a feature often requires opening many files.

Small supporting types become scattered.



---

One concept per file

Advantages

High locality.

Related code is read together.

Small helper types remain close to their consumers.


Disadvantages

Files may become larger.

Requires judgment about conceptual boundaries.

Less predictable navigation.



---

Hybrid

Advantages

Balances locality with discoverability.

Supports gradual evolution.


Disadvantages

Depends on consistent engineering judgment rather than rigid rules.



---

When to choose each option

Use one class per file when types have independent responsibilities or public APIs.

Keep related declarations together when separating them would force readers to constantly switch between files without improving understanding.

Do not split solely because another declaration was added.

Split when cohesion decreases, not when declaration count increases.


---

Common failure modes

Developers mechanically create a new file for every enum, exception, helper type, or small interface.

A feature requiring ten minutes to understand now requires opening fifteen files.

The organizational rule has increased navigation cost without increasing clarity.

The opposite failure is equally common: unrelated types accumulate indefinitely because "they already happen to be in the same file."


---

Example

Java conventionally encourages one public class per file.

Go, Python, and JavaScript more commonly treat a file as the implementation of one cohesive concept, allowing several related declarations to coexist naturally.

Neither convention is inherently superior. Each reflects different assumptions about how developers navigate code.


---

Decision: Split When Cohesion Falls

What it is

Files should be split because they contain multiple concerns, not because they exceed an arbitrary length.

Low cohesion is the primary signal.


---

Why it exists

Length alone rarely makes code difficult.

Unrelated ideas do.

A thousand-line parser may remain understandable if it implements one cohesive algorithm.

A two-hundred-line file mixing authentication, logging, configuration, and business rules is already difficult to reason about.


---

Options

Split by responsibility

Each cohesive concern receives its own file.


---

Keep related responsibilities together

Closely cooperating code remains together despite increasing size.


---

Split by arbitrary thresholds

Maximum line counts or declaration counts determine when files are divided.


---

Trade-offs

Responsibility-driven splitting

Advantages

Reflects conceptual boundaries.

Improves maintainability.

Aligns with cohesion.


Disadvantages

Requires judgment.

Different engineers may choose different boundaries.



---

Keeping cohesive code together

Advantages

Preserves locality.

Reduces navigation overhead.


Disadvantages

Large files require careful organization.



---

Threshold-based splitting

Advantages

Easy to automate.

Simple code review rules.


Disadvantages

Ignores actual design.

Encourages meaningless decomposition.



---

When to choose each option

Split when readers naturally think, "This part is about something different."

Do not split simply because a file crossed an arbitrary number of lines.

The Rule of Three applies here as well. Wait until multiple independent responsibilities have clearly emerged before introducing additional files.


---

Common failure modes

A growing file reaches five hundred lines.

Rather than identifying conceptual boundaries, developers divide it into "Part1," "Part2," and "Helpers."

The original cohesion problem remains unchanged.


---

Example

Many long-lived legacy systems contain enormous files that accumulated unrelated functionality over years.

The difficulty is rarely their size alone—it is that no single organizing concept remains.


---

Decision: Keep Tightly Coupled Pieces Together

What it is

Some code loses clarity when separated.

Small helper types, finite state machines, or tightly coordinated internal logic are often easier to understand when viewed together.


---

Why it exists

Navigation has a cost.

Every additional file introduces another place readers must search.

When two pieces are almost always read together, separation may reduce rather than increase clarity.


---

Options

Separate every declaration

Each piece receives an independent file.


---

Keep tightly related declarations together

Supporting code remains adjacent to the primary implementation.


---

Merge aggressively

Everything related to a feature remains in one growing file.


---

Trade-offs

Separate everything

Advantages

Predictable organization.

Independent editing.


Disadvantages

Excessive navigation.

Fragmented understanding.



---

Keep tightly related code together

Advantages

Excellent locality.

Easier comprehension.

Less context switching.


Disadvantages

Files become moderately larger.



---

Merge aggressively

Advantages

Maximum locality.


Disadvantages

Risks creating a god file.



---

When to choose each option

Keep code together when understanding one piece almost always requires reading the others.

Small state machines are a common example. States, transitions, and transition logic frequently make more sense when presented together than when distributed across multiple files.

Similarly, tiny helper types used exclusively by one implementation often belong beside it.


---

Common failure modes

An implementation becomes fragmented into dozens of tiny files.

Every edit requires repeatedly navigating between them because none contains enough context to stand alone.


---

Example

Many parser implementations deliberately keep token definitions, parser state, and transition logic in one file because the three concepts are inseparable during maintenance.


---

Decision: Apply the Rule of Three Before Splitting

What it is

Do not split files preemptively.

Allow the code to demonstrate that separate responsibilities actually exist.


---

Why it exists

Premature organization produces the same problems as premature abstraction.

Every additional file introduces another organizational boundary that must later be maintained.


---

Options

Split immediately

Assume future growth.


---

Wait for evidence

Split after repeated pressure demonstrates distinct concerns.


---

Never split

Keep everything together indefinitely.


---

Trade-offs

Immediate splitting

Advantages

Future growth appears accommodated.


Disadvantages

Frequently predicts the wrong evolution.

Produces unnecessary structure.



---

Evidence-driven splitting

Advantages

Reflects actual maintenance patterns.

Avoids speculative organization.


Disadvantages

Requires occasional refactoring later.



---

Never splitting

Advantages

Minimal organization overhead.


Disadvantages

Files eventually become unmanageable.



---

When to choose each option

Prefer evidence over prediction.

If developers repeatedly modify different portions of a file independently, or if unrelated responsibilities continue growing in parallel, the boundary has likely emerged naturally.


---

Common failure modes

Engineers anticipate future complexity that never arrives.

Years later, dozens of tiny files each contain ten lines of code because the expected growth never occurred.


---

Example

The Rule of Three from Chapter 4 applies equally here. Organizational boundaries should emerge from recurring maintenance needs rather than speculation.


---

Decision: Co-Locate Closely Related Assets

What it is

The question of cohesion extends beyond source files.

Tests, stylesheets, templates, and configuration may either live beside the implementation they support or inside centralized directories.


---

Why it exists

Developers typically modify these artifacts together.

Physical proximity can therefore improve maintainability.


---

Options

Co-location

A feature contains its implementation and directly related assets.

Button/
    Button.tsx
    Button.css
    Button.test.ts


---

Parallel trees

components/
styles/
tests/


---

Trade-offs

Co-location

Advantages

High cohesion.

Easier feature maintenance.

Clear ownership.


Disadvantages

More directories.

Shared assets require deliberate extraction.



---

Parallel trees

Advantages

Uniform organization.

Easy browsing by artifact type.


Disadvantages

Feature changes span multiple directories.

Ownership becomes less obvious.



---

When to choose each option

Co-location generally works well when assets primarily belong to one feature.

Shared infrastructure may justify centralized organization.

Test placement itself is discussed in Part V, but it follows the same cohesion-first reasoning.


---

Common failure modes

A simple UI change requires editing four unrelated directory trees because implementation, styling, tests, and assets have all been organized independently of one another.


---

Example

Modern frontend frameworks commonly organize components, styles, documentation, and tests within a single feature directory.

This represents a deliberate shift toward cohesion over artifact-type organization.


---

Why Smart Engineers Disagree

Most experienced engineers agree that cohesion should determine file boundaries.

The disagreement concerns how much locality is desirable before navigation becomes difficult.

Developers from Java ecosystems often value highly predictable file layouts where every major type has an obvious location.

Developers from Go or Python ecosystems frequently prefer reading several closely related declarations in one place rather than navigating through numerous small files.

These preferences are shaped partly by language conventions and tooling, but both camps are ultimately optimizing the same goal: minimizing the effort required to understand and modify a cohesive piece of software.


---

Summary

A file is a unit of comprehension, not a unit of syntax.

The decision to split or combine files should be driven by cohesion rather than arbitrary conventions or line counts. A file containing one coherent concept may legitimately be large, while a much smaller file containing unrelated responsibilities already deserves decomposition.

One-class-per-file conventions provide predictable navigation and suit some ecosystems well. Concept-oriented files emphasize locality and are common in languages such as Go, Python, and JavaScript. Both approaches succeed when they preserve cohesion.

The Rule of Three applies here just as it does to abstraction. Resist speculative decomposition. Split files when repeated maintenance demonstrates distinct concerns, not because future growth is imagined.

Finally, cohesion extends beyond source code. The same reasoning that keeps related types together often justifies co-locating tests, styles, and other feature-specific assets. The principle remains consistent across all of these decisions: keep together the things that are understood and changed together.
