Chapter 42 — Issue Tracking: What Makes a Good Issue

Purpose

An issue is the smallest planning artifact in an engineering process. It exists to describe a piece of work that someone can understand, perform, verify, and close. Every larger planning construct—iterations, milestones, roadmaps, and release plans—is ultimately composed of issues. If the unit is poorly defined, every higher-level process inherits that ambiguity.

This chapter defines what makes an issue useful, how to scope one appropriately, and why maintaining a healthy issue tracker requires continual deletion as well as creation.

Prerequisites

This chapter assumes familiarity with:

Part I, Chapter 1's principle that every engineering activity should optimize for an explicit objective.

Part I, Chapter 2's treatment of complexity as a cost that must be justified.

Part I, Chapter 9's decision frameworks and the distinction between reversible and irreversible decisions.

Part IV, Chapter 30's distinction between implementation documentation and design rationale.


Larger design work discussed here may reference specifications (expanded in Chapter 46), but the issue itself is not a specification.

Key Concepts

The central ideas are:

An issue describes work, not intent.

Issues describe problems before solutions.

Every issue has an objectively checkable definition of done.

Every issue should represent one bounded unit of work.

Collections of issues decay unless actively maintained.



---

Actionable Issues

What it is

An actionable issue contains enough information that someone other than its author can complete the work without requiring hidden context.

The issue is a shared engineering artifact, not a personal reminder.

Why it exists

Engineering teams change over time. People go on vacation, switch projects, or leave the organization entirely. If only the original author understands what an issue means, the tracker ceases to function as shared knowledge.

The issue should answer:

What problem exists?

Why does it matter?

What observable outcome constitutes completion?


Everything else is secondary.

Options

Minimal reminder

Examples:

"Fix login page."

"Improve caching."

"Clean up parser."


Useful only for the person who wrote it.

Actionable issue

Contains:

problem statement

relevant context

constraints

acceptance criteria

references if necessary


Trade-offs

Minimal issues

Pros:

fast to create

little process overhead


Cons:

cannot be delegated

context is lost over time

difficult to prioritize accurately



---

Detailed issues

Pros:

transferable

easier to estimate

easier to review after completion


Cons:

higher upfront effort

unnecessary for trivial work



---

Issue templates

Pros:

improve consistency

reduce forgotten information

scale well across teams


Cons:

encourage mechanical form-filling

can become excessively long


When to choose each option

A private TODO list can tolerate minimal reminders because the author already possesses the missing context.

A shared issue tracker should almost always contain actionable issues. If another engineer cannot reasonably begin the work after reading it, the issue is incomplete.

Issue templates are most valuable for recurring work such as bug reports, feature requests, and operational incidents.

Common failure modes

The most common symptom is an issue that generates a comment thread asking basic questions:

"What exactly is broken?"

"How do I reproduce this?"

"Is this still relevant?"


The discussion gradually becomes the real issue while the original description becomes obsolete.

Example

GitHub Issues deliberately uses a simple issue model while allowing repositories to define templates that enforce required information. Rather than making every field mandatory in the software, the repository chooses which information matters for its workflow.


---

Describe the Problem Before the Solution

What it is

An issue should begin with the problem being solved rather than immediately prescribing an implementation.

Why it exists

The first proposed solution is often not the best one.

By recording the problem independently, engineers remain free to discover simpler or more robust implementations during development.

Options

Solution-first issue

"Replace Redis with PostgreSQL."

Problem-first issue

"Current cache invalidation occasionally serves stale data after writes. Reduce stale reads to zero under normal operation."

Trade-offs

Solution-first

Pros:

immediately actionable

useful when the solution is already constrained


Cons:

discourages better alternatives

hides assumptions

difficult to revisit later



---

Problem-first

Pros:

preserves engineering flexibility

easier to evaluate success

enables alternative designs


Cons:

sometimes requires additional design work

may require a separate specification for large changes


When to choose each option

State the problem first unless the implementation is externally mandated—for example, migrating to an unsupported library version or complying with a regulatory requirement.

Common failure modes

Teams become committed to implementations that no longer make sense because the issue itself encodes the solution instead of the underlying objective.

Example

Many successful open-source projects describe observable deficiencies rather than prescribing patches. Contributors frequently solve the stated problem in ways different from what the original reporter imagined.


---

Acceptance Criteria

What it is

Acceptance criteria define the observable conditions that determine whether an issue is complete.

Why it exists

Without objective completion criteria, "done" becomes a matter of opinion.

Acceptance criteria convert completion into something another engineer can independently verify.

Options

Implicit completion

"The feature works."

Explicit completion

Examples:

User can reset a password without contacting support.

API returns HTTP 404 for nonexistent resources.

Operation completes within two seconds under the stated workload.


Trade-offs

Implicit criteria

Pros:

quick to write


Cons:

inconsistent interpretations

recurring reopenings

difficult QA



---

Explicit criteria

Pros:

objective closure

easier testing

easier review


Cons:

requires more thought

occasionally needs revision as understanding improves


When to choose each option

Explicit acceptance criteria should be the default for shared work.

Large design questions may additionally reference a specification (covered in Chapter 46), but the issue should still contain enough criteria to determine whether it can be closed.

Common failure modes

Issues repeatedly reopen because different engineers disagree about whether the intended behavior has actually been achieved.

Example

Many production teams define issue completion around observable system behavior rather than implementation details, allowing internal refactoring without changing the definition of done.

Bug, Feature, and Task Are Different Kinds of Work

What it is

Not every issue represents the same category of engineering work.

Why it exists

Different kinds of work require different information and have different definitions of completion.

Options

Bug

Requires:

reproduction steps

expected behavior

observed behavior

environment

logs or screenshots when appropriate


Definition of done:

The incorrect behavior no longer occurs.


---

Feature

Requires:

problem statement

desired capability

acceptance criteria

constraints


Definition of done:

The new capability exists and satisfies the stated acceptance criteria.


---

Task or chore

Requires:

objective

scope

completion condition


Definition of done:

The maintenance activity has been completed.

Trade-offs

Using a single generic issue type simplifies tooling but loses information.

Distinct issue types require slightly more discipline while producing much better issue quality.

When to choose each option

Choose the issue type based on the nature of the work rather than organizational preference.

A bug is not merely "important work." It describes incorrect existing behavior.

A feature introduces new behavior.

A task changes engineering assets without directly changing externally visible functionality.

Common failure modes

Feature requests are reported as bugs to receive higher priority.

Maintenance work is disguised as features.

Bug reports omit reproduction information because they reuse feature templates.

Example

Joel Spolsky argued in Painless Bug Tracking that a useful bug report begins with reproducible behavior rather than speculation about the cause. Reproduction steps, expected behavior, observed behavior, and environment remain the standard structure more than two decades later.


---

Scope Issues Around Deliverable Work

What it is

One issue should represent one bounded piece of work that can realistically be completed and closed.

Why it exists

Issues exist to move from open to closed.

An issue with no natural completion point becomes permanent inventory.

Options

Bounded issue

Examples:

Add request timeout to authentication service.

Fix race condition in cache invalidation.

Support CSV export.


Theme

Examples:

Improve performance.

Clean up technical debt.

Modernize infrastructure.


Themes are collections of future issues, not issues themselves.

Trade-offs

Large issues

Pros:

fewer tracker entries

simpler planning initially


Cons:

unclear progress

difficult estimation

rarely finish completely



---

Small issues

Pros:

easier delegation

measurable completion

predictable progress


Cons:

more tracking overhead

excessive fragmentation if taken too far


When to choose each option

If completion cannot be objectively recognized, the work should be decomposed.

If multiple engineers could independently complete different portions simultaneously, multiple issues probably exist.

Common failure modes

Issues remain open for years while continually accumulating unrelated work.

Eventually nobody remembers what completion would even mean.

Example

Long-running infrastructure programs inside large organizations are typically managed as collections of independently closable work items rather than one perpetual "Improve infrastructure" issue.


---

The Tracker Is a Living Collection

What it is

Creating issues is inexpensive.

Maintaining a useful issue tracker is not.

Why it exists

Every open issue imposes maintenance cost:

prioritization

duplicate detection

searching

planning

cognitive overhead


As the backlog grows, signal decreases.

Options

Complete historical backlog

Never delete anything.


---

Continuously pruned backlog

Regularly close:

obsolete issues

duplicate reports

superseded work

intentionally rejected proposals


Trade-offs

Complete backlog

Pros:

nothing is forgotten

complete historical record


Cons:

overwhelming search results

misleading priorities

stale information dominates active work



---

Pruned backlog

Pros:

reflects actual intentions

easier planning

better prioritization


Cons:

requires continuous maintenance

occasionally closes something later reconsidered


When to choose each option

Treat "won't do" as a legitimate engineering outcome rather than a failure.

An issue tracker is a decision-making tool, not an archive of every idea ever proposed.

Common failure modes

The classic write-only tracker contains hundreds or thousands of issues that nobody has reviewed in years.

Engineers stop trusting the backlog because it no longer represents planned work.

Eventually new issues become difficult to distinguish from abandoned ones.

Example

Linear explicitly embraces backlog decay, automatically archiving stale issues unless they receive renewed attention. This is a tooling decision that reflects a process philosophy: inactive work should disappear unless someone still intends to do it.

The same reasoning appeared earlier in Chapter 39 regarding obsolete tests. Removal is sometimes the correct maintenance action.


---

Mechanical Enforcement Improves Process

What it is

Whenever issue quality depends entirely on memory, quality varies significantly.

Mechanically enforcing minimum requirements reduces this variability.

Why it exists

Principle 8 established that systems should enforce important invariants automatically whenever practical. The same principle applies to engineering process.

Options

Manual discipline

Everyone remembers what good issues look like.

Tool-supported discipline

Templates require fields appropriate for each issue type.

Trade-offs

Manual discipline

Pros:

flexible

low tooling complexity


Cons:

inconsistent quality

repeated reviewer effort



---

Templates

Pros:

standardized information

easier triage

easier automation


Cons:

can become bureaucratic

excessive required fields discourage reporting


When to choose each option

Templates work best when they request only information that is routinely necessary.

Every mandatory field should justify its existence.

Common failure modes

Organizations gradually expand templates until reporters spend more time satisfying the process than describing the actual problem.

Example

GitHub supports repository-specific issue templates through its ISSUE_TEMPLATE mechanism, allowing teams to require reproduction steps for bugs while using a different structure for feature requests. The mechanism is intentionally lightweight, leaving workflow decisions to repository maintainers.


---

Why Smart Engineers Disagree

Much of the disagreement around issue tracking is philosophical rather than technical.

One end of the spectrum is represented by GitHub Issues: a deliberately minimal model with few built-in workflow assumptions. Teams adopt conventions only when they find them valuable.

The opposite end is represented by Jira, whose configurable workflows, custom fields, permissions, and state machines allow organizations to encode highly structured processes directly into the tracker.

Neither philosophy is universally correct.

Minimal systems reduce process overhead and are well suited to smaller teams or projects that value flexibility.

Highly configurable systems provide consistency, traceability, and governance for organizations whose coordination costs dominate individual productivity.

The important decision is not which tool is "better," but whether the structure imposed by the tracker removes more coordination cost than it introduces. That broader question—when process justifies its own complexity—is deferred until Chapter 49.
