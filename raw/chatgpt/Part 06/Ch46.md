Chapter 46 — Spec-First Development

Purpose

A specification is a proposal for work that has not yet been implemented. Its purpose is not to document code, but to expose design flaws before code exists.

The central economic argument is simple: defects discovered in a specification cost dramatically less to correct than defects discovered after implementation. Spec-first development moves design review to the point where changes are cheapest.

Prerequisites

This chapter assumes familiarity with:

Part I, Chapter 1's principle of explicit optimization targets.

Part I, Chapter 6's principle that process should exist only when it provides more value than it costs.

Part I, Chapter 9's reversibility × blast-radius framework.

Chapter 42's definition of issues.

Chapter 45's distinction between proposals and Architecture Decision Records.


Documentation taxonomy is covered in Part VIII.

Key Concepts

The central ideas are:

A specification is a proposal, not a historical record.

The purpose of a spec is to surface disagreement before implementation.

Non-goals are as important as goals.

The weight of a specification should match the weight of the decision.

Once implementation begins, the code—not the spec—becomes the authoritative description of behavior.



---

A Specification Is a Design Proposal

What it is

A specification describes proposed work before implementation begins.

It explains:

the problem

the constraints

the proposed design

why this design is preferable to the alternatives


It is not an implementation guide, nor is it intended to become permanent documentation.

Why it exists

The cheapest design review occurs before code is written.

A reviewer can reject or improve an idea in minutes when it exists as a document.

The same disagreement discovered during code review may require days of redesign, reimplementation, and retesting.

Options

Spec-first development

Review the design before implementation.

Implementation-first development

Allow design review to occur primarily through code review.

Trade-offs

Spec-first

Pros:

inexpensive design iteration

shared understanding

clearer scope

fewer architectural surprises


Cons:

upfront documentation effort

unnecessary for trivial work



---

Implementation-first

Pros:

immediate progress

little initial overhead


Cons:

expensive late feedback

architectural disagreements surface after substantial work

higher rework cost


When to choose each option

When implementation represents a meaningful design decision rather than straightforward execution, write the proposal first.

Small bug fixes and routine implementation work rarely justify a separate specification.

Common failure modes

A reviewer recognizes a flawed design only after thousands of lines of code have already been written.

Technically the review succeeded.

Economically it occurred at the worst possible time.

Example

Amazon's "working backwards" process deliberately begins with written artifacts rather than implementation, forcing discussion before engineering investment.


---

A Spec Exists to Surface Disagreement

What it is

The primary function of a specification is to expose disagreement while changing direction is still inexpensive.

Why it exists

Most significant engineering mistakes are not implementation mistakes.

They are design mistakes.

Specifications provide a place for reviewers to challenge assumptions before implementation creates inertia.

Options

Early design review

Disagreement occurs during proposal review.

Late design review

Disagreement occurs during implementation review.

Trade-offs

Early review

Pros:

inexpensive revisions

collaborative exploration

reduced implementation churn


Cons:

additional planning before coding



---

Late review

Pros:

immediate implementation

less initial documentation


Cons:

costly redesign

wasted implementation effort

discourages substantial reviewer feedback


When to choose each option

Design questions should be resolved before code exists whenever the answers materially affect architecture, interfaces, or system behavior.

Implementation review should focus on whether the implementation faithfully realizes an already-understood design.

Common failure modes

Reviewers recognize better approaches but hesitate to request substantial redesign because doing so would invalidate weeks of implementation work.

The review becomes an evaluation of sunk cost rather than engineering quality.

Example

Google's published design document practices emphasize reaching agreement on architecture before implementation begins, reducing large design debates during code review.


---

What Belongs in a Specification

What it is

A useful specification communicates enough information for reviewers to evaluate the proposal without prescribing implementation details.

A typical specification includes:

problem statement

goals

non-goals

constraints

proposed design

interfaces and data flow

alternatives considered

open questions


Why it exists

Each section answers a different engineering question.

Together they allow reviewers to evaluate both the proposal and its assumptions.

Options

Problem-focused specification

Explain why the design exists and how it behaves.

Implementation-focused specification

Describe classes, functions, algorithms, and internal code organization.

Trade-offs

Problem-focused

Pros:

implementation flexibility

architecture review

durable reasoning


Cons:

requires disciplined abstraction



---

Implementation-focused

Pros:

concrete

easy to visualize


Cons:

duplicates implementation

becomes obsolete quickly

encourages premature design detail


When to choose each option

Describe interfaces, responsibilities, and information flow.

Leave implementation details to the implementation itself unless those details materially affect the architectural decision.

Common failure modes

Specifications become pseudocode.

Reviewers spend time debating implementation details before agreeing on whether the overall design is appropriate.

Example

Google design documents generally emphasize system behavior, interfaces, goals, non-goals, and alternatives while intentionally avoiding unnecessary implementation detail.


---

Non-Goals Prevent Scope Creep

What it is

A non-goal explicitly states what the proposed work will not attempt.

Why it exists

Scope expands naturally.

Without explicit boundaries, reviewers and stakeholders continuously suggest adjacent improvements until the proposal no longer represents a coherent piece of work.

Options

Explicit non-goals

The specification identifies work intentionally excluded.

Implicit boundaries

Readers infer what is outside scope.

Trade-offs

Explicit non-goals

Pros:

protects schedule

reduces misunderstanding

simplifies review


Cons:

requires deliberate prioritization



---

Implicit boundaries

Pros:

shorter documents


Cons:

continual scope expansion

repeated clarification

planning instability


When to choose each option

Every substantial specification should contain a non-goals section.

A reviewer should immediately understand not only what the proposal intends to accomplish, but what it deliberately leaves for future work.

Common failure modes

Review discussions gradually accumulate unrelated feature requests because nothing explicitly states where the proposal ends.

Example

Many successful design document templates place goals and non-goals adjacent to one another because together they define the proposal's true scope.

Specification Weight Should Match Decision Weight

What it is

Not every engineering task deserves the same level of design documentation.

The effort invested in writing a specification should be proportional to the cost of making the wrong decision.

Why it exists

Documentation is process.

Like every process, it should justify its own cost.

Options

Heavyweight design document

Comprehensive proposal with formal review.

Lightweight proposal

Brief design captured in the issue.

No specification

Implementation proceeds directly.

Trade-offs

Heavyweight specifications

Pros:

thorough review

broad visibility

appropriate for major architectural decisions


Cons:

slower progress

greater documentation effort



---

Lightweight proposals

Pros:

quick

sufficient for bounded work

low overhead


Cons:

less architectural exploration



---

No specification

Pros:

fastest execution

no documentation cost


Cons:

design review deferred to implementation

unsuitable for significant changes


When to choose each option

Use the decision framework introduced in Chapter 9.

Hard-to-reverse decisions with broad blast radius deserve comprehensive design review.

Bounded feature work often needs only a short proposal within the issue itself.

Routine bug fixes generally need no separate specification.

Common failure modes

Teams requiring extensive design documents for every change gradually produce documentation few people read.

Conversely, teams writing no specifications eventually conduct architecture review through pull requests, where changing direction is most expensive.

Example

The Internet Engineering Task Force's RFC process, Python Enhancement Proposals, and Rust RFCs all deliberately require substantial proposal review because programming language evolution is expensive to reverse.


---

A Specification Is Not Living Documentation

What it is

A specification describes a proposed future.

Once implementation is complete, it has served its purpose.

Why it exists

Implementation inevitably evolves.

If specifications attempt to remain synchronized with code indefinitely, they eventually become inaccurate or require continuous maintenance.

Neither outcome serves readers.

Options

Proposal artifact

The specification records the proposal that guided implementation.

Living design document

The specification is continuously updated to match the code.

Trade-offs

Proposal artifact

Pros:

historically accurate

low maintenance

preserves original design intent


Cons:

no longer describes the current implementation



---

Living specification

Pros:

theoretically current


Cons:

expensive maintenance

frequently diverges

unclear authority when disagreements arise


When to choose each option

Treat specifications as proposal artifacts.

Long-term truth lives in:

the implementation

the tests

Architecture Decision Records documenting important decisions


A specification may provide useful historical context, but it should not become the authoritative description of the running system.

Common failure modes

The frozen spec.

The software evolves while the specification quietly remains unchanged.

Eventually readers trust a document that no longer reflects reality.

Example

Many engineering organizations archive completed design documents rather than continually updating them, recognizing that their primary value lies in recording the proposal that shaped the implementation.


---

Commit Once Enough Is Known

What it is

Specifications exist to improve decisions, not to postpone them indefinitely.

Why it exists

Analysis has diminishing returns.

Once the major uncertainties have been resolved, additional discussion often produces little additional value.

Options

Decision-oriented specification process

Review until sufficient confidence exists, then implement.

Open-ended specification process

Continue refining the proposal without committing.

Trade-offs

Decision-oriented

Pros:

maintains momentum

captures useful feedback

avoids unnecessary delay


Cons:

some uncertainty remains



---

Open-ended

Pros:

explores additional alternatives

increases confidence


Cons:

delayed execution

continual redesign

no natural stopping point


When to choose each option

For reversible decisions, prefer implementation once the proposal is good enough.

Reserve lengthy specification cycles for decisions whose consequences are genuinely expensive to reverse.

Common failure modes

Analysis paralysis.

The specification becomes an excuse to avoid making a decision.

Review cycles continue long after they stop producing meaningful new information.

Example

The RFC processes used by language communities intentionally move slowly because language changes affect millions of users and are difficult to reverse. Applying the same process to ordinary application features would impose costs far exceeding the associated risk.


---

Why Smart Engineers Disagree

Most engineers agree that major architectural work benefits from written proposals. The disagreement concerns where the threshold lies.

Some organizations encourage writing a specification before nearly every feature, believing the discipline improves engineering quality across the board.

Others reserve specifications for only the largest changes, arguing that documentation overhead grows faster than its value.

Both approaches become dysfunctional when applied indiscriminately. Heavyweight specifications for every bug fix create spec theater: documents written only because the process requires them. No specifications at all create the opposite failure: architecture review migrates into pull requests, where disagreement arrives after implementation has made change expensive.

The appropriate question is not whether a specification is "good practice." The question is whether the expected reduction in implementation risk exceeds the cost of writing and reviewing it. As with every engineering process, the artifact should earn its existence.
