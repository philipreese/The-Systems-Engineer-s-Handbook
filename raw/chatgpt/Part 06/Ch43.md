Chapter 43 — Issue as Tracking Unit vs. PR as Review Unit

Purpose

An engineering process has two fundamentally different units of work:

the issue, which records intent and defines the desired outcome

the pull request (PR), which presents a bounded implementation for review


Many teams implicitly treat these as the same thing. They are not. Confusing them produces either planning artifacts that no longer represent the work being done or code reviews too large to review effectively.

This chapter establishes the distinction and explains why the relationship between issues and PRs is naturally one-to-many.

Prerequisites

This chapter assumes familiarity with:

Chapter 42's definition of an issue as a bounded, verifiable unit of work.

Part I, Chapter 1's distinction between optimizing for outcomes versus optimizing for proxy metrics.

Part I, Chapter 2's discussion of complexity as a cost.

Part I, Chapter 9's decision frameworks around reversibility and incremental change.


The mechanics of branches, merging, and Git workflows are covered in Part VII. Review quality and review standards are covered in Chapter 47.

Key Concepts

The central ideas are:

Issues and PRs solve different problems.

Issues are units of intent; PRs are units of review.

The natural mapping is one issue to many PRs.

Review quality declines as changes become larger.

Mechanical linkage between issues and PRs preserves engineering history.



---

Issue: The Unit of Intent

What it is

An issue records why work exists.

It defines:

the problem

the desired outcome

acceptance criteria

the conditions under which the work is complete


It is closed when those outcomes have been achieved, regardless of how many implementation steps were required.

Why it exists

Engineering work begins with intent rather than code.

The issue answers questions such as:

Why are we making this change?

What problem are we solving?

How do we know when we're finished?


These questions remain meaningful long after the implementation details have changed.

Options

Intent recorded in an issue

The motivation exists independently of implementation.

Intent recorded only in code history

The only explanation is scattered across commits and review discussions.

Trade-offs

Issue-centered tracking

Pros:

preserves rationale

survives refactoring

supports planning

supports delegation


Cons:

requires maintaining another artifact



---

Code-history-only tracking

Pros:

less process

fewer artifacts


Cons:

poor discoverability

difficult historical reconstruction

no explicit definition of completion


When to choose each option

Any work that multiple people may plan, prioritize, or revisit should begin with an issue.

Very small personal projects may reasonably skip formal issue tracking because the author's memory provides the missing context.

Common failure modes

Teams begin treating merged code as documentation of intent.

Months later, engineers can reconstruct how something changed but not why the change was made.

Example

Many mature open-source projects use issues as the long-lived record of motivation, while implementation evolves across multiple review cycles.


---

Pull Request: The Unit of Review

What it is

A PR presents a specific implementation change for human review.

It answers different questions:

Is this implementation correct?

Is it understandable?

Is it safe to merge?

Does it satisfy the intended behavior?


Its lifecycle ends when the change is merged or abandoned.

Why it exists

Review is fundamentally a human activity.

Human attention is limited.

A review unit should therefore be sized for human comprehension rather than project planning.

Options

Small reviewable PR

One coherent implementation step.

Large feature PR

Entire feature delivered in a single review.

Trade-offs

Small PRs

Pros:

easier review

faster feedback

fewer merge conflicts

simpler rollback


Cons:

more individual reviews

additional coordination



---

Large PRs

Pros:

fewer review requests

fewer intermediate merges


Cons:

reduced review quality

slower turnaround

higher review fatigue

increased integration risk


When to choose each option

A PR should contain approximately one coherent implementation idea.

If reviewers cannot reasonably understand the entire change during one review session, the change is probably too large.

Common failure modes

Reviewers skim very large diffs because reviewing every line carefully becomes impractical.

Approval gradually shifts from verification toward trust.

Example

Google's engineering organization deliberately encourages relatively small changes with rapid review turnaround. Their published work on modern code review describes this as an organizational practice rather than merely a tooling preference.


---

One Issue, Many Pull Requests

What it is

The natural relationship between issues and PRs is one-to-many.

An issue may require several implementation steps before its acceptance criteria are satisfied.

Why it exists

Planning and review optimize for different objectives.

Issues optimize for representing complete work.

PRs optimize for reviewing understandable changes.

Those objectives rarely produce identical boundaries.

Options

One issue → one PR

Planning and review share identical boundaries.

One issue → multiple PRs

Implementation proceeds incrementally while preserving a single statement of intent.

Trade-offs

One-to-one mapping

Pros:

simple bookkeeping

easy mental model


Cons:

encourages oversized reviews

discourages incremental delivery



---

One-to-many mapping

Pros:

reviewable changes

continuous integration

easier testing

safer deployment


Cons:

requires explicit linkage

slightly more coordination


When to choose each option

A one-to-one mapping is reasonable only when both the implementation and review naturally fit into a single bounded change.

Otherwise, preserve the issue and divide the implementation into multiple PRs.

Common failure modes

Two opposite failures appear repeatedly.

The first is the monster PR.

The issue represents appropriate work, but the entire implementation is delivered as one enormous review containing thousands of changed lines.

The second is the fragmented issue tracker.

Instead of dividing implementation, teams divide intent into dozens of artificial issues whose only purpose is matching convenient review boundaries.

The tracker gradually stops representing meaningful engineering goals.

Example

Large infrastructure migrations frequently involve dozens of independently reviewable changes while remaining one logical engineering objective.


---

Review Quality Declines as Changes Grow

What it is

Empirical research consistently shows that reviewers become less effective as review size increases.

Why it exists

Human attention does not scale linearly with diff size.

Beyond a certain point, reviewers cannot maintain detailed scrutiny.

Options

Small reviews with rapid turnaround

Large reviews performed infrequently

Trade-offs

Small reviews

Pros:

higher defect detection

lower reviewer fatigue

quicker iteration


Cons:

more review events



---

Large reviews

Pros:

reduced administrative overhead


Cons:

substantially lower review effectiveness

delayed feedback

increased approval by exhaustion


When to choose each option

Prefer splitting changes before review rather than expecting reviewers to compensate through greater effort.

The burden belongs on change authors, not reviewers.

Common failure modes

Large reviews often receive only superficial comments despite containing many independent logical changes.

The apparent review coverage remains high while actual scrutiny falls dramatically.

Example

The SmartBear and Cisco code review studies found that defect detection falls sharply once reviews reach roughly 200–400 lines. Google's published code review practices similarly emphasize relatively small changes and fast review cycles rather than infrequent large reviews.


---

Techniques for One-Issue, Many-PR Development

What it is

Several engineering practices allow work to be divided into reviewable increments while preserving one coherent objective.

Why it exists

Incremental development reduces review complexity without sacrificing overall progress.

Options

Incremental PRs

Each merge leaves the system in a working state.


---

Feature flags

Implementation is merged before the functionality is enabled.


---

Stacked PRs

Dependent implementation steps are reviewed as a sequence of smaller changes rather than one large diff.

Trade-offs

Incremental PRs

Pros:

continuous integration

straightforward review

reduced merge conflicts


Cons:

requires careful decomposition



---

Feature flags

Pros:

merge incomplete work safely

reduce long-lived branches


Cons:

temporary complexity

flags require eventual removal



---

Stacked PRs

Pros:

preserve logical decomposition

manageable reviews

maintain development momentum


Cons:

additional tooling or workflow discipline

dependency management between reviews


When to choose each option

Use incremental PRs whenever work can leave the system functional.

Use feature flags when implementation must land before activation.

Use stacked PRs when later changes depend directly on earlier implementation.

Common failure modes

Teams postpone merging until every dependent change is complete, allowing branches and review size to grow continuously.

Example

The popularity of tools such as Graphite reflects the reality that many engineering tasks naturally require sequences of dependent, reviewable PRs rather than one monolithic change.


---

Link Intent to Implementation

What it is

Every PR should explicitly reference the issue that motivated it.

The issue records intent.

The PR records implementation.

The linkage connects the two.

Why it exists

Without explicit linkage, engineering history fragments.

Readers must reconstruct relationships by searching commit history, review discussions, and timestamps.

Options

Explicit issue linkage

PR references the motivating issue.

Implicit relationship

Reviewers infer the relationship manually.

Trade-offs

Explicit linkage

Pros:

complete traceability

easier navigation

automatic status updates

better historical archaeology


Cons:

minor author discipline required



---

Implicit linkage

Pros:

slightly less effort


Cons:

disconnected history

difficult maintenance

lost rationale


When to choose each option

Use explicit linkage by default.

The overhead is negligible compared with the long-term value of preserving engineering intent.

Common failure modes

The PR-only workflow.

The organization gradually abandons issues altogether.

Months later, engineers can find implementation history but cannot determine what problem the work was originally intended to solve.

Example

GitHub supports issue-closing keywords such as fixes #123 and closes #123, allowing merged PRs to automatically close the corresponding issue. This is a mechanical enforcement of the relationship between intent and implementation rather than relying on manual bookkeeping.


---

Why Smart Engineers Disagree

The disagreement is usually not about whether small PRs are desirable. It is about where the cost should be paid.

Some engineers prefer one PR per issue because it minimizes bookkeeping and keeps planning artifacts simple.

Others accept additional PRs because they view review quality as the limiting resource. From this perspective, splitting implementation is cheaper than asking reviewers to understand increasingly complex changes.

The empirical evidence generally favors the second position. Human review effectiveness degrades with change size, while planning quality suffers when issues are artificially divided to match implementation boundaries.

The useful compromise is to preserve stable issue boundaries while allowing implementation to proceed through however many reviewable PRs the work naturally requires. Issues represent engineering intent; PRs represent engineering execution. Treating them as different units allows each to optimize for the purpose it actually serves.
