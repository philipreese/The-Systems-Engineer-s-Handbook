Chapter 47 — Code Review

Purpose

Code review is one of the most common engineering practices and one of the least examined. Most teams perform it because they always have, without agreeing on what it is actually intended to accomplish.

That purpose matters. A review process optimized to catch bugs looks very different from one optimized to improve code health and build shared ownership.

This chapter defines what code review is for, what standards reviewers should apply, and how teams should spend their limited review attention.

Prerequisites

This chapter assumes familiarity with:

Part I, Chapter 8's principle that anything a machine can verify should be automated.

Part I, Chapter 9's discussion of proxy metrics.

Chapter 43's distinction between issues and pull requests, including the evidence that smaller PRs receive better review.

Chapter 46's role of specifications in resolving design disagreements before implementation.


Branching, merge mechanics, commit conventions, and CI policy are covered in Part VII.

Key Concepts

The central ideas are:

Code review primarily improves code health and shared understanding.

The review standard is improvement, not perfection.

Human attention should be reserved for human judgment.

Small, fast reviews outperform large, slow ones.

Approval is not the same as scrutiny.



---

The Primary Purpose of Code Review

What it is

Code review is a structured human examination of a proposed implementation before it becomes part of the shared codebase.

Why it exists

Code review serves several purposes, but not equally.

Its most durable contributions are:

improving long-term code health

spreading knowledge across the team

maintaining shared ownership


It also detects defects, but many implementation errors are already found by types, tests, static analysis, and automated verification.

Review contributes most where human judgment remains necessary.

Options

Review optimized for code health

Treat review as collaborative improvement.

Review optimized for defect detection

Treat review primarily as manual bug finding.

Trade-offs

Code-health focus

Pros:

improves maintainability

builds shared understanding

encourages constructive discussion

scales over time


Cons:

may miss some implementation defects that tests also miss



---

Bug-hunting focus

Pros:

emphasizes correctness

encourages careful examination


Cons:

encourages adversarial reviews

overestimates what humans reliably detect

discourages collaborative improvement


When to choose each option

Every review should consider correctness.

However, the review process itself should primarily optimize for long-term code quality and team understanding rather than assuming humans are the primary defect-detection mechanism.

Common failure modes

Reviewers spend substantial effort searching for isolated implementation mistakes while overlooking architectural problems, confusing APIs, or unclear abstractions that will affect the codebase for years.

Example

Google's published research on modern code review identifies code health and knowledge transfer as central long-term benefits of review, with defect detection representing only part of its overall value.


---

Approve Improvements, Not Perfection

What it is

A reviewer should approve a change when it clearly improves the overall health of the codebase.

The question is not:

"Is this exactly how I would have implemented it?"

The question is:

"Is the codebase better after this change than before?"

Why it exists

Perfect code does not exist.

Waiting for perfection prevents continuous improvement.

Review should increase quality, not demand an unattainable endpoint.

Options

Improvement standard

Approve changes that clearly improve the system.

Perfection standard

Require every concern to be resolved before approval.

Trade-offs

Improvement standard

Pros:

continuous progress

faster iteration

healthier engineering culture


Cons:

some imperfections remain



---

Perfection standard

Pros:

maximizes individual change quality


Cons:

stalled reviews

discourages authors

encourages endless revisions

delays valuable improvements


When to choose each option

Block changes only when unresolved concerns materially affect correctness, maintainability, readability, or future evolution.

Preference-level observations should remain suggestions.

Common failure modes

Authors spend multiple review rounds satisfying increasingly subjective comments that produce little measurable improvement.

The discussion gradually shifts from engineering quality to personal preference.

Example

Google's engineering practices explicitly instruct reviewers to approve changes that improve overall code health even when they are not perfect. Incremental improvement is preferred over waiting indefinitely for an ideal solution.


---

Human Review Should Focus on Human Judgment

What it is

Review should examine questions that cannot be answered reliably by automation.

Why it exists

Human attention is the most expensive resource in the review process.

Every minute spent discussing formatting is a minute unavailable for evaluating architecture, correctness, or readability.

Options

Automate mechanical checks

Formatting, style, and simple correctness rules are verified automatically.

Review everything manually

Humans examine both stylistic and architectural concerns.

Trade-offs

Automation-first

Pros:

consistent enforcement

faster reviews

preserves reviewer attention

fewer repetitive discussions


Cons:

requires tooling investment



---

Manual enforcement

Pros:

no automation setup


Cons:

repetitive review comments

inconsistent application

poor use of reviewer time


When to choose each option

Anything that can be verified mechanically should not consume review attention.

Reviewers should focus on:

correctness

design

readability

maintainability

test adequacy


Common failure modes

The reviewer leaves twenty comments about formatting while making no observations about the architecture, interface design, or testing strategy.

The review appears thorough while contributing little value.

Example

Tools such as gofmt, Black, and Prettier largely removed formatting discussions from entire language communities, allowing reviewers to concentrate on questions requiring human judgment.


---

Review Size and Review Latency Matter More Than Most Policies

What it is

Two variables dominate review quality:

change size

review turnaround time


Why it exists

Large reviews overwhelm human attention.

Slow reviews interrupt engineering flow and increase the cost of every subsequent change.

Options

Small, rapidly reviewed PRs

Frequent reviews completed quickly.

Large, slow reviews

Substantial changes reviewed over multiple days.

Trade-offs

Small and fast

Pros:

better review quality

lower context switching

faster feedback

easier iteration


Cons:

more review events



---

Large and slow

Pros:

fewer review requests


Cons:

reduced scrutiny

review fatigue

rebasing

context loss

slower delivery


When to choose each option

As discussed in Chapter 43, review quality declines as change size increases.

The remaining variable under team control is latency.

Treat review turnaround as a team service-level objective measured in hours rather than days.

Common failure modes

An engineer waits several days for review.

By the time comments arrive, they have already switched contexts, forgotten implementation details, and accumulated additional changes.

Every review round becomes progressively slower.

Example

Google's engineering practices emphasize rapid review turnaround because review delay compounds throughout the development process rather than affecting only a single change.


---

Distinguish Blocking Issues from Preferences

What it is

Not every review comment carries equal weight.

Some observations identify problems that must be corrected.

Others simply express reviewer preference.

Why it exists

Conflating preferences with correctness unnecessarily blocks engineering progress.

Reviewers should communicate the severity of comments explicitly.

Options

Severity-aware feedback

Differentiate blocking concerns from optional suggestions.

Uniform feedback

Treat every comment as equally important.

Trade-offs

Severity-aware

Pros:

clearer expectations

faster reviews

less unnecessary debate


Cons:

reviewers must exercise judgment



---

Uniform comments

Pros:

simple process


Cons:

uncertainty for authors

unnecessary revisions

inconsistent review outcomes


When to choose each option

Blocking comments should identify issues that materially affect the quality of the change.

Minor wording, naming, or stylistic preferences should be labeled accordingly—commonly with conventions such as "nit:"—so authors understand they are optional.

Common failure modes

Authors cannot distinguish mandatory changes from casual suggestions.

Every comment effectively becomes a review blocker regardless of importance.

Example

Many engineering organizations adopt explicit conventions for non-blocking comments to separate engineering requirements from personal preference.


---

Review Is Collaborative, Not Gatekeeping

What it is

Review is a collaborative engineering activity whose purpose is improving the shared codebase.

It is not an opportunity to assert authority.

Why it exists

Every engineer both reviews and is reviewed.

The process succeeds only if participants trust that criticism serves the code rather than individual status.

Options

Collaborative review

Reviewer and author jointly improve the implementation.

Gatekeeping review

Approval becomes a demonstration of authority or expertise.

Trade-offs

Collaborative review

Pros:

encourages learning

builds trust

spreads ownership

improves long-term quality


Cons:

requires constructive communication



---

Gatekeeping

Pros:

may produce strict compliance


Cons:

discourages contribution

slows delivery

concentrates knowledge

creates adversarial relationships


When to choose each option

Review comments should explain engineering reasoning rather than personal preference or organizational hierarchy.

The discussion should remain centered on improving the code.

Common failure modes

Experienced reviewers repeatedly require changes because they would have written the code differently, even when the proposed implementation is already clear, correct, and maintainable.

The review becomes an exercise in stylistic conformity rather than engineering improvement.

Example

Healthy review cultures routinely accept multiple valid implementations while focusing discussion on objective engineering concerns.


---

Why Smart Engineers Disagree

Most experienced engineers support code review. The disagreement concerns how demanding the review process should be.

Some reviewers believe every imperfection should be corrected before approval because merged code is difficult to revisit.

Others argue that insisting on perfection slows engineering progress while providing diminishing returns.

The evidence generally favors the second position. Codebases improve through continuous incremental refinement rather than isolated perfect changes. Waiting for perfect code delays improvements that are already objectively beneficial.

There is a similar disagreement around approval requirements. Organizations often measure the number of required approvals because it is easy to count. Chapter 9 warned that proxy metrics become unreliable once they become mandatory targets. The same applies here. Requiring two approvals guarantees only that two approvals exist—not that meaningful review occurred. Rubber-stamping is the natural outcome when approval itself becomes the objective rather than careful engineering judgment.

A healthy review culture optimizes for thoughtful, timely evaluation that leaves the codebase in better condition than before. It treats reviewer attention as scarce, reserves that attention for questions requiring human judgment, and measures success by improved software rather than completed process steps.
