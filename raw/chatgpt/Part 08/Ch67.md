Chapter 67 — API Documentation: What Consumers Need

Purpose

API documentation serves a fundamentally different audience from every other documentation artifact in this Part. Most documentation assumes the reader can inspect the implementation if something is unclear. API documentation often cannot make that assumption.

An API consumer may be using a published library, integrating with a remote service, or working outside the organization entirely. The documentation is not supplemental to the implementation—it is the primary interface through which the API is understood.

This chapter examines what information API consumers actually need, how that information should be maintained, and which portions should be generated rather than written by hand.


---

Prerequisites

This chapter assumes familiarity with:

Chapter 20's API modeling principles.

Chapter 21's error contract design.

Chapter 24's authentication boundaries.

Chapter 25's distinction between internal and external APIs.

Chapter 66's discussion of keeping living documentation accurate.



---

Key Concepts

API documentation is part of the public interface.

Reference documentation should be mechanically generated whenever possible.

Narrative documentation complements generated reference rather than replacing it.

Precise documentation reduces the risks described by Hyrum's Law.

Examples are often more valuable than descriptions.



---

API Documentation Has a Different Reader

Unlike an engineer reading internal documentation, an API consumer frequently lacks three important advantages:

access to the implementation,

knowledge of internal conventions,

the ability to ask the development team for clarification.


The documentation must therefore answer questions that internal documentation can safely leave to the source code.

This difference explains why API documentation deserves separate treatment instead of being another instance of a README.


---

Decision: Document the Contract, Not the Implementation

What it is

API documentation should describe the externally observable behavior of the interface rather than its internal implementation.

Why it exists

Consumers depend on the API contract.

They should not need to understand the implementation to integrate successfully.

The documentation should therefore explain everything the consumer is expected to rely upon—and nothing that they are not.

The design of those contracts was established in Part III. This chapter concerns communicating them.


---

Options

A complete API reference typically documents:

each operation,

request structure,

response structure,

parameters,

authentication requirements,

error behavior,

expected status codes,

usage examples.


Internal algorithms and implementation details generally do not belong in the reference.


---

Trade-offs

Documenting the public contract

Advantages:

Stable consumer expectations

Clear integration guidance

Reduced support burden


Disadvantages:

Requires maintenance alongside interface evolution



---

Documenting implementation details

Advantages:

May satisfy technical curiosity


Disadvantages:

Encourages consumers to depend on non-contract behavior

Becomes obsolete quickly



---

Minimal documentation

Advantages:

Lower documentation effort


Disadvantages:

Higher integration cost

Greater likelihood of incorrect assumptions



---

When to choose each option

Document everything consumers are expected to depend upon.

Do not document internal implementation choices unless they intentionally form part of the public contract.


---

Common failure modes

Reference documentation describing internal caching strategies.

Omitting authentication requirements.

Leaving error responses undocumented.

Explaining implementation while neglecting observable behavior.


---

Example

The public APIs of GitHub describe requests, responses, authentication, and observable behavior. Consumers interact with the documented contract rather than GitHub's internal implementation.


---

Decision: Prefer Generated Reference Documentation

What it is

Generate mechanical aspects of API documentation directly from machine-readable interface definitions or source code whenever tooling permits.

Why it exists

Reference documentation duplicates information already present elsewhere.

Duplicated information inevitably drifts.

Generated documentation eliminates much of this duplication by deriving documentation directly from the authoritative interface.

This is another application of Principle 8: keep information as close as possible to its source of truth.


---

Options

Generated reference

Derived from:

OpenAPI specifications,

source annotations,

type definitions,

documentation comments.


Hand-written reference

Maintained independently from implementation.

Hybrid approach

Generate structural reference while writing explanatory material manually.


---

Trade-offs

Generated documentation

Advantages:

Cannot drift from signatures

Consistent formatting

Lower maintenance cost

Accurate parameter and type information


Disadvantages:

Limited explanatory capability

Often lacks integration guidance



---

Hand-written reference

Advantages:

Flexible presentation

Rich explanation


Disadvantages:

Higher maintenance burden

Greater risk of drift

Duplicates interface information



---

Hybrid documentation

Advantages:

Accurate reference

Useful explanation


Disadvantages:

Requires maintaining the narrative layer



---

When to choose each option

Generate:

endpoints,

methods,

parameters,

request schemas,

response schemas,

types,

status codes,


whenever tooling supports it.

Write prose only for information generation cannot produce.


---

Common failure modes

Parameter descriptions inconsistent with implementation.

Documentation listing removed fields.

Examples using obsolete endpoints.

Reference pages maintained manually despite machine-readable interface definitions already existing.


---

Example

The OpenAPI Specification has become the dominant mechanism for generating REST API reference documentation. Tooling built around OpenAPI produces accurate endpoint descriptions directly from the machine-readable contract, substantially reducing documentation drift.


---

Decision: Write Narrative Only Where Generation Cannot Help

What it is

Reserve hand-written documentation for information that requires human explanation.

Why it exists

Consumers need more than a list of endpoints.

They need context.

Generated documentation can explain what exists.

It cannot explain:

why an API is organized a particular way,

how features fit together,

common integration workflows,

recommended usage patterns,

where new users should begin.


Those remain narrative documents.


---

Options

Typical narrative content includes:

getting started guides,

authentication walkthroughs,

complete integration examples,

migration guides,

architectural overviews.


Reference material should remain generated whenever possible.


---

Trade-offs

Narrative documentation

Advantages:

Easier onboarding

Explains workflows

Connects isolated endpoints into complete integrations


Disadvantages:

Cannot be mechanically verified for conceptual accuracy

More susceptible to documentation drift



---

Reference-only documentation

Advantages:

Low maintenance

Mechanically accurate


Disadvantages:

Difficult for newcomers

Lacks practical guidance



---

When to choose each option

Use generated reference for precise interface information.

Use narrative documentation to teach consumers how those interfaces work together.


---

Common failure modes

Large tutorials duplicating API reference.

Reference pages attempting to become programming guides.

Examples no longer reflecting recommended practices.


---

Example

Stripe is widely regarded as an industry benchmark because its documentation combines precise API reference with realistic, copyable examples and end-to-end integration guides. The generated reference answers "what does this endpoint require?" while the surrounding narrative answers "how do I actually build a payment flow?"


---

Decision: Favor Working Examples Over Additional Prose

What it is

A runnable example often communicates API usage more effectively than paragraphs describing individual fields.

Why it exists

Consumers usually arrive with a practical objective:

> "How do I make this request?"



A working example answers that question immediately.

Long textual descriptions frequently repeat information already visible in parameter tables.


---

Options

Examples may include:

complete HTTP requests,

complete HTTP responses,

SDK examples,

command-line examples,

minimal end-to-end workflows.



---

Trade-offs

Working examples

Advantages:

Immediately usable

Easy to adapt

Reduce ambiguity


Disadvantages:

Must remain synchronized with the API



---

Descriptive prose

Advantages:

Explains concepts


Disadvantages:

Often slower to understand

Frequently duplicates structural reference



---

No examples

Advantages:

Less documentation effort


Disadvantages:

Higher consumer friction

More support requests



---

When to choose each option

Whenever practical, provide examples that consumers can copy with minimal modification.

Prefer complete examples over isolated fragments.


---

Common failure modes

Pseudo-code that cannot actually run.

Requests missing required headers.

Responses inconsistent with current schemas.

Examples using deprecated SDKs.


---

Example

Many successful public APIs provide complete curl examples because they can be executed immediately and translated easily into other programming environments.


---

Decision: Document the Entire Contract

What it is

Document every behavior consumers are expected to rely upon.

Why it exists

Chapter 25 introduced Hyrum's Law:

> With enough users, every observable behavior becomes someone's dependency.



Incomplete documentation does not prevent those dependencies from forming.

It merely makes them invisible.

Consumers observe actual behavior, not documentation boundaries.

If behavior consistently exists but the documentation omits it, some consumers will still depend upon it.

The API owner simply loses awareness that the dependency exists.

Precise documentation therefore reduces Hyrum's Law risk by making the intended contract explicit.


---

Options

Document:

guarantees,

limitations,

ordering requirements,

consistency expectations,

error behavior,

documented edge cases.


Avoid relying on consumers to infer contract details from experimentation.


---

Trade-offs

Complete documentation

Advantages:

Clear contract boundaries

Predictable integrations

Fewer accidental dependencies


Disadvantages:

Greater documentation effort



---

Ambiguous documentation

Advantages:

Faster initially


Disadvantages:

Encourages consumers to reverse-engineer behavior

Increases compatibility surprises

Makes contract evolution more difficult



---

When to choose each option

If consumers are expected to rely on a behavior, document it.

If they are not expected to rely on it, avoid exposing it consistently enough that it becomes an accidental contract.


---

Common failure modes

Pagination behavior left unspecified.

Rate limiting undocumented.

Ordering guarantees implied but never stated.

Timeout behavior differing across implementations without documentation.


---

Example

Large public APIs frequently discover that clients have come to rely on undocumented response ordering, optional fields, or timing behavior. The absence of documentation did not prevent those dependencies—it merely prevented the API provider from recognizing them until changing the behavior caused widespread breakage.


---

Why Smart Engineers Disagree

Experienced API designers generally agree that structural reference should be generated. The disagreement concerns how much narrative should accompany it.

One philosophy minimizes hand-written documentation, arguing that examples and generated reference are sufficient and substantially easier to keep accurate.

The other invests heavily in tutorials, integration guides, and conceptual documentation, arguing that successful APIs are measured not by completeness alone but by how quickly new consumers become productive.

Both approaches recognize the same maintenance constraint described in Chapter 66: every manually written page can drift. They differ in how much onboarding value they believe that additional narrative provides.


---

Summary

API documentation is unique because it frequently serves readers who cannot consult the implementation. As a result, it becomes part of the interface itself rather than merely describing it.

The most reliable approach separates documentation into two layers. The reference layer—operations, parameters, schemas, types, status codes, and similar structural information—should be generated mechanically whenever possible so it remains synchronized with the interface. The narrative layer—getting-started guides, integration patterns, and practical examples—must be written by humans because it explains relationships and intent that generation cannot infer.

Finally, API documentation should describe the complete public contract. Precise documentation is not merely a convenience for consumers; it helps define the boundaries of the interface itself and reduces the accidental dependencies predicted by Hyrum's Law.
