Chapter 64 — What to Document vs. What to Leave to the Code

Purpose

Documentation exists to communicate information the code cannot. This chapter establishes the first decision in the documentation lifecycle: whether information should be written down outside the source code at all.

Part IV established that comments must justify their maintenance cost by explaining why rather than merely restating what the code already says. This chapter applies that reasoning at repository scale. Before choosing whether something belongs in a README, specification, architecture decision record, or operational guide, first determine whether external documentation is warranted in the first place.

Documentation is an engineering asset with an ongoing maintenance cost. Every page becomes another source of truth that future engineers must locate, read, trust, and eventually update. As established in Chapter 2, reducing complexity is valuable only when the reduction exceeds the complexity introduced. Principle 6 applies unchanged: documentation should exist only when its long-term value exceeds its long-term cost.


---

Prerequisites

This chapter assumes familiarity with:

Chapter 2's treatment of complexity as an engineering cost.

Principle 6 from Part VI: processes and artifacts justify themselves only when their benefits exceed their overhead.

Chapter 27 on expressing intent through code organization.

Chapter 28 on naming as documentation.

Chapter 30 on the WHY-versus-WHAT distinction for comments.

Chapters 45 and 46, which established the purpose of specifications and ADRs without discussing their long-term documentation role.



---

Key Concepts

Documentation is information that cannot be naturally expressed in executable code.

Code should be improved before documentation is added.

Documentation exists primarily for information that spans files, services, teams, or time.

Documentation is maintained under the same engineering standards as production code.

Documentation quantity is a poor proxy for documentation quality.



---

Decision: Should This Information Exist Outside the Code?

What it is

A decision framework for determining whether information deserves external documentation before deciding what form that documentation should take.

Why it exists

Most unnecessary documentation exists because engineers begin with the assumption that documentation is inherently good. The more useful assumption is the opposite:

> If the code can communicate it, external documentation is unnecessary.



External documentation should only exist when no reasonable improvement to the code itself could communicate the same information.


---

Options

Option 1: Express it in the code

Improve naming, structure, module boundaries, or types so the code explains itself.

Option 2: Express it with an inline comment

When the information belongs to one location and explains intent that cannot be encoded directly.

(Expanded in Chapter 30.)

Option 3: Write external documentation

Only when the information has no natural location inside the codebase.


---

Trade-offs

Favoring code

Advantages:

Single source of truth

Automatically updated during implementation changes

Impossible to forget while editing nearby code


Disadvantages:

Cannot express decisions spanning multiple modules

Cannot explain historical reasoning

Cannot orient readers before they understand the codebase



---

Favoring external documentation

Advantages:

Can describe whole-system behavior

Can capture rationale invisible from implementation

Can describe relationships across repositories or services


Disadvantages:

Must be maintained separately

Can become incorrect while code remains correct

Creates another source engineers must discover and trust



---

Duplicating information

Advantages:

Sometimes improves discoverability


Disadvantages:

Creates synchronization problems

Encourages conflicting explanations

Usually produces one outdated copy



---

When to choose each option

Improve the code when the information is recoverable from clearer names, simpler control flow, stronger types, or better module boundaries.

Use an inline comment when only one location requires additional explanation and the explanation concerns intent rather than implementation.

Choose external documentation only when the information:

spans multiple files,

spans multiple services,

spans multiple teams,

spans time,

or exists before a reader can reasonably understand the code.



---

Common failure modes

Documentation explaining poorly named code instead of improving the names.

Architecture documents that merely restate the directory structure.

README files that duplicate installation commands already automated by the build system.

Comments copied into wiki pages and updated independently.

Design documents explaining implementation details that changed months ago.


---

Example

The source code of the Linux explains how the scheduler operates. It cannot explain decades of design evolution, rejected scheduler algorithms, or why particular fairness properties were prioritized. Those belong in design documents and mailing-list discussions rather than comments scattered throughout the scheduler source.


---

Decision: Fix the Code Before Writing Documentation

What it is

Documentation should not compensate for code that could simply be made clearer.

Why it exists

Documentation is often used as a cheaper alternative to refactoring.

That choice usually creates two maintenance problems instead of solving one.


---

Options

Improve:

names,

structure,

abstractions,

interfaces,


before introducing documentation.

Only after those improvements fail should documentation be considered.


---

Trade-offs

Refactoring

Pros:

Permanently improves readability

Benefits every future reader

Removes the need for explanation


Cons:

May require broader changes

Can temporarily increase implementation effort



---

Adding documentation

Pros:

Faster in the short term

Useful when code genuinely cannot express intent


Cons:

Leaves confusing code untouched

Creates future maintenance obligations



---

Doing both

Sometimes both are necessary.

A complicated distributed algorithm may deserve better naming and a design document describing assumptions invisible from the implementation.


---

When to choose each option

Refactor when better code eliminates the need for explanation.

Document when the remaining information is fundamentally outside the expressive power of source code.


---

Common failure modes

Large comments preceding simple functions.

Repository wikis explaining confusing module layouts instead of reorganizing the modules.

Long onboarding guides that compensate for inconsistent project organization.


---

Example

Many modern libraries reduced extensive API documentation simply by improving naming conventions and type systems. Languages such as Rust encourage expressive types that eliminate paragraphs of explanatory prose previously required in C APIs.


---

Decision: Document Information That Has Nowhere Else to Live

What it is

Some information cannot be attached to any individual file because it exists above the level of implementation.

Why it exists

Code is inherently local.

Many engineering decisions are not.


---

Options

Information that often requires external documentation includes:

project onboarding context,

architectural constraints,

historical design decisions,

operational procedures,

organization-wide conventions.


The exact artifact chosen is deferred until Chapter 65.


---

Trade-offs

Keeping cross-cutting knowledge external

Pros:

Accessible before understanding implementation

Provides system-wide perspective

Avoids duplicating explanations


Cons:

Requires ongoing maintenance

Risks drifting from implementation



---

Trying to embed everything in code

Pros:

Reduces documentation surface


Cons:

Some information has no meaningful location

Forces important context into arbitrary comments



---

Distributing explanations across many files

Pros:

Close to implementation


Cons:

Readers must reconstruct the larger picture themselves

Context becomes fragmented



---

When to choose each option

External documentation is justified when information concerns:

repository entry points,

architectural intent,

historical decisions,

operational behavior,

assumptions spanning multiple components.


Later chapters determine which document type is appropriate.


---

Common failure modes

The same architecture explanation copied into dozens of source files.

Critical deployment assumptions hidden inside one implementation comment.

New engineers forced to reverse-engineer repository organization because no onboarding context exists.


---

Example

Large distributed systems such as Kubernetes cannot reasonably explain cluster architecture, control-plane responsibilities, or extension mechanisms through source comments alone. Those concepts necessarily exist as higher-level documentation because they span thousands of files and numerous independently evolving components.


---

Information Code Cannot Express Well

Several categories consistently justify external documentation.

Onboarding context

A newcomer needs orientation before reading the first source file.

Typical examples include:

repository purpose,

build prerequisites,

development workflow,

high-level architecture.


Chapter 65 discusses where this information belongs.


---

Design rationale

Code records what was built.

It rarely records why competing approaches were rejected.

That historical reasoning belongs in permanent decision records rather than implementation comments, as introduced in Chapters 45 and 46.


---

Cross-cutting constraints

Some assumptions span multiple services, repositories, or teams.

No individual file can adequately explain them because no single file owns them.


---

Operational knowledge

Many facts matter only during incidents:

recovery procedures,

escalation guidance,

diagnostic workflows.


Those belong with operational documentation, discussed in Chapter 68 rather than alongside implementation.


---

Decision: Treat Documentation Like Production Code

What it is

Documentation should receive the same engineering scrutiny as code.

Why it exists

Documentation that is exempt from review, ownership, or maintenance eventually becomes less trustworthy than the implementation it describes.

The "docs as code" movement, championed by the Write the Docs, argues that documentation should be version-controlled, reviewed, tested where practical, and evolved alongside the software rather than maintained as an afterthought.

Similarly, publicly discussed engineering practices from Google distinguish between comments that belong with code and broader engineering documentation maintained separately, reinforcing that different information belongs at different levels rather than accumulating indiscriminately.


---

Options

Treat documentation as:

production engineering artifacts,

disposable project notes,

or compliance paperwork.


Only the first consistently produces documentation engineers trust.


---

Trade-offs

Engineering-quality documentation requires additional review effort.

The alternative is cheaper initially but frequently results in documents that become ignored because their accuracy is uncertain.


---

When to choose each option

Documentation intended for repeated use over months or years should be maintained with the same discipline as code.

Temporary personal notes generally should not become repository documentation.


---

Common failure modes

Wiki pages abandoned after major refactoring.

Architecture diagrams updated only before presentations.

Design documents that describe systems which no longer exist.


---

Example

Many large open-source projects keep documentation within the same repository as the implementation so changes can be reviewed together, reducing—but not eliminating—the risk of divergence.


---

Decision: Never Optimize for Documentation Coverage

What it is

Documentation coverage measures whether something was written.

It does not measure whether anything useful was communicated.

Why it exists

Like test coverage discussed in Chapter 41, documentation coverage is a proxy metric.

Proxy metrics become dangerous when treated as objectives instead of observations.


---

Options

Organizations commonly measure:

comment coverage,

docstring coverage,

API documentation coverage.


These metrics are useful only as indicators of missing documentation, never as indicators of documentation quality.


---

Trade-offs

Coverage targets increase consistency.

They also incentivize engineers to satisfy the measurement rather than the reader.

The result is documentation that technically exists but communicates almost nothing.


---

When to choose each option

Use coverage reports to discover obvious omissions.

Never evaluate documentation quality solely from coverage percentages.

Review usefulness directly instead.


---

Common failure modes

Auto-generated stubs such as:

> Returns the current user.



attached to a function named GetCurrentUser().

Docstrings repeating parameter names without explaining semantics.

Large repositories reporting nearly perfect documentation coverage while engineers continue reading implementation to understand behavior.

Linters that encourage one-line summaries for every trivial method, producing thousands of mechanically generated descriptions that no one reads.


---

Example

Many organizations have experienced waves of automatically generated docstrings created solely to satisfy lint rules. The resulting repositories technically achieved high documentation coverage while providing almost no additional understanding beyond function names and signatures. The metric improved; the documentation did not.


---

Why Smart Engineers Disagree

Experienced engineers disagree primarily about the threshold for writing documentation, not its value.

One school prefers documenting aggressively, arguing that writing is cheaper than repeatedly answering the same questions.

The other documents conservatively, arguing that every unnecessary document becomes future maintenance debt and competes with genuinely valuable documentation for attention.

Both positions recognize documentation's cost. They differ only in estimating how frequently that cost is justified.

Teams with stable architectures often benefit from more permanent documentation because the maintenance burden is relatively low. Teams changing rapidly often benefit from documenting less, since every additional page is another artifact that must evolve alongside the code.

Neither philosophy supports documenting information already evident from well-written code.


---

Summary

External documentation should be treated as the final option, not the first. Improve the code where possible, use local comments when the explanation belongs beside a specific implementation, and reserve external documents for information that genuinely cannot live within the source itself because it spans modules, services, readers, or time.

This chapter answered whether information deserves external documentation. The next chapter answers where that information belongs by establishing the documentation taxonomy: README, specification, ADR, and other long-lived engineering artifacts.
