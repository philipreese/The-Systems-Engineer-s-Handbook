Chapter 65 — README vs. Spec vs. ADR vs. Inline Comment

Purpose

The previous chapters established that documentation should exist only when it communicates information the code cannot express. This chapter answers the next question: once information deserves to be written down, where does it belong?

Throughout the handbook, four documentation artifacts have already appeared:

Inline comments (Chapter 30)

Specifications (Chapter 46)

Architecture Decision Records (Chapter 45)

README files (introduced in Chapter 64)


This chapter places them into a single taxonomy using two questions:

1. Who is the intended reader?


2. Should this artifact describe current reality, or preserve a moment in time?



Those two axes determine nearly every documentation decision.


---

Prerequisites

This chapter assumes familiarity with:

Chapter 30's distinction between inline comments and code.

Chapter 45's treatment of Architecture Decision Records.

Chapter 46's treatment of specifications.

Chapter 64's decision framework for determining whether external documentation is warranted at all.



---

Key Concepts

Documentation artifacts serve different audiences.

Some artifacts are expected to stay current; others intentionally record history.

Staleness has different meanings depending on the artifact.

Choosing the wrong artifact creates predictable maintenance failures.



---

The Two Axes

Every documentation artifact can be classified by two independent dimensions.

Audience

Who is asking the question?

Someone reading one function?

Someone joining the repository?

Someone proposing future work?

Someone wondering why an architectural decision was made?


Different readers require different documents.


---

Time

Should the document describe the system today, or should it permanently preserve what was true at a particular point in history?

This distinction is frequently overlooked.

Many documentation problems arise because engineers expect historical documents to remain current or expect living documents to preserve historical reasoning.

They cannot do both well.


---

The Documentation Taxonomy

Artifact	Primary Reader	Tracks Current Reality?	Primary Purpose

Inline comment	Someone reading nearby code	Yes	Explain local intent
README	Someone approaching the repository	Yes	Orient and onboard
Specification	Engineers planning future work	No	Describe proposed work
ADR	Future engineers revisiting a decision	No	Record why a decision was made


Everything in this Part builds on this table.


---

Decision: Use an Inline Comment for Local Intent

What it is

An explanation attached directly to specific lines of source code.

Why it exists

Some intent cannot be expressed naturally through code alone but applies only to one implementation location.

The explanation should live with the implementation itself.


---

Options

Inline comment

External documentation

Better code


Chapter 30 established how to choose among these.

This chapter only establishes where inline comments fit within the broader taxonomy.


---

Trade-offs

Inline comments

Advantages:

Visible where needed

Updated alongside nearby code

Closely tied to implementation


Disadvantages:

Cannot describe system-wide behavior

Cannot explain repository-level concepts

Become meaningless when copied elsewhere



---

External documentation

Advantages:

Broader perspective


Disadvantages:

Too distant from implementation for local explanations



---

When to choose each option

Use inline comments when the reader is already inside the implementation and needs local context.

If understanding requires stepping back beyond the surrounding code, another documentation artifact is probably appropriate.


---

Common failure modes

Repository-level architecture described inside one source file.

Comments explaining deployment procedures.

Historical design rationale scattered across implementations.


---

Example

The source of SQLite contains carefully placed comments explaining subtle implementation decisions near complex algorithms. Those comments exist because the information is inseparable from the code they annotate rather than because they describe the database as a whole.


---

Decision: Use a README as the Front Door

What it is

A repository's entry point for engineers who have not yet understood the code.

Why it exists

Readers need orientation before implementation details become meaningful.

The README answers questions such as:

What is this project?

How do I build it?

How do I run it?

Where should I read next?


It is not an architectural history.

It is not a design proposal.

It is not a tutorial covering every subsystem.


---

Options

README

Wiki

Design documents

Specifications


The README should point toward these resources rather than replace them.


---

Trade-offs

Keeping the README focused

Advantages:

Easy to maintain

Easy for newcomers to read

Clearly establishes project purpose


Disadvantages:

Cannot contain every detail



---

Expanding the README indefinitely

Advantages:

Fewer separate documents


Disadvantages:

Difficult to navigate

Frequently outdated

Important onboarding information becomes buried



---

Splitting documentation appropriately

Advantages:

Each artifact has a clear responsibility


Disadvantages:

Requires careful organization



---

When to choose each option

Choose a README whenever the audience is someone approaching the repository for the first time.

Unlike historical documents, the README must always describe the repository as it exists today.


---

Common failure modes

Installation instructions that no longer work.

References to deleted directories.

Describing architecture that has significantly evolved.

Treating the README as an archive of every design discussion.


---

Example

GitHub automatically renders a repository's README on its landing page. That convention has effectively established the README as the industry's front door for software projects. New contributors encounter it before any source code, making its accuracy critical.


---

Decision: Use a Specification to Describe Proposed Reality

What it is

A document describing work that has not yet been completed.

Why it exists

A specification coordinates implementation before implementation exists.

Once implementation begins, the specification becomes a historical proposal rather than an authoritative description of the running system.

This behavior is intentional.


---

Options

Specification

Direct implementation

ADR


Each serves a different stage of engineering work.


---

Trade-offs

Keeping specifications as historical proposals

Advantages:

Preserves original intent

Explains planning decisions

Avoids rewriting history


Disadvantages:

Cannot be treated as current documentation



---

Continuously updating specifications

Advantages:

May appear more current


Disadvantages:

Erases historical context

Creates duplicate maintenance alongside implementation



---

When to choose each option

Use a specification when discussing work that is proposed but not yet implemented.

Once implementation diverges, readers should consult the code or current documentation rather than expect the specification to evolve indefinitely.


---

Common failure modes

Assuming an old specification accurately reflects today's system.

Editing historical proposals until they resemble documentation instead of design history.

Treating specifications as permanent reference manuals.


---

Example

The Kubernetes Enhancement Proposal (KEP) process exemplifies this category. A KEP records the motivation, design, alternatives, and rollout plan for a proposed feature. It is distinct from Kubernetes' user and API documentation, which describe the implemented system.


---

Decision: Use an ADR to Explain Why a Decision Was Made

What it is

A permanent record of an architectural decision and the alternatives considered.

Why it exists

Implementation shows what was built.

An ADR records why that choice won.

Like specifications, ADRs preserve a moment in time rather than describing the current implementation.

An ADR remains correct even if the system later evolves beyond it.


---

Options

ADR

Updating the README

Inline comments


The README explains what exists.

The ADR explains why it came to exist.


---

Trade-offs

Preserving historical decisions

Advantages:

Future engineers understand original reasoning

Prevents repeatedly revisiting settled questions

Documents rejected alternatives


Disadvantages:

May no longer describe today's architecture



---

Editing ADRs to match current implementation

Advantages:

Appears more up to date


Disadvantages:

Destroys the historical record

Removes evidence of why earlier choices were made



---

When to choose each option

Choose an ADR when the primary question is:

> "Why was this decision made?"



not:

> "How does the system currently work?"




---

Common failure modes

Updating old ADRs to match current architecture.

Treating ADRs as architecture documentation.

Expecting every ADR to remain descriptively accurate years later.


---

Example

Projects following the Nygard ADR convention typically maintain an /adr directory alongside a README. The README introduces today's project. The ADR directory explains why yesterday's important architectural decisions were made. They complement rather than replace one another.


---

Decision: Classify Documentation by Question, Audience, and Time

What it is

A practical selection process for choosing the correct documentation artifact.

Why it exists

Engineers often begin by asking:

> "Where should I write this?"



The better first question is:

> "What question is someone trying to answer?"




---

Options

Ask three questions.

Who is asking?

A new contributor?

Someone reading one function?

Someone reviewing a historical decision?

Someone planning future work?


---

What question are they asking?

How do I start?

Why is this implemented this way?

What are we proposing?

Why was this chosen?


---

Must the answer remain current?

If yes, choose a living document.

If no, choose a historical record.


---

Trade-offs

Choosing based on audience improves discoverability.

Choosing based on lifespan reduces maintenance confusion.

Ignoring either usually results in documents serving multiple incompatible purposes.


---

When to choose each option

If the document must always describe the system as it exists today, it belongs to the living documentation category.

Examples include:

README

API reference documentation (covered in Chapter 67)


If the document exists to preserve a decision or proposal, it belongs to the historical documentation category.

Examples include:

specifications

ADRs



---

Common failure modes

Specifications rewritten after implementation to become reference manuals.

READMEs filled with historical design discussions.

ADRs describing current architecture instead of recording past reasoning.

Inline comments explaining repository organization.


---

Example

A newcomer opening a repository asks:

> "What is this project and how do I run it?"



A README answers that.

An engineer investigating a controversial architectural choice asks:

> "Why did we choose this approach instead of the alternatives?"



An ADR answers that.

Those are fundamentally different questions requiring different artifacts.


---

Why Smart Engineers Disagree

Experienced teams disagree primarily about whether historical artifacts should continue evolving after implementation.

Some organizations periodically revise specifications to match reality, treating them as long-lived design documentation.

Others archive specifications once implementation begins, preserving them as records of what was proposed rather than what ultimately shipped.

Similarly, some teams amend ADRs with later outcomes, while others treat them as immutable records supplemented by new ADRs when circumstances change.

Either approach can work if applied consistently. The critical mistake is mixing expectations—treating one historical artifact as immutable while expecting another, with the same purpose, to continuously track reality.

For this handbook, the distinction is explicit:

Living documents (such as READMEs and, later, API reference documentation) are expected to remain current.

Historical documents (specifications and ADRs) preserve a point in time. Their value comes from accurately recording that moment, not from continuously describing the present.



---

The Central Taxonomy

The taxonomy can be summarized with two questions:

Artifact	Who is it for?	Current or Historical?

Inline comment	Someone reading this code	Current
README	Someone entering this repository	Current
Specification	Someone evaluating proposed work	Historical snapshot
ADR	Someone understanding a past decision	Historical snapshot


The distinction between current and historical is the foundation for the remainder of this Part.

A README that no longer reflects the repository is a documentation failure because it has stopped fulfilling its purpose.

An old specification or ADR that no longer describes today's implementation is not inherently a failure. It continues to fulfill its purpose as a faithful record of what was proposed or decided at the time.

The next chapter builds directly on this distinction by examining how living documentation is kept accurate over time without imposing unnecessary maintenance overhead.
