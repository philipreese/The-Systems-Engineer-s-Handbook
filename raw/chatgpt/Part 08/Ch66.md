Chapter 66 — Keeping Documentation Honest

Purpose

Writing documentation is only the beginning of its lifecycle. For documentation intended to describe the current system, long-term value depends on continued accuracy.

This chapter addresses the primary failure mode of living documentation: drift. A README, API reference, or operational guide that no longer matches reality is often worse than having no documentation at all because it encourages readers to trust incorrect information.

Chapter 49 established that every engineering artifact incurs maintenance overhead throughout its lifetime. This chapter applies that principle specifically to documentation that is expected to track current reality.


---

Prerequisites

This chapter assumes familiarity with:

Chapter 30's discussion of comment rot.

Chapter 49's framework for evaluating process overhead and periodically auditing engineering artifacts.

Chapter 64's decision for determining whether documentation should exist.

Chapter 65's distinction between living documentation (README, API documentation) and historical documentation (specifications and ADRs).



---

Key Concepts

Living documentation must evolve with the system.

Documentation drift is a structural problem before it is an individual failure.

Some documentation correctness can be verified automatically; much cannot.

Deleting obsolete documentation is often healthier than preserving it indefinitely.

Documentation repositories can become write-only systems.



---

Decision: Keep Documentation Close to the Code

What it is

Store documentation as near as possible to the implementation it describes, preferably within the same repository and reviewed through the same change process.

Why it exists

The greatest cause of documentation drift is organizational separation.

When code lives in one system and documentation in another, changes naturally occur independently. Engineers modify implementation, merge the pull request, and never visit the separate documentation platform.

Keeping documentation adjacent to the implementation does not guarantee synchronization, but it gives documentation changes the opportunity to occur during the same review.

This is an application of Principle 8: information should live as close as practical to the system it describes.


---

Options

Documentation in the same repository

Markdown files, documentation comments, and guides reviewed alongside code.

Documentation in a separate documentation repository

Useful when documentation serves many independent codebases.

Documentation in a separate wiki or knowledge base

Suitable primarily for information that is intentionally independent of implementation.


---

Trade-offs

Same repository

Advantages:

Reviewed together with implementation

Shared version history

Documentation changes naturally appear in pull requests


Disadvantages:

Repository becomes larger

Documentation contributors must interact with the source repository



---

Separate repository

Advantages:

Independent publication

Shared across multiple projects


Disadvantages:

Easier for implementation and documentation to diverge

Requires explicit coordination



---

Separate wiki

Advantages:

Easy to edit

Accessible to non-developers


Disadvantages:

No structural relationship to implementation

Rarely reviewed alongside behavioral changes

Frequently becomes abandoned



---

When to choose each option

Keep implementation-specific documentation with the implementation whenever practical.

Use separate systems only when the documentation genuinely has a broader scope than any individual repository.


---

Common failure modes

Behavior changes merged without corresponding documentation updates.

Repository README maintained while an external wiki quietly becomes obsolete.

Multiple contradictory onboarding guides existing simultaneously.


---

Example

Both GitHub and GitLab maintain much of their public documentation as Markdown within version-controlled repositories. Documentation changes undergo review alongside other changes rather than existing as an entirely separate publishing workflow.


---

Decision: Mechanically Verify What Can Be Verified

What it is

Convert documentation correctness into automated verification wherever possible.

Why it exists

Some forms of documentation can be objectively tested.

Others cannot.

Automating the former reduces silent documentation drift by transforming documentation failures into build failures.


---

Options

Common automated verification includes:

compiling code examples,

executing documentation examples,

validating hyperlinks,

checking formatting and structure,

verifying generated API references.


Other forms of documentation require human review.


---

Trade-offs

Mechanical verification

Advantages:

Immediate feedback

Consistent enforcement

Detects many regressions automatically


Disadvantages:

Only applicable to objectively testable content

Cannot determine whether an explanation remains appropriate



---

Human review

Advantages:

Evaluates meaning and clarity

Detects conceptual drift


Disadvantages:

Subjective

Easy to overlook during implementation changes



---

No verification

Advantages:

Minimal maintenance effort


Disadvantages:

Drift accumulates silently



---

When to choose each option

Automate everything that can be objectively checked.

Accept that explanatory correctness remains a human responsibility.


---

Common failure modes

Documentation examples that no longer compile.

Broken links throughout developer guides.

Installation instructions referencing removed tools.

Guides recommending workflows no longer considered best practice.


---

Example

Rust includes the rustdoc doctest system, which compiles and executes runnable code examples embedded in documentation comments during normal test execution. Examples that no longer work fail the build rather than silently misleading readers.


---

Decision: Accept That Documentation Review Is Partly Human

What it is

Recognize the boundary between objective correctness and engineering judgment.

Why it exists

Automation can determine whether an example compiles.

It cannot determine whether the explanation is still the best explanation.

It cannot determine whether architectural guidance still reflects organizational consensus.

Those remain engineering decisions.


---

Options

Documentation can be reviewed:

opportunistically during implementation changes,

through periodic audits,

or not at all.


Only the first two reliably preserve trust.


---

Trade-offs

Continuous review

Advantages:

Drift corrected quickly

High reader confidence


Disadvantages:

Additional review effort



---

Periodic review

Advantages:

Lower ongoing cost


Disadvantages:

Drift may persist for months



---

Ignoring documentation

Advantages:

No immediate effort


Disadvantages:

Readers eventually stop trusting documentation altogether



---

When to choose each option

Review documentation whenever related behavior changes.

Periodically revisit important living documentation even when implementation remains relatively stable.

Historical documents such as specifications and ADRs are outside the scope of this review because Chapter 65 established that they are not intended to track current implementation.


---

Common failure modes

README files describing workflows the team abandoned years earlier.

Developer guides recommending deprecated tools.

Architecture overviews reflecting previous system boundaries.


---

Example

Successful engineering organizations frequently treat documentation review as a normal component of code review rather than an independent documentation project, reducing the chance that documentation becomes detached from implementation.


---

Decision: Delete Documentation That No Longer Deserves to Exist

What it is

Removing obsolete documentation is a maintenance activity, not a failure.

Why it exists

Documentation imposes cost whether or not anyone still benefits from it.

Keeping obsolete documents "just in case" increases search time, creates conflicting guidance, and reduces confidence in the documentation corpus as a whole.

Deletion is often the correct maintenance decision.


---

Options

Update the document.

Archive it if historical value exists.

Delete it.


The appropriate choice depends on whether the document is intended to describe current reality.


---

Trade-offs

Updating

Advantages:

Preserves continuity


Disadvantages:

Requires ongoing maintenance



---

Archiving

Advantages:

Preserves historical context


Disadvantages:

Must be clearly distinguished from current guidance



---

Deleting

Advantages:

Reduces maintenance burden

Improves overall documentation quality


Disadvantages:

Removes potentially useful historical information if done carelessly



---

When to choose each option

Update documentation that remains valuable.

Archive documentation whose value is historical.

Delete documentation that serves neither purpose.


---

Common failure modes

Multiple onboarding guides for obsolete development environments.

Deprecated workflows preserved alongside current ones.

Engineers unable to determine which document should be trusted.


---

Example

Many organizations eventually accumulate abandoned internal knowledge bases—often informally called a "Confluence graveyard"—containing years of obsolete procedures, architecture diagrams, and onboarding instructions. Although some pages remain accurate, engineers increasingly ignore the entire knowledge base because determining which pages are trustworthy costs more than rediscovering the information directly.


---

Decision: Prevent Documentation from Becoming a Write-Only System

What it is

Documentation should accumulate only if obsolete documentation is removed at roughly the same rate.

Why it exists

Chapter 42 described the write-only tracker: information enters continuously but is rarely removed or reorganized.

The same pattern appears in documentation.

Writing documentation feels productive.

Deleting documentation feels risky and rarely receives recognition.

Over time, the corpus grows while its average accuracy declines.

Eventually engineers stop consulting it.

At that point, even accurate documents lose value because readers no longer know which pages can be trusted.


---

Options

Treat documentation as:

an actively curated knowledge base,

or an append-only archive.


Living documentation requires the former.


---

Trade-offs

Active curation

Advantages:

Higher trust

Easier navigation

Lower cognitive overhead


Disadvantages:

Requires regular pruning



---

Append-only growth

Advantages:

Little immediate effort

Nothing appears to be lost


Disadvantages:

Trust steadily declines

Search becomes increasingly difficult

Contradictory guidance accumulates



---

When to choose each option

Living documentation should be continuously curated.

Historical records belong in explicitly historical artifacts such as specifications and ADRs rather than accumulating inside living guides.


---

Common failure modes

Hundreds of wiki pages with unknown ownership.

Multiple conflicting deployment guides.

Search results dominated by obsolete documents.

New engineers asking experienced developers because documentation is no longer considered trustworthy.


---

Example

Large internal wiki installations frequently demonstrate this pattern. Years of incremental additions create a corpus where every search returns dozens of partially outdated pages. Engineers begin treating the documentation as a starting point for investigation rather than as an authoritative source, defeating the purpose of maintaining living documentation in the first place.


---

Why Smart Engineers Disagree

Experienced teams disagree about how aggressively documentation should be maintained.

One philosophy favors extensive documentation supported by regular review and dedicated ownership. The expectation is that high-quality documentation reduces repeated support costs enough to justify the maintenance effort.

The other philosophy deliberately limits the amount of living documentation, arguing that every additional page is another potential source of drift. Instead, it relies more heavily on expressive code, concise READMEs, and mechanically generated reference material.

Both approaches acknowledge the same constraint: documentation loses value once readers doubt its accuracy. They differ primarily in where they believe the sustainable maintenance boundary lies.


---

Summary

Living documentation succeeds only when readers can assume it accurately reflects the current system. Keeping documentation close to the implementation improves the likelihood that code and documentation evolve together. Automated verification catches objective regressions such as broken examples and links, while human review remains necessary for explanatory accuracy and recommended practices.

Just as importantly, documentation must be treated as a curated body of knowledge rather than an append-only archive. Updating, archiving, and deleting are all legitimate maintenance actions. A smaller corpus that engineers consistently trust is more valuable than a larger one whose accuracy is uncertain.

The next chapter examines a specialized form of living documentation—API reference documentation—whose external audience, tooling, and maintenance constraints are distinct enough to warrant separate treatment.
