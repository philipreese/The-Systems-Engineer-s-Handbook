Chapter 07 — Reliability as a Design Principle

1. Purpose

Reliability is often treated as an operational concern: monitoring, alerts, retries, incident response.

That framing is backwards.

Reliability is mostly determined at design time — by what kinds of failure are possible, how visible they are, and what states the system is allowed to enter.

This chapter defines reliability as a structural property of systems, not an operational layer added on top.

---

2. Prerequisites

From earlier chapters:

* complexity comes from state, control flow, and code volume
* coupling determines how failures propagate
* cost models determine latency and bottlenecks
* abstractions hide implementation details but not failure modes

Assumes familiarity with:

* distributed systems basics (services, networks, partitions)
* databases (PostgreSQL conceptually)
* logs / queues (Kafka conceptually)
* consensus systems (ZooKeeper / etcd at a high level)

---

3. Key Concepts

* Reliability is emergent, not configurable
* Systems fail in structurally different ways, not just “more often” or “less often”
* Not all failures are equal; some are silently catastrophic
* Partial failure is the default mode in distributed systems
* “Fail fast” is about bounding damage, not convenience
* CAP theorem is about unavoidable trade-offs under partition, not a design checklist

---

4. Failure Mode Taxonomy (What Actually Goes Wrong)

What it is:
Failures are not homogeneous. The *type* of failure determines how dangerous it is, not just its frequency.

---

4.1 Crash Failures

What it is:
The system stops operating in a visible, detectable way.

Why it exists:
Explicit errors, process termination, unhandled exceptions, node crashes.

Options:

* Crash immediately on invalid state
* Attempt recovery before crashing
* Mask failure and continue operating

Trade-offs:

* Crashing is visible and bounded
* Recovery attempts can introduce inconsistent state
* Masking failures hides damage and delays detection

When to choose each option:

* Crash fast: systems with strong correctness requirements (databases, kernels)
* Recover: stateless services, idempotent operations
* Mask: rarely justified; only for non-critical UX layers

Common failure modes:

* Kubernetes restarting crashed pods repeatedly without fixing root cause
* Silent retries masking persistent downstream failures
* Partial initialization leading to undefined runtime state

Example:
PostgreSQL prefers to fail fast on corruption or WAL inconsistency rather than continue operating in a potentially corrupted state.

---

4.2 Corruption Failures

What it is:
The system continues operating, but internal state is incorrect or inconsistent.

Why it exists:
Partial writes, race conditions, deserialization errors, disk corruption, or buggy migrations.

Options:

* Detect and halt on corruption
* Attempt repair in-place
* Continue operating with degraded assumptions

Trade-offs:

* Halting preserves correctness but reduces availability
* Repair introduces operational complexity and risk of further corruption
* Continuing increases risk of cascading incorrect behavior

When to choose each option:

* Halt: databases, financial systems, distributed logs (Kafka, PostgreSQL)
* Repair: file systems, storage layers with recovery tooling
* Continue: user-facing services where partial degradation is acceptable

Common failure modes:

* Silent data corruption in caches diverging from source of truth
* Filesystem corruption not detected until later reads
* Schema mismatch after partial migration rollback

Example:
PostgreSQL uses checksums and WAL consistency checks to detect corruption early, preferring to stop rather than serve potentially incorrect data.

---

4.3 Wrong Answer Failures (Most Dangerous)

What it is:
The system produces incorrect outputs while appearing fully healthy.

Why it exists:
Incorrect logic, stale state, inconsistent reads, misconfigured dependencies, or hidden coupling.

This is the most dangerous class because:

* there is no obvious failure signal
* monitoring often shows “healthy” systems
* downstream systems may silently propagate incorrect results

Options:

* Strong validation (invariants, checksums, assertions)
* Redundant computation (cross-checking, quorum systems)
* Accept risk (common in low-criticality systems)

Trade-offs:

* Validation reduces throughput but increases correctness
* Redundancy increases cost and complexity
* Acceptance increases risk but reduces system overhead

When to choose each option:

* Validation: financial systems, databases, consensus systems
* Redundancy: distributed systems with quorum (ZooKeeper, etcd)
* Acceptance: non-critical UI or caching layers

Common failure modes:

* Cache returning stale but valid-looking data
* Incorrect business logic silently producing wrong results
* Distributed state divergence without detection

Example:
A microservice returning correct HTTP responses but based on stale or inconsistent cache data is not “failing” operationally, but is functionally incorrect — this is often worse than a crash.

5. Fail Fast Principle

What it is:
A system should detect invalid or inconsistent states as early as possible and terminate execution rather than continue operating incorrectly.

Why it exists:
Continuing execution after entering an invalid state increases the blast radius of failure.

Options:

* Fail immediately on detection
* Attempt recovery before failing
* Continue operating in degraded mode

Trade-offs:

* Immediate failure reduces damage propagation
* Recovery attempts risk compounding errors
* Degraded mode hides failure and increases systemic ambiguity

When to choose each option:

* Fail fast: databases (PostgreSQL), consensus systems (etcd), kernel-level code
* Recover: stateless services, retryable operations
* Degraded mode: user-facing services where partial functionality is acceptable

Common failure modes:

* Systems continuing with corrupted in-memory state
* Retry loops masking persistent failures
* Partial initialization leaving systems in undefined states

Example:
PostgreSQL aggressively terminates transactions on invariant violation rather than attempting to continue, preserving global consistency at the cost of local operation failure.

---

6. Partial Failure in Distributed Systems

What it is:
In distributed systems, different components can fail independently. Some nodes succeed while others fail, producing inconsistent global state.

Why it exists:
Network partitions, node crashes, latency variance, and asynchronous replication guarantee that systems cannot fail atomically.

Options:

* Assume partial failure (design explicitly for it)
* Attempt to hide partial failure (retry, coordination layers)
* Ignore it (leads to inconsistency bugs)

Trade-offs:

* Explicit handling increases design complexity but improves correctness
* Hiding failure improves simplicity but increases risk of inconsistency
* Ignoring failure produces unpredictable system behavior under load

When to choose each option:

* Explicit handling: Kubernetes, Kafka, distributed databases
* Hiding: small-scale services with strong infrastructure guarantees
* Ignoring: prototypes or tightly controlled environments

Common failure modes:

* Split-brain conditions in clustered systems
* Inconsistent writes across replicated services
* Partial deployment rollouts causing mixed-version behavior

Example:
Kubernetes pods may be partially scheduled across nodes while some nodes are degraded, leading to a system state where the “desired state” exists but is not fully realized anywhere.

7. CAP Theorem (Correct Interpretation)

What it is:
In the presence of a network partition, a distributed system must choose between:

* Consistency (all nodes see the same data)
* Availability (every request receives a response)

Partition tolerance is not optional in real distributed systems — partitions will occur.

Why it exists:
Network failures are not rare edge cases; they are normal operational behavior in distributed environments.

What it actually means:
Under partition, you cannot simultaneously guarantee both consistency and availability.

It does NOT mean:

* you can freely choose two properties in general
* systems are permanently “CA” or “CP” or “AP”
* design is a static classification problem

Options:

* CP systems (consistency prioritized)
* AP systems (availability prioritized)
* Hybrid systems (context-dependent behavior)

Trade-offs:

* CP systems may reject requests during partitions (e.g., etcd, ZooKeeper)
* AP systems may return stale or conflicting data (e.g., Cassandra)
* Hybrid systems require complex reconciliation logic

When to choose each option:

* CP: coordination systems, leader election, configuration stores
* AP: high-availability user-facing systems, caching layers
* Hybrid: large-scale distributed databases with eventual consistency models

Common failure modes:

* assuming availability during partitions in CP systems
* assuming consistency in AP systems
* misinterpreting stale reads as bugs rather than design choices

Example:
etcd prioritizes consistency over availability; during a partition, it will refuse writes rather than risk divergent cluster state. This is a deliberate design decision, not a limitation.

---

8. Real Systems: Reliability Design Choices

PostgreSQL (WAL)

* Write-Ahead Logging ensures durability before commit acknowledgment
* Trades latency (fsync overhead) for correctness guarantees
* Fails fast on corruption rather than continuing with uncertain state

ZooKeeper / etcd

* Use consensus protocols (Raft / Zab) to ensure single-writer consistency
* High coordination cost, but strong guarantees
* Sacrifice availability during partitions to preserve correctness

Kafka (Log-based durability)

* Appends to immutable log segments
* Trades immediate consistency for high throughput and durable ordering
* Consumers handle replay, making system resilient to partial failure

---

9. Why Smart Engineers Disagree

Some engineers optimize for consistency and correctness, accepting reduced availability during failure modes.

Others prioritize availability and resilience, accepting eventual consistency and temporary divergence.

Others focus on minimizing failure detection latency, preferring fail-fast systems even at the cost of reduced uptime.

The disagreement is fundamentally about what constitutes “correct behavior under failure”:

* CP-oriented engineers prioritize correctness invariants
* AP-oriented engineers prioritize system responsiveness
* fail-fast engineers prioritize detectability and bounded damage

None of these eliminate failure. They only choose what shape failure takes.
