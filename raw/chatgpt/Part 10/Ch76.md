Chapter 76 — Async vs. Threads vs. Processes

Purpose

Chapters 74 and 75 answered one question: how do concurrent units coordinate?

This chapter answers a different one: what is the concurrent unit?

These decisions are independent. Choosing message passing does not imply async. Choosing shared state does not imply threads. A system can combine any coordination model with any execution model.

Confusing these two axes leads to poor architectural decisions. A team debating "threads versus async" when the real issue is shared-state coordination is solving the wrong problem.

This chapter separates these concepts before examining the execution models themselves.

Concurrency and Parallelism Are Different Concepts

Concurrency and parallelism are often used interchangeably.

They are not synonyms.

Concurrency is about the structure of a program.

A concurrent program can make progress on multiple units of work during overlapping periods of time.

Nothing in that definition requires multiple CPU cores.

A single-core machine can execute a concurrent program by switching between tasks.

Parallelism is about execution.

Multiple units of work execute simultaneously on different processing resources.

Parallelism requires hardware capable of executing multiple instructions at the same instant.

Concurrency is a software property.

Parallelism is a hardware execution property.

A program may be:

concurrent but not parallel,

parallel but not meaningfully concurrent,

both,

or neither.


This distinction is foundational for the rest of this chapter.


---

Decision / Principle — OS Processes

What it is

A process is an operating system's unit of resource ownership.

Each process has its own virtual address space, file descriptors, security context, and operating-system state.

Processes communicate through explicit operating-system mechanisms such as pipes, sockets, shared memory, or other forms of inter-process communication (IPC).


---

Why it exists

Processes provide isolation.

A memory error in one process generally cannot corrupt another process's address space.

This isolation improves robustness and security while allowing true parallel execution across CPU cores.


---

Options

Single-process application

Multi-process architecture

Worker processes

Process pools



---

Trade-offs

Advantages:

Strong fault isolation

Separate memory spaces

True parallelism

Operating-system-enforced protection boundaries


Costs:

Higher creation cost

More expensive context switches

Explicit communication required

Data sharing requires IPC or serialization


Processes maximize isolation at the expense of communication efficiency.


---

When to choose each option

Processes are appropriate when:

fault isolation is important,

security boundaries matter,

components should fail independently,

CPU-intensive work scales across cores,

separate runtimes or languages are required.



---

Common failure modes

Excessive serialization

Large IPC overhead

Duplicated memory usage

Treating process communication as though it were shared memory



---

Example

Modern web browsers isolate tabs into separate processes so that a crash in one page does not terminate the entire browser.


---

Decision / Principle — OS Threads

What it is

Threads are execution contexts within a process.

All threads share the same address space while maintaining independent call stacks and execution state.

The operating system schedules threads preemptively.


---

Why it exists

Threads combine inexpensive communication through shared memory with true parallel execution across multiple CPU cores.

Because they share an address space, threads can access common data structures directly.


---

Options

Single worker thread

Thread pool

Dedicated long-running threads

One thread per request (rarely appropriate today)



---

Trade-offs

Advantages:

True CPU parallelism

Lower overhead than processes

Efficient shared-memory communication

Mature operating-system support


Costs:

Shared-memory synchronization

Context-switch overhead

Memory consumed by thread stacks

Greater reasoning complexity than isolated execution


Threads balance communication efficiency against synchronization complexity.


---

When to choose each option

Threads are appropriate when:

CPU-bound work must scale across cores,

shared memory is advantageous,

workloads justify true parallel execution.



---

Common failure modes

Creating excessive numbers of threads

Oversubscribing available CPU cores

Using threads for workloads dominated by waiting

Paying synchronization costs unnecessarily



---

Example

Java's traditional concurrency model uses operating-system threads together with synchronization primitives such as synchronized blocks or ReentrantLock.

Decision / Principle — Async and Coroutines

What it is

Async systems execute many logical tasks using cooperative scheduling.

Instead of blocking while waiting for I/O, a task suspends itself, allowing another task to execute.

An event loop resumes suspended tasks when the requested operation completes.

Many async runtimes operate primarily on a single operating-system thread, although some use a small pool of worker threads internally.


---

Why it exists

Waiting for disk or network operations wastes CPU time.

Async execution allows thousands of waiting operations to share a small number of operating-system threads efficiently.

The goal is concurrency rather than CPU parallelism.


---

Options

Single-threaded event loop

Multi-reactor event loop

Async runtime with worker thread pool

Coroutine scheduler



---

Trade-offs

Advantages:

Extremely low per-task overhead

Excellent scalability for I/O-bound workloads

Minimal memory consumption

Fewer operating-system context switches


Costs:

No CPU parallelism by itself

Blocking operations stall the event loop

Cooperative scheduling requires discipline

Async call chains can complicate control flow


Async systems optimize waiting, not computation.


---

When to choose each option

Async execution is appropriate when:

workloads spend most of their time waiting,

network or disk I/O dominates execution,

many concurrent operations must remain active,

responsiveness matters more than raw computation.



---

Common failure modes

Blocking inside the event loop

Long-running CPU computations preventing other tasks from running

Mixing blocking APIs into asynchronous code

Assuming async automatically improves CPU throughput



---

Example

Node.js is built around a single-threaded event loop.

Its architecture is optimized for handling large numbers of concurrent network requests efficiently.

CPU-intensive work is typically delegated to worker threads or external processes rather than performed directly on the event loop.


---

Decision / Principle — Match the Execution Model to the Workload

What it is

Execution models should be selected according to the dominant resource limiting progress.

The first question is not "Which concurrency model is fastest?"

It is "What is the program waiting for?"


---

Why it exists

CPU-bound and I/O-bound workloads have fundamentally different bottlenecks.

Optimizing the wrong bottleneck wastes complexity without improving throughput.


---

Options

Threads

Processes

Async execution

Hybrid designs



---

Trade-offs

CPU-bound workloads

The processor is fully occupied performing computation.

Adding concurrency without additional CPU cores cannot increase execution speed.

Threads or processes capable of running simultaneously on multiple cores are the appropriate solution.


---

I/O-bound workloads

Most execution time is spent waiting.

Adding more operating-system threads often increases memory usage and scheduling overhead without reducing waiting time.

Async execution allows many operations to overlap efficiently while using relatively few threads.


---

Hybrid workloads

Many production systems combine both.

An async front end manages network connections while a thread pool performs expensive computation.

The models complement rather than replace one another.


---

When to choose each option

Choose threads or processes when computation dominates.

Choose async when waiting dominates.

Choose hybrid architectures when both occur in different parts of the system.


---

Common failure modes

Using threads for I/O-bound work:

unnecessary context switches,

unnecessary memory consumption,

scheduling overhead without increased throughput.


Using async for CPU-bound work:

event-loop starvation,

stalled request handling,

reduced responsiveness for every concurrent task.


The second failure is often worse because the entire event loop becomes unavailable while one computation monopolizes it.


---

Example

A web server handling thousands of idle network connections benefits greatly from async execution.

A video encoder benefits from parallel execution across CPU cores.

Attempting to solve both problems with the same execution model usually produces an inferior design.


---

Real Systems Illustrate Different Priorities

No single execution model dominates every environment.

Different runtimes optimize for different workloads.

Python

Python's traditional implementation includes the Global Interpreter Lock (GIL).

Multiple operating-system threads may exist simultaneously, but only one thread executes Python bytecode at a time within a single interpreter.

Threads remain valuable for overlapping I/O operations, but they generally do not provide CPU parallelism for ordinary Python code.

CPU-intensive Python workloads frequently use multiple processes instead.


---

Node.js

Node.js embraces asynchronous execution.

Its single-threaded event loop efficiently handles large numbers of concurrent network operations while avoiding the overhead of one operating-system thread per connection.

This design deliberately targets I/O-bound services rather than CPU-heavy computation.


---

Go

Go intentionally blurs the distinction.

Goroutines are lightweight execution contexts managed by the Go runtime rather than directly by the operating system.

The runtime schedules many goroutines onto a comparatively small pool of operating-system threads using an M:N scheduler.

This provides much of the programming simplicity associated with coroutines while still allowing true CPU parallelism when goroutines execute on multiple operating-system threads across multiple cores.


---

Why Smart Engineers Disagree

Experienced engineers often disagree because they optimize different workloads.

Engineers building high-performance web servers frequently default to async because waiting dominates execution time.

Engineers building numerical software, compilers, rendering engines, or scientific simulations naturally favor threads or processes because computation dominates.

Neither approach is universally superior.

The disagreement usually reflects different optimization targets rather than different concurrency philosophies.

Attempting to standardize on one execution model for every subsystem often forces parts of the system into a model poorly suited to their workload.


---

Coordination and Execution Are Independent Axes

The previous chapters and this one together define two independent architectural decisions.

The first asks:

How do concurrent units coordinate?

Shared state

Message passing


The second asks:

What executes those units?

Processes

Threads

Async tasks


These choices combine freely.

An async application may still require locks around shared state.

Threads may communicate exclusively through channels.

Processes may share memory segments protected by mutexes.

Go commonly combines goroutines with channels.

Java frequently combines threads with shared-memory synchronization.

Distributed systems often combine processes with message passing.

Keeping these decisions separate avoids treating unrelated trade-offs as though they were one architectural choice.
