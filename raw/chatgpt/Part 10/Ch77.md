Chapter 77 — Deadlock, Livelock, and Starvation

Purpose

The previous chapters introduced the two coordination models (Chapter 74), the primary synchronization mechanism for shared state (Chapter 75), and the execution models that run concurrent work (Chapter 76).

This chapter answers the next question:

How does coordination fail?

Concurrency bugs are difficult not because their definitions are complicated, but because their occurrence depends on timing. The same program may execute correctly thousands of times before one particular interleaving exposes a latent failure.

This is the concrete manifestation of the state space explosion introduced in Chapter 2. Every additional concurrent execution path multiplies the number of possible instruction interleavings, making exhaustive reasoning and testing increasingly impractical.

Coordination Failures Depend on Interleaving

Most software failures occur because the program executes incorrect logic.

Concurrency failures are different.

The logic may be completely correct.

The failure emerges only because independently executing operations happen to interleave in a particular order.

Changing processor count, operating-system scheduling, compiler optimizations, system load, or network latency may completely change program behavior.

This is precisely why concurrent failures are notoriously difficult to reproduce.

The program does not have one execution order.

It has an enormous number of possible execution orders.


---

Decision / Principle — Deadlock

What it is

A deadlock occurs when a set of execution units each waits indefinitely for a resource held by another member of the same set.

No participant can proceed.

No participant voluntarily releases the resource it already holds.

The system reaches permanent standstill.


---

Why it exists

Deadlock arises because independently reasonable resource-acquisition decisions combine into a cycle.

Each participant is waiting correctly.

Collectively, they make progress impossible.


---

Options

Deadlocks can be addressed through:

Prevention

Detection and recovery

Avoidance algorithms

Ignoring the possibility (rarely acceptable)



---

Trade-offs

Prevention

Advantages:

Eliminates entire categories of failures

Easier operational behavior

No runtime detection overhead


Costs:

Places constraints on program structure

May reduce flexibility



---

Detection and recovery

Advantages:

Allows greater scheduling flexibility

Simplifies some application logic


Costs:

Runtime overhead

One participant must usually be interrupted or rolled back

Recovery logic becomes part of normal operation



---

Avoidance algorithms

Advantages:

Permit higher resource utilization


Costs:

Considerably greater implementation complexity

Rare outside specialized systems



---

When to choose each option

Prefer structural prevention whenever practical.

Runtime detection is appropriate when preventing every possible deadlock would impose unacceptable restrictions, as in many database systems.

Avoidance algorithms are uncommon outside operating-system and academic contexts.


---

Common failure modes

Inconsistent lock acquisition order

Nested locking without ordering rules

Resource cycles spanning multiple subsystems

Independently developed components acquiring shared resources differently



---

Example

The Dining Philosophers problem remains the canonical illustration.

Each philosopher acquires one fork and waits indefinitely for the other.

Every philosopher behaves correctly according to the local rules.

Collectively, none can eat.

The failure is not individual behavior.

It is circular waiting.


---

The Coffman Conditions

Four conditions are required for deadlock to occur.

All four must be present simultaneously.

1. Mutual Exclusion

At least one resource cannot be shared simultaneously.

A mutex is the obvious example.


---

2. Hold and Wait

An execution unit already holding one resource waits while requesting another.


---

3. No Preemption

Resources cannot simply be taken away.

Their owner must release them voluntarily.


---

4. Circular Wait

A cycle exists:

A waits for B.

B waits for C.

C waits for A.


No participant can make progress.


---

Breaking any one of these four conditions is sufficient to eliminate deadlock.

In practice, the least disruptive technique is almost always eliminating circular wait.


---

Decision / Principle — Enforce a Global Lock Order

What it is

Every lock in a system is assigned a consistent acquisition order.

If two locks may ever be held simultaneously, every execution path acquires them in the same sequence.

No exceptions exist.


---

Why it exists

Deadlock requires circular waiting.

Consistent ordering makes circular waiting structurally impossible.

If every execution path acquires:

Lock A
then Lock B
then Lock C

no execution path can legally acquire:

Lock C
then Lock A

The cycle cannot form.


---

Options

Global lock hierarchy

Numeric lock ordering

Hierarchical ownership

Runtime deadlock detection



---

Trade-offs

Advantages:

Simple reasoning

Minimal runtime cost

Eliminates the most common deadlock cause

Easy to review during code review


Costs:

Requires discipline across the codebase

May require refactoring existing code

Can become cumbersome in very large systems



---

When to choose each option

Use global lock ordering whenever multiple locks may be acquired simultaneously.

Reserve runtime detection for systems where strict ordering is impractical.


---

Common failure modes

Local ordering rules that conflict globally

Introducing new locks without updating the hierarchy

Temporary exceptions becoming permanent

Third-party libraries acquiring locks unexpectedly



---

Example

Large codebases often document explicit lock hierarchies so reviewers can verify acquisition order mechanically.

The rule prevents deadlock before execution rather than attempting to recover afterward.


---

Decision / Principle — Livelock

What it is

A livelock occurs when execution units continue changing state and responding to one another, yet no useful work completes.

Unlike deadlock, the system remains active.

It simply never makes forward progress.


---

Why it exists

Well-intentioned retry or avoidance logic can continually interfere with itself.

Each participant repeatedly backs off to allow the other to proceed.

Unfortunately, both make the same decision every time.

The result is endless motion without completion.


---

Options

Common mitigation techniques include:

Randomized backoff

Priority rules

Explicit ownership

Retry limits



---

Trade-offs

Advantages of aggressive retry:

High responsiveness under light contention


Costs:

Oscillation under heavy contention

Wasted CPU time

No useful progress



---

When to choose each option

Introduce randomized behavior or asymmetric decision rules whenever identical participants repeatedly compete for the same resource.


---

Common failure modes

Endless retry loops

Simultaneous yielding

Repeated conflict detection

Protocols continually restarting themselves



---

Example

Two network protocols repeatedly detecting a collision, both backing off identically, then retransmitting simultaneously can produce livelock rather than deadlock.

Activity continues.

Progress does not.


---

Decision / Principle — Starvation

What it is

Starvation occurs when an execution unit is permanently denied the opportunity to make progress despite the system as a whole continuing to function.

Other execution units complete work normally.

One participant waits indefinitely.


---

Why it exists

Scheduling policies and lock-acquisition rules are not automatically fair.

A scheduler may continually prefer newly arriving work.

A lock implementation may repeatedly favor one class of requester.

Correctness alone does not guarantee eventual progress.


---

Options

Starvation can be reduced through:

Fair scheduling

Fair lock implementations

Aging mechanisms

Priority adjustments



---

Trade-offs

Advantages of fairness:

Better progress guarantees

Predictable latency


Costs:

Lower maximum throughput

Less scheduling flexibility

Additional implementation complexity


Maximizing throughput and maximizing fairness are often competing goals.


---

When to choose each option

Favor fairness whenever latency guarantees matter.

Favor throughput-oriented scheduling only when occasional delay is acceptable.


---

Common failure modes

High-priority work permanently dominating lower-priority work

Readers continually delaying writers

Writers continually delaying readers

Continuous arrival of new work preventing older requests from executing



---

Example

Some read-write lock implementations may repeatedly admit new readers while a waiting writer never acquires exclusive access.

The system continues making progress.

One participant never does.


---

Runtime Detection Versus Structural Prevention

Not every system prevents deadlock entirely.

Database systems often choose a different trade-off.

Rather than forbidding every potentially dangerous lock sequence, they monitor active transactions.

A wait-for graph represents which transaction is waiting for which resource.

If the graph develops a cycle, a deadlock exists.

Rather than allowing every transaction to wait forever, systems such as PostgreSQL and MySQL's InnoDB detect the cycle, select one transaction as the deadlock victim, abort it, release its locks, and allow the remaining transactions to continue.

This approach accepts that deadlocks may occur while ensuring they do not become permanent.

It illustrates an important engineering principle: prevention and recovery are distinct strategies.


---

Why Smart Engineers Disagree

The disagreement is usually about where to spend complexity.

One approach favors strict prevention through lock hierarchies and carefully constrained synchronization rules.

The other allows more flexible resource acquisition while investing in runtime detection and recovery.

Operating systems, application servers, and embedded software often prefer prevention because unexpected stalls are unacceptable.

Database systems frequently accept detection because rigid global ordering would unnecessarily restrict transaction execution and reduce concurrency.

Neither strategy is universally superior.

They optimize different costs.


---

State Space Explosion Makes These Bugs Different

Deadlock, livelock, and starvation all derive their difficulty from the same source introduced in Chapter 2.

Each concurrent execution path multiplies the number of possible interleavings.

Most of those interleavings execute correctly.

Only a small subset expose the failure.

A bug that appears once every several million executions may still occur regularly in production systems processing billions of operations.

This is why concurrency failures often disappear when logging is added, when a debugger is attached, or when timing changes for unrelated reasons.

The underlying logic has not changed.

Only the execution order has.
