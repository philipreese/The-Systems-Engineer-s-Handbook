Chapter 74 — Shared State vs. Message Passing

Purpose

Part I introduced one of the fundamental optimization tensions in systems engineering: throughput versus reasoning simplicity. Concurrency is where that tension becomes concrete.

This chapter establishes the two primary coordination models used in concurrent systems:

Shared state, where multiple execution contexts access the same memory and correctness depends on synchronization.

Message passing, where each execution context exclusively owns its own state and communicates by exchanging messages instead of sharing memory.


This decision is the conceptual spine of Part X. Later chapters assume this framing already exists:

Chapter 75 explains how to make shared state safe.

Chapter 76 separates coordination models from execution models (threads, async, and processes).

Chapter 77 explores the failure modes that arise when coordination fails.

Chapter 78 presents the actor model as the strongest expression of message passing.

Coordination Model Is a Separate Design Decision

One of the most common mistakes in concurrent software is treating "threads versus async" as the primary architectural decision.

It is not.

Before deciding how work executes, a system must decide how concurrent work coordinates.

A program may use:

many threads with shared memory,

many threads with message passing,

asynchronous tasks with shared state,

asynchronous tasks communicating over channels,

multiple OS processes sharing memory,

multiple processes exchanging IPC messages.


Execution units and coordination mechanisms are independent dimensions.

Choosing one does not imply the other.

This chapter concerns only the coordination model.


---

Decision / Principle — Shared State

What it is

Multiple concurrent execution contexts access the same mutable memory.

Correctness depends entirely on ensuring that every access obeys synchronization rules.

Without synchronization, concurrent modification becomes undefined behavior.


---

Why it exists

Sharing memory is often the cheapest way to exchange information.

No serialization is required.

No copies are necessary.

Both participants can directly observe the same object.

This minimizes communication overhead and can achieve extremely high throughput when contention is low.

Most traditional operating-system APIs, including POSIX threads, are built around this model.


---

Options

Unsynchronized shared memory (almost never correct)

Shared memory protected by mutexes

Shared memory protected by reader-writer locks

Lock-free shared structures using atomic operations

Compiler-enforced safe sharing (such as Rust)



---

Trade-offs

Advantages:

Minimal communication overhead

Zero-copy data sharing

Efficient access to large in-memory structures

Natural fit for caches, memory pools, and shared indexes


Costs:

Every shared object becomes a synchronization problem.

Correctness depends on discipline.

Reasoning becomes global rather than local.

Small mistakes may only appear under rare timing conditions.

Bugs are frequently nondeterministic.


The performance benefit comes at the cost of substantially greater reasoning complexity.


---

When to choose each option

Shared state is appropriate when:

copying data dominates execution cost,

extremely low latency matters,

concurrent workers genuinely need the same object,

synchronization overhead is lower than serialization overhead.


It becomes less attractive as ownership boundaries become unclear or as the number of interacting components increases.


---

Common failure modes

Unsynchronized writes

Hidden aliasing

Accidental sharing

Lock omissions

Inconsistent invariants during updates


The defining failure mode is the data race, introduced later in this chapter.


---

Example

POSIX threads expose shared-memory concurrency directly.

Every thread may access any shared object.

Correctness depends on programmers consistently protecting shared memory using synchronization primitives such as mutexes.

The operating system provides the mechanisms.

It does not guarantee they are used correctly.


---

Decision / Principle — Message Passing

What it is

Each concurrent execution context exclusively owns its own mutable state.

Other execution contexts cannot directly modify it.

Communication occurs by sending messages through channels, queues, sockets, pipes, or other communication mechanisms.

Ownership moves through communication rather than simultaneous access.


---

Why it exists

Instead of preventing concurrent modification with synchronization, message passing avoids concurrent modification entirely.

If only one execution context owns an object, no second execution context can race with it.

The coordination problem becomes message ordering rather than shared-memory correctness.


---

Options

In-process channels

Queues

Mailboxes

Pipe-based IPC

Network messaging

Actor mailboxes (expanded in Chapter 78)



---

Trade-offs

Advantages:

Eliminates data races by construction

Localizes reasoning

Naturally supports isolation

Clear ownership boundaries

Easier failure containment


Costs:

Messages must be copied or ownership transferred.

Serialization may become expensive.

Communication introduces latency.

Message ordering becomes a design concern.

Large object graphs are more expensive to exchange than to share.


The reasoning model is dramatically simpler, but communication is no longer free.


---

When to choose each option

Message passing is appropriate when:

components have natural ownership boundaries,

isolation is more valuable than maximum throughput,

failures should remain localized,

distributed execution is likely,

concurrency complexity dominates raw performance.


Many modern distributed systems naturally adopt this model because network communication already requires messages.


---

Common failure modes

Although data races disappear, different problems emerge:

mailbox growth,

excessive message copying,

backpressure failures,

protocol mismatches,

message ordering assumptions.


These are coordination problems rather than memory-correctness problems.


---

Example

The Go programming language deliberately encourages this model through its well-known proverb:

> "Do not communicate by sharing memory; instead, share memory by communicating."



Channels provide language-level support for safely transferring information between goroutines without exposing shared mutable state.


---

Decision / Principle — Data Races

What it is

A data race occurs when:

two concurrent accesses target the same memory location,

at least one access is a write,

and no synchronization establishes an ordering between them.


This is the defining failure mode of unsafely shared memory.


---

Why it exists

Processors execute instructions concurrently.

Compilers reorder instructions.

Modern CPUs reorder memory operations.

Without synchronization, neither the compiler nor the processor guarantees which access occurs first—or even whether another execution context observes intermediate states.

A data race is therefore not merely an incorrect result.

It removes the assumptions under which the language defines program behavior.

As discussed in Chapter 33, this is the concurrency-specific instance of undefined behavior.


---

Options

Prevent data races by:

mutexes,

atomic operations,

ownership systems,

immutable data,

message passing.


Ignoring them is not a viable option.


---

Trade-offs

Synchronization:

preserves correctness,

increases contention,

requires careful design.


Ownership systems:

eliminate entire classes of bugs,

impose restrictions on program structure.


Message passing:

removes races entirely,

increases communication cost.



---

When to choose each option

If multiple execution contexts must mutate the same object, synchronization is mandatory.

If they do not need simultaneous ownership, transferring ownership through messages is generally simpler.


---

Common failure modes

Real production symptoms include:

crashes that disappear under debugging,

corrupted internal state,

impossible invariant violations,

intermittent memory corruption,

nondeterministic test failures,

failures reproducible only under production load.


These failures are notoriously difficult to isolate because timing changes alter behavior.


---

Example

A shared counter incremented simultaneously by multiple POSIX threads without synchronization exhibits a classic data race.

The counter may appear correct during testing and fail unpredictably under higher concurrency.


---

Shared State Across FFI Boundaries

Chapter 26 deferred one particularly dangerous concurrency case: shared memory crossing a language boundary.

An FFI boundary removes many of the guarantees provided by a language runtime.

A garbage-collected language may expose a raw pointer into native memory.

Native code may mutate that memory concurrently.

Neither runtime necessarily understands the synchronization assumptions of the other.

A data race across an FFI boundary combines:

undefined behavior from unsafe memory access,

undefined behavior from concurrent mutation,

two independent memory models,

two independent runtimes.


This is one of the highest-risk forms of concurrency in production systems.

Whenever ownership can instead be transferred through explicit messages or immutable values, the boundary becomes substantially easier to reason about.


---

Compiler-Enforced Safety Versus Programmer Discipline

Not all shared-memory systems rely equally on programmer discipline.

This represents another important design choice.

POSIX Threads

POSIX provides synchronization primitives.

Whether they are used correctly is entirely the programmer's responsibility.

The compiler generally cannot detect incorrect synchronization.


---

Rust

Rust takes a fundamentally different approach.

Ownership rules, borrowing rules, and trait requirements such as Send and Sync allow the compiler to reject many programs that could otherwise contain data races.

This is a concrete realization of Principle 8 from earlier chapters: move correctness checking earlier whenever possible.

The programmer still writes concurrent software.

The compiler rejects many invalid concurrent programs before they execute.


---

Go

Go encourages message passing while still permitting shared memory.

The language makes channels first-class, but mutexes remain available.

Go does not eliminate shared-state concurrency.

It intentionally nudges programmers toward message passing as the default design.


---

Why Smart Engineers Disagree

The disagreement is rarely about correctness.

It is about optimization priorities.

Engineers building databases, kernels, high-frequency trading systems, or memory allocators often accept the complexity of shared memory because copying data would dominate execution time.

Engineers building distributed services, cloud platforms, and business systems often prefer message passing because operational simplicity outweighs the communication overhead.

Both positions are rational.

The disagreement reflects different optimization targets, not different definitions of correctness.

The mistake is assuming one model universally dominates the other.


---

Shared State and Message Passing Are Not Opposites

Production systems frequently combine both models.

A service may internally use shared memory protected by synchronization while communicating with other services through messages.

A thread pool may coordinate through lock-free queues while each worker uses ordinary shared objects internally.

A language runtime may use fine-grained locks to implement channels whose public API exposes only message passing.

The architectural question is therefore not which model exists.

It is where each model's boundaries belong.

Well-designed systems keep shared mutable state local and expose message passing across larger architectural boundaries.
