Chapter 75 — Locks: When to Use Them

Purpose

Chapter 74 established that shared mutable state requires synchronization. This chapter examines the primary synchronization mechanism: the lock.

Locks are neither inherently good nor inherently bad. They are a tool for preserving correctness when multiple concurrent execution contexts must access the same mutable state. Like every coordination mechanism, they involve trade-offs.

The central question is not whether to use locks, but how much state each lock should protect. This chapter argues for a deliberately conservative default: prefer coarse-grained locking until evidence shows lock contention is a genuine performance bottleneck. Only then should finer-grained locking or more sophisticated synchronization strategies be considered.

Decision / Principle — Use Locks Only to Protect Shared Mutable State

What it is

A lock guarantees that only one execution context may enter a protected critical section at a time.

Any access that could violate shared-state invariants must occur while holding the appropriate lock.

The lock protects the invariant, not merely the variable.


---

Why it exists

Without synchronization, concurrent modification produces data races (Chapter 74).

A lock establishes an ordering between accesses, ensuring that updates appear atomic to other execution contexts.

Instead of relying on timing, the program explicitly defines when shared state may be modified.


---

Options

Exclusive mutex

Read-write lock

Optimistic concurrency

Lock-free algorithms

Versioning approaches such as MVCC (Chapter 6)



---

Trade-offs

Advantages:

Straightforward correctness model

Well-understood implementation

Broad language and operating-system support

Preserves complex multi-variable invariants


Costs:

Waiting under contention

Reduced parallelism

Risk of programming errors involving lock management

Scalability limits when many workers compete for the same lock


Locks simplify correctness but serialize execution.


---

When to choose each option

Use an exclusive lock whenever:

shared mutable state exists,

updates must preserve multiple related invariants,

correctness matters more than maximum parallelism.


Avoid introducing locks where ownership or immutability can eliminate shared state entirely (Chapter 74).


---

Common failure modes

Protecting only part of an invariant

Forgetting to acquire the lock

Holding locks longer than necessary

Using different locks to guard related state

Assuming operations are atomic when they are not



---

Example

Java's synchronized keyword associates mutual exclusion with an object monitor.

Go's sync.Mutex provides explicit mutual exclusion around critical sections.

Both implement the same fundamental guarantee: only one execution context may mutate protected state at a time.


---

Decision / Principle — Prefer Coarse-Grained Locking First

What it is

Coarse-grained locking protects a relatively large portion of shared state with a single lock.

Instead of many independently synchronized objects, an entire subsystem or data structure may be guarded by one mutex.


---

Why it exists

The primary cost of concurrent software is rarely the lock itself.

It is the reasoning complexity introduced by many independently synchronized objects.

Every additional lock increases the number of possible execution interleavings.

A single lock minimizes those interleavings.

This follows the handbook's broader Principle 6: do not optimize complexity before measuring the need.


---

Options

One lock protecting an entire subsystem

One lock per major data structure

One lock per object

One lock per field


These represent progressively finer granularity.


---

Trade-offs

Advantages:

Easy to reason about

Simple correctness proofs

Lower maintenance burden

Lower risk of synchronization mistakes


Costs:

More contention

Less available parallelism

Reduced throughput under heavy concurrent load


The performance cost is visible.

The reasoning benefit is often much larger.


---

When to choose each option

Begin with coarse-grained locking when:

building a new system,

contention has not been measured,

correctness is still evolving,

maintenance simplicity is a priority.


Only reduce lock scope after profiling demonstrates that lock contention is the limiting factor. Measuring and profiling contention is discussed in Part XII (Chapter 86).


---

Common failure modes

Prematurely introducing dozens of locks

Optimizing for theoretical scalability

Losing track of which lock protects which invariant

Creating synchronization rules too complex for reviewers to verify



---

Example

Many mature systems begin with a single mutex protecting an internal cache or shared map.

Only after production measurements reveal substantial contention do those structures evolve toward partitioned or sharded locking.


---

Decision / Principle — Fine-Grained Locking Is a Performance Optimization

What it is

Fine-grained locking divides shared state among multiple independently protected regions.

Different execution contexts may update different regions simultaneously.


---

Why it exists

If unrelated operations continually block each other despite touching different data, reducing lock scope increases parallelism.

Instead of one large critical section, many smaller ones may execute concurrently.


---

Options

Partitioned locks

Per-object locks

Bucket-level locks in hash tables

Independent locks for unrelated subsystems



---

Trade-offs

Advantages:

Higher throughput under contention

Better CPU utilization

Increased parallel execution


Costs:

More synchronization logic

Many more possible execution interleavings

Higher maintenance cost

Significantly greater risk of deadlock when multiple locks must be acquired (discussed in Chapter 77)


Fine-grained locking improves scalability at the cost of substantially greater reasoning complexity.


---

When to choose each option

Move toward finer granularity only after demonstrating:

measurable lock contention,

stable correctness requirements,

sufficient operational evidence that synchronization—not computation—is limiting throughput.


Do not optimize based on intuition alone.


---

Common failure modes

Excessive lock proliferation

Inconsistent protection boundaries

Complex acquisition rules

Increased debugging difficulty

Synchronization becoming the dominant maintenance burden



---

Example

Large concurrent hash tables often protect each bucket independently rather than protecting the entire table.

This improves concurrent insert and lookup performance, but substantially complicates correctness.


---

Decision / Principle — Read-Write Locks Only Help Read-Heavy Workloads

What it is

A read-write lock distinguishes between readers and writers.

Multiple readers may proceed simultaneously.

Writers still require exclusive access.


---

Why it exists

Many systems perform vastly more reads than writes.

Serializing readers unnecessarily wastes available parallelism.

A read-write lock allows concurrent observation while preserving exclusive modification.


---

Options

Ordinary mutex

Read-write lock



---

Trade-offs

Advantages:

Higher read concurrency

Better throughput for read-dominated workloads


Costs:

More complicated implementation

Greater overhead than a normal mutex

Writers may experience increased waiting depending on fairness policy

No benefit—and often worse performance—when writes are frequent


Read-write locks are specialized tools rather than universal upgrades.


---

When to choose each option

Choose a read-write lock when:

reads vastly outnumber writes,

read operations are long enough for concurrent execution to matter,

measurements show reader contention.


Otherwise, an ordinary mutex is usually simpler and often faster.


---

Common failure modes

Using read-write locks for write-heavy workloads

Assuming every read-mostly structure benefits

Ignoring implementation overhead

Creating starvation problems through poor fairness policies



---

Example

Go exposes both sync.Mutex and sync.RWMutex.

The existence of both primitives reflects different workload assumptions rather than a progression from "basic" to "advanced."


---

Decision / Principle — Consider Lock-Avoiding Alternatives Before Lock-Free Algorithms

What it is

Not every synchronization problem requires increasingly sophisticated locking.

Many systems avoid contention by changing how conflicts are handled rather than making locks more efficient.


---

Why it exists

Reducing synchronization is often more valuable than optimizing synchronization.

Several mature techniques accomplish this by allowing operations to proceed independently and resolving conflicts only when necessary.


---

Options

Optimistic concurrency control

Read-write locks

MVCC

Lock-free algorithms using atomic operations



---

Trade-offs

Optimistic concurrency

Advantages:

No waiting when conflicts are rare

Excellent scalability under low contention


Costs:

Failed operations require retry

Performance degrades rapidly under frequent conflicts



---

MVCC

Chapter 6 examined PostgreSQL's Multi-Version Concurrency Control in detail.

Rather than blocking readers with locks, MVCC presents each transaction with a consistent snapshot while writers create new versions of rows.

This avoids many forms of read-write contention entirely, at the cost of maintaining multiple row versions and performing later cleanup.

The database-specific trade-offs remain exactly those described in Chapter 6.


---

Lock-free algorithms

Advantages:

Avoid blocking caused by traditional locks

Excellent scalability for carefully chosen problems


Costs:

Extremely difficult to design correctly

Depend on subtle atomic-memory semantics

Hard to review, debug, and maintain

Often solve narrower problems than conventional locking


Lock-free code should not be viewed as the natural successor to locks. It is a specialized technique appropriate only after simpler approaches have proven insufficient.


---

When to choose each option

Use optimistic concurrency when conflicts are genuinely uncommon.

Use MVCC when versioning naturally fits the storage model (as discussed in Chapter 6).

Reserve lock-free algorithms for performance-critical components whose bottlenecks have been demonstrated through measurement and whose correctness can be rigorously validated.


---

Common failure modes

Excessive retry loops under contention

Assuming "lock-free" means "simple"

Replacing understandable synchronization with unnecessarily complex atomic algorithms

Choosing sophisticated concurrency mechanisms before measuring actual contention



---

Example

PostgreSQL avoids much reader-writer blocking through MVCC rather than increasingly elaborate lock hierarchies.

The Linux kernel employs a wider range of synchronization mechanisms—including mutexes, spinlocks, and Read-Copy-Update (RCU)—selecting each according to the latency and contention characteristics of the subsystem rather than imposing one universal locking strategy.


---

Why Smart Engineers Disagree

Experienced engineers often disagree about how aggressively to optimize locking.

One school favors introducing finer-grained synchronization early, arguing that redesigning synchronization after deployment is expensive.

Another favors deliberately coarse locking until profiling demonstrates contention, arguing that synchronization complexity becomes a permanent maintenance cost even if the anticipated scalability problem never appears.

This handbook adopts the second position.

Most systems fail from unnecessary complexity far more often than from insufficient lock granularity. Premature synchronization optimization follows the same pattern as premature performance optimization elsewhere: it spends complexity before evidence justifies the expense.


---

Matching Lock Strategy to the Problem

No mature operating system or runtime uses a single synchronization primitive everywhere.

The Linux kernel illustrates this well:

Mutexes protect longer critical sections where sleeping is acceptable.

Spinlocks protect extremely short critical sections where sleeping would cost more than waiting.

RCU enables readers to proceed without traditional locking in read-mostly structures while updates occur through controlled version replacement.


The lesson is not that every system should adopt all three techniques.

The lesson is that synchronization mechanisms are chosen according to workload characteristics, latency requirements, and contention profiles—not according to fashion or perceived sophistication.


---

Looking Ahead

This chapter treated a lock as though it worked correctly in isolation.

Real systems become substantially harder once multiple locks interact. Acquiring locks in inconsistent orders, circular waiting, starvation, livelock, and other coordination failures arise from those interactions rather than from the mechanics of a single lock. Those system-level failure modes are the subject of Chapter 77.

Before examining those failures, however, the next chapter separates another commonly confused axis of concurrent design: the choice of execution unit itself—threads, asynchronous tasks, or processes—which is independent of whether coordination uses locks or message passing.
