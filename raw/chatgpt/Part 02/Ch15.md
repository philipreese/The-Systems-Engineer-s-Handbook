Chapter 15 — API Surface Design: What to Expose, What to Hide

1. Purpose

This chapter defines how to design API surfaces so that they minimize long-term coordination cost while remaining useful to consumers. Every exposed element is a contract. The central constraint is simple: once an API surface is used, it becomes a stability boundary, not a design convenience.

The goal is not expressive APIs, but constrained, evolvable ones.

2. Prerequisites

Readers are assumed to understand:

3. Key Concepts

An API surface is not a reflection of internal structure. It is a selective projection of a system’s capabilities into a stable contract that external systems depend on.

The dominant failure mode in API design is accidental exposure: leaking internal data models into external contracts.

---

Minimal API Surface Principle

What it is:
A principle stating that APIs should expose only the smallest possible set of fields and operations required to satisfy legitimate consumer use cases.

Why it exists:
Every exposed field or operation becomes a long-term commitment. Once consumers depend on it, removing or changing it requires coordination, migration, or deprecation cycles.

Options:

1. Expose full internal data model (“just return the entity”)
2. Expose minimal required subset per use case
3. Over-generalized API exposing everything “for flexibility”

Trade-offs:

* Full exposure maximizes short-term convenience but creates strong coupling to internal structure.
* Minimal surfaces reduce coupling and improve long-term flexibility but require more upfront design discipline.
* Over-generalized APIs increase perceived flexibility but produce unstable, inconsistent consumer usage patterns.

When to choose each option:

* Full exposure: internal-only systems or prototypes with no external consumers.
* Minimal surfaces: stable production systems, especially those with external consumers.
* Over-generalized: rarely justified; only when consumer requirements are truly unknown and iteration speed outweighs stability concerns.

Common failure modes:

* Returning database entities directly in API responses.
* Adding fields “just in case” they might be useful later.
* Allowing internal naming conventions to leak into external contracts.
* APIs that grow without constraint, becoming effectively schema dumps.

Example:
A service exposing a `User` object with internal fields like `internalFlags`, `cacheState`, or `dbVersion` couples external consumers to implementation details. Any internal refactor becomes a breaking API change. In contrast, exposing only `id`, `name`, and `publicProfile` decouples consumers from internal evolution.

Progressive Disclosure in APIs

What it is:
A design approach where APIs expose a simple, minimal interface by default, while advanced or optional capabilities are available through additional parameters, endpoints, or expansion mechanisms.

Why it exists:
Most consumers only need a small subset of functionality. Exposing everything equally creates cognitive overload and increases accidental coupling to advanced features.

Options:

1. Flat API surface (everything exposed equally)
2. Progressive disclosure (simple default + optional advanced features)
3. Segmented APIs (separate endpoints or resources per capability level)

Trade-offs:

* Flat APIs maximize discoverability but overwhelm consumers and encourage misuse of advanced fields.
* Progressive disclosure reduces cognitive load but requires careful design of defaults and optionality.
* Segmented APIs improve clarity but can fragment the mental model of the system.

When to choose each option:

* Flat: small APIs with low complexity and few features.
* Progressive disclosure: most production APIs with mixed consumer sophistication.
* Segmented: large systems with distinct use cases or strongly separated domains.

Common failure modes:

* Optional parameters accumulating until the API becomes unreadable.
* Hidden coupling where “advanced” fields are required in practice but undocumented.
* Inconsistent behavior between default and advanced modes.
* Consumers relying on advanced features as de facto defaults.

Example:
Stripe’s API consistently uses progressive disclosure: simple requests require minimal fields, while advanced options are available but not required. This allows most integrations to remain simple while still supporting complex financial workflows when needed.

Internal vs External API Surfaces

What it is:
A distinction between APIs intended for internal system communication and those exposed to external consumers. Internal APIs optimize for evolution; external APIs optimize for stability.

Why it exists:
Internal consumers can be coordinated and migrated. External consumers cannot. This changes the acceptable rate and cost of change.

Options:

1. Unified API model (same interface for internal and external use)
2. Separate internal and external APIs
3. External API as a strict subset of internal capabilities

Trade-offs:

* Unified APIs reduce duplication but force internal systems to obey external stability constraints.
* Separate APIs increase flexibility but require translation layers and duplication of logic.
* Subset models simplify external guarantees but can restrict internal design evolution.

When to choose each option:

* Unified: small systems or systems without external consumers.
* Separate APIs: mature systems with public-facing interfaces.
* Subset model: systems where external API stability is critical but internal evolution is frequent.

Common failure modes:

* Internal APIs leaking into external contracts unintentionally.
* External constraints constraining internal refactoring.
* Divergence between internal and external models becoming difficult to reconcile.
* Maintaining multiple APIs without clear ownership boundaries.

Example:
Many large systems maintain strict separation between internal service-to-service APIs and public APIs. Internal APIs evolve rapidly to support optimization and refactoring, while public APIs remain stable for years to avoid breaking external integrations.

GraphQL as Consumer-Driven Surface Design

What it is:
A query language for APIs where consumers specify exactly what data they want, rather than relying on server-defined response shapes.

Why it exists:
It attempts to solve over-fetching and under-fetching problems in fixed REST-like APIs by shifting control of data shape to the consumer.

Options:

1. Fixed response APIs (REST-style)
2. Consumer-driven query APIs (GraphQL-style)
3. Hybrid APIs (fixed endpoints + selective query parameters)

Trade-offs:

* Fixed APIs provide predictability and easier caching but may return unnecessary data or require multiple calls.
* GraphQL provides flexibility and reduces over-fetching but increases backend complexity and can obscure system-level coupling.
* Hybrid approaches balance performance and flexibility but increase API surface complexity.

When to choose each option:

* Fixed APIs: stable, performance-critical systems with predictable access patterns.
* GraphQL: systems with highly variable consumer requirements or complex UI-driven data needs.
* Hybrid: systems transitioning between paradigms or with mixed workloads.

Common failure modes:

* N+1 query problems in GraphQL implementations due to naive resolvers.
* Overly flexible queries leading to unpredictable backend load.
* Loss of clear API boundaries, making system behavior harder to reason about.
* Schema becoming overly complex to accommodate all consumer needs.

Example:
GraphQL is widely used in UI-driven systems where clients require different subsets of data depending on context. However, poorly implemented GraphQL layers often shift complexity from the API boundary into backend resolution logic, creating performance and observability challenges.

POSIX as Minimal Stable API Design

What it is:
A long-lived API standard defining a minimal set of operating system interfaces that have remained stable for decades.

Why it exists:
POSIX demonstrates that extreme minimalism and careful constraint of surface area can produce long-lived stability across radically different implementations.

Options:

1. Rich, evolving OS-specific APIs
2. Minimal standardized API surface (POSIX)
3. Hybrid approach with POSIX baseline + extensions

Trade-offs:

* Rich OS-specific APIs provide power and performance but reduce portability.
* POSIX standardization reduces feature exposure but enables portability and long-term compatibility.
* Hybrid models balance portability with innovation but require careful boundary management.

When to choose each option:

* Rich APIs: systems tightly bound to a single platform.
* POSIX-style minimalism: systems requiring portability and long-term stability.
* Hybrid: modern operating systems balancing standard compliance with innovation.

Common failure modes:

* Treating POSIX as restrictive rather than intentionally minimal.
* Building APIs that exceed necessary surface area and then struggling to maintain compatibility.
* Extending APIs without discipline, breaking the minimal stability guarantees.

Example:
POSIX has remained stable for decades because it exposes only the core abstractions needed for portable system programming. Everything else is left to implementation-specific extensions, preserving a stable core contract across diverse systems.

---

Why smart engineers disagree

Disagreement centers on whether API design should prioritize expressive flexibility or strict minimalism.

One view argues that APIs should expose rich, flexible surfaces to minimize friction for consumers, especially in rapidly evolving product environments. This perspective values developer velocity and iteration speed over long-term contract stability.

The opposing view argues that every additional exposed surface becomes a long-term liability. APIs should be minimal, conservative, and stable by default, with complexity pushed inward rather than outward.

Both views are valid depending on consumer scale and stability requirements. Internal APIs with known consumers can tolerate more flexibility. External APIs with unknown or uncoordinated consumers require strict minimalism. The failure mode is applying the wrong exposure strategy to the wrong stability domain.
