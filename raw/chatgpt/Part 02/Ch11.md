Chapter 11 — Layered, Hexagonal, and Ports-and-Adapters Architecture

1. Purpose

This chapter explains how code should be structured inside a single service or application boundary once the decision to build a monolith or service has already been made. The focus is internal architecture: how dependencies are organized, how business logic is isolated, and how infrastructure is prevented from dictating system shape over time.

The central idea: most architectural failures inside a service are not algorithmic failures, but dependency direction failures.

2. Prerequisites

Readers are assumed to understand:

* Coupling and cohesion (Part I)
* Information hiding (Ch 04)
* Monolith vs service decomposition boundaries (Ch 10)
* Basic failure modes of layered systems under evolution
* The concept of infrastructure volatility (databases, frameworks, external APIs)

3. Key Concepts

Inside any non-trivial system, the primary structural problem is not “where code goes,” but “what depends on what.”

Three dominant internal architectures exist because they encode different answers to that dependency question:

* Layered architecture: dependencies flow downward through conceptual layers
* Hexagonal architecture: dependencies flow inward toward a domain core
* Ports-and-adapters: a concrete implementation of hexagonal architecture emphasizing explicit interfaces at boundaries

Each is a different strategy for controlling dependency direction under change.

---

Layered Architecture (N-tier)

What it is:
A structure where code is organized into horizontal layers, typically presentation → business logic → data access, with each layer depending only on the one directly below it.

Why it exists:
Layered architecture became dominant because it matches how engineers initially reason about systems: UI talks to logic, logic talks to data. It is easy to teach, easy to scaffold, and maps cleanly onto early enterprise frameworks like Java EE and early .NET stacks.

Options:

1. Strict layered dependency flow (top-down only)
2. Relaxed layered flow (allowed cross-layer calls for convenience)
3. “Layered in name only” (logical layers but no enforcement)

Trade-offs:

* Strict layering provides clear dependency structure but often slows development due to indirection.
* Relaxed layering improves productivity but introduces hidden coupling between layers.
* “Layered in name only” maximizes short-term speed but eventually collapses into dependency entanglement.

When to choose each option:

* Strict layering: small to medium systems where discipline can be enforced and domain complexity is moderate.
* Relaxed layering: systems under rapid iteration where enforcement overhead is too costly.
* Name-only layering: only acceptable in prototypes or throwaway systems.

Common failure modes:

* Business logic leaking into controllers or data access code, bypassing intended boundaries.
* Repositories becoming “god objects” that contain both query logic and domain rules.
* Circular dependencies between layers as abstractions erode over time.
* “Fat service layer” where all meaningful logic collapses into one unstable layer.

Example:
Django’s MTV pattern behaves like a layered architecture in practice: views (presentation), models (data + some logic), and templates (rendering). Over time, complex Django systems often see business logic drift into views or models depending on developer pressure, demonstrating classic layer leakage.

Hexagonal Architecture (Ports-and-Adapters)

What it is:
An architecture where the core business logic is isolated from external systems. The core defines “ports” (interfaces), and external systems interact through “adapters” that implement those interfaces. The dependency direction always points inward toward the domain.

Why it exists:
It exists to solve a specific problem: infrastructure changes faster than business logic. Databases, message queues, external APIs, and frameworks evolve independently. Hexagonal architecture prevents these changes from contaminating domain logic.

Options:

1. Core defines ports, infrastructure implements adapters (strict hexagonal)
2. Partial hexagonal (only critical boundaries abstracted)
3. Infrastructure-first (core depends directly on external systems)

Trade-offs:

* Strict hexagonal isolates business logic and improves testability but introduces abstraction overhead and interface maintenance cost.
* Partial hexagonal reduces complexity but selectively preserves only some invariants.
* Infrastructure-first minimizes upfront design but leads to tight coupling and expensive future migrations.

When to choose each option:

* Strict hexagonal: systems with long-lived domain logic and high infrastructure churn (e.g., payment systems, core business engines).
* Partial hexagonal: most production systems; apply only where external volatility is high.
* Infrastructure-first: early prototypes or systems tightly bound to a single stable platform.

Common failure modes:

* Over-abstraction: defining ports for stable infrastructure that never changes, creating unnecessary indirection.
* Leakage through adapters: infrastructure-specific behavior creeping into domain logic via convenience shortcuts.
* Interface explosion: too many ports for trivial interactions, increasing cognitive overhead.
* False independence: assuming adapters make systems portable when core assumptions still depend on infrastructure semantics.

Example:
A REST API service refactored into hexagonal structure typically moves database and HTTP frameworks to the outer layer. Business rules (validation, workflows, invariants) are moved into a core service module that only interacts through interfaces like `UserRepository` or `PaymentGateway`, allowing unit tests to run without database or network access.

---

Ports-and-Adapters as a Constraint Model

What it is:
A more explicit framing of hexagonal architecture where all external interactions must pass through declared ports, and all implementations are replaceable adapters.

Why it exists:
It enforces information hiding at the system boundary: the core does not know whether data comes from PostgreSQL, Redis, or an external API.

Options:

1. Explicit ports for all external dependencies
2. Ports only for high-risk or high-volatility dependencies
3. Direct coupling for all external dependencies

Trade-offs:

* Full port isolation maximizes testability and substitution flexibility but increases boilerplate and interface maintenance.
* Selective ports balance abstraction cost with risk management.
* Direct coupling minimizes code overhead but locks the system into infrastructure choices.

When to choose each option:

* Full ports: systems where long-term maintainability and substitution are critical.
* Selective ports: most real-world systems, focusing only on volatile boundaries.
* Direct coupling: stable, low-risk integrations or internal tools with short lifespans.

Common failure modes:

* Mock complexity: test suites become more complex than production logic due to excessive interface abstraction.
* Accidental framework dependence leaking through ports (e.g., ORM entities bleeding into domain layer).
* Misaligned abstraction layers where ports reflect infrastructure shape rather than domain concepts.

Example:
Spring applications often illustrate both layered and hexagonal variants. In traditional layered Spring systems, service classes directly depend on repositories and frameworks. In cleaner hexagonal-style Spring configurations, repositories are abstracted behind interfaces, and domain logic is moved into framework-agnostic services, improving test isolation at the cost of additional configuration.

---

Layered vs Hexagonal: Structural Difference in Dependency Direction

The fundamental distinction is not organizational layout but dependency control:

* Layered architecture: dependency flows downward through functional strata.
* Hexagonal architecture: dependency flows inward toward a domain core.

This difference determines how systems evolve under change:

* In layered systems, change propagates downward unpredictably once leakage occurs.
* In hexagonal systems, change is forced outward, isolating core logic from infrastructure volatility.

However, hexagonal architecture only works when discipline is enforced. Without strict adherence to port boundaries, it degenerates into a layered system with extra indirection.

---

Why smart engineers disagree

The disagreement is not about correctness but about cost distribution.

One perspective argues that layered architecture is sufficient for most systems, and that strict architectural separation introduces unnecessary abstraction overhead. This view is common in teams optimizing for delivery speed and where infrastructure stability is high.

The opposing perspective argues that isolating domain logic from infrastructure is essential for long-lived systems, and that early investment in ports and adapters reduces long-term migration cost and testing complexity.

In practice, both are correct depending on volatility. If infrastructure changes frequently or domain logic is complex and long-lived, hexagonal architecture pays off. If the system is simple, short-lived, or tightly bound to a single platform, layered architecture is often more efficient.

The failure mode is not choosing the wrong pattern—it is applying either pattern without regard to how fast external systems and internal domain rules actually change.
