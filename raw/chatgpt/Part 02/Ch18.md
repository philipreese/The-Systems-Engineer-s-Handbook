Chapter 18 — Data Ownership Boundaries

1. Purpose

This chapter defines how data ownership works in a decomposed system and why it is one of the primary sources of hidden coupling. Once systems are split into services, data becomes the strongest coordination constraint. If ownership is unclear or shared, the system will re-couple at the database layer even if service boundaries appear clean.

The central rule: every piece of mutable data must have a single authoritative owner.

2. Prerequisites

Readers are assumed to understand:

* Service decomposition and boundary design (Ch 10)
* Coupling at system level (Ch 13)
* API surface design (Ch 15)
* Synchronous vs asynchronous communication (Ch 17)
* Event-driven decoupling and partial failure (Ch 13, 17)

3. Key Concepts

Data ownership is not about where data is stored. It is about who is allowed to define its meaning and mutate its state.

In distributed systems, ambiguity in ownership collapses into implicit coupling.

---

Single Data Ownership Principle

What it is:
A rule stating that each domain entity or dataset has exactly one service responsible for its lifecycle and authoritative state transitions.

Why it exists:
Without a single owner, multiple services will encode conflicting assumptions about the same data, leading to schema drift, inconsistent invariants, and cross-service coordination requirements.

Options:

1. Shared ownership (multiple services read/write same data store)
2. Single owner with controlled access via API
3. Single owner with event-driven propagation of derived data

Trade-offs:

* Shared ownership reduces initial friction but creates strong schema coupling and coordination overhead.
* Single ownership enforces clear invariants but requires API or event-based access patterns for other services.
* Event-driven propagation adds flexibility for read models but increases system complexity and eventual consistency.

When to choose each option:

* Shared ownership: almost never appropriate in decomposed systems; acceptable only in transitional monolith-to-service phases.
* Single ownership: default for all stable service architectures.
* Event-driven propagation: when multiple consumers require independent read models or performance isolation.

Common failure modes:

* Multiple services directly writing to the same table with no clear authority boundary.
* Divergent interpretations of shared fields (e.g., “status” meaning different things across services).
* Schema changes requiring coordinated deployments across multiple teams.
* Hidden ownership where one service “technically owns” data but other services mutate it indirectly.

Example:
In a user system, if both a billing service and a profile service directly modify a `users` table, neither service has true ownership. Changes to user schema or lifecycle rules immediately require coordination, recreating a distributed monolith at the persistence layer.

---

Shared Database Anti-Pattern

What it is:
A structural failure where multiple services directly access the same database schema, violating ownership boundaries even if service boundaries exist at the application layer.

Why it exists:
It emerges from convenience: shared storage avoids designing explicit ownership interfaces and avoids designing inter-service communication. It feels efficient locally but collapses system independence globally.

Options:

1. Shared database across services
2. Database-per-service with strict ownership
3. Shared physical infrastructure but logically isolated databases per service

Trade-offs:

* Shared database minimizes initial engineering effort but destroys service autonomy and creates hidden coupling.
* Database-per-service enforces clear ownership and isolation but requires explicit data exchange patterns.
* Logical isolation allows infrastructure sharing while preserving ownership boundaries, but requires discipline to maintain separation.

When to choose each option:

* Shared database: transitional migration states or tightly controlled monolith refactoring phases.
* Database-per-service: default for true service decomposition.
* Logical isolation: infrastructure-constrained environments where full separation is not yet feasible.

Common failure modes:

* Cross-service joins across shared tables.
* Schema changes breaking multiple services simultaneously.
* Services bypassing APIs to access data directly for convenience.
* Emergence of implicit contracts around shared schema that are never formally defined.

Example:
A common microservices failure pattern is multiple services reading and writing to a shared PostgreSQL schema. Although services are deployed independently, any schema evolution becomes a coordinated release problem, effectively eliminating the autonomy the architecture was supposed to provide.

CQRS as Ownership Separation Mechanism

What it is:
Command Query Responsibility Segregation is a pattern where write operations (commands) are handled by the owning service, while read operations are served through separate read-optimized models derived from the authoritative source.

Why it exists:
Directly querying ownership-bound data across services creates coupling and performance bottlenecks. CQRS allows separation of write correctness from read scalability.

Options:

1. Unified read/write model in owner service
2. CQRS with derived read models
3. External services directly querying owner’s database (breaking ownership)

Trade-offs:

* Unified model is simplest but can become a performance bottleneck under read-heavy workloads.
* CQRS improves scalability and decoupling but introduces eventual consistency and synchronization complexity.
* Direct cross-service querying breaks ownership boundaries and recreates shared-database coupling indirectly.

When to choose each option:

* Unified model: low-scale systems or simple domains with balanced read/write load.
* CQRS: high-scale systems or systems requiring specialized query performance.
* Direct querying: rarely acceptable outside of tightly controlled internal systems.

Common failure modes:

* Read models lagging behind write models, causing stale or inconsistent views.
* Overcomplicated projection pipelines that are difficult to debug.
* Treating CQRS as mandatory rather than workload-driven.
* Proliferation of read models without clear ownership or lifecycle management.

Example:
In an order management system, the order service owns the authoritative order state. A separate read model may be built for dashboards or analytics, derived from order events, allowing optimized queries without compromising the integrity of the source of truth.

---

Event Sourcing as Ownership Reinforcement

What it is:
A data model where the source of truth is the sequence of events that led to the current state, rather than the current state itself. The owning service defines and appends events; state is derived from replaying them.

Why it exists:
It strengthens ownership by making all state transitions explicit and append-only, preventing silent mutation of shared state.

Options:

1. State-based persistence (current snapshot only)
2. Event sourcing (append-only event log)
3. Hybrid (state + event log for critical domains)

Trade-offs:

* State-based persistence is simple but loses historical causality.
* Event sourcing preserves full history but increases complexity in querying and reconstruction.
* Hybrid approaches balance simplicity and auditability but require careful consistency design.

When to choose each option:

* State-based: most CRUD systems with low audit requirements.
* Event sourcing: systems requiring auditability, traceability, or complex state evolution.
* Hybrid: systems with selective domains requiring strict traceability.

Common failure modes:

* Event schema evolution becoming difficult to manage.
* Rebuilding state projections becoming expensive at scale.
* Misusing event sourcing for systems that do not require historical reconstruction.
* Confusing event logs with integration events, leading to ambiguous ownership.

Example:
In financial or order systems, event sourcing ensures that every state change (order created, payment authorized, shipment dispatched) is recorded as an immutable event, making the owning service the sole authority over state transitions.

---

Determining Ownership of Shared Entities

What it is:
A rule for resolving ambiguity when multiple services appear to “own” the same entity.

Why it exists:
Entities like “User”, “Order”, or “Product” often span multiple business domains, leading to confusion about which service should own them.

Options:

1. Split ownership across services (partial ownership)
2. Assign ownership to lifecycle authority (single owning domain)
3. Duplicate entity across services with independent lifecycles

Trade-offs:

* Split ownership creates ambiguous invariants and coordination overhead.
* Lifecycle-based ownership provides clear authority but may require other services to adapt to the owner’s model.
* Duplication avoids coordination but risks divergence and inconsistency.

When to choose each option:

* Split ownership: rarely appropriate; usually indicates incorrect decomposition.
* Lifecycle ownership: default approach in well-structured systems.
* Duplication: acceptable when entities are context-specific and do not require global consistency.

Common failure modes:

* Multiple services attempting to define authoritative state transitions.
* Services assuming ownership based on usage frequency rather than lifecycle control.
* Inconsistent entity definitions across bounded contexts.
* Hidden coupling through shared identifiers without shared semantics.

Example:
In most systems, the “User” entity is owned by the identity or authentication service, because it controls creation, deletion, and identity lifecycle. Other services (billing, analytics, profiles) reference the user but do not mutate its core identity state.

---

Why smart engineers disagree

Disagreement centers on how strictly data ownership boundaries should be enforced versus how pragmatically shared access should be allowed.

One view argues for strict single ownership of all mutable data, with strict prohibition of shared database access. This view prioritizes long-term system integrity and avoids hidden coupling at all costs.

The opposing view argues that strict ownership can introduce unnecessary indirection and performance overhead, especially in systems where data is naturally shared or where operational simplicity is more important than strict isolation.

Both views are valid depending on scale and volatility. Strict ownership is essential in large, long-lived distributed systems. Looser ownership models can be acceptable in smaller systems or systems with tightly coordinated teams. The failure mode is allowing convenience-driven shared access to evolve into structural coupling without explicit acknowledgment.
