Chapter 13 — Coupling and Cohesion at the Architecture Level

1. Purpose

This chapter extends coupling and cohesion from code-level concerns (Part I) to system-level boundaries. Once systems are decomposed into services, the dominant failure mode is no longer local structure but inter-service dependency structure. Poor boundary design turns independent services into a tightly coupled distributed system that behaves like a monolith with network latency.

The goal is to define what “good boundaries” actually mean in practice and how to recognize when a system has violated them.

2. Prerequisites

Readers are assumed to understand:

* Coupling and cohesion at the code level (Part I Ch 03)
* Information hiding (Ch 04)
* Partial failure and distributed systems behavior (Ch 07)
* Service decomposition rationale (Ch 10)
* Dependency direction and inversion (Ch 12)

3. Key Concepts

At the architectural level, coupling is not about function calls but about shared state, shared assumptions, and coordination requirements across boundaries.

Cohesion becomes the property of a bounded context: whether a subsystem’s internal model is self-consistent without external knowledge.

A system is well-designed when internal cohesion is high and external coupling is minimal and explicit.

---

Bounded Context (Domain-Driven Design)

What it is:
A bounded context is a self-consistent domain model within which terminology, rules, and data semantics are internally consistent and unambiguous. It defines the maximum scope of high cohesion in a system.

Why it exists:
Large systems inevitably develop multiple interpretations of the same concept (“User”, “Order”, “Account”). Without explicit boundaries, these meanings collapse into inconsistent global models that create hidden coupling.

Options:

1. Single global domain model across the system
2. Multiple bounded contexts with explicit translation at boundaries
3. Implicit boundaries (accidental context separation via code structure or teams)

Trade-offs:

* Global model maximizes conceptual simplicity but breaks down under domain complexity and organizational scale.
* Explicit bounded contexts increase upfront design and translation cost but preserve local consistency and autonomy.
* Implicit boundaries reduce design effort but produce semantic drift and hidden coupling between modules.

When to choose each option:

* Global model: small systems with simple, stable domain semantics.
* Explicit bounded contexts: most production systems with multiple domains or teams.
* Implicit boundaries: prototypes or systems not expected to evolve significantly.

Common failure modes:

* “Shared vocabulary illusion” where the same term is used across services but means different things in each context.
* Over-normalized domain models that attempt to unify incompatible business rules.
* Translation leakage where one bounded context directly depends on another’s internal model.

Example:
In an e-commerce system, “Order” in the payment context refers to financial authorization state, while in fulfillment it refers to shipment preparation state. Treating these as the same object across services creates incorrect coupling and forces conflicting lifecycle rules into a single model.

---

Shared Database Anti-Pattern

What it is:
An architectural failure where multiple services share a single database schema or directly depend on each other’s tables. Despite independent deployment, the services are coupled through shared persistence.

Why it exists:
It emerges as a shortcut to avoid designing explicit service interfaces or data contracts. It preserves convenience at the cost of boundary integrity.

Options:

1. Shared database across services
2. Single service owning the database (no cross-service access)
3. Separate databases per service with explicit data exchange mechanisms

Trade-offs:

* Shared database reduces initial development friction but creates hidden coupling at the schema level.
* Single ownership enforces clear boundaries but requires careful API design for data access.
* Separate databases maximize autonomy but introduce coordination complexity and consistency challenges.

When to choose each option:

* Shared database: almost never appropriate in true multi-service architectures; acceptable only in transitional phases.
* Single ownership: standard for well-defined service boundaries.
* Separate databases: default in mature distributed systems with strong autonomy requirements.

Common failure modes:

* Schema changes requiring coordinated deployments across multiple services.
* Accidental coupling via foreign keys across service boundaries.
* “Read-your-neighbor’s-table” behavior that bypasses service APIs entirely.
* Difficulty evolving services independently due to hidden schema dependencies.

Example:
Early microservice migrations in large systems often retain a shared PostgreSQL database for convenience. Over time, schema changes become tightly coordinated releases, eliminating the independence that justified the service split in the first place. Many such systems eventually either enforce strict ownership rules or collapse back into a monolith.

Event-Driven Decoupling

What it is:
A pattern where services communicate by emitting and consuming events rather than making synchronous calls, reducing temporal coupling between producers and consumers.

Why it exists:
Synchronous communication couples systems in both time and availability. If one service is down, others may block or fail. Event-driven communication breaks this dependency by decoupling execution timing.

Options:

1. Synchronous request/response communication
2. Event-driven asynchronous communication
3. Hybrid approach (synchronous for queries, events for state changes)

Trade-offs:

* Synchronous communication is simple and easy to reason about but creates tight runtime coupling.
* Event-driven systems improve resilience and scalability but introduce eventual consistency and debugging complexity.
* Hybrid approaches balance clarity and resilience but require careful boundary design to avoid inconsistency.

When to choose each option:

* Synchronous: low-scale systems or interactions requiring immediate consistency.
* Event-driven: systems with independent workflows, high scale, or resilience requirements.
* Hybrid: most production systems with mixed consistency needs.

Common failure modes:

* Event explosion where every state change becomes an event without clear semantic boundaries.
* Debugging difficulty due to indirect causal chains across services.
* Event ordering assumptions breaking under distributed execution.
* “Ghost state” where system state is reconstructed from incomplete event histories.

Example:
Kafka-based architectures in large-scale systems often use events like `OrderPlaced`, `PaymentProcessed`, and `ShipmentDispatched` to decouple services. This allows each subsystem to operate independently, but requires careful handling of eventual consistency, such as cases where payment succeeds but fulfillment is delayed or fails.

---

Coupling as a System Property

What it is:
Coupling at the architectural level is the degree to which a change in one service requires changes in another service, measured across data, time, and coordination boundaries.

Why it exists:
Unlike code-level coupling, architectural coupling is often indirect and emerges from shared assumptions rather than explicit references.

Options:

1. Tight coupling (shared state, synchronous calls, shared schemas)
2. Loose coupling (event-driven, independent data models, explicit APIs)
3. Apparent decoupling (services appear independent but share hidden dependencies)

Trade-offs:

* Tight coupling simplifies reasoning locally but destroys independent evolution.
* Loose coupling enables scalability and autonomy but increases system complexity.
* Apparent decoupling creates false confidence while preserving hidden dependency chains.

When to choose each option:

* Tight coupling: systems that must behave as a single consistency unit.
* Loose coupling: systems requiring independent scaling or evolution.
* Apparent decoupling: never intentionally chosen; always a design failure state.

Common failure modes:

* Hidden coupling through shared identifiers or implicit contract assumptions.
* Cascading failure propagation due to synchronous dependency chains.
* Inconsistent data states across services due to partial updates.
* Misaligned evolution where services drift into incompatible assumptions.

Example:
A system where multiple services independently store “User” data but assume consistent schema fields (email, status, preferences) without a shared contract often experiences subtle divergence. Updates in one service silently break assumptions in others, producing inconsistent behavior without obvious runtime errors.

---

Why smart engineers disagree

Disagreement centers on where the boundary between services should sit and how aggressively coupling should be eliminated.

One view argues that strict separation of bounded contexts with asynchronous communication is essential for long-term scalability and organizational autonomy. This perspective prioritizes independent evolution over immediate simplicity.

The opposing view argues that excessive decoupling introduces operational complexity, debugging difficulty, and consistency problems that outweigh the benefits unless system scale or organizational complexity truly demands it.

Both views are valid under different conditions. The key variable is whether coordination cost or consistency cost dominates the system. If coordination cost dominates, decoupling pays off. If consistency and observability dominate, tighter coupling is often more stable. The failure mode is treating decoupling as universally beneficial rather than context-dependent.
