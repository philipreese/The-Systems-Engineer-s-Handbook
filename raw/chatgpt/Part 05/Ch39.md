Chapter 39 — When Not to Test

Purpose

Most discussions of testing ask, "What should we test?"

An equally important engineering question is, "What should we not test?"

Every test has a cost. It must be written, understood, maintained, debugged, and executed. A test suite is an engineering system with its own complexity budget. If tests consume more effort than the confidence they provide, they reduce rather than improve software quality.

This chapter is about recognizing those cases and making the deliberate decision not to write—or to remove—a test.

Key Concepts

This chapter introduces four ideas:

Tests are investments, not obligations.

The goal is confidence in behavior, not maximum test count.

Difficult-to-test code and unnecessary tests are different problems.

Removing low-value tests is sometimes the correct engineering decision.



---

Decision: Don't Test Trivial Code

What it is

Some code is so simple that a test adds little or no additional confidence.

Typical examples include:

trivial getters

trivial setters

one-line delegation methods

simple constructors with no logic


If the test is longer and more complex than the implementation, the implementation is probably not where risk resides.

Why it exists

Testing effort should be concentrated where failures are likely and consequences are significant.

Testing code with essentially no decision-making logic rarely improves confidence meaningfully.

Options

Test every method.

Test only methods containing meaningful behavior.

Rely on higher-level tests to exercise trivial code incidentally.


Trade-offs

Testing everything

Pros

Uniform rules

No judgment required


Cons

Large, noisy test suites

More maintenance

Little additional confidence



---

Testing meaningful behavior

Pros

Higher return on effort

Smaller suites

Easier maintenance


Cons

Requires engineering judgment

Borderline cases exist


When to choose each option

If removing the code would obviously break meaningful behavior, it is probably worth testing.

If removing the code simply eliminates a pass-through function with no decisions, higher-level tests usually provide sufficient coverage.

Common failure modes

Codebases sometimes accumulate thousands of tests whose only purpose is verifying field assignment.

Developers spend time maintaining these tests while genuine business logic receives comparatively little attention.

Example

Testing that a getter returns the value previously assigned by a setter is primarily testing the programming language rather than the application.


---

Decision: Don't Test Someone Else's Framework

What it is

Avoid writing tests whose primary purpose is verifying that established frameworks, libraries, or language runtimes function correctly.

Why it exists

Application tests should verify application behavior.

Framework vendors already test their own software extensively.

Duplicating those tests rarely improves confidence.

Options

Test framework behavior directly.

Test your integration with the framework.

Test only your application logic.


Trade-offs

Testing framework behavior

Pros

Can reveal configuration mistakes


Cons

Frequently duplicates upstream testing

Brittle when frameworks evolve

Low value



---

Testing your integration

Pros

Verifies application behavior

Detects incorrect configuration

Focuses on code you control


Cons

Requires understanding where your responsibility begins


When to choose each option

Test the code you wrote.

Do not test that dependency injection works.

Test that your application behaves correctly after dependencies have been injected.

Do not test that an ORM generates SQL exactly as documented.

Test that your repository correctly stores and retrieves application data.

Common failure modes

Teams sometimes write tests asserting that:

dependency injection creates objects

framework annotations function

routing behaves according to framework documentation


Such tests often fail only when the framework itself changes rather than when the application is incorrect.

Example

A Spring application should test that a service produces correct business results.

It generally should not test that Spring successfully injects constructor dependencies.


---

Decision: Test Public Behavior, Not Private Implementation

What it is

Tests should verify the externally observable behavior of a component rather than its internal structure.

Why it exists

Implementation changes frequently.

Public behavior changes much less often.

Coupling tests to internal implementation dramatically increases maintenance cost without improving user-visible correctness.

Options

Test public interfaces.

Test private methods directly.

Verify internal interaction sequences.


Trade-offs

Behavior-oriented tests

Pros

Stable under refactoring

Reflect user-visible correctness

Encourage encapsulation


Cons

May require broader execution scope



---

Implementation-oriented tests

Pros

Fine-grained diagnostics

Easy to target individual methods


Cons

Fragile

Discourage refactoring

Couple tests to internal design


When to choose each option

If changing internal implementation while preserving externally observable behavior requires rewriting the test, the test is probably verifying the wrong thing.

Common failure modes

Tests inspect:

private helper methods

internal caches

call ordering

intermediate variables


The software continues behaving correctly.

The tests fail anyway.

The failures reflect refactoring rather than regression.

Example

Changing an algorithm from two helper functions to one combined implementation should not invalidate tests if callers observe identical behavior.


---

Decision: Distinguish Hard-to-Test from Not-Worth-Testing

What it is

These are fundamentally different engineering signals.

Hard-to-test code often indicates poor design.

Easy-to-test but low-value code indicates poor test selection.

Why it exists

The wrong response depends on which problem exists.

Poor design should be improved.

Poor tests should be removed.

Options

Refactor difficult code.

Write complicated tests anyway.

Decide the behavior is not worth testing.


Trade-offs

Refactoring

Pros

Improves architecture

Reduces long-term testing cost

Benefits production code


Cons

Requires engineering effort



---

Complicated tests

Pros

Immediate verification


Cons

Expensive maintenance

Difficult debugging

Reinforces poor design



---

Skipping low-value tests

Pros

Smaller suite

Faster execution

Better focus


Cons

Requires thoughtful judgment


When to choose each option

If business logic requires enormous setup because it mixes domain rules with databases, networking, and configuration, improve the design.

If a trivial delegating method requires no meaningful verification, simply do not test it.

Common failure modes

Teams confuse these two situations.

Complex tests become permanent workarounds for architectural problems.

Meanwhile trivial methods receive exhaustive coverage simply because they are easy to test.

The result is effort invested exactly where it produces the least value.

Example

A pricing engine tightly coupled to SQL queries should usually be refactored until pricing logic becomes independently testable.

A simple forwarding method rarely deserves its own test.


---

Decision: Consider the Cost of the Test

What it is

Tests consume engineering resources throughout the lifetime of a project.

Maintenance cost is part of the design decision.

Why it exists

A test is not finished when it is merged.

It must continue evolving alongside the software.

Every unnecessary test increases future maintenance work.

Options

Preserve every historical test.

Periodically remove low-value tests.

Continuously evaluate testing return on investment.


Trade-offs

Keeping every test

Pros

Maximum historical coverage


Cons

Slower execution

Higher maintenance

Increased noise



---

Removing low-value tests

Pros

Faster feedback

Smaller suites

Lower maintenance burden


Cons

Requires periodic review

Some judgment calls are subjective


When to choose each option

A test should justify its ongoing existence.

If a test consistently fails because of harmless refactoring rather than behavioral regression, reconsider whether it is providing useful confidence.

Deleting that test may improve the quality of the suite.

Common failure modes

Over time, suites accumulate:

obsolete tests

duplicated verification

fragile implementation checks

excessive setup


Execution time grows.

Developers stop running the suite locally.

Confidence declines despite increasing test counts.

Example

A continuous integration pipeline requiring forty-five minutes to execute largely because of thousands of low-value tests slows feedback enough to reduce the effectiveness of the entire testing strategy.


---

Decision: Recognize Tests with Negative Return on Investment

What it is

Some tests cost more to maintain than the confidence they provide.

These tests have negative engineering value.

Why it exists

Confidence and maintenance do not increase at the same rate.

Beyond a certain point, additional tests primarily create work.

Options

Keep every existing test.

Replace brittle tests with higher-value ones.

Remove tests that no longer justify their cost.


Trade-offs

Removing tests feels risky.

Keeping ineffective tests creates ongoing costs that are often invisible because they are paid gradually through slower development and more difficult refactoring.

When to choose each option

Ask:

Does this test catch realistic regressions?

Does it survive reasonable refactoring?

Would anyone notice if it disappeared?


If the answer to all three is "no," reconsider whether the test belongs in the suite.

Common failure modes

Mock-heavy suites sometimes reach the point where most of each test consists of mock configuration while the actual behavioral assertion occupies only a few lines.

Such tests frequently verify interaction sequences instead of software correctness.

They become expensive to maintain and surprisingly poor at detecting real defects.

Example

Many public critiques of excessive mocking demonstrate suites that continue passing even after production behavior is broken because mocked collaborators return values that the real systems never would.


---

Why Smart Engineers Disagree

The debate surrounding testing became particularly visible during the 2014 "TDD is dead" discussion, led publicly by David Heinemeier Hansson and others. The criticism was not that testing lacks value, but that some testing practices produce brittle suites, excessive mocking, and designs optimized for testability rather than clarity.

Supporters of strict test-driven development responded that these problems reflect poor testing practices rather than flaws in testing itself.

Both perspectives identify real failure modes.

Poor tests can become a maintenance burden.

Poorly tested systems accumulate regressions.

The engineering challenge is balancing those costs rather than treating either extreme as universally correct.

The Go community provides a useful counterpoint. Many Go projects favor relatively straightforward integration tests for simple packages rather than exhaustive unit tests for every exported function. This reflects a cultural preference for testing meaningful behavior instead of maximizing test counts.


---

Position

Do not measure the quality of a test suite by the number of tests it contains.

Measure it by the confidence it provides relative to its cost.

Test behavior with meaningful risk.

Refactor code that is difficult to test because of poor design.

Avoid writing tests that merely verify trivial implementation, framework behavior, or programming language guarantees.

Finally, recognize that deleting a test can be as valuable an engineering decision as adding one. A smaller suite that developers trust and run continuously provides more value than a larger suite that slows development and is ignored.

Relationship to Later Chapters

This chapter established that testing is an economic decision as well as a technical one.

The remaining chapters assume that the tests which remain are worth keeping:
