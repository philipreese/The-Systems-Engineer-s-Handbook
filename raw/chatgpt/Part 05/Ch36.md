Chapter 36 — When to Mock vs. Use Real Dependencies

Purpose

Choosing between real dependencies and test doubles is one of the oldest and most persistent disagreements in software testing. Nearly every engineering organization develops conventions around the question, yet many teams use inconsistent terminology or adopt practices without understanding the trade-offs that motivated them.

This chapter establishes a shared vocabulary, explains the major schools of thought, and provides a practical decision framework. Later chapters assume these definitions when discussing fixtures, property-based testing, and test organization.

Key Concepts

This chapter introduces four ideas:

Test doubles are different tools with different purposes.

Replacing a dependency changes what a test actually verifies.

Different testing philosophies optimize for different goals.

Modern tooling has shifted the cost-benefit analysis toward using more real dependencies.



---

The Test Double Taxonomy

Before discussing trade-offs, the terminology must be precise. Teams routinely use "mock" to describe every replacement object, but the distinctions matter because they describe fundamentally different testing techniques.

The canonical reference is Martin Fowler's essay "Mocks Aren't Stubs" (2004), which formalized the taxonomy and the philosophical divide surrounding it.

A stub supplies predetermined responses.

It exists solely to provide data required by the test.

The test does not verify whether the stub was called.

Example:

A currency exchange service always returns an exchange rate of 1.25 because the test requires deterministic input.


---

A mock verifies interactions.

The test declares that particular methods must be called, often with specific arguments and in a particular order.

If those expectations are not satisfied, the test fails.

Example:

A payment processor mock expects exactly one call to Charge() with a particular amount.


---

A fake is a genuine implementation with simplified behavior.

Unlike a stub or mock, it performs real work while avoiding production infrastructure.

Examples include:

in-memory repositories

embedded databases

local SMTP servers

temporary object stores


The interface remains the same as production.

The implementation differs.


---

A spy records interactions for later inspection.

Unlike a mock, it does not usually declare expectations before execution.

The test examines the recorded interactions afterward.

Example:

A notification service records every email request, allowing the test to verify that one message was produced with the expected recipient.


---

These distinctions are not academic.

Saying "mock the repository" could reasonably mean four different testing techniques.

Shared terminology reduces misunderstanding.

Decision: Verify Behavior or Verify Interactions?

What it is

Replacing collaborators changes the question a test answers.

A test using real collaborators primarily verifies behavior.

A test using mocks often verifies interactions.

These are related but different forms of confidence.

Why it exists

Software consists of collaborating components.

A test may care about:

the final observable outcome

the sequence of internal collaboration

both


Choosing a test double determines which is emphasized.

Options

Prefer real collaborators.

Replace collaborators with mocks.

Use lightweight fakes.

Use stubs or spies for narrowly defined purposes.


Trade-offs

Real collaborators

Pros

Verify actual behavior

Survive internal refactoring

Better represent production


Cons

May require additional setup

May execute more slowly



---

Mocks

Pros

Strong isolation

Precise verification of interactions

Fast execution


Cons

Couple tests to implementation details

Frequently break during harmless refactoring

Can verify incorrect behavior if expectations are wrong



---

Fakes

Pros

Exercise realistic behavior

Fast execution

Stable over time

Encourage interface-oriented design


Cons

Must remain behaviorally consistent with production

Require maintenance as interfaces evolve



---

Stubs and spies

Pros

Simple

Focused

Useful for deterministic inputs or observation


Cons

Solve narrower problems

Not appropriate as general replacements


When to choose each option

Choose the replacement that removes only the dependency you actually need to remove.

Avoid replacing collaborators simply because a mocking framework makes it easy.

Common failure modes

Large mock-heavy suites often become collections of interaction specifications rather than behavior specifications.

The implementation changes.

The externally visible behavior does not.

Hundreds of tests fail anyway.

The failures reflect rewritten implementation rather than broken software.

Example

A pricing service that internally changes from calling three helper methods to calling one combined helper should not invalidate dozens of tests if the calculated price remains identical.

Behavior remained correct.

Only implementation changed.


---

Decision: Classicist or Mockist Testing?

What it is

Two major schools of thought dominate unit testing.

The classicist (often called the Detroit school) prefers real collaborators whenever practical.

The mockist (often called the London school) replaces collaborators aggressively with mocks, emphasizing interaction verification.

Neither philosophy is irrational.

They optimize for different priorities.

Why it exists

The disagreement centers on what constitutes a "unit."

Classicists define the unit as a cohesive piece of behavior.

Mockists define the unit as a single object isolated from all collaborators.

Those definitions naturally produce different testing strategies.

Options

Classicist testing.

Mockist testing.

A hybrid approach.


Trade-offs

Classicist approach

Pros

Tests observable behavior

More tolerant of refactoring

Less coupled to implementation

Encourages cohesive object models


Cons

May involve more collaborating objects

Slightly larger execution scope



---

Mockist approach

Pros

Strong isolation

Precise interaction verification

Can simplify testing of complex dependency graphs


Cons

High maintenance

Tests implementation rather than outcomes

Difficult to refactor safely


When to choose each option

Classicist testing works particularly well when collaborators are inexpensive and deterministic.

Mockist techniques become more useful when collaborators are genuinely external, non-deterministic, or prohibitively expensive.

Common failure modes

Organizations that aggressively mock every collaborator frequently discover that refactoring becomes unexpectedly expensive because every internal interaction has become part of the testing contract.

The implementation becomes frozen by its own tests.

Example

Martin Fowler's discussion of mockist and classicist testing illustrates how two teams can obtain similar coverage while verifying fundamentally different properties of the software.


---

Decision: Prefer Real or Near-Real Dependencies

What it is

When a dependency is inexpensive, deterministic, and under your control, prefer using the real implementation or a behaviorally equivalent fake.

Replace dependencies only when doing so removes a genuine obstacle to effective testing.

Why it exists

Every replacement object introduces assumptions about production behavior.

Real implementations eliminate those assumptions.

Options

Real dependency.

Fake implementation.

Stub.

Mock.


Trade-offs

Real dependency

Pros

Highest behavioral confidence

Detects integration problems naturally


Cons

May require infrastructure

Can increase execution time



---

Fake implementation

Pros

Fast

Exercises realistic behavior

Maintains the same interface


Cons

Must accurately model production semantics



---

Mocks

Pros

Eliminate external systems completely


Cons

Risk diverging from reality


When to choose each option

Prefer the real implementation when:

the dependency belongs to your system

startup cost is reasonable

deterministic execution is achievable


Prefer a fake when:

infrastructure is unnecessarily heavy

equivalent behavior can be implemented simply


Use stubs or mocks when the dependency is:

a third-party payment processor

an external SaaS platform

a licensed service

a legacy mainframe

a real-time clock

a random number generator

an unreliable network dependency


These systems are external to your control or inherently non-deterministic.

Common failure modes

Tests pass because mocked dependencies return values that production systems never would.

Deployment immediately exposes assumptions hidden by the mocks.

The tests verified the mocks.

They never verified reality.

Example

A domain-driven design application frequently uses an in-memory repository implementing the same repository interface as the production database implementation.

Business logic exercises real repository behavior without requiring database infrastructure.


---

Decision: Reconsider What Counts as "Too Expensive"

What it is

Modern infrastructure has substantially lowered the cost of using real dependencies in automated testing.

The economic assumptions behind extensive mocking have changed.

Why it exists

Historically, creating a fresh PostgreSQL instance for every integration test was impractical.

Container technology dramatically reduced that cost.

Today, disposable infrastructure is often cheaper than maintaining elaborate mocks.

Options

Continue mocking infrastructure.

Run real infrastructure in automated environments.

Mix both approaches where appropriate.


Trade-offs

Real containerized infrastructure

Pros

High confidence

Production-like behavior

Detects configuration issues

Less behavioral drift


Cons

Slower than isolated unit tests

Requires container support



---

Infrastructure mocks

Pros

Fast

Minimal setup


Cons

Can diverge from production

Require ongoing maintenance

Hide integration defects


When to choose each option

If spinning up a real dependency takes only seconds and fits comfortably within the integration testing budget established in Chapter 35, use the real dependency.

Reserve mocking for systems whose operational cost remains genuinely prohibitive.

Common failure modes

Teams continue mocking databases because doing so was once necessary, even though modern tooling makes running real databases inexpensive.

The organization inherits the maintenance cost of mocks without receiving a corresponding reduction in execution cost.

Example

Testcontainers, available across Java, Go, Python, .NET, and other ecosystems, has made disposable PostgreSQL, MySQL, Redis, and similar services practical in continuous integration. Many integration tests that once relied on mocks now execute against the actual software they will encounter in production.


---

Why Smart Engineers Disagree

The disagreement persists because both philosophies solve real problems.

Mock-heavy testing emphasizes isolation and fast execution.

Behavior-oriented testing emphasizes confidence that the software actually works.

Different organizations place different values on those outcomes.

Historically, infrastructure costs favored aggressive mocking.

Modern tooling has shifted that balance by making real infrastructure significantly cheaper to provision and automate.

The underlying trade-off remains, but the economic assumptions have changed.


---

Position

Prefer real or near-real dependencies whenever the cost is acceptable.

Replace dependencies when they are outside your control, introduce unavoidable non-determinism, or are genuinely impractical to run during automated testing.

When a replacement is necessary, prefer the least artificial option.

A fake that implements the same contract usually provides stronger confidence than a mock that merely verifies method calls.

Mocks remain valuable tools, but they should solve specific problems rather than define the default testing strategy.

Tests should primarily verify that the system behaves correctly, not that its internal implementation followed one particular sequence of method calls.
