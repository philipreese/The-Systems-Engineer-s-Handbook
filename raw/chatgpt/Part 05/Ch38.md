Chapter 38 — Property-Based Testing

Purpose

Most tests are example-based: they specify a particular input and verify the expected output. This works well for many kinds of software, but it has an obvious limitation—the engineer must think of every interesting example.

Property-based testing (PBT) approaches the problem differently. Instead of enumerating examples, it defines properties that should hold for entire classes of inputs and lets the testing framework search for counterexamples automatically.

Property-based testing is not a replacement for example-based testing. It is a complementary technique that is particularly effective for algorithmic code with large input spaces and well-defined invariants.

Key Concepts

This chapter introduces four ideas:

Properties describe universal correctness, not individual examples.

Random generation explores input spaces humans rarely consider.

Shrinking makes randomly discovered failures understandable.

Property-based testing is valuable only when correctness can be expressed as an invariant.



---

Example-Based vs. Property-Based Testing

An example-based test answers the question:

"Does this input produce the expected output?"

For example:

sort([3,1,2]) produces [1,2,3]

encode("abc") produces "YWJj"


Each test describes one specific example.

A property-based test instead asks:

"What must always be true?"

For example:

decoding an encoded value returns the original value

sorting preserves the collection's length

sorting preserves every element

compressing and then decompressing returns the original data


The framework generates hundreds or thousands of inputs automatically and attempts to find one that violates the stated property.

Instead of writing hundreds of examples, the engineer specifies one invariant.


---

Decision: Test Properties, Not Just Examples

What it is

Property-based testing verifies that a universal statement holds across a large generated input space.

Rather than asserting:

f(3) == 7

it asserts something like:

"For every valid input x, decode(encode(x)) == x."

Why it exists

Large input spaces cannot be explored manually.

Humans naturally choose representative examples.

Computers excel at exploring unusual, unexpected, and pathological combinations.

Property-based testing combines human insight about correctness with automated exploration of inputs.

Options

Example-based testing.

Property-based testing.

Combining both.


Trade-offs

Example-based testing

Pros

Easy to understand

Documents expected behavior

Excellent for edge cases discovered through experience


Cons

Limited input coverage

Relies on human imagination

Easily misses unexpected combinations



---

Property-based testing

Pros

Explores large input spaces

Finds unexpected edge cases

Encourages precise specifications

Produces broad confidence from relatively little code


Cons

Requires defining meaningful properties

Less suitable for behavior without clear invariants

Generated failures can initially appear unfamiliar


When to choose each option

Use example-based tests to communicate expected behavior and document important cases.

Use property-based tests when correctness is naturally expressed as something that should hold for every valid input.

Most systems benefit from both.

Common failure modes

Teams sometimes replace readable example-based tests with property-based tests entirely.

The result is a suite that verifies correctness but communicates very little about expected behavior.

Property-based testing supplements examples.

It does not replace them.

Example

A Base64 encoder should satisfy:

decode(encode(x)) == x

for every valid byte sequence.

Writing a handful of example strings provides useful documentation.

Generating thousands of random byte arrays provides confidence that the implementation handles cases humans would never think to write.


---

Decision: Let the Framework Search for Counterexamples

What it is

Property-based testing frameworks generate inputs automatically.

When they discover a failing input, they repeatedly simplify it until the smallest reproducing case remains.

This process is known as shrinking.

Why it exists

A random failure involving a 5,000-element list is difficult to understand.

A minimal failure involving two elements often reveals the defect immediately.

Shrinking transforms random exploration into practical debugging.

Options

Manual examples only.

Random generation without shrinking.

Random generation with shrinking.


Trade-offs

Manual examples

Pros

Predictable

Easy to debug


Cons

Limited exploration



---

Random generation without shrinking

Pros

Broad exploration


Cons

Difficult debugging

Large failing inputs



---

Shrinking

Pros

Minimal counterexamples

Faster diagnosis

Repeatable failures


Cons

More sophisticated framework implementation

Can obscure the original generated input if misunderstood


When to choose each option

Whenever property-based testing is used, shrinking should be considered an essential capability rather than an optional convenience.

It is one of the reasons modern property-based testing is practical.

Common failure modes

Without shrinking, engineers often spend more time simplifying generated inputs than fixing the underlying defect.

Large random failures discourage adoption because debugging becomes unnecessarily difficult.

Example

Suppose a parser fails on a randomly generated document containing several thousand tokens.

A shrinking algorithm may reduce that failure to:

[]

or

"("

The minimal input frequently exposes the underlying bug immediately.


---

Decision: Use Property-Based Testing Where Input Spaces Are Large

What it is

Property-based testing is particularly effective when manually enumerating interesting inputs is unrealistic.

Why it exists

Some algorithms operate over effectively infinite input spaces.

Writing representative examples provides only limited confidence.

Universal properties provide stronger guarantees.

Options

Apply property-based testing to:

parsers

serializers

encoders

decoders

compressors

search algorithms

mathematical functions

data structures


Or rely solely on manually written examples.

Trade-offs

Property-based testing excels where correctness has mathematical or structural meaning.

It provides much less value where correctness is primarily contextual or procedural.

When to choose each option

Property-based testing is especially valuable for:

Round-trip invariants

encode/decode

serialize/deserialize

compress/decompress



---

Algebraic properties

commutativity

associativity

distributivity

idempotency



---

Data structure invariants

balanced trees remain balanced

heaps preserve heap ordering

hash tables maintain lookup correctness



---

Transformations

sorting preserves elements

normalization is stable

formatting and parsing remain consistent


These properties naturally describe correctness across an enormous number of inputs.

Common failure modes

Teams often write hundreds of individual examples for algorithmic code when a handful of carefully chosen properties would exercise vastly more behavior.

The result is larger test suites with weaker coverage of the actual input space.

Example

A sorting algorithm should satisfy all of the following:

output is ordered

output length equals input length

output contains exactly the same elements

sorting an already sorted list changes nothing


These properties collectively verify far more behavior than dozens of handwritten examples.


---

Decision: Recognize When Property-Based Testing Is the Wrong Tool

What it is

Not every problem has useful universal properties.

Property-based testing depends on the ability to define correctness independently of individual examples.

Why it exists

Many systems derive their complexity from interaction rather than computation.

The challenge is not generating inputs.

It is specifying expected behavior.

Options

Property-based testing.

Example-based testing.

A combination.


Trade-offs

Property-based testing becomes less useful as defining the invariant becomes more difficult.

Example-based testing becomes more valuable when specific scenarios communicate requirements more clearly than abstract properties.

When to choose each option

Property-based testing is usually not the right primary tool for:

user interfaces

multi-step workflows

complex business processes

interaction-heavy state machines

domains dominated by exceptional cases

behavior defined by policy rather than mathematical rules


These systems often require carefully selected examples because correctness depends on context rather than universal laws.

Common failure modes

Engineers sometimes attempt to force property-based testing onto domains where no meaningful invariant exists.

The resulting properties become vague or trivial:

"the application should not crash"

"the result should be valid"


Such properties rarely provide useful confidence because they do not describe correctness precisely.

Example

Testing an e-commerce checkout flow is usually better served by representative business scenarios.

Testing a JSON parser is an excellent candidate for property-based testing because correctness can be expressed independently of individual documents.


---

Why Smart Engineers Disagree

The primary disagreement concerns effort.

Supporters argue that defining good properties produces stronger specifications and exposes subtle defects impossible to discover manually.

Skeptics observe that many business systems lack clean mathematical invariants, making property design more difficult than writing examples.

Both observations are correct.

Property-based testing provides exceptional value when correctness has a clear structural definition.

It provides much less value when correctness depends on policy, workflow, or human expectations.

The question is not whether property-based testing is superior.

The question is whether the software naturally admits useful properties.


---

Real Systems

Property-based testing originated with QuickCheck, introduced for Haskell in 1999 by John Hughes and Koen Claessen. QuickCheck established the core model of random input generation coupled with automatic shrinking, and it demonstrated that many subtle defects could be discovered by specifying properties instead of enumerating examples.

One of the most influential production successes came from Ericsson, where QuickCheck uncovered defects in Erlang protocol implementations that had survived years of conventional testing. This remains one of the canonical demonstrations of property-based testing's ability to expose edge cases that manual example selection missed.

Modern implementations have extended the approach across many ecosystems.

Hypothesis has become the dominant Python implementation, notable for combining shrinking with a persistent example database that remembers previously discovered failures and interesting inputs.

fast-check brings the same model to JavaScript and TypeScript, providing generators and shrinkers for modern web applications and libraries.

Despite differences in language and implementation, these frameworks all embody the same principle: engineers define correctness, and the framework searches aggressively for counterexamples.


---

Position

Property-based testing should be viewed as a force multiplier for example-based testing, not a replacement.

Use examples to document expected behavior and important business scenarios.

Use properties to explore input spaces that humans cannot realistically enumerate.

When a function has a large input domain and a precise invariant, property-based testing often provides more confidence per line of test code than writing dozens of additional examples.

When no clear invariant exists, writing good examples remains the better engineering choice.
