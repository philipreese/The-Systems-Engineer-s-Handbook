Chapter 35 — What Belongs at Each Layer

Purpose

Chapter 34 established that a healthy test suite is structured as a pyramid. This chapter answers the next practical question: given a particular piece of code, where should it be tested?

This is not a classification exercise. It is a design exercise. The layer that naturally tests a piece of code reveals what kind of responsibility that code has. If it is difficult to determine where something belongs, the design itself is often the problem.

The goal is not to maximize the number of tests at any layer, but to place each behavior at the lowest layer capable of verifying it correctly.

Key Concepts

This chapter introduces four core ideas:

Every responsibility has a natural testing layer.

Business logic and infrastructure should be tested differently.

Architectural boundaries determine testing boundaries.

Test organization should reflect code organization.



---

Decision: Unit-Test Pure Behavior

What it is

Unit tests verify behavior whose correctness depends only on inputs, outputs, and the logic between them.

These tests execute without databases, networks, filesystems, clocks, or other external systems.

Typical examples include:

business rules

domain logic

validation

parsing

formatting

calculations

state transitions

data transformations


If the same inputs always produce the same outputs, the behavior naturally belongs at the unit layer.

Why it exists

Pure behavior is inexpensive to execute, deterministic, and easy to reason about.

Because no external systems participate, failures usually identify the defect immediately.

This layer provides the highest confidence per unit of maintenance effort.

Options

Test logic as isolated units.

Test the same logic indirectly through integration.

Test it only through end-to-end workflows.


Trade-offs

Unit tests

Pros

Extremely fast

Precise diagnostics

Minimal setup

Stable over time

Encourage small, composable designs


Cons

Cannot verify infrastructure

Cannot detect integration failures



---

Higher-level verification

Pros

Exercises more realistic execution paths


Cons

Slower

Harder to debug

More expensive to maintain

Often duplicates lower-level verification


When to choose each option

Choose unit tests whenever correctness depends solely on computation.

If adding a database, HTTP request, filesystem, or message queue contributes nothing to verifying the behavior, they should not be present.

Common failure modes

Business rules frequently migrate into controllers, repositories, or service adapters.

The resulting code requires databases or web servers simply to verify tax calculations, validation rules, or pricing logic.

The tests become integration tests by accident rather than necessity.

Example

A pricing engine, discount calculator, or parser should be almost entirely unit-tested. The logic is independent of how inputs are obtained or outputs are stored.


---

Decision: Integration-Test Boundaries

What it is

Integration tests verify that multiple components collaborate correctly across architectural boundaries.

The interaction itself is the subject under test.

Typical examples include:

database access

ORM mappings

serialization

message queues

HTTP clients

cache implementations

adapter implementations

external service integrations


Why it exists

Many failures emerge only when independently correct components interact.

A repository may contain perfectly valid SQL syntax yet map columns incorrectly.

An HTTP client may serialize requests incorrectly despite correct business logic.

A unit test cannot detect these failures because they occur at the boundary.

Options

Test against real infrastructure.

Test against lightweight substitutes that closely match production behavior.

Replace the boundary entirely.


Trade-offs

Real infrastructure

Pros

High confidence

Detects configuration issues

Verifies actual behavior


Cons

Slower

Requires environment setup

May complicate CI



---

Near-real infrastructure

Examples include ephemeral databases or containerized dependencies.

Pros

Much of the confidence of production systems

Easier automation


Cons

Still slower than unit tests

May differ subtly from production



---

Replacing the boundary

Pros

Fast execution


Cons

Risks testing assumptions instead of reality


When to choose each option

Whenever the correctness of an adapter depends on communicating with another component, that communication should be exercised.

A database repository should be tested against an actual database engine or a very close equivalent.

An HTTP adapter should communicate with an implementation that behaves like the real service.

The boundary is precisely what is being verified.

Common failure modes

Testing ORMs exclusively with mocked repositories.

Testing SQL without executing SQL.

Testing serialization without parsing serialized output.

Testing HTTP clients without exchanging HTTP messages.

These tests often pass while production immediately fails.

Example

A repository using PostgreSQL should execute real queries against PostgreSQL or an equivalent temporary instance. Modern tooling such as Testcontainers makes disposable infrastructure practical, allowing adapters to be validated against real systems rather than simulated behavior.


---

Decision: End-to-End Test Complete User Journeys

What it is

End-to-end tests verify complete workflows spanning the assembled system.

They answer the question:

"Can a real user successfully accomplish this task?"

Typical examples include:

user registration

checkout

authentication

password reset

order fulfillment

document upload


These workflows intentionally cross multiple architectural boundaries.

Why it exists

Some failures only emerge after every component participates.

Configuration mistakes.

Deployment problems.

Routing errors.

Authentication failures.

Incorrect interaction between independently correct services.

Only an end-to-end test can reveal these issues.

Options

Test every workflow.

Test only critical workflows.

Rely entirely on lower layers.


Trade-offs

Testing every workflow provides broad confidence but becomes expensive very quickly.

Testing only critical workflows minimizes maintenance while preserving confidence in the application's essential capabilities.

Relying exclusively on lower layers risks missing deployment and configuration failures.

When to choose each option

Reserve end-to-end tests for workflows whose value comes from validating the assembled system.

Individual business rules should not require browser automation simply because users eventually encounter them.

Common failure modes

Large suites gradually accumulate hundreds of nearly identical workflows differing only in small variations.

Execution time grows.

Flakiness increases.

Developers stop trusting failures.

Eventually unstable tests are disabled rather than repaired.

Example

An online store might maintain end-to-end tests for:

customer login

purchasing an item

payment processing

order confirmation


It does not require browser tests for every discount rule, shipping calculation, or validation function. Those belong lower in the pyramid.


---

Decision: Let Testability Reveal Design Quality

What it is

Difficulty writing a unit test often indicates a design problem rather than a testing problem.

The test is exposing coupling.

Why it exists

Well-designed software naturally separates computation from interaction.

Poorly designed software mixes them together.

When business logic depends directly on databases, filesystems, clocks, network calls, or UI frameworks, isolating behavior becomes difficult because the responsibilities are already intertwined.

Options

Accept tightly coupled code.

Refactor toward explicit architectural boundaries.

Push more verification upward into integration tests.


Trade-offs

Accepting coupling avoids immediate refactoring but steadily increases testing cost.

Refactoring improves long-term maintainability but requires architectural discipline.

Moving tests upward postpones the design problem rather than solving it.

When to choose each option

If writing a unit test requires constructing half the application, reconsider the design before reconsidering the test.

Code with explicit dependencies, clear interfaces, and isolated domain logic naturally becomes easy to verify.

Common failure modes

Classes that:

create database connections internally

perform HTTP requests during business logic

read configuration directly

construct collaborators internally

depend on global state


These responsibilities make unit testing difficult because they eliminate architectural seams.

Example

Hexagonal architecture provides a natural testing strategy.

The domain core contains business rules and is verified almost entirely with unit tests.

Adapters are verified with integration tests against real infrastructure.

Ports define the contracts between them.

The architecture itself determines the testing layers rather than requiring engineers to invent them afterward.


---

Decision: Organize Tests Alongside the Code They Verify

What it is

Test organization should mirror the structure of the codebase.

The placement of tests is an architectural decision, not merely a filesystem convention.

Two common approaches dominate.

Co-location

Tests live next to the implementation they verify.

Parallel test tree

Production code and tests exist in separate directory hierarchies with mirrored structures.

Why it exists

Tests evolve together with the code they verify.

Keeping related artifacts close improves discoverability and reduces maintenance overhead.

Options

Co-locate tests with source files.

Maintain a separate test tree.


Trade-offs

Co-location

Pros

Strong locality

Easier navigation

Reinforces package-by-feature organization

Encourages maintaining tests alongside implementation


Cons

Production directories contain additional files

Some organizations prefer stricter separation



---

Parallel test tree

Pros

Clear distinction between production and test code

Familiar in many ecosystems


Cons

Mirrored structures drift over time

Navigation becomes more cumbersome

Feature ownership becomes less obvious


When to choose each option

Projects organized by feature generally benefit from co-location because tests become part of the feature itself.

Projects with strong ecosystem conventions around separate test directories may reasonably follow those conventions.

Consistency across the codebase matters more than the specific choice.

Common failure modes

A parallel test tree gradually diverges from production structure.

Features are reorganized.

Tests remain behind.

Eventually locating the relevant tests requires searching rather than navigation.

Example

The Go toolchain strongly endorses co-location by discovering _test.go files that reside alongside the code they test. Go also supports black-box testing through packages with the _test suffix, allowing tests to exercise only the public API while remaining co-located.

By contrast, pytest commonly discovers tests from dedicated tests/ directory trees, demonstrating that parallel organization can also work effectively when applied consistently.

The important architectural principle is that the organization of tests should reinforce, rather than obscure, the organization of the system.


---

Why Smart Engineers Disagree

The strongest disagreement is usually not about whether different behaviors belong at different testing layers, but about where particular boundaries should exist.

One team may consider a repository part of the domain and test it with lightweight integration tests.

Another may treat the repository as a thin adapter and keep nearly all business behavior in a domain core composed entirely of unit tests.

Both approaches can succeed if the architectural boundaries are explicit and consistently applied.

The testing strategy follows the architecture. If responsibilities are clearly separated, the correct testing layer is usually obvious.
