Chapter 21 — Error Handling Contracts

1. Purpose

Errors are not “edge behavior.” They are part of the API contract.

If a client can rely on a success response shape but not an error response shape, the system is only half-specified. In production systems, that asymmetry is where most integration failures actually come from — not the happy path, but inconsistent or underspecified error surfaces.

This chapter defines what a correct, stable, machine-readable error contract looks like across REST and REST-like APIs, independent of retry policy or transport mechanics.


---

2. Prerequisites

Assumed:

API surface area includes both success and failure responses (Ch 15)

Backward compatibility applies to all wire-level contracts (Ch 16)

HTTP semantics for status codes at a basic level (4xx vs 5xx)

Resource or RPC-style API already chosen (Ch 19–20)


Not covered here:

Retry strategy or idempotency behavior (Ch 22)

Auth-specific semantics (Ch 24)

System-level reliability design (Part I)

3. Key Concepts

Error responses are first-class contract artifacts

Status codes alone are insufficient as error semantics

Structured error payloads are required for predictable clients

Partial failure invalidates single-status-code assumptions



---

Structured Error Response Model

Machine-Readable Error Contract

What it is: A consistent, structured format for representing failure across all endpoints.

Why it exists: To ensure clients can reliably branch on errors without parsing free-form text or guessing from HTTP status alone.

Options:

Minimal HTTP status-only errors

Unstructured message-only JSON errors

Structured error objects (code + message + metadata)


Trade-offs:

Structured: predictable client behavior, SDK generation possible


Minimal: simplest possible implementation


− Minimal: forces clients to infer meaning from brittle signals

− Unstructured: impossible to automate or standardize handling


When to choose each:

Structured error objects: default for any public or multi-client API

Minimal errors: internal prototypes or throwaway services

Unstructured: avoid in production systems


Common failure modes:

Returning only "error": "something broke" without machine-readable classification

Inconsistent error shapes across endpoints

Encoding error semantics in human messages instead of structured fields


Example:

Stripe error object:

type: high-level category (e.g. invalid_request_error)

code: precise machine-readable reason

message: human-readable explanation

param: field-level attribution when applicable

HTTP Status Codes as a Semantic Layer

Status Code Semantics Are a Coarse Signal, Not the Full Model

What it is: HTTP status codes classify broad failure categories, not domain-specific error states.

Why it exists: To give a transport-level signal for caching, retry logic, and routing behavior.

Options:

Strict semantic use (precise 4xx/5xx classification)

Loose usage (generic 400/500 for most failures)

Misuse (always 200 with embedded error payloads)


Trade-offs:

Strict: enables reliable retry and observability behavior


Loose: faster implementation, less thought required


− Loose: destroys client ability to differentiate failure types

− Misuse: breaks intermediaries, monitoring, and SDK assumptions


When to choose each:

Strict semantic usage: production APIs with multiple clients

Loose usage: internal systems with tightly controlled clients

Misuse: never; it eliminates HTTP-level reasoning entirely


Common failure modes:

Returning 200 OK with an embedded error body

Mapping all client errors to 400 Bad Request

Mapping all server issues to 500 Internal Server Error

Using status codes as application logic switches instead of coarse classification


Example:

Stripe uses HTTP status codes correctly as a first-pass classifier, then refines meaning in the error body

GitHub similarly uses 4xx/5xx properly while encoding specific reasons in JSON payloads

Client Error vs Server Error Boundary

4xx vs 5xx as a Retry Semantics Contract

What it is: A strict contract about who is responsible for failure and whether retrying is meaningful.

Why it exists: To prevent clients from blindly retrying failures that will never succeed, or failing to retry transient server conditions.

Options:

Strict classification (4xx = do not retry, 5xx = retry may succeed)

Soft classification (client heuristics decide retry behavior)

Ignored classification (clients retry everything or nothing)


Trade-offs:

Strict: reliable automation of retry/backoff logic


Soft: flexibility in ambiguous cases


− Soft: inconsistent behavior across clients

− Ignored: either wasted load or missed recovery opportunities


When to choose each:

Strict: any system with automated retries (most production APIs)

Soft: specialized clients with domain-specific retry intelligence

Ignored: only acceptable in very simple systems


Common failure modes:

Treating all 5xx as permanent failures

Retrying 4xx errors like 429 or validation failures (causing load amplification)

Misclassifying validation errors as server failures or vice versa

Losing distinction between “bad request” and “temporary unavailability”


Example:

Google APIs clearly separate retryable vs non-retryable errors in their gRPC status model

Stripe distinguishes invalid requests (non-retryable) from rate limits and server issues (retryable)

Partial Failure in Batch Operations

The Single Status Code Problem

What it is: Batch operations often contain multiple independent sub-operations, each of which may succeed or fail independently.

Why it exists: Single HTTP status codes assume atomic success/failure, which breaks under partial execution.

Options:

All-or-nothing batch (fail entire request on any error)

Partial success with per-item results

Split into multiple independent requests


Trade-offs:

All-or-nothing: simple semantics, easy to reason about


Partial success: realistic representation of distributed execution


− All-or-nothing: wastes successful work, poor efficiency at scale

− Partial success: more complex client logic and response parsing


When to choose each:

All-or-nothing: when atomicity is required (financial transfers, strict invariants)

Partial success: bulk ingestion, bulk updates, analytics pipelines

Split requests: when failure isolation is more important than efficiency


Common failure modes:

Returning 200 OK for batch requests but silently dropping failed items

Returning 400 for entire batch when only one item fails

Not exposing per-item error details, making partial debugging impossible

Clients incorrectly assuming batch success implies full success


Example:

Stripe bulk operations expose per-item error results where applicable

Many data ingestion APIs return mixed result arrays with success/failure per element

AWS batch APIs commonly return structured per-item success/failure maps

Industry Convergence Models

RFC 7807 — Problem Details for HTTP APIs

What it is: A standardized structure for HTTP error responses: type, title, status, detail, instance.

Why it exists: To unify error representation across heterogeneous HTTP APIs.

Trade-offs:

Standardization across ecosystems


Clear machine-readable baseline


− Not universally adopted

− Too generic for rich domain-specific error modeling


When to choose:

Good default baseline for HTTP APIs without existing error contract

Often extended rather than used directly


Common failure modes:

Treating RFC 7807 as sufficient without domain-specific extensions

Ignoring its structure entirely and reinventing incompatible formats



---

Google / gRPC Error Model

What it is: A structured error system with:

status (canonical error category)

code (machine-readable enum)

details (typed structured metadata)


Why it exists: To support strongly typed RPC systems with consistent error semantics across services.

Trade-offs:

Strong schema enforcement across distributed systems


Works well with code generation


− Less human-readable without tooling

− More rigid evolution constraints than JSON-based models


When to choose:

Internal service architectures using gRPC or similar RPC systems

Systems where tooling and schema consistency dominate flexibility concerns


Common failure modes:

Overloading details with unstructured data

Failing to map domain-specific errors cleanly into canonical codes


Example:

Google Cloud APIs use this model extensively across services

Kubernetes API errors follow similar structured patterns internally



---

Why Smart Engineers Disagree

The disagreement is not about whether errors should be structured — that is settled in practice.

The disagreement is about where semantics belong:

HTTP purists argue:

Status codes should carry as much meaning as possible

Error bodies are secondary metadata


API designers (Stripe-style) argue:

Status codes are a coarse transport signal

Real semantics belong in structured payloads


RPC systems argue:

HTTP is irrelevant entirely; error is a typed return value



The tension is between:

leveraging transport semantics (HTTP)

vs. defining transport-agnostic contracts (RPC/event-style thinking)


In practice, robust systems treat HTTP status as a classifier, not a full semantic system.


---

Closing Structural Insight

A system without a stable error contract is not actually an API — it is a set of probabilistic behaviors that sometimes fail in consistent ways.

The real contract is not “what happens when things work,” but “what failure means, how it is classified, and what a client can deterministically know from it.”

Everything else in API design assumes this layer is already correct.
