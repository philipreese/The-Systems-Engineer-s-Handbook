Chapter 19 — REST vs. RPC vs. Event-Driven

1. Purpose

Once you’ve decided whether an interaction is synchronous or asynchronous (Ch 17), you still haven’t decided what the API is. That next layer is the wire model: whether the system exposes resources, procedures, or events.

This chapter is about the structural shape of communication on the wire. Not semantics like idempotency or pagination, and not transport details like HTTP vs TCP, but the mental model encoded in the interface itself.

REST, RPC, and event-driven systems are not stylistic variations. They encode fundamentally different assumptions about ownership, coupling, and what a system considers a “call”.

2. Prerequisites

Assumed from Part I and II:

Information hiding and minimal surface area

Temporal coupling

Sync vs async coupling decision (Ch 17)

Internal vs external API stability

Backward compatibility and versioning

Synchronous vs asynchronous failure semantics


Also assumed context:

You are designing APIs for distributed systems under partial failure

You already know whether each interaction is request/response or event-based at a macro level



---

3. Key Concepts

REST vs RPC vs Event-Driven as “API Ontologies”

These are not protocols first. They are ontologies:

REST: system as a set of resources with state

RPC: system as a set of functions with inputs/outputs

Event-driven: system as a stream of facts about state changes


The biggest engineering mistake is mixing these ontologies unintentionally (e.g., RPC over REST-shaped endpoints, or events used as commands).


---

REST (Resource-Oriented APIs)

RESTful Resource Model

What it is: A system exposes nouns (resources) identified by stable URIs. Clients manipulate resource state using a small, uniform set of verbs (HTTP methods like GET, POST, PUT, DELETE).

Why it exists: To decouple client logic from server behavior by standardizing interaction patterns around state rather than actions.

Options:

Strict REST (uniform interface, HATEOAS, hypermedia-driven navigation)

Pragmatic REST (resource URLs + JSON + conventional HTTP methods)

REST-in-name-only (HTTP used as transport for RPC-style endpoints)


Trade-offs:

High cacheability (especially GET semantics in HTTP)


Stable, long-lived public interfaces (Stripe, GitHub APIs)


Easy client consumption without code generation


− Weak typing at the protocol level (JSON over HTTP ambiguity)

− Encourages “anemic REST” where resources are just RPC endpoints in disguise

− Hard to model workflows cleanly without introducing state machines externally


When to choose each option:

Strict REST: rare; only when hypermedia-driven discoverability is a hard requirement

Pragmatic REST: default for public APIs with many independent clients

REST-in-name-only: avoid unless compatibility with legacy HTTP tooling is primary constraint


Common failure modes:

“Verb endpoints” disguised as resources (/createUser, /getUser)

Resource explosion due to over-normalization

Hidden coupling through client assumptions about response shape evolution

Misuse of HTTP status codes as application control flow


Example:

Stripe API uses pragmatic REST resource modeling for charges, customers, and invoices

GitHub API similarly exposes repositories, issues, and pull requests as resources


Why HATEOAS is rarely implemented

What it is: Hypermedia As The Engine Of Application State — responses contain links that drive navigation.

Why it exists: To eliminate hardcoded client knowledge of API structure.

Options:

Full HATEOAS compliance

Partial hypermedia (links for pagination or next actions only)

No hypermedia (static documentation-driven APIs)


Trade-offs:

True client/server decoupling in theory


Evolvable APIs without breaking clients (in principle)


− High implementation complexity on both sides

− Poor tooling support compared to OpenAPI/gRPC

− Debugging becomes indirect (state is embedded in responses, not schema)


When to choose each option:

Full HATEOAS: almost never outside academic or highly dynamic systems

Partial hypermedia: reasonable compromise for pagination/navigation

No hypermedia: default in most production APIs


Common failure modes:

Teams claim “we are RESTful” but ignore hypermedia entirely

Clients hardcode URL structures anyway, defeating the model

Overengineering link relations that no client uses


Example:

GitHub includes limited hypermedia links in API responses (partial adoption)

Most Stripe API usage ignores hypermedia entirely (pragmatic non-HATEOAS REST)

RPC (Procedure-Oriented APIs)

RPC as Typed Function Calls Over the Network

What it is: A system exposes procedures (functions) that accept strongly typed inputs and return strongly typed outputs. The network is treated as an extension of a local function call boundary.

Why it exists: To optimize service-to-service communication with strong typing, performance, and developer ergonomics through code generation.

Options:

gRPC (protobuf + HTTP/2)

Thrift (older but similar model)

Twirp (simplified HTTP-based RPC)

JSON-RPC (lightweight but less strict typing)


Trade-offs:

Strong schema enforcement (protobuf contracts)


Excellent performance (binary encoding, multiplexed HTTP/2 streams)


Code generation reduces client/server mismatch risk


− Tight coupling to service contracts (schema evolution must be managed carefully)

− Poor human debuggability compared to REST (binary payloads)

− Harder to expose directly to external consumers without SDKs


When to choose each option:

gRPC: internal microservices at scale (Google, Kubernetes ecosystem)

Twirp: simpler systems where HTTP compatibility matters but RPC ergonomics are desired

JSON-RPC: lightweight tooling or scripting-heavy environments


Common failure modes:

Leaky abstractions where RPC makes network calls feel “local” and hides latency costs

Overuse of fine-grained RPC calls leading to chatty systems

Schema evolution breaking clients due to overly strict protobuf changes

Mixing RPC and REST semantics in the same service boundary


Example:

Google uses gRPC internally across large-scale distributed systems

Kubernetes APIs increasingly rely on gRPC-style mechanisms under the hood

etcd uses gRPC for cluster communication



---

Event-Driven APIs

Event Streams as Facts, Not Commands

What it is: A system publishes immutable events representing facts that something already happened. Consumers subscribe and react independently.

Why it exists: To decouple producers and consumers temporally and structurally by shifting from request/response to shared event log semantics.

Options:

Kafka-style event streaming (partitioned logs with consumer groups)

Pub/Sub systems (e.g., Google Pub/Sub)

Event buses (RabbitMQ, NATS in event mode)

CDC-based event streams (database change capture)


Trade-offs:

Extreme decoupling between producers and consumers


Natural audit log / replayability (Kafka-style logs)


Scales well for fan-out architectures


− Event schema evolution is hard to coordinate across consumers

− Debugging requires reconstructing state from distributed logs

− Event ordering is often only partially guaranteed (partition-level ordering only)


When to choose each option:

Kafka-style logs: when replayability and durability matter (payments, analytics)

Pub/Sub: when simplicity and fan-out distribution matter more than replay

CDC streams: when database state changes are the primary source of truth


Common failure modes:

“Event explosion” where every internal state change becomes a public contract

Ambiguous event naming (“UserUpdated” meaning 12 different things)

Consumers coupling to incidental fields in event payloads

Rebuilding request/response logic on top of events, recreating RPC indirectly


Example:

Kafka at LinkedIn and Uber for large-scale event pipelines

Stripe uses event streams heavily for external integration (webhooks as events)

GitHub webhooks represent event-driven external integrations

4. Why Smart Engineers Disagree

The disagreement is not about correctness. It is about where coupling is acceptable.

REST advocates optimize for:

Stability of public interfaces

Tooling simplicity

Human readability and long-lived external contracts


RPC advocates optimize for:

Developer velocity in internal systems

Strong contracts and refactoring safety

Performance and network efficiency


Event-driven advocates optimize for:

System decoupling over time

Scalability of integration surface area

Auditability and replay


The deeper disagreement is this:

REST assumes the API is a shared interface between autonomous systems

RPC assumes the API is a boundary between cooperating services

Events assume the API is a historical record that multiple systems interpret independently


Each is correct in its domain. Problems occur when one ontology is forced into another’s role.


---

5. Cross-Model Failure Modes (Critical Misuse Patterns)

RPC-shaped REST APIs

Example: /createUser, /deleteUser, /resetPassword

Symptoms:

URL path becomes verb soup

HTTP methods are ignored or misused

Clients treat endpoints as functions anyway


Consequence: You lose REST’s uniform interface benefits without gaining RPC’s typing benefits.


---

REST-shaped RPC systems

Example: gRPC services designed as “UserService.GetUser(UserIdRequest)”

Symptoms:

Overloaded services acting like resource containers

Poor separation of procedural boundaries

Confusing schema evolution patterns


Consequence: You lose RPC’s clarity and end up with awkward hybrid abstractions.


---

Event systems used as hidden RPC

Example: “UserUpdateRequested” event that always triggers exactly one service synchronously

Symptoms:

Events behave like commands

Tight coupling between producer and single consumer

No real fan-out or independence


Consequence: You inherit event-driven complexity without its decoupling benefits.


---

6. Practical Selection Heuristics (Non-Idealized Reality)

REST tends to dominate when:

The API is public-facing

You cannot control all clients

Human debuggability matters (curl, browser inspection)

You expect long-lived backward compatibility constraints


RPC tends to dominate when:

Services are owned by the same organization

Performance and strict contracts matter

You control both client and server generation

Internal refactoring velocity is high priority


Event-driven dominates when:

Multiple downstream consumers must react independently

You need durable audit/history semantics

Workflows span multiple bounded contexts

You explicitly want temporal decoupling



---

7. Closing Structural Insight

These three models are not competing implementations of the same idea. They are different answers to the question:

“What is an API?”

REST: a state interface

RPC: a function boundary

Events: a recorded history of state transitions


The mistake is not choosing the wrong one. The mistake is pretending they are interchangeable representations of the same abstraction.
