Chapter 20 — Resource Modeling

1. Purpose

Once you’ve decided to use a resource-shaped API (REST or REST-like RPC), the real work begins: deciding what a “resource” actually is in your domain.

This chapter is about turning domain concepts into stable, addressable nouns on the wire. Not endpoints, not controllers, not database tables — resources as externally visible entities with identity, lifecycle, and relationships.

Bad resource modeling is one of the fastest ways to accidentally turn a REST API into RPC with JSON formatting.

3. Key Concepts

A resource is not a database table

A resource is not an object model

A resource is a stable, externally meaningful entity with identity and state


Most design failures come from confusing those three.


---

Resource as Domain Noun (Not Schema Projection)

What it is: A resource represents a concept that the consumer understands, not how the system stores it.

Why it exists: To decouple API evolution from internal persistence and service structure.

Options:

Domain-first resources (customer-facing nouns like Order, Invoice)

Schema-driven resources (mirroring database tables or ORM models)

Hybrid façade resources (domain concepts backed by multiple internal models)


Trade-offs:

Domain-first: stable APIs, aligns with business language, easier client reasoning


Schema-driven: fast to implement, minimal translation layer


− Schema-driven: leaks internal design decisions, brittle to refactoring

− Domain-first: requires careful modeling work and sometimes non-trivial aggregation logic


When to choose each:

Domain-first: public APIs, long-lived contracts (Stripe, GitHub)

Schema-driven: internal prototypes, throwaway services

Hybrid: transitional systems evolving from monoliths


Common failure modes:

Exposing tables as resources (/user_table, /order_rows)

Resources that change shape whenever internal schema changes

Clients depending on accidental fields that were never meant to be stable


Example:

Stripe’s PaymentIntent is a domain construct, not a DB table projection

GitHub’s Issue represents a user-facing concept, not a single storage entity



---

URI Design and Resource Identity

What it is: URIs are stable identifiers for resources, not paths to functions or database lookups.

Why it exists: To give resources global identity that survives internal refactoring.

Options:

Flat resource space (/orders/{id})

Hierarchical resource space (/users/{id}/orders/{id})

Hybrid with shallow nesting (/repos/{owner}/{repo})


Trade-offs:

Flat: simplicity, easier caching and reuse of identifiers


Hierarchical: expresses relationships explicitly


− Deep hierarchy: couples URL structure to traversal logic

− Flat: loses contextual relationship clarity in some domains


When to choose each:

Flat: most resources that are globally addressable

Shallow hierarchy: when parent context is required for identity resolution (GitHub repos)

Avoid deep nesting unless the child resource is meaningless without the parent


Common failure modes:

URLs that mirror ORM joins or SQL schemas

Deep nesting like /orgs/{id}/teams/{id}/members/{id}/permissions

Changing URL structure breaking client assumptions about traversal paths


Example:

GitHub uses shallow nesting: /repos/{owner}/{repo}/issues

Stripe avoids deep nesting entirely for most resources



---

The Depth Boundary Problem

What it is: The practical limit of hierarchical modeling before the URL becomes a traversal algorithm.

Why it exists: Every additional level encodes a dependency chain, not just a relationship.

Options:

0–1 level nesting (flat model)

2-level nesting (contextual grouping)

Deep nesting (multi-level hierarchy)


Trade-offs:

Flat: maximum flexibility, simplest routing


Shallow: good balance of clarity and independence


− Deep: brittle, tightly couples client navigation logic to backend structure


When to choose each:

Flat: when resources are globally unique and independently meaningful

Shallow (recommended default): when parent context matters but is not structural dependency

Deep: almost never; only when resource identity is genuinely path-dependent


Common failure modes:

“Navigation API design” where clients must walk trees of endpoints

Refactoring backend structure forces breaking URL changes

Clients treat URL structure as a graph traversal contract


Example:

GitHub issues under repositories is shallow and justified

File system APIs that expose full path hierarchies often suffer from deep coupling problems



---

Handling the Action Problem (Non-CRUD Operations)

What it is: Real systems have operations that don’t map cleanly to CRUD:

“publish a post”

“cancel an order”

“refund a charge”

“rebuild index”


Why it exists: Resource state machines are richer than CRUD semantics.

Options:

Sub-resource action endpoint (POST /orders/{id}/cancel)

Status field mutation (PATCH /orders/{id} with status: "cancelled")

Generic command endpoint (POST /commands)

Separate resource creation (POST /order_cancellations)


Trade-offs:

Sub-resource action: preserves resource identity and intent clarity


Status mutation: keeps pure CRUD but can obscure business intent


− Generic command endpoint: collapses everything into RPC disguised as REST

− Separate resources: can over-model simple state transitions


When to choose each:

Sub-resource action: best default for explicit state transitions

Status mutation: when cancellation is truly just a state field change

Separate resources: when the action itself has lifecycle (e.g. refunds, disputes)

Avoid generic command endpoints unless you are explicitly building RPC over HTTP


Common failure modes:

/doCancelOrder, /executeRefund, /runJob

Clients unaware of idempotency or lifecycle semantics hidden behind verbs

Loss of discoverability of allowed transitions


Example:

Stripe uses explicit resources for operations like refunds (Refund is a resource)

GitHub uses actions like POST /repos/{owner}/{repo}/issues/{issue_number}/comments

Many internal APIs degenerate into /api/doSomething endpoints over time



---

Resource Relationships: Embedding vs Linking

What it is: Resources often relate to other resources. The API must decide whether to:

embed related data directly

return references (links/IDs) requiring follow-up requests


Why it exists: This is a fundamental trade-off between network efficiency and decoupled normalization.

Options:

Embedded representation (eager loading)

Linked reference (lazy loading)

Hybrid (partial embedding with optional expansion)


Trade-offs:

Embedded: fewer client round-trips, simpler client logic


Linked: smaller payloads, independent resource evolution


− Embedded: payload bloat, tight coupling to current response shape

− Linked: chatty APIs, higher latency in distributed calls


When to choose each:

Embedded: small, stable sub-resources (e.g. issue labels)

Linked: large or frequently changing resources (e.g. user profiles in large systems)

Hybrid: default in most production APIs (explicit expansion flags in practice, though covered later in Ch 23)


Common failure modes:

Over-embedding leading to massive response payloads

Over-linking leading to N+1 request patterns

Clients assuming embedded fields are always present

Hidden coupling to nested object structure stability


Example:

GitHub often embeds limited user info while linking full user resources

Stripe uses IDs for related objects but allows expansion when needed



---

POST as the Verb of Last Resort

What it is: When an operation does not map cleanly to GET/PUT/PATCH/DELETE semantics, POST becomes the escape hatch for creating or triggering something.

Why it exists: HTTP defines POST as “submit data for processing,” making it the only verb flexible enough for arbitrary operations.

Options:

POST to sub-resource (preferred action modeling)

POST to collection (resource creation)

POST to generic endpoint (last resort)

Avoid POST entirely by forcing unnatural REST modeling


Trade-offs:

Flexible and extensible


Works for non-idempotent operations


− Weak semantic clarity if overused

− Can become RPC disguised as REST if not carefully constrained


When to choose each:

Sub-resource POST: default for actions tied to a resource

Collection POST: standard resource creation

Generic POST endpoint: only when no resource abstraction exists yet

Avoid forced avoidance of POST (leads to awkward PUT/PATCH abuse)


Common failure modes:

Using PUT for actions that are not idempotent

Overloading PATCH with semantic meaning it was not designed to carry

Turning POST into a universal “do anything” endpoint without structure


Example:

GitHub uses POST for comments and actions on issues

Stripe uses POST for most state transitions (charges, refunds, intents)



---

Why Smart Engineers Disagree

The disagreement comes from where they believe structure should live.

REST purists argue:

Resources should be primary

Actions should emerge from state transitions

URLs should represent stable domain nouns


Pragmatic engineers argue:

Some operations are inherently procedural

Forcing everything into nouns creates awkward or leaky models

Explicit action endpoints improve clarity


RPC-influenced REST engineers argue:

Sub-resource actions are just RPC with worse naming

A command model is more honest in many cases


The real divide is this:

One side optimizes for long-term API stability

The other optimizes for direct expressiveness of domain behavior


Both are correct depending on whether the resource is stable enough to deserve abstraction.


---

Closing Structural Insight

Resource modeling is where REST either becomes a clean domain interface or silently collapses into RPC over HTTP.

A good resource model has:

stable nouns (not verb endpoints in disguise)

shallow, meaningful hierarchy (not traversal graphs)

explicit handling of actions without abandoning resource identity

controlled embedding that avoids both payload bloat and chatty clients


At this layer, the system stops being “an API” and becomes a domain graph with addressable state.
