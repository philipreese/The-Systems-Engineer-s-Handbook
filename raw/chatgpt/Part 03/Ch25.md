Chapter 25 — Internal vs. External API Design

1. Purpose

Ch 15 defined the distinction between internal and external API stability. That distinction is structural. This chapter is operational: what actually changes once an API has external consumers who cannot be directly coordinated, migrated, or fixed.

An API stops being “just another service interface” the moment someone outside your control depends on it. At that point, behavior becomes governance, not implementation detail.

3. Key Concepts

Internal APIs are coordination problems; external APIs are governance problems

External APIs require formalized change control and communication channels

SDKs act as insulation layers between wire format and consumer code

Documentation becomes a system dependency, not optional metadata

Hyrum’s Law turns undocumented behavior into implicit contract



---

Governance Shift: From Coordination to Constraint

Internal vs External Operational Reality

What it is: Internal APIs can be changed by talking to consumers directly. External APIs cannot; consumers are no longer reachable as a coordinated group.

Why it exists: The presence of uncoordinated consumers removes the possibility of synchronous migration.

Options:

Internal-only API governance (informal coordination)

External API governance (formal contracts and deprecation policy)

Hybrid APIs (internally owned but externally consumed accidentally)


Trade-offs:

Internal governance: high agility, low overhead


External governance: stability and predictability for unknown consumers


− Internal governance: fragile if external consumption appears

− External governance: slower iteration due to compatibility constraints


When to choose each:

Internal governance: services with strictly controlled ownership boundaries

External governance: public APIs, partner integrations, SDK consumers

Hybrid: transitional systems (most dangerous state in practice)


Common failure modes:

Assuming internal APIs remain internal after exposure

Changing behavior without realizing external dependency exists

Treating “no official consumers” as equivalent to “no consumers”



---

Consumer Uncoordination and Change Management

The Loss of Direct Migration Paths

What it is: Once consumers are external, you can no longer coordinate atomic upgrades across all users of an API.

Why it exists: External consumers operate on independent release cycles, unknown deployment windows, and untracked usage patterns.

Options:

Coordinated migrations (internal systems)

Versioned gradual deprecation (external systems)

Break-fix communication model (emergency-driven evolution)


Trade-offs:

Coordinated migrations: clean transitions, no long-lived legacy behavior


Gradual deprecation: safe evolution under uncertainty


− Coordinated migrations: impossible at scale externally

− Gradual deprecation: long-term maintenance overhead


When to choose each:

Coordinated migrations: internal service refactors

Gradual deprecation: public APIs with unknown client populations

Break-fix: legacy or ungoverned exposure scenarios (avoid if possible)


Common failure modes:

Removing behavior assuming no one relies on it

Introducing silent breaking changes without deprecation windows

Failing to track actual consumer usage


Example:

GitHub explicitly deprecates API endpoints with notice periods before removal

Internal microservices often change interfaces rapidly without migration plans



---

SDKs as Stability Buffers

Abstracting Wire Changes from Consumers

What it is: SDKs provide a stable programming interface that shields consumers from changes in underlying API structure.

Why it exists: To decouple consumer code from wire-level representation, reducing the blast radius of API evolution.

Options:

No SDK (direct HTTP usage)

Thin SDK (minimal wrapping over API calls)

Full SDK (strongly abstracted client library with typed models)


Trade-offs:

Full SDK: strong stability, easier consumer adoption


No SDK: simplicity, no maintenance burden


− Full SDK: maintenance overhead, version synchronization issues

− No SDK: clients directly coupled to wire format changes


When to choose each:

No SDK: internal APIs or highly technical consumers

Thin SDK: most public APIs with moderate complexity

Full SDK: platforms where developer experience is a core product (Stripe, AWS-style ecosystems)


Common failure modes:

SDK lagging behind API changes, creating version mismatch

SDK hiding important API semantics, leading to incorrect usage

Treating SDK updates as optional rather than part of API lifecycle


Example:

Stripe SDKs abstract API calls but maintain strict version alignment

GitHub maintains SDKs across multiple languages to stabilize API consumption

Internal services often skip SDKs entirely and rely on direct RPC calls



---

Documentation as a First-Class System Dependency

External APIs Require Published Truth

What it is: Documentation becomes a binding contract surface, not just explanation.

Why it exists: External consumers cannot infer intent from code or internal communication channels.

Options:

Internal docs only

Public static documentation

Living documentation tied to API schema and SDKs


Trade-offs:

Public docs: discoverability, onboarding efficiency


Internal docs: low overhead, fast iteration


− Internal docs: unusable for external consumers

− Public docs: maintenance burden and synchronization risk


When to choose each:

Internal docs: internal services

Public docs: any externally consumed API

Living docs: mature external APIs with SDK ecosystems


Common failure modes:

Documentation drift from actual API behavior

Missing edge-case behavior in docs leading to consumer bugs

Treating docs as optional rather than authoritative


Example:

GitHub API documentation is a core part of its developer ecosystem

Stripe maintains tightly synchronized docs, often treated as canonical alongside SDKs



---

Hyrum’s Law: The Hidden Contract Explosion

Everything Observable Becomes a Dependency

What it is: With enough consumers, every observable behavior of a system — even unintended or undocumented — becomes part of the effective contract.

Why it exists: Consumers optimize for working behavior, not intended design. Once behavior is relied upon, it becomes impossible to change safely.

Options:

Strict contract enforcement (break non-compliant usage)

Soft contract evolution (preserve behavior even if unintended)

Mixed approach (deprecate but maintain compatibility)


Trade-offs:

Strict enforcement: cleaner long-term system design


Soft evolution: fewer breaking changes


− Strict enforcement: breaks unknown consumers

− Soft evolution: accumulated legacy behavior and technical debt


When to choose each:

Strict enforcement: internal APIs or controlled environments

Soft evolution: external APIs with unknown consumer bases

Mixed: most real-world public APIs


Common failure modes:

Removing “undocumented” behavior that external clients depend on

Assuming unused fields or endpoints are safe to delete

Underestimating implicit coupling through error messages, timing, or ordering


Example:

Google systems observed that any visible behavior eventually becomes relied upon (Hyrum’s Law origin context)

Public APIs often preserve “accidental” behaviors due to external dependency risk



---

API Gateways and External Infrastructure

The Operational Layer That Emerges With Exposure

What it is: Once APIs are external, infrastructure emerges to manage traffic, access, and lifecycle:

API gateways

Developer portals

Rate limiting layers

Deprecation tracking systems


Why it exists: External APIs require coordination, observability, and controlled evolution mechanisms that internal APIs do not.

Options:

Direct service exposure

Gateway-mediated exposure

Full API management layer


Trade-offs:

Gateway: centralized control, observability, policy enforcement


Direct exposure: simplicity, lower latency


− Gateway: additional complexity and potential bottleneck

− Direct exposure: no governance or usage control


When to choose each:

Direct exposure: internal systems or tightly coupled clients

Gateway: all external APIs

Full API management layer: large ecosystems (platform APIs)


Common failure modes:

Exposing internal services directly without governance layer

Using gateways without lifecycle management (docs, deprecation, analytics)

Treating gateway as optional rather than structural requirement


Example:

GitHub uses API gateways plus developer portals for public API management

Large cloud providers expose APIs through managed gateway infrastructure

Internal RPC systems often bypass gateways entirely



---

Why Smart Engineers Disagree

The disagreement is about how much structure external exposure justifies.

Internal-first engineers argue:

APIs should remain simple services

Externalization is just a distribution detail

Too much process slows iteration


Platform engineers argue:

External APIs are products, not services

Without governance, accidental breakage is inevitable

Infrastructure and process are required for stability


Product platform engineers argue:

SDKs and documentation are part of the API surface itself

The API is not just the service, but the entire developer experience layer



The real divide:

“API as code boundary” vs

“API as public product surface”


Once external consumers exist, the second interpretation becomes unavoidable.


---

Closing Structural Insight

Internal APIs are optimized for change. External APIs are optimized for continuity.

The moment an API becomes externally consumed, it stops being a technical interface and becomes a long-lived dependency contract supported by infrastructure, documentation, SDKs, and governance systems.

At that point, the API is no longer just what the system does — it is what the ecosystem depends on it doing.
