Chapter 22 — Idempotency

1. Purpose

At-least-once delivery is not an edge case in distributed systems — it is the default shape of reality once you cross a process or network boundary (Ch 17). Packets are retried, connections drop, clients time out, and servers may execute the same request more than once.

This chapter defines how systems survive that without corrupting state: idempotency.

Idempotency is not about preventing duplication. It is about making duplication irrelevant.

At-least-once delivery as baseline reality (Ch 17)

3. Key Concepts

Natural idempotency vs designed idempotency

Idempotency keys as deduplication contracts

Server-side uniqueness enforcement as the correctness mechanism

Idempotency windows and retention semantics

Safe retries as a property of the API, not the client



---

Natural Idempotency

Some Operations Are Idempotent by Construction

What it is: An operation is naturally idempotent if repeating it produces the same end state.

Why it exists: Certain state transitions collapse duplicates into the same result by definition.

Options:

Idempotent operations (PUT, DELETE, some PATCH cases)

Non-idempotent operations (POST, event creation, charge initiation)


Trade-offs:

Idempotent: safe to retry without coordination


Non-idempotent: simpler conceptual model for “do a thing”


− Idempotent: requires careful state design and overwrite semantics

− Non-idempotent: requires external deduplication mechanisms


When to choose each:

PUT/DELETE: when target state is fully known and replaceable

POST: when each invocation produces a distinct new entity or effect


Common failure modes:

Assuming all PUT operations are safe when they actually trigger side effects

Treating POST as idempotent accidentally, causing silent overwrites

Building workflows that rely on implicit uniqueness rather than explicit state


Example:

PUT /users/123 with the same payload is idempotent by design

DELETE /users/123 repeated has no additional effect after first deletion

POST /orders typically creates a new order each time unless protected



---

Designed Idempotency (Idempotency Keys)

Making Non-Idempotent Operations Safe

What it is: A mechanism where the client supplies a unique key representing a logical operation, and the server ensures that multiple executions of the same key produce the same result.

Why it exists: Because many critical operations (payments, order creation, job submission) are inherently non-idempotent but must still survive retries safely.

Options:

No idempotency (unsafe retries)

Client-side deduplication only (fragile, race-prone)

Server-side idempotency keys (robust deduplication contract)


Trade-offs:

Server-side keys: correctness under retries, safe network behavior


No idempotency: simple implementation, but unsafe in distributed systems


− Server-side keys: storage overhead and retention complexity

− Client-only deduplication: unreliable across crashes and concurrent retries


When to choose each:

Server-side idempotency keys: default for any operation with side effects that must not duplicate (payments, provisioning, mutations with cost)

Client-only deduplication: rare, internal tools with controlled retry behavior

No idempotency: only for truly safe duplicate operations (rare in practice)


Common failure modes:

Generating idempotency keys per HTTP request instead of per logical operation

Not persisting key-to-result mapping atomically with the side effect

Allowing key reuse across unrelated operations (causing accidental collisions)

Treating idempotency as optional metadata instead of a correctness constraint



---

Server-Side Deduplication Mechanism

The Core Pattern: Key + Uniqueness Constraint

What it is: The server stores each idempotency key alongside the result of the first successful execution. Subsequent requests with the same key return the cached result instead of re-executing logic.

Why it exists: To transform retry duplication from “double execution” into “cache hit.”

Options:

In-memory cache (fast, volatile)

Persistent store with uniqueness constraint (correct, durable)

Distributed cache without strict consistency (fast but unsafe under race conditions)


Trade-offs:

DB-backed UNIQUE constraint: strongest correctness guarantee


In-memory: fast, simple, but unsafe across crashes/restarts


− Distributed cache: can produce duplicate execution under race conditions

− No persistence: loses idempotency guarantee under failure


When to choose each:

PostgreSQL UNIQUE constraint: production-grade correctness (Stripe-style systems)

In-memory cache: short-lived or low-stakes deduplication

Distributed cache: only when eventual consistency is acceptable and duplicates are non-critical


Common failure modes:

Writing side effect first, then storing idempotency key (race window creates duplicates)

Not atomically binding key insertion and state mutation

Allowing partial execution before deduplication check completes

Assuming cache hit prevents all downstream side effects


Example:

PostgreSQL table with UNIQUE(idempotency_key) ensures only one execution commits

Stripe stores idempotency keys per endpoint and replays original response on retry

Payment processors rely on DB-level constraints to prevent duplicate charges

Idempotency Window and Retention

How Long a System Must Remember a Request

What it is: The server must retain idempotency key mappings long enough to cover realistic retry windows.

Why it exists: Because retries are not instantaneous — they may occur seconds, minutes, or even hours later depending on client behavior and network conditions.

Options:

Short retention window (minutes)

Medium retention (hours to days)

Long retention (weeks or more)


Trade-offs:

Short window: reduced storage cost and cleanup overhead


Long window: stronger retry safety guarantees


− Short window: late retries incorrectly treated as new operations

− Long window: storage growth and key management complexity


When to choose each:

Short (minutes–hours): high-frequency, low-cost operations where retry delays are short

Medium (days): payment systems, order creation, job submission

Long (weeks): financial systems where duplicate execution is catastrophic


Common failure modes:

Retrying after window expiration causes duplicate side effects

Cleaning keys too aggressively underestimating real-world retry delays

Treating expiration as safe re-execution without client awareness

Not aligning retention with upstream retry/backoff policies


Example:

Stripe retains idempotency keys long enough to safely cover payment retry scenarios

Many job systems use TTL-based deduplication windows aligned with queue retry policies



---

Idempotency as a Substitute for Exactly-Once

What it is: Idempotency does not prevent multiple executions. It ensures that multiple executions produce the same observable result.

Why it exists: Because exactly-once execution across distributed systems is not reliably achievable under real failure modes (crashes, retries, partial commits).

Options:

Exactly-once semantics (theoretical or tightly constrained systems)

At-least-once + idempotency (practical distributed systems)

At-most-once (safe but lossy systems)


Trade-offs:

Idempotency approach: robust under failure, scalable across systems


Exactly-once: simpler mental model, but rarely achievable end-to-end


− Idempotency approach: requires careful design of state transitions

− Exactly-once: breaks under network partitions and partial failure modes


When to choose each:

Idempotency + at-least-once: default for distributed APIs

Exactly-once: limited to controlled environments (single database transaction scope)

At-most-once: only where loss is acceptable (logging, telemetry sampling)


Common failure modes:

Assuming transport guarantees prevent duplicate execution

Building business logic that cannot tolerate replayed requests

Relying on client correctness instead of server enforcement

Ignoring race conditions between retry and execution completion



---

Why Smart Engineers Disagree

The disagreement is about where correctness should live.

Client-centric engineers argue:

Idempotency should be a client responsibility

Servers should remain stateless and simple


Platform engineers argue:

Clients are unreliable under failure conditions

Correctness must be enforced server-side


Database-centric engineers argue:

Idempotency is just a transactional uniqueness problem

Proper constraints eliminate most distributed complexity



The real divide:

Stateless purity vs.

Practical correctness under partial failure


In production systems that move money or mutate shared state, server-side idempotency almost always wins.


---

Closing Structural Insight

Idempotency is the mechanism that converts unreliable networks into usable systems.

Without it, retries are dangerous. With it, retries are neutral.

The key shift is this:

Without idempotency: “Did this request run once?”

With idempotency: “It doesn’t matter — it has a stable identity and outcome”


That single shift is what makes real-world distributed APIs survivable.
