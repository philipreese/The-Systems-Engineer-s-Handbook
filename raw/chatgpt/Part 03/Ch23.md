Chapter 23 — Pagination and Streaming

1. Purpose

Any API that returns collections eventually hits a hard boundary: you cannot return “all rows” without either breaking latency, memory limits, or correctness under change.

Pagination is not a UI feature. It is a contract about consistency under mutation and cost under scale.

This chapter is about how that contract is defined and what breaks when it is defined poorly.

3. Key Concepts

Pagination is a consistency model, not just a slicing mechanism

Offset pagination trades simplicity for instability and performance degradation

Cursor (keyset) pagination trades flexibility for correctness and scale stability

Streaming replaces pagination when the dataset is conceptually unbounded or real-time



---

Offset / Limit Pagination

The Simple but Fragile Default

What it is: Pagination via LIMIT N OFFSET M, returning a slice of a sorted result set.

Why it exists: It is the most straightforward mapping from “give me page N of results” to SQL-like queries.

Options:

Offset-based pagination

Limit-only (rare, implicit infinite stream truncation)

Hybrid offset + ordering


Trade-offs:

Extremely simple to implement and understand


Supports arbitrary page access (jump to page N)


− Breaks under concurrent writes (page drift)

− Performance degrades linearly with offset size

− Requires careful ordering discipline to avoid nondeterminism


When to choose each:

Offset pagination: small datasets, internal tools, or read-mostly tables

Limit-only: streaming-like consumption where only “next chunk” matters

Hybrid: rarely justified; usually a transitional artifact


Common failure modes:

Page drift: new inserts shift items between pages, causing duplicates or missing items

Hidden O(N) performance cost: large offsets force database to scan and discard rows

Inconsistent ordering due to missing deterministic sort keys

Client assumptions that page boundaries are stable over time


Example:

PostgreSQL OFFSET 10000 LIMIT 50 on large tables becomes increasingly slow

Many early REST APIs used offset pagination and later had to retrofit cursors



---

Cursor-Based (Keyset) Pagination

Stable Pagination via State Anchors

What it is: Pagination based on a stable cursor derived from a sorted, unique field (e.g., created_at + id), where each request asks for “items after this position.”

Why it exists: To eliminate instability caused by mutable offsets and to maintain consistent performance regardless of dataset size.

Options:

Forward-only cursors (starting_after)

Bidirectional cursors (starting_after / ending_before)

Encoded opaque cursors (base64-encoded composite keys)


Trade-offs:

Stable under concurrent inserts and deletes


Constant performance regardless of dataset depth


Avoids page drift entirely


− Cannot jump to arbitrary page N

− Requires stable, sortable, unique ordering field

− More complex client logic and state handling


When to choose each:

Cursor pagination: default for production APIs with large or changing datasets

Bidirectional cursors: when clients need back/forward navigation

Opaque cursors: when you want to hide internal sort keys or schema details


Common failure modes:

Using non-unique sort keys causing skipped or duplicated items

Failing to define deterministic ordering (e.g., created_at without tie-breaker ID)

Treating cursors as arbitrary tokens instead of encoding a stable ordering position

Mixing offset and cursor pagination in the same API surface


Example:

Stripe uses starting_after / ending_before with stable IDs for pagination

GitHub API uses Link headers with cursor-like tokens under the hood

Many modern SaaS APIs have migrated from offset to cursor pagination for correctness



---

Cursor Stability and Ordering Discipline

The Hidden Requirement Behind Keyset Pagination

What it is: Cursor pagination only works if the ordering is fully deterministic and stable.

Why it exists: Any ambiguity in ordering leads to lost or duplicated records between pages.

Options:

Single-field ordering (simplest but fragile)

Composite ordering (e.g., timestamp + unique ID)

Opaque server-generated cursors (encapsulated ordering logic)


Trade-offs:

Composite ordering: correctness under concurrency


Opaque cursors: hides schema changes from clients


− Single-field ordering: prone to duplicates under identical timestamps

− Opaque cursors: harder to debug and inspect


When to choose each:

Composite ordering: default for most systems

Opaque cursors: when internal ordering logic may evolve independently

Single-field ordering: only when uniqueness is guaranteed by design


Common failure modes:

Using created_at alone, causing unstable pagination when timestamps collide

Changing sort order between requests

Assuming natural ordering exists in the underlying store

Breaking pagination contracts when schema evolves


Example:

GitHub uses stable identifiers in combination with timestamps for ordering issues and events

Stripe ensures pagination stability via consistent object IDs

Streaming as a Non-Pagination Model

When the Collection Is Not a Collection

What it is: Streaming returns results incrementally over time rather than as discrete pages.

Why it exists: Some datasets are unbounded, real-time, or too large to materialize as paginated snapshots.

Options:

NDJSON (newline-delimited JSON over HTTP)

Server-Sent Events (SSE)

gRPC server streaming

WebSocket-based streams (less common for pure data delivery)


Trade-offs:

Eliminates pagination overhead entirely


Lower latency for first results


Natural fit for real-time or continuous data


− Harder to retry or resume mid-stream

− More complex client consumption model

− Requires backpressure handling or buffering strategies


When to choose each:

NDJSON: batch-like streams, log exports, large dataset dumps

SSE: real-time updates in HTTP-native environments

gRPC streaming: internal high-performance service-to-service pipelines

WebSockets: interactive systems (beyond pure data APIs)


Common failure modes:

Treating streaming like paginated requests and expecting rewind capability

Losing stream position on disconnect without resumable tokens

Overusing streaming for datasets that are actually finite and better paginated

Ignoring backpressure, causing memory buildup on clients or servers


Example:

Kafka-style consumers treat partitions as infinite streams

GitHub event feeds can be consumed as streams of changes rather than paginated snapshots

Stripe webhooks represent event-driven streaming of state changes



---

Why Smart Engineers Disagree

The disagreement is about what a “collection” fundamentally is.

REST purists tend to prefer offset pagination:

aligns with mental model of “a list you can slice”

easier for humans to reason about


Distributed systems engineers prefer cursor pagination:

correctness under mutation matters more than convenience

performance must not degrade with dataset size


Streaming advocates reject pagination entirely:

the abstraction of “pages” is incorrect for real-time systems

systems should model flows, not snapshots



The real distinction:

Pagination assumes a stable snapshot with navigation

Streaming assumes a continuously evolving sequence



---

Closing Structural Insight

Pagination is where APIs quietly encode their assumptions about time.

Offset pagination assumes a mostly static dataset. Cursor pagination assumes a changing but ordered dataset. Streaming assumes no stable snapshot exists at all.

Choosing incorrectly does not just degrade performance — it changes what “a result set” means in your system.
