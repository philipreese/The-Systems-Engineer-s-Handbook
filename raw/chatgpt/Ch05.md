Chapter 05 — Designing for Change

1. Purpose



Most systems don’t fail because they were “badly designed.” They fail because they made the wrong things hard to change.

This chapter is about separating stability from flexibility in a deliberate way:

what should remain fixed over time

what should be designed to evolve cheaply


This is not “future-proofing.” It’s constraint selection: deciding which dimensions of the system are allowed to move and which are explicitly locked.

3. Key Concepts



Good design makes specific changes cheap, not all changes possible

Stability and flexibility are orthogonal concerns

Open/Closed Principle is useful but incomplete as a general law

“Designing for change” is about identifying axes of variation

“Future-proofing” is usually undirected speculation with high upfront cost



---

4. What Should Be Stable vs What Should Be Flexible



What it is: System design is fundamentally about choosing invariants. Some parts of a system should not change often; others should be explicitly designed to change frequently.

Why it exists: Change is the dominant cost in software systems. Every modification propagates through dependencies, tests, deployments, and mental models.

Options:

Stable interfaces, flexible implementations

Flexible interfaces, unstable implementations

Everything flexible (maximal adaptability, minimal guarantees)

Everything stable (rigid but predictable system)


Trade-offs:

Stable interfaces reduce coordination cost but require upfront correctness in modeling

Flexible interfaces allow rapid evolution but increase downstream churn

Fully flexible systems degrade into unpredictability and tight coupling over time

Fully stable systems become resistant to necessary evolution


When to choose each option:

Stable interface / flexible implementation: most production APIs, OS abstractions, databases

Flexible interface: exploratory systems, early-stage products

Fully stable: low-level protocols, kernel ABIs, long-lived infrastructure contracts

Fully flexible: prototypes and experimental systems


Common failure modes:

Locking in incorrect API shapes that become impossible to evolve without breaking clients

Over-stabilizing internal logic, making feature changes expensive

Systems where every component changes frequently, causing systemic instability


Example: The Linux syscall ABI is intentionally stable. The kernel implementation behind it changes constantly, but user-space programs depend on the syscall interface remaining consistent across decades.

5. Open/Closed Principle (OCP) — Useful but Limited



What it is: Software entities should be open for extension but closed for modification.

Why it exists: It aims to reduce change risk by allowing behavior extension without modifying existing code.

What it really means in practice: You are trying to separate stable contracts from variable behavior so that new behavior does not invalidate existing assumptions.

Options:

Strict OCP adherence (plugins, inheritance hierarchies, dependency injection)

Partial OCP (stable core + controlled extension points)

No OCP (direct modification as primary mechanism)


Trade-offs:

Strict OCP reduces modification risk but introduces abstraction overhead and indirection

Partial OCP balances flexibility and simplicity but requires careful boundary design

No OCP maximizes clarity but increases risk of regression in shared code paths


When to choose each option:

Strict OCP: frameworks, plugin systems, long-lived platforms

Partial OCP: most backend services, databases, OS-level abstractions

No OCP: small systems, scripts, tightly owned codebases


Common failure modes:

Over-engineered plugin systems that never gain additional extensions

Deep inheritance trees that are difficult to reason about under real change pressure

“Generic” extension points that do not match actual future needs


Example: PostgreSQL extensions provide a controlled OCP-like model: core engine remains stable, while extensions (e.g., PostGIS) add behavior without modifying the database kernel.

6. Designing for Change vs Future-Proofing



What it is:

Designing for change: identify known axes of variation and make them cheap to modify.

Future-proofing: attempt to anticipate unknown future requirements and design generality upfront.

Why the distinction matters: One is grounded in observed variability; the other is speculative generalization.

Options:

Axis-based design (explicit variability points)

General-purpose design (broad, abstract interfaces)

Minimal design (no anticipation of change)


Trade-offs:

Axis-based design reduces change cost for known variations but does not cover unknown ones

General-purpose design increases upfront flexibility but adds accidental complexity

Minimal design is fast initially but may incur refactoring cost later


When to choose each option:

Axis-based: production systems with known evolution patterns (e.g., authentication methods, storage backends)

General-purpose: platform systems, frameworks (carefully)

Minimal: early-stage systems, experimental products


Common failure modes:

Over-generalized “configurable everything” systems that are hard to use

Premature abstraction for requirements that never materialize

Rigid systems that require invasive refactoring when real requirements emerge


Example: A system that anticipates “multiple database backends” but only ever uses PostgreSQL often incurs unnecessary abstraction layers (repository patterns, ORM indirection) that do not reduce real change cost.

7. Axes of Variation (Practical Design Tool)



What it is: An axis of variation is a dimension along which a system is expected to change.

Examples:

authentication method (password, OAuth, SSO)

storage backend (PostgreSQL, object storage)

execution environment (local, Kubernetes, serverless)

data format (JSON, protobuf)


Why it exists: Not all parts of a system change equally. Identifying likely change dimensions allows targeted abstraction.

Options:

Explicit axis separation (interfaces per variation point)

Implicit coupling (changes propagate through system)

Over-generalization (abstract everything as variable)


Trade-offs:

Explicit axes reduce change cost but require correct identification

Implicit coupling increases fragility under change

Over-generalization increases complexity without proportional benefit


When to choose each option:

Explicit axes: systems with known domain variability (payment systems, authentication layers)

Implicit: small systems where change is infrequent

Over-generalization: rarely justified, except in foundational libraries


Common failure modes:

Abstracting variability that never occurs

Missing real axes (e.g., treating “storage backend” as fixed when it becomes variable later)

Creating abstraction layers that encode incorrect variability assumptions


Example: Git cleanly isolates variation in storage representation (content-addressable objects) from operations (branching, merging), allowing new features without changing the core object model.

8. Git’s Content-Addressable Storage (Stable Core Design)



What it is: Git stores all data as immutable content-addressed objects (blobs, trees, commits). Everything else is built on top.

Why it works: The core data model is extremely stable. Changes in features (branching models, merge strategies, protocols) do not require changing the underlying object model.

What is stable:

object identity (hash-based addressing)

immutability of stored objects

basic graph structure of commits


What is flexible:

ref management (branches, tags)

transfer protocols

UI/command-level features


Trade-offs:

Extremely robust historical integrity

High conceptual clarity at storage level

Complexity shifts to higher layers (indexing, refs, plumbing commands)

Steeper learning curve due to layered model


Failure modes:

Misunderstanding refs as data instead of pointers to immutable objects

Performance tuning complexity in large repositories

Confusion between working tree state and object database state


Why it survives: Because it locks in the correct invariant (immutable history graph) and leaves everything else adjustable.

9. Linux Syscall ABI Stability



What it is: The Linux kernel maintains a stable system call interface (ABI) for user-space programs, even as internal kernel code changes dramatically.

Why it exists: User-space binaries must continue running across kernel upgrades without recompilation.

What is stable:

syscall numbers and signatures

behavior contracts for system calls

basic process and memory model semantics


What is flexible:

kernel internals (scheduler, memory management, drivers)

performance optimizations

hardware-specific implementations


Trade-offs:

Strong backward compatibility guarantees

Heavy internal implementation freedom

Significant complexity in maintaining legacy behavior

Long-term accumulation of edge-case behavior


Failure modes:

Old system call behavior preserved even when suboptimal

Kernel complexity increases due to backward compatibility constraints

Subtle behavioral differences across architectures or versions


Why it works: Because it stabilizes the contract with user-space, not the implementation. Everything else is allowed to evolve.


---

10. PostgreSQL Wire Protocol Stability



What it is: PostgreSQL maintains a stable wire protocol across major versions, allowing clients to interact with newer server versions without rewriting client logic.

Why it exists: Client-server decoupling depends on stable communication contracts.

What is stable:

query/response protocol structure

authentication handshake mechanisms

basic session semantics


What is flexible:

query planner internals

storage engine improvements

indexing strategies

execution optimizations


Trade-offs:

Long-lived client compatibility

Ability to evolve internals without breaking ecosystem

Complex backward compatibility constraints in server behavior

Slow evolution of protocol-level improvements


Failure modes:

Clients relying on undocumented behavior of query responses

Version-specific performance differences causing unexpected regressions

Misinterpretation of protocol stability as implementation stability


Why it works: Because it isolates change inside the server while preserving a strict external contract.

11. Why Smart Engineers Disagree



Some engineers prioritize strict stability of interfaces above all else, minimizing change risk even if internal complexity increases.

Others prioritize flexibility, preferring to redesign interfaces when system understanding improves.

Others focus on minimizing abstraction layers entirely, accepting change propagation as a cost of simplicity.

The disagreement reduces to:

whether stability should be enforced at the interface boundary or allowed to evolve

whether change cost should be pushed inward (implementation) or distributed outward (API)

whether correctness of early modeling is assumed or continuously corrected


Git, Linux, and PostgreSQL represent a consistent philosophy:

stabilize the contract

allow everything else to change freely


This is not universal. It is a design choice with strong consequences.
