# Chapter 87 — Latency vs. Throughput Trade-offs

## 1. Purpose

This chapter introduces the first major application of the profiling discipline established in Chapter 86. Once a measured bottleneck has been identified, one of the first questions is what kind of performance problem actually exists.

Engineers often say a system should be "faster." That request is incomplete. It may mean reducing the time required to complete a single request, increasing the number of requests completed per second, or both. Those goals are related but not identical, and improving one frequently degrades the other.

This chapter establishes the distinction between latency and throughput, the mechanisms that trade one for the other, and how to reason about those trade-offs using the capacity models introduced earlier in the handbook.

# 3. Key Concepts

* Latency and throughput measure different properties.
* Fixed costs create opportunities for batching.
* Batching increases throughput by increasing latency.
* Tail latency often matters more than average latency.
* Every capacity optimization changes queueing behavior somewhere.

# Latency vs. Throughput

**What it is:**

Latency is the time required to complete a single unit of work.

Throughput is the number of units completed per unit of time.

Although related, they answer different questions.

A service responding in 20 milliseconds has low latency.

A service processing one million requests per second has high throughput.

Neither measurement implies the other.

**Why it exists:**

Many engineering discussions use "performance" without specifying which property is being optimized.

Treating latency and throughput as interchangeable leads teams to optimize for the wrong objective.

**Options:**

* Optimize latency
* Optimize throughput
* Balance both against explicit requirements

**Trade-offs:**

Reducing latency often requires performing work immediately, reducing opportunities to share fixed costs.

Increasing throughput frequently delays individual work so more operations can be processed together.

Attempting to maximize both simultaneously usually increases implementation complexity while improving neither substantially.

**When to choose each option:**

Optimize latency when user response time is the primary requirement.

Optimize throughput when maximizing completed work dominates, such as analytics pipelines or batch processing.

Balance both when explicit SLOs require acceptable response time while sustaining target load.

**Common failure modes:**

* Reporting requests per second when users complain about slow responses.
* Optimizing median latency while overload causes unacceptable queue growth.
* Assuming higher throughput automatically improves user experience.

**Example:**

Interactive web applications generally prioritize latency.

Large-scale log processing systems prioritize throughput.

These represent different optimization targets, not different definitions of performance.

# Batching

**What it is:**

Batching accumulates multiple units of work before processing them together.

Instead of paying fixed overhead for every operation individually, the overhead is amortized across many operations.

**Why it exists:**

Most systems contain fixed costs independent of request size:

* Network round trips
* System calls
* Disk seeks
* Transaction setup
* Lock acquisition

Batching reduces the average cost of these fixed expenses.

Little's Law and the Theory of Constraints explain why this improves system capacity without needing to re-derive them here.

**Options:**

* Immediate processing
* Fixed-size batches
* Time-based batches
* Adaptive batching

**Trade-offs:**

Immediate processing minimizes waiting but repeats fixed costs.

Large batches maximize throughput but increase latency.

Adaptive batching improves utilization but introduces additional scheduling complexity.

**When to choose each option:**

Immediate execution is appropriate when latency dominates.

Fixed-size batches suit predictable workloads.

Time-based batching prevents indefinite waiting under light load.

Adaptive batching is appropriate when workload characteristics fluctuate significantly.

**Common failure modes:**

* Waiting too long for batches to fill.
* Creating bursty resource utilization.
* Increasing queueing delays until latency SLOs are violated.

**Example:**

Bulk inserts in **PostgreSQL** execute dramatically more efficiently than thousands of individual insert statements because transaction setup, logging, and synchronization costs are shared across many rows.

# Tail Latency Matters

**What it is:**

Tail latency describes the slowest fraction of requests rather than the average request.

Systems are typically evaluated using latency percentile SLIs, already defined in Chapter 73.

**Why it exists:**

Users experience individual requests rather than statistical averages.

A service with excellent average latency may still produce unacceptable user experience if a small percentage of requests are extremely slow.

**Options:**

* Optimize average latency
* Optimize median latency
* Optimize latency percentiles

**Trade-offs:**

Average latency is easy to calculate but hides variability.

Median latency reflects typical behavior while ignoring extreme delays.

Percentile-based measurements expose tail behavior but are more difficult to improve because they frequently arise from contention rather than computation.

**When to choose each option:**

Average latency is useful for internal diagnostics.

Median latency describes typical performance.

Operational SLOs should generally be expressed using latency percentiles because they capture user-visible degradation.

**Common failure modes:**

* Celebrating improved averages while the slowest requests worsen.
* Ignoring queueing delays affecting only peak load.
* Measuring idle-system latency instead of production behavior.

**Example:**

Many large web services publish p95 or p99 latency targets rather than average latency because occasional multi-second requests dominate user perception despite excellent averages.

# Connection Pooling

**What it is:**

Connection pooling reuses previously established network or database connections instead of creating a new connection for every request.

**Why it exists:**

Connection establishment incurs fixed costs including authentication, negotiation, and resource allocation.

Reusing existing connections allows more useful work to be completed with the same resources.

**Options:**

* One connection per request
* Shared connection pool
* Persistent long-lived connections

**Trade-offs:**

Per-request connections minimize shared state but waste setup overhead.

Pools improve throughput by amortizing connection creation.

Oversized pools increase memory usage and contention while providing diminishing returns.

**When to choose each option:**

Per-request connections are acceptable only for infrequent operations.

Connection pools should be the default for high-volume services.

Long-lived persistent connections suit systems with continuous communication.

**Common failure modes:**

* Pool exhaustion creating long request queues.
* Oversized pools overwhelming downstream services.
* Mistaking larger pools for unlimited scalability.

**Example:**

Database-backed application servers routinely use pooled connections because repeatedly establishing TCP and database sessions becomes the bottleneck long before query execution itself.

---

# Request Coalescing

**What it is:**

Request coalescing combines multiple similar operations into a single larger operation.

Rather than performing identical work repeatedly, the system performs it once on behalf of multiple requesters.

**Why it exists:**

Repeated fixed overhead wastes processing capacity.

Combining work increases throughput by reducing duplicated execution.

**Options:**

* Independent processing
* Opportunistic coalescing
* Scheduled aggregation

**Trade-offs:**

Independent execution minimizes waiting.

Coalescing increases throughput but introduces coordination delay.

Aggressive aggregation risks increasing tail latency under light load.

**When to choose each option:**

Independent execution suits latency-sensitive operations.

Coalescing is effective when many similar requests arrive together.

Scheduled aggregation works best for predictable workloads.

**Common failure modes:**

* Waiting excessively for additional requests.
* Increasing coordination complexity beyond the saved work.
* Creating latency spikes during low traffic periods.

**Example:**

Many storage systems combine adjacent writes before issuing disk operations, increasing throughput while delaying individual write completion.

# Network Batching: Nagle's Algorithm and TCP_NODELAY

**What it is:**

Nagle's algorithm batches small TCP writes into larger packets before transmission.

Setting `TCP_NODELAY` disables this behavior, transmitting data immediately.

**Why it exists:**

Sending numerous tiny packets wastes bandwidth and increases protocol overhead.

Batching reduces packet count and improves throughput.

Immediate transmission reduces latency.

The two goals compete directly.

**Options:**

* Nagle enabled
* `TCP_NODELAY` enabled

**Trade-offs:**

Nagle's algorithm:

* Higher throughput
* Better bandwidth utilization
* Increased per-write latency

`TCP_NODELAY`:

* Lower latency
* More packets
* Higher protocol overhead

Neither setting is universally correct.

**When to choose each option:**

Interactive protocols typically favor `TCP_NODELAY`.

Bulk data transfer usually benefits from Nagle's algorithm.

**Common failure modes:**

* Enabling `TCP_NODELAY` everywhere without measuring network overhead.
* Leaving Nagle enabled for latency-sensitive interactive traffic.
* Treating protocol defaults as universally optimal.

**Example:**

Interactive remote shells often disable Nagle's algorithm because individual keystrokes should appear immediately, whereas bulk file transfers benefit from batching.

# Throughput-Oriented System Design

**What it is:**

Some systems deliberately sacrifice request latency to maximize sustained throughput.

This is a conscious architectural objective rather than a limitation.

**Why it exists:**

For sufficiently large workloads, maximizing completed work often produces greater overall value than minimizing the latency of individual operations.

**Options:**

* Low-latency architecture
* High-throughput architecture
* Hybrid architecture

**Trade-offs:**

Low-latency systems minimize waiting but often waste capacity.

High-throughput systems maximize utilization while accepting additional delay.

Hybrid systems attempt to balance competing objectives but increase operational complexity.

**When to choose each option:**

Interactive applications generally prioritize latency.

Large-scale ingestion, analytics, and event processing often prioritize throughput.

Hybrid designs should be justified by explicit requirements rather than optimism.

**Common failure modes:**

* Optimizing throughput for workloads where users care primarily about responsiveness.
* Treating batching delays as implementation details instead of product behavior.
* Assuming higher utilization always represents better performance.

**Example:**

**Kafka** is explicitly designed around throughput. Log-structured storage, sequential disk I/O, and batching allow extremely high sustained write rates. Individual messages may wait briefly before being flushed, but this latency is an intentional trade for dramatically higher overall throughput. Kafka does not attempt to hide this trade-off; it embraces it as part of its design philosophy.

---

# Why Smart Engineers Disagree

Engineers often disagree about whether latency or throughput should be the primary optimization target because the answer depends on what the system is trying to accomplish, not on which metric is inherently better.

Teams building interactive products naturally prioritize responsiveness because users perceive delays directly.

Teams operating data pipelines, streaming platforms, or large-scale storage systems often prioritize sustained throughput because overall work completed is the dominant constraint.

The mistake is not choosing one objective over the other. The mistake is attempting to optimize both without explicitly deciding which requirement governs the system.

---

## Conclusion

Latency measures how long one operation takes. Throughput measures how much work the system completes over time. They are complementary but distinct properties, and improving one commonly requires sacrificing some of the other.

Batching, connection pooling, request coalescing, bulk database operations, and network packet aggregation all exploit the same principle: amortize fixed costs across more work to increase throughput while accepting additional latency. Tail latency provides the necessary counterbalance by ensuring that improvements in average performance do not conceal unacceptable delays experienced by users.

Once the optimization target has been identified through profiling, the engineer's task is not to maximize performance in the abstract, but to improve the property that actually limits the system's stated requirements.
