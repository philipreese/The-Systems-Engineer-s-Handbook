# Chapter 88 — Caching: Layers, Invalidation, Failure Modes

## 1. Purpose

Caching is one of the most effective techniques for improving system performance, but it is also one of the easiest ways to introduce subtle correctness problems. This chapter examines caching as a family of architectural layers rather than a single mechanism and focuses on the decisions that determine whether a cache simplifies a system or becomes a source of operational complexity.

Chapter 06 introduced the hardware end of the latency hierarchy and deferred application caching to this chapter. Chapter 08 likewise deferred caching strategy after establishing Little's Law and system constraints. Chapter 02 used distributed cache invalidation as the canonical example of accidental complexity that cannot be solved by intuition alone. This chapter resolves those forward references.

The question is no longer whether optimization is justified—that decision was made in Chapter 85—nor how the bottleneck was identified—that was Chapter 86. The question here is how to introduce caching without creating a system that is fast, incorrect, or both.

# 3. Key Concepts

* Caches exist at multiple layers.
* Storage is the easy part; invalidation is the difficult part.
* Every cache trades freshness for speed.
* Cache population and invalidation are independent decisions.
* Cache failures frequently occur under load rather than during normal operation.

## Caching as a Layered System

**What it is:**

Caching stores previously computed or retrieved data closer to where it will be needed again, reducing access to slower resources.

Caching exists as a hierarchy rather than a single implementation.

Typical layers include:

* CPU cache (covered in Ch 06)
* In-process application cache
* Distributed cache
* CDN edge cache

Each layer serves the same purpose while operating at different scales and latencies.

**Why it exists:**

Most systems repeatedly access the same data.

Avoiding unnecessary recomputation or remote access reduces latency and increases system capacity.

**Options:**

* No cache
* In-process cache
* Distributed cache
* CDN edge cache
* Multiple layers simultaneously

**Trade-offs:**

In-process caches are extremely fast but isolated to one process.

Distributed caches allow sharing across many instances but introduce network latency and consistency concerns.

CDN caches reduce latency for globally distributed users but provide the least immediate control over freshness.

Multiple cache layers provide the greatest performance improvements while significantly increasing operational complexity.

**When to choose each option:**

In-process caches suit immutable or process-local data.

Distributed caches are appropriate when many application instances require shared cached state.

CDNs are appropriate for static assets and cacheable HTTP responses delivered to geographically distributed users.

Layering multiple caches is justified only when profiling demonstrates that each additional layer addresses a measured bottleneck.

**Common failure modes:**

* Multiple cache layers holding conflicting versions of the same data.
* Assuming lower layers compensate for incorrect higher-layer behavior.
* Forgetting that every additional cache introduces another consistency boundary.

**Example:**

A typical web application might serve images through a CDN, session data from **Redis**, application configuration from an in-memory cache, while relying on CPU caches transparently within the processor. Each layer reduces access to a slower one beneath it.

## Cache Invalidation

Phil Karlton famously observed:

> "There are only two hard things in Computer Science: cache invalidation and naming things."

The quotation has endured because it identifies the true difficulty.

**What it is:**

Cache invalidation is the process of determining when cached data should no longer be trusted.

**Why it exists:**

Stale data is often worse than slow data.

Reading outdated account balances, permissions, inventory counts, or pricing can produce incorrect behavior even if responses are extremely fast.

**Options:**

* Time-based expiration (TTL)
* Explicit invalidation
* Event-driven invalidation

**Trade-offs:**

TTL expiration requires no coordination between writers and the cache.

Its simplicity comes at the cost of a bounded staleness window.

Explicit invalidation provides fresher data but requires every write path to invalidate every affected cache entry correctly.

Event-driven invalidation automates explicit invalidation through the system's write pipeline, reducing reliance on developers remembering invalidation calls while introducing additional infrastructure.

**When to choose each option:**

TTL is appropriate when temporary staleness is acceptable.

Explicit invalidation suits systems where freshness is more important than implementation simplicity.

Event-driven invalidation should be preferred whenever cache consistency spans multiple services or teams because it mechanically enforces correctness rather than depending on human discipline.

**Common failure modes:**

* Forgotten invalidation calls.
* Invalidating only one of several affected keys.
* Stale data persisting indefinitely because expiration was omitted.
* Different services maintaining inconsistent cache contents.

**Example:**

Many systems using **Redis** publish invalidation events whenever authoritative data changes so every application instance removes affected entries automatically instead of relying on individual developers to remember cache-busting logic in every write path.

## Cache Population Strategies

**What it is:**

Population strategy determines how cached data enters the cache.

This decision is independent of invalidation.

**Why it exists:**

Different workloads value freshness, write latency, and implementation simplicity differently.

No single strategy dominates all systems.

**Options:**

* Cache-aside
* Write-through
* Write-behind

**Trade-offs:**

### Cache-aside

The application checks the cache first.

On a miss, it loads data from the backing store and inserts it into the cache.

Advantages:

* Simple
* Lazy population
* Stores only requested data

Disadvantages:

* First request incurs full latency.
* Misses can produce burst load.

---

### Write-through

Every write updates both the backing store and cache before completion.

Advantages:

* Cache always contains recently written values.
* Reads have high hit rates.

Disadvantages:

* Increased write latency.
* Every write depends on cache availability.

---

### Write-behind

Writes update the cache immediately while persistence occurs asynchronously.

Advantages:

* Very low write latency.
* High write throughput.

Disadvantages:

* Risk of data loss before persistence.
* Greater implementation complexity.
* Eventual rather than immediate consistency.

**When to choose each option:**

Cache-aside should generally be the default because of its simplicity and broad applicability.

Write-through is appropriate when read consistency outweighs additional write latency.

Write-behind should be reserved for workloads where high write throughput justifies accepting asynchronous persistence.

**Common failure modes:**

* Cache-aside systems experiencing repeated cold misses.
* Write-through systems becoming unavailable because the cache failed.
* Write-behind systems losing recently accepted writes after crashes.

**Example:**

Many database-backed web applications use cache-aside with **Redis**, while storage engines and messaging systems frequently employ write-behind techniques internally to improve sustained throughput.

## Cache Stampede

**What it is:**

A cache stampede occurs when many concurrent requests miss the same cache entry simultaneously and all attempt to regenerate it.

This failure mode is also commonly called the **thundering herd** problem.

**Why it exists:**

Expiration often occurs simultaneously for every requester.

Without coordination, one expired key becomes hundreds or thousands of expensive backend operations.

**Options:**

* No coordination
* Request coalescing
* Short-lived regeneration lock

**Trade-offs:**

No coordination minimizes implementation complexity but risks overwhelming the backing store.

Request coalescing allows one computation to satisfy many waiting requests.

Short-lived locks prevent duplicate work while introducing brief synchronization delays.

**When to choose each option:**

No coordination is acceptable only for inexpensive computations.

Request coalescing should be the default when cache misses are expensive.

Short-lived locks are appropriate when regeneration cost is substantial and backend capacity is limited.

**Common failure modes:**

* Database overload immediately after popular keys expire.
* Cascading failures from synchronized cache expiration.
* Latency spikes despite otherwise healthy infrastructure.

**Example:**

High-traffic services using **Redis** frequently implement per-key locking or request coalescing so only one request repopulates an expired entry while others wait briefly for the result instead of independently querying the database.

## CDN Edge Caching

**What it is:**

A Content Delivery Network caches responses at servers geographically close to users.

Rather than every request reaching the origin infrastructure, many are satisfied at the network edge.

**Why it exists:**

Physical distance contributes significantly to latency.

Serving content nearer to users reduces network travel while decreasing load on origin servers.

**Options:**

* Origin-only serving
* Partial CDN caching
* Aggressive edge caching

**Trade-offs:**

Origin-only serving provides immediate consistency but scales poorly.

Partial caching balances freshness with reduced infrastructure load.

Aggressive caching minimizes origin traffic while increasing invalidation complexity.

**When to choose each option:**

Origin-only serving suits highly dynamic content.

Partial caching works well for mixed workloads.

Aggressive edge caching is appropriate for static assets and content whose freshness requirements tolerate explicit invalidation or expiration.

**Common failure modes:**

* Serving outdated assets after deployment.
* Invalidating origin caches while edge caches remain stale.
* Assuming CDN propagation is instantaneous.

**Example:**

CDNs such as Cloudflare and Fastly intentionally place caches close to end users, reducing latency by serving cached responses from edge locations instead of repeatedly contacting origin servers.

## Why Smart Engineers Disagree

The largest disagreement about caching is not whether it improves performance—it clearly does—but whether the operational complexity it introduces is justified.

One philosophy caches aggressively, arguing that modern systems cannot achieve required performance without extensive caching.

The other caches conservatively, arguing that every cache creates another copy of the truth and therefore another opportunity for inconsistency.

Both perspectives reflect legitimate engineering priorities.

If latency dominates the system's requirements, additional caching layers may be entirely justified.

If correctness, simplicity, and operational predictability dominate, fewer caches with carefully defined responsibilities often produce a more maintainable system.

The appropriate question is not "Can this be cached?" but "Is the performance benefit worth the additional consistency problem?"

---

## Conclusion

Caching is not a single optimization but a hierarchy of techniques operating at different layers of a system. CPU caches, application caches, distributed caches such as **Redis**, and CDN edge caches all exploit locality to avoid slower operations, but each introduces its own consistency boundary.

The difficult problem is not storing data—it is knowing when stored data is no longer valid. TTL expiration favors simplicity while accepting bounded staleness. Explicit invalidation improves freshness but depends on every write path remaining correct. Event-driven invalidation applies Principle 8 by replacing human discipline with mechanical enforcement. Population strategies such as cache-aside, write-through, and write-behind further determine how data enters the cache, while failure modes such as cache stampedes require explicit mitigation through request coalescing or regeneration locks.

Caching is therefore not merely a performance technique. It is a consistency strategy whose operational complexity must be designed as carefully as its performance characteristics.
