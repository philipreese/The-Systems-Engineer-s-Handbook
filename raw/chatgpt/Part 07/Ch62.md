Chapter 62 — Environment Promotion


---

1. Purpose

This chapter defines how a released artifact moves from development to production. The central principle is that deployment should promote an already-built artifact through progressively higher-trust environments, not repeatedly rebuild the same source code.

A successful staging deployment only provides evidence about production if both environments are running the identical artifact.


---

This chapter assumes the reader understands:

Release automation and artifact generation (Ch 61)

Release tags (Ch 56)

3. Key Concepts

Environment promotion separates two concerns:

producing an artifact

deciding where that artifact runs


Only the second should change as software progresses toward production.


---

Build Once, Promote Many

What it is:

Build a single immutable artifact exactly once, then deploy that identical artifact through development, staging, and production.

Examples include:

container images

compiled executables

packaged libraries

deployment bundles


Why it exists:

Testing establishes confidence in a specific artifact, not in the source code that produced it. Rebuilding creates a different artifact, invalidating that confidence.

Options:

Build once and promote the same artifact

Rebuild separately for each environment

Hybrid workflows that rebuild only for selected environments


Trade-offs:

Build-once promotion preserves confidence that tested software is the software deployed.

Per-environment rebuilds simplify some deployment pipelines but introduce artifact drift.

Hybrid approaches reduce some operational complexity while retaining portions of rebuild risk.


When to choose each option:

Build once, promote many should be the default for production systems.

Per-environment rebuilds are acceptable only where builds are themselves the object under test, not the deployed artifact.

Hybrid approaches should be rare and justified explicitly.


Common failure modes:

Production artifact differs from the one validated in staging.

Dependency resolution changes between environments.

Different compiler versions produce different binaries.

"Works in staging" no longer predicts production behavior because staging never tested the deployed artifact.


Example:

A container image is built once, assigned an immutable image digest, and stored in a registry. Development, staging, and production all deploy that same digest rather than rebuilding from source.


---

Build, Release, Run

What it is:

The Twelve-Factor App separates deployment into three distinct phases:

Build

Release

Run


The build creates the artifact.

The release combines that artifact with environment-specific configuration.

The run phase executes the released system.

Why it exists:

Separating these phases prevents environment-specific behavior from becoming part of the build itself.

Options:

Strict build/release/run separation

Environment-specific builds

Mixed workflows where configuration leaks into build time


Trade-offs:

Strict separation produces reproducible deployments.

Environment-specific builds increase flexibility but reduce reproducibility.

Mixed approaches often appear convenient initially but accumulate deployment complexity over time.


When to choose each option:

Strict separation should be the default.

Environment-specific builds should exist only when technically unavoidable.

Mixed approaches should be treated as transitional rather than permanent architecture.


Common failure modes:

Environment values compiled into binaries.

Separate production and staging builds diverging over time.

Configuration changes requiring recompilation.

Deployment pipelines coupling build and runtime concerns.


Example:

The Twelve-Factor App methodology formalizes build, release, and run as separate stages. A release changes configuration, not the executable itself.


---

Configuration Varies, Artifacts Do Not

What it is:

Environment-specific behavior is provided through configuration while the deployed artifact remains identical.

Typical configuration includes:

database endpoints

API keys

feature flags

service URLs


Why it exists:

Environments differ operationally, not functionally. Configuration should express those operational differences without changing executable code.

Options:

Environment variables

Configuration files injected at deployment

Configuration compiled into artifacts


Trade-offs:

Runtime configuration preserves artifact immutability.

Embedded configuration simplifies deployment but destroys reproducibility.

External configuration introduces operational management but preserves deployment integrity.


When to choose each option:

Runtime configuration should be the standard approach.

Embedded configuration is appropriate only for fixed-purpose systems such as firmware.

External configuration systems are appropriate for distributed applications.


Common failure modes:

Different binaries produced for staging and production.

Secrets embedded during compilation.

Configuration changes requiring rebuilds.

Environment behavior unintentionally diverging because artifacts differ.


Example:

A web application deploys the same container image everywhere while supplying different database connection strings through environment variables.


---

Progressive Promotion Within Production

What it is:

Production deployment strategies that gradually increase exposure rather than replacing the entire production system immediately.

Common approaches include:

canary deployments

blue-green deployments

phased rollouts


Why it exists:

Limiting initial exposure reduces the blast radius of deployment failures while allowing validation under real production traffic.

Options:

Immediate full deployment

Canary release

Blue-green deployment

Incremental regional rollout


Trade-offs:

Immediate deployment is operationally simple but maximizes deployment risk.

Canary releases reduce risk but require rollout management.

Blue-green deployments simplify rollback but duplicate production infrastructure.

Regional rollouts reduce geographic blast radius at the cost of longer deployment windows.


When to choose each option:

Immediate deployment is appropriate for low-risk internal systems.

Canary releases are appropriate for continuously deployed services.

Blue-green deployments are valuable when rollback speed is critical.

Regional rollout is appropriate for globally distributed systems.


Common failure modes:

Promoting different artifacts through different rollout stages.

Mixing rebuilt artifacts into the rollout.

Assuming rollout completion without validating production behavior.

Inconsistent deployment state across production regions.


Example:

A service deploys a single container image to 5% of production traffic before expanding to the remaining fleet. The metrics determining whether rollout proceeds belong to Part IX, Ch 73.


---

Why Smart Engineers Disagree

The disagreement is about what should vary between environments.

One position argues that rebuilding for each environment ensures every deployment is freshly produced and therefore trustworthy.

The other argues that rebuilding changes the thing being validated, making previous testing evidence less meaningful.

Both approaches can produce functioning systems.

The difference is whether environments validate source code or validate artifacts.

Modern deployment systems overwhelmingly optimize for artifact reproducibility. Confidence comes from promoting the same tested artifact rather than repeatedly recreating it.


---

Position

Production deployments should follow one simple rule:

> Build once. Promote many.



The correct default is:

Build a single immutable artifact.

Test that artifact.

Promote the identical artifact through every environment.

Allow only configuration—not executable code—to vary between environments.


Progressive rollout strategies such as canary and blue-green deployment then reduce the risk of promotion within production. The signals that determine whether rollout should continue or roll back are an observability concern and are covered in Part IX, Ch 73.
