Chapter 58 — Fail-Fast vs. Fail-Safe Pipeline Design


---

1. Purpose

This chapter defines how CI pipeline stages behave when failures occur: whether execution stops immediately or continues running remaining checks to collect full diagnostic output.

This is a direct application of the fail-fast principle from Part I, Ch 07, applied specifically to CI pipeline structure. The principle is reused, not redefined: early failure detection is valuable because it reduces wasted computation and feedback latency.

Here, the question is not whether to fail fast — but where and how aggressively to apply it inside an automated pipeline.


---

2. Prerequisites

Assumed concepts:

CI scope is limited to fast deterministic checks (Ch 57)

Feedback latency determines system usability (Part V, Ch 34)

Pipeline checks are structured into logical stages

Fail-fast principle is already established (Part I, Ch 07)



---

3. Key Concepts

Pipeline failure behavior operates at two levels:

Intra-stage behavior: what happens inside a single logical check unit

Inter-stage behavior: how multiple independent check categories are coordinated


Confusing these two leads to either wasted compute or poor developer feedback.


---

Fail-Fast Within a Single Stage

What it is: A stage stops execution immediately when a fatal condition is encountered within that stage.

Why it exists: Because continuing execution after a known invalid state is wasted computation.

Options:

Stop on first error within stage

Attempt partial continuation with degraded output

Always run to completion regardless of errors


Trade-offs:

Fail-fast minimizes wasted compute and reduces noise from cascading errors

Full continuation may produce more diagnostic context but risks irrelevant downstream errors

Hybrid approaches require error classification logic that increases system complexity


When to choose each option:

Fail-fast: compilation, build steps, or dependency resolution stages

Full continuation: loosely coupled validation steps where errors are independent

Hybrid: systems that explicitly separate fatal vs non-fatal diagnostics


Common failure modes:

Build continues after a fundamental failure (e.g., compilation error), producing misleading secondary errors

Excessive stopping that hides useful additional diagnostics within the same stage

Over-engineered error classification systems that are harder to maintain than they are worth


Example: If a TypeScript compilation fails, continuing to lint generated artifacts or run dependent build steps produces noise rather than useful signal. The correct behavior is to terminate the stage immediately.


---

Fail-Safe (Run-to-Completion) Across Independent Checks

What it is: All independent pipeline categories run regardless of individual failures, and results are aggregated into a single report.

Why it exists: To maximize diagnostic density per pipeline run.

Options:

Fail-fast across entire pipeline (stop on first failure anywhere)

Run-to-completion across all independent check categories

Hybrid grouping (fail-fast within groups, run-to-completion across groups)


Trade-offs:

Fail-fast pipeline minimizes compute usage but increases developer round-trips

Run-to-completion increases compute cost but reduces iteration cycles

Hybrid models balance both but require explicit grouping semantics


When to choose each option:

Fail-fast across pipeline: small systems or extremely expensive checks

Run-to-completion: most modern CI systems with parallelized execution

Hybrid: large systems with clearly separable check domains (lint, test, integration)


Common failure modes:

Developers fixing issues one at a time due to sequential failure reporting

Hidden secondary failures discovered only after primary fix

Excessive compute usage when running expensive downstream checks after early failures in unrelated domains

Lack of clear grouping leading to inconsistent behavior across pipelines


Example: A CI pipeline runs lint, unit tests, and type checks in parallel. Even if lint fails, unit tests and type checks still run, producing a full report of all issues in a single run. This reduces repeated CI cycles and improves developer feedback efficiency.


---

Ordering as a Complementary Mechanism

What it is: The sequence in which checks execute, independent of whether the pipeline is fail-fast or run-to-completion.

Why it exists: To reduce wasted computation by ensuring cheap checks run before expensive ones.

Options:

Strict sequential ordering (lint → type-check → unit → integration)

Parallel execution with early gating

Hybrid staged pipelines with dependency-aware execution


Trade-offs:

Strict ordering ensures minimal wasted work but reduces parallelism

Parallel execution increases speed but may run expensive checks unnecessarily

Hybrid staging balances efficiency and throughput


When to choose each option:

Strict ordering: simple pipelines or resource-constrained environments

Parallel execution with gating: modern CI systems with sufficient compute resources

Hybrid staging: large systems optimizing both cost and latency


Common failure modes:

Running expensive integration tests before basic static checks, wasting time on doomed builds

Over-parallelization leading to resource contention and slow overall throughput

Inconsistent ordering leading to unpredictable feedback latency


Example: A well-structured pipeline runs lint and type checks first. If either fails, no integration or E2E tests are triggered. This ensures that expensive tests are only run on code that has already passed basic correctness gates.


---

Fail-Fast vs Fail-Safe as a Design Choice

What it is: A spectrum between minimizing wasted computation and maximizing diagnostic completeness.

Options:

Global fail-fast (stop pipeline immediately on any failure)

Stage-level fail-fast with pipeline-level aggregation

Full run-to-completion with post-hoc reporting


Trade-offs:

Global fail-fast minimizes resource usage but increases iteration count

Stage-level fail-fast reduces waste while preserving good diagnostics

Full run-to-completion improves debugging signal but increases compute cost


When to choose each option:

Global fail-fast: high-cost pipelines or extremely slow checks

Stage-level fail-fast (default): most production CI systems

Full run-to-completion: debugging-heavy environments or when compute cost is negligible


Common failure modes:

Overusing global fail-fast and forcing multiple CI cycles for independent failures

Running all checks regardless of early catastrophic failures

Failing to distinguish between dependent and independent failure domains


Example: GitHub Actions supports fail-fast behavior in matrix jobs, where a failure in one job can cancel others. This directly encodes the trade-off between compute efficiency and full diagnostic collection at the orchestration level.


---

Why Smart Engineers Disagree

The disagreement is about what CI output is optimized for.

One view:

CI should minimize wasted computation and stop immediately on failure

Developers should fix issues incrementally, one at a time

Simplicity of execution model is more important than full error visibility


Another view:

CI should maximize feedback per run

Developers should see all failures at once to reduce iteration cycles

Compute cost is secondary to developer time


Both are valid under different constraints:

In resource-constrained or expensive compute environments, fail-fast behavior reduces cost significantly

In developer productivity–optimized environments, run-to-completion reduces cognitive and scheduling overhead


The real axis of disagreement is whether the dominant cost is compute or human iteration time.


---

Position

Within a single logical stage, fail-fast is the correct default: continuing after a known fatal failure is wasteful.

Across independent pipeline stages, run-to-completion with aggregation is the correct default: developers should not pay multiple CI cycles to discover independent failures.

Ordering then becomes the optimization layer on top of this split: cheap checks first, expensive checks later, with fail-fast behavior preventing unnecessary downstream execution when early invariants already fail.
