Chapter 59 — Caching Strategy in CI


---

1. Purpose

This chapter defines how CI systems reduce runtime cost through caching without compromising correctness of build and test results. The core tension is simple: caching makes CI fast by reusing work, but incorrect caching makes CI lying fast instead of truthfully fast.

A CI cache is safe only if it is strictly tied to the exact inputs that determine build output. Anything looser turns CI into a probabilistic system.


---

2. Prerequisites

Assumed concepts:

CI must remain a fast feedback system (Ch 57)

Pipeline behavior must preserve correctness under failure conditions (Ch 58)

Build and test steps are deterministic given a fixed dependency state

CI’s goal is reproducible verification, not just speed optimization



---

3. Key Concepts

Caching in CI exists to eliminate repeated work across identical dependency and build states.

The system is dominated by three cache targets:

Dependency resolution (package managers, lockfiles)

Build artifacts (compiled binaries, intermediate outputs)

Container layers (Docker / BuildKit caches)


All of them share one constraint: correctness depends entirely on cache key precision.


---

Dependency and Build Artifact Caching

What it is: Reusing previously computed results for dependency installation and build steps when inputs have not changed.

Why it exists: Most CI time is spent rebuilding identical states across commits that only modify small portions of a system.

Options:

Cache dependencies only (package manager caches)

Cache build outputs (incremental compilation, artifact reuse)

Cache container layers (Docker/BuildKit layered images)

No caching (fully clean builds every run)


Trade-offs:

Aggressive caching drastically reduces CI runtime

Weak caching increases rebuild cost but guarantees correctness

No caching maximizes correctness assurance but is usually too slow for practical CI


When to choose each option:

Dependency + build caching: default for most production systems

Minimal caching: security-sensitive or highly dynamic build environments

No caching: debugging cache-related issues or verifying build determinism


Common failure modes:

Cache serving artifacts from incompatible dependency states

Build systems incorrectly reusing compiled outputs across version changes

CI appearing green due to reused artifacts that would fail on clean build

Hidden coupling between unrelated commits due to shared cache state


Example: A Node.js project uses npm ci with caching keyed on package-lock.json. When dependencies do not change, the cached node_modules is reused, dramatically reducing install time while preserving correctness guarantees.


---

Cache Key Design as the Core Correctness Constraint

What it is: The cache key determines when cached artifacts are valid. It must encode exactly the inputs that affect the output.

Why it exists: A cache is only correct if identical outputs are produced for identical inputs. Any mismatch between key and actual inputs produces silent correctness failures.

Options:

Lockfile-based keying (recommended baseline)

Source-hash-based keying (broader but safer)

Over-broad keys (branch name, date, CI run ID)

Over-narrow keys (individual file hashes or partial dependency graphs)


Trade-offs:

Lockfile-based keys maximize reuse while preserving correctness boundaries

Source-hash keys increase safety but reduce cache hit rate

Over-broad keys increase reuse but risk stale or incorrect artifacts

Over-narrow keys reduce reuse efficiency without improving correctness meaningfully


When to choose each option:

Lockfile hash: standard for dependency caching in most ecosystems

Full source hash: systems where build outputs depend on more than dependencies

Avoid broad keys entirely for dependency caches

Avoid overly granular keys unless debugging cache invalidation issues


Common failure modes:

Cache reuse across dependency changes due to missing lockfile in key

False cache invalidation leading to poor CI performance

Cross-branch cache contamination where unrelated builds reuse incompatible artifacts

Non-deterministic builds masked by valid-looking cached outputs


Example: GitHub Actions and GitLab CI commonly use cache keys derived from package-lock.json, Cargo.lock, or similar lockfiles. This ensures cache reuse only occurs when the dependency graph is identical.


---

Cache Poisoning and Stale Cache Masking

What it is: A failure mode where cached artifacts are valid under the cache key but incorrect under the actual build semantics, causing CI to pass when a clean build would fail.

Why it exists: Because caches operate outside the compiler or test system’s awareness of correctness semantics.

Options:

Strict key enforcement with lockfile hashing

Periodic cache invalidation

Full cache bypass on scheduled runs

No caching in critical validation pipelines


Trade-offs:

Strict keying reduces risk but does not eliminate structural caching bugs

Periodic invalidation limits drift but reduces cache efficiency

Full bypass guarantees correctness but removes performance benefit

No caching eliminates risk but makes CI slow


When to choose each option:

Strict keying: baseline requirement for all systems using caching

Scheduled invalidation: large systems with long-lived caches

Full bypass: periodic verification (nightly or pre-release)

No caching: debugging or high-assurance builds


Common failure modes:

CI passing due to cached artifacts that hide broken dependency builds

Developers unaware that cache state differs from clean environment state

Production failures that cannot be reproduced in CI due to cache dependence

Long-lived caches accumulating subtle divergence from upstream dependencies


Example: A build passes in CI due to cached compiled dependencies. A clean build in a fresh environment fails because a transitive dependency update was not reflected in cached artifacts. This discrepancy is only exposed when caches are bypassed.


---

Scheduled Clean Builds as a Correctness Backstop

What it is: A CI job that runs without cache reuse, typically on a nightly schedule or pre-release pipeline.

Why it exists: To ensure that cached CI success reflects real build correctness, not just cached state consistency.

Options:

Nightly full clean builds

Pre-release clean builds

On-demand cache bypass builds

No clean builds (risk accepted)


Trade-offs:

Clean builds detect cache-related correctness issues

They are expensive relative to cached runs

They introduce delayed feedback rather than immediate detection

They increase CI infrastructure cost but reduce systemic risk


When to choose each option:

Nightly clean builds: standard for most production systems using caching

Pre-release clean builds: systems with strict release gates

On-demand builds: debugging cache inconsistencies

No clean builds: small systems or environments where caching risk is negligible


Common failure modes:

Cache bugs persisting undetected for long periods

Production builds failing despite green CI history

Over-reliance on cached CI results as proof of correctness

Ignoring divergence between cached and clean build environments


Example: Many large CI systems run a nightly job that disables caching entirely to validate that the build system remains deterministic. This catches cases where cached success masks broken dependency resolution or build assumptions.


---

Real Systems and Implementation Patterns

What it is: The standard industry implementations of caching strategies in CI systems.

Why it exists: To provide reproducible, high-performance CI while maintaining correctness guarantees.

Examples:

GitHub Actions cache keyed on lockfile hashes to ensure dependency alignment

GitLab CI cache configuration using explicit key derivation from dependency state

Docker BuildKit layer caching, where each image layer is reused only if its inputs (Dockerfile + context) remain unchanged


Trade-offs:

Significant CI speed improvements through reuse of deterministic build artifacts

Requires strict discipline in defining cache boundaries

Misconfiguration leads to subtle correctness failures that are hard to debug


When to use:

Any CI system with non-trivial build or dependency cost

Containerized build pipelines

Large monorepos with repeated dependency resolution


Common failure modes:

Incorrect layer invalidation in Docker leading to stale build artifacts

Cross-environment cache reuse causing environment-specific bugs

Misaligned cache keys between CI and local development environments


Example: Docker BuildKit caches each layer of an image build based on the Dockerfile instructions and input context. If any upstream layer changes, all dependent layers are invalidated, preserving correctness while maximizing reuse.


---

Why Smart Engineers Disagree

The disagreement is about whether CI cache is an optimization layer or a correctness risk surface.

One view:

Caching is purely an optimization problem

Strict keying and occasional invalidation are sufficient

Clean builds are unnecessary overhead if cache design is correct


Another view:

Caching introduces a new failure mode class that must be actively validated

CI correctness must be periodically verified without caching assumptions

Speed gains are secondary to reproducibility guarantees


Both are valid under different constraints:

In mature systems with well-understood build graphs, caching is a safe optimization layer

In complex or rapidly changing dependency ecosystems, caching becomes a correctness hazard without periodic validation


The key distinction is whether build determinism is structurally guaranteed or merely assumed.


---

Position

Caching is mandatory for practical CI performance, but it is not part of the correctness model.

The correct default is:

Cache aggressively using lockfile-derived keys

Treat cache correctness as a separate invariant from test correctness

Run periodic cache-bypassed builds to validate system determinism

Never allow cache behavior to influence logical test outcomes


A CI system that cannot pass a clean build is broken. A CI system that only passes due to cache is worse — it is indistinguishable from correctness until it fails in production.
