Chapter 52 — Squash vs. Preserve History


---

1. Purpose

This chapter defines what happens to a branch’s internal commit history at merge time: whether it collapses into a single synthetic commit (squash merge) or is preserved as a sequence of individual commits.

This is not a stylistic choice. It determines how much of development-time structure becomes permanent system history, and therefore directly affects debugging, bisecting, and long-term traceability.

It also depends heavily on Ch 51: commit quality determines whether preserving history is valuable or just archival noise.


---

2. Prerequisites

Assumed concepts:

Commit messages are structured artifacts with semantic meaning (Ch 51)

Branches are integration topologies, not semantic units of work (Ch 50)

PRs are transient integration vehicles, not the final form of history (Part VI, Ch 43)

git bisect and git blame operate on commit granularity, not PR granularity



---

3. Key Concepts

Merge-time history policy has two dominant models:

Squash merge: collapse all commits in a branch into a single commit

Preserve history: keep the branch’s commit sequence intact in the main history


A third hybrid exists (squash locally, preserve selectively via curated commits), but it behaves operationally like preserve-history with stricter hygiene requirements.

The real decision is not “which is better,” but:

> Do you trust the commits inside a PR to be individually meaningful units of history?




---

Squash Merge Model

What it is: All commits in a branch are combined into a single commit on the target branch, typically with a manually edited final message.

Why it exists: To decouple development noise from long-term history.

Options:

Squash and merge with auto-generated message

Squash and merge with curated final commit message

Squash and merge with PR title-derived message


Trade-offs:

Produces clean, linear history in main

Removes intermediate “messy development state” commits from permanent record

Eliminates visibility into step-by-step evolution of a change


When to choose each option:

Default choice for teams with inconsistent commit hygiene

High-churn feature development where intermediate commits are not meaningful

Systems prioritizing readable long-term history over micro-level traceability


Common failure modes:

Loss of debugging granularity when regression is introduced mid-branch

Over-reliance on PR discussion history as the only record of reasoning

Accumulation of large squashed commits that mix multiple conceptual changes

False sense of cleanliness masking poor development discipline upstream


Example: GitHub’s “Squash and merge” workflow is the most widely used default in modern software teams. It assumes that commit history inside a PR is not reliable enough to be preserved as system history and instead delegates meaning to the final merged commit.

This works well in environments where CI + review validate correctness at PR level, not commit level.


---

Preserve History Model

What it is: All commits from a branch are integrated into the main branch as a sequence, preserving their original structure (often after rebasing to ensure linearity).

Why it exists: To retain fine-grained historical signals for debugging, auditing, and bisecting.

Options:

Preserve full raw branch history

Preserve but require atomic commits (curated history)

Linearize via rebase before merge (single-parent history without squashing)


Trade-offs:

Enables git bisect at sub-PR granularity

Preserves development narrative as a sequence of incremental states

Increases noise in git log if commit discipline is weak


When to choose each option:

Full preservation: mature teams with strict commit discipline (Ch 51 enforced)

Curated preservation: large systems where debugging requires step-level traceability

Linearized preservation: teams that want clean graph structure but retain commit granularity


Common failure modes:

“Noise history” where commits like “fix typo” and “WIP” pollute production history

Developers gaming discipline locally but not maintaining consistency under pressure

Merge commits or rebases creating confusing graph structures without clear policy

git log becoming unusable without filtering tools due to volume of low-signal commits


Example: The Linux kernel and Git project both preserve commit history rigorously. Each commit is expected to be meaningful and reviewable independently. Maintainers rely on this granularity for git bisect, subsystem analysis, and regression tracking.

This model only works because commit quality is enforced upstream — without Ch 51-level discipline, preserve-history systems collapse into noise.


---

Bisectability as the Real Cost Function

What it is: The effectiveness of git bisect depends on how informative each commit is as a binary state in system evolution.

Options:

Squashed history: one PR = one bisect step

Preserved noisy history: many low-signal steps

Preserved curated history: high-resolution, high-signal steps


Trade-offs:

Squash merges reduce bisect resolution to PR granularity

Noisy preserved history increases search space without increasing signal quality

Curated history maximizes debugging efficiency but requires strict discipline


When to choose each option:

Squash: when PRs are already small and well-isolated

Preserve noisy history: almost never justified in production systems

Preserve curated history: large-scale systems with high debugging cost (kernel-like environments)


Common failure modes:

Assuming squash merge removes debugging capability entirely (it does not; it shifts granularity upward)

Assuming preserved history is automatically useful (it is only as good as commit discipline)

Ignoring PR size effects, making bisect useless regardless of strategy


Example: In large systems, a regression introduced in a squashed PR means bisect will identify the PR but not the internal step. In a kernel-style preserved-history model, bisect may identify the exact patch that introduced the regression, sometimes down to a single-line change. The difference is not merge strategy alone — it is commit granularity upstream.


---

Why Smart Engineers Disagree

The disagreement is not philosophical — it is about where structure is enforced.

Squash advocates assume:

Developers cannot reliably maintain high-quality commit history

PR-level review is the correct granularity of truth

Simpler history is worth loss of internal granularity


Preserve-history advocates assume:

Commit-level structure is enforceable and valuable

Debugging at sub-PR granularity is necessary at scale

History is a dataset, not just a narrative


Both assumptions are valid under different constraints:

If commit discipline is weak, preserving history preserves noise

If commit discipline is strong, squashing throws away useful signal


This is a dependency relationship: merge policy depends on commit hygiene, not the other way around.


---

Position

Squash vs preserve is not a preference axis. It is a consistency check.

If commit discipline is not enforced at Ch 51 level, squash merge is the correct default.

If commit discipline is enforced and meaningful, preserving history provides measurable debugging and analysis benefits.


The worst possible outcome is preserving unstructured history, because it maximizes long-term cognitive load without providing bisect or readability benefits.

In practice, most production systems converge on squash or curated linear history, because very few teams sustain kernel-grade commit hygiene at scale.
