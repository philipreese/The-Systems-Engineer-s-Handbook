Chapter 56 — Tagging and Release Markers


---

1. Purpose

This chapter defines how Git marks a specific commit as a release point. A tag is the simplest stable reference in the entire system: a named pointer to a commit that is intended to be treated as immutable once published.

This chapter does not define what a version means (Ch 16) or how releases are deployed (Ch 62). It defines the Git-level artifact that anchors those concepts in the repository history.


---

2. Prerequisites

Assumed concepts:

Branches are ephemeral execution structures (Ch 50, Ch 53)

Commit history is an ordered system of immutable snapshots (Ch 51)

Merge/rebase affects history shape but not commit identity semantics (Ch 55)

Release strategy defines version semantics separately from Git mechanics (Part II, Ch 16)

CI/CD pipelines consume release markers but do not define them (Ch 44, Ch 61–62 deferred)



---

3. Key Concepts

A Git tag is a permanent reference to a specific commit. Unlike branches, tags do not move.

There are two fundamental types:

Lightweight tag: a simple pointer to a commit

Annotated tag: a full Git object with metadata and optional cryptographic signature


The distinction is not cosmetic — it determines whether a tag is an informational bookmark or a formal release artifact.


---

Lightweight Tags

What it is: A direct reference to a commit hash, stored as a named pointer (e.g., v1.2.3 → commit abc123).

Why it exists: To provide a minimal, low-overhead way to label a commit.

Options:

Lightweight tag as a local bookmark

Lightweight tag as a temporary reference in scripts or experimentation

Avoidance in production workflows entirely in favor of annotated tags


Trade-offs:

Extremely simple and fast to create

No metadata about who created it or why

No built-in integrity or provenance guarantees

Easy to misuse as a “release marker” without accountability


When to choose each option:

Local development: marking experimental states or checkpoints

Automation internals: ephemeral tagging in scripts

Avoid in formal release workflows


Common failure modes:

Treating lightweight tags as production release markers

Loss of accountability (no tagger identity or timestamp)

Ambiguity about whether a tag was intentional or accidental

Inability to verify provenance in distributed systems


Example: A developer creates a lightweight tag v1.0-test locally to mark a build state. It is pushed accidentally and later mistaken for an official release. There is no metadata to determine origin or intent.


---

Annotated Tags (Release-Grade Markers)

What it is: A full Git object containing:

Tagger identity

Timestamp

Tag message

Reference to a commit

Optional GPG signature


Why it exists: To turn a simple pointer into an auditable, verifiable release artifact.

Options:

Annotated tag without signature

Annotated + signed tag (cryptographically verified)

Annotated tag with structured release message (common in automated systems)


Trade-offs:

Adds metadata overhead, but minimal in practice

Enables provenance verification

Distinguishes intentional releases from accidental references

Slightly more operational complexity than lightweight tags


When to choose each option:

Default for all production releases

Required for public or distributed software

Mandatory in systems with integrity or supply-chain concerns


Common failure modes:

Creating annotated tags but never signing or validating them

Treating annotation as documentation rather than enforcement boundary

Inconsistent tagging practices across release channels

Mixing lightweight and annotated tags for the same semantic version


Example: The Git project itself uses annotated and signed tags for official releases. Each release tag includes a message describing the release and is signed by maintainers, allowing consumers to verify authenticity.


---

Semantic Versioning as Tag Encoding

What it is: A versioning scheme (SemVer, Tom Preston-Werner) commonly encoded into tag names (e.g., v1.4.2).

Why it exists: To standardize how version numbers communicate compatibility expectations.

Options:

Semantic Versioning (dominant standard)

Calendar versioning (time-based releases)

Internal sequential versioning (build numbers, hashes)


Trade-offs:

SemVer encodes compatibility intent in a structured way

Calendar versioning encodes release timing but not compatibility semantics

Sequential or hash-based versions maximize uniqueness but lose human interpretability


When to choose each option:

SemVer: libraries, APIs, and consumer-facing interfaces (default in most ecosystems)

Calendar versioning: time-driven release cycles (e.g., enterprise software suites)

Sequential versions: internal systems or CI artifacts where human interpretation is irrelevant


Common failure modes:

Mislabeling breaking changes as minor or patch releases

Treating version numbers as purely chronological rather than semantic contracts

Decoupling tag meaning from actual API behavior (version becomes cosmetic)


Example: A release tag v2.3.0 typically encodes SemVer meaning: new features added without breaking backward compatibility. This chapter does not define what “breaking” means — that belongs to Ch 16, which defines version semantics at the system-contract level.


---

Signed Tags and Provenance

What it is: A cryptographically signed annotated tag, typically using GPG, which allows verification that a tag was created by an authorized maintainer.

Why it exists: To ensure that release artifacts have verifiable origin and have not been tampered with or impersonated.

Options:

Unsigned annotated tags

GPG-signed tags

External signature systems (in more complex supply-chain setups)


Trade-offs:

Signed tags increase trust guarantees at minimal operational cost

Require key management discipline

Verification must be integrated into release consumption workflows to be meaningful


When to choose each option:

Unsigned: internal systems with trusted deployment pipelines

Signed: public releases, open-source software, or distributed artifacts

Stronger signing infrastructure: high-security or regulated environments


Common failure modes:

Generating signatures but never verifying them downstream

Key rotation mismanagement invalidating historical trust chains

Assuming signing alone guarantees full supply-chain integrity (it does not)


Example: The Git project signs its official release tags. Consumers can verify that a given version was produced by an authorized maintainer, providing a baseline integrity guarantee against tampering or impersonation.


---

Why Smart Engineers Disagree

The disagreement is about whether tags are metadata or artifacts.

One view:

Tags are convenience pointers for humans

Lightweight tags are sufficient for most workflows

Additional metadata is unnecessary overhead


Another view:

Tags are the root of release integrity

They must be auditable, signed, and structured

A tag is not just a pointer — it is a declaration of system state


Both are valid under different assumptions:

In internal systems with controlled CI/CD pipelines, lightweight or minimal tagging may be sufficient because trust is enforced elsewhere

In distributed or public systems, tags become part of the trust boundary and require stronger structure


The key distinction is whether a tag is a private navigation aid or a public system contract.


---

Position

For production systems:

Annotated tags are the default release mechanism

Lightweight tags are reserved for local or ephemeral use

Signed tags are strongly recommended for any externally consumed software


A tag is not just a label — it is a commitment that a specific commit represents a defined system state. If that commitment is exposed beyond a single machine, it should be treated as an artifact, not a shortcut.
