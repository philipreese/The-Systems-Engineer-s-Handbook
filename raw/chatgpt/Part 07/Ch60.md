Chapter 60 — Matrix Builds


---

1. Purpose

This chapter defines how CI pipelines execute the same verification logic across multiple environments. A matrix build is not a different pipeline — it is a Cartesian expansion of a single pipeline across dimensions like OS, runtime version, or dependency set.

The goal is to detect environment-specific failures that a single execution context cannot reveal.


---

2. Prerequisites

Assumed concepts:

CI scope is defined by fast deterministic checks (Ch 57)

Pipeline execution behavior under failure is governed by fail-fast vs fail-safe rules (Ch 58)

Caching strategies may differ per environment but are not part of matrix logic (Ch 59)

CI checks assume deterministic behavior given a fixed environment



---

3. Key Concepts

A matrix build expands one logical pipeline into multiple independent execution cells.

Each dimension represents a variability axis:

Operating system (Linux, macOS, Windows)

Language/runtime version (e.g., Node 18 vs Node 20)

Dependency set (different lockfiles, optional feature sets)


The key structural property is multiplicative scaling:

> Total jobs = product of all dimension cardinalities




---

Matrix Explosion and Combinatorial Cost

What it is: The number of CI jobs grows multiplicatively as dimensions are added.

Why it exists: Each additional axis represents an independent source of runtime variability, and coverage requires full Cartesian sampling.

Options:

Full Cartesian matrix (all combinations)

Pruned matrix (only supported or meaningful combinations)

Minimal matrix (single representative per major dimension)


Trade-offs:

Full matrix maximizes environment coverage but scales cost multiplicatively

Pruned matrix reduces cost while preserving meaningful risk coverage

Minimal matrix optimizes speed but risks missing environment-specific regressions


When to choose each option:

Full matrix: libraries or frameworks with strict cross-platform guarantees

Pruned matrix: most production applications with known supported environments

Minimal matrix: internal tools or systems deployed in controlled environments


Common failure modes:

Treating minor version differences as independent risk axes when they are not

Expanding matrices to include unsupported or unofficial environments

CI cost growing exponentially without corresponding increase in defect detection

Developers disabling matrix dimensions due to slow CI, losing real coverage unintentionally


Example: A CI pipeline defined as:

3 operating systems

4 runtime versions

2 dependency configurations


produces 3 × 4 × 2 = 24 independent jobs per commit. Adding one more runtime version doubles the cost of every pipeline run along that axis.


---

Matrix Design as a Support Contract Encoding

What it is: A matrix build is a formal encoding of the environments a system claims to support.

Why it exists: Testing is only meaningful if it reflects real-world execution environments. Anything beyond that is synthetic coverage with no production relevance.

Options:

Strict support matrix (only officially supported environments)

Extended matrix (includes near-future or experimental support targets)

Over-broad matrix (tests many irrelevant environments “for safety”)


Trade-offs:

Strict matrices reduce CI cost and align testing with actual deployment reality

Extended matrices allow forward-looking validation but increase cost

Over-broad matrices waste compute and dilute signal-to-noise ratio


When to choose each option:

Strict matrix: most production systems with defined deployment targets

Extended matrix: libraries preparing for upcoming platform support

Over-broad matrix: generally a design smell indicating unclear support policy


Common failure modes:

Testing OS patch versions that do not change observable behavior

Running matrix combinations that are not actually deployable configurations

Confusing theoretical coverage with real compatibility guarantees

CI slowdowns causing teams to disable matrix dimensions entirely


Example: A cross-platform library may support Windows, macOS, and Linux on two major Node.js versions. Adding every patch version of Node or every minor OS update creates combinatorial explosion without meaningful increase in coverage fidelity.


---

Fail-Fast Behavior in Matrix Execution

What it is: A policy governing whether remaining matrix jobs continue after one or more jobs fail.

Why it exists: To balance compute efficiency against diagnostic completeness across multiple environment runs.

Options:

Fail-fast enabled: cancel remaining matrix jobs on first failure

Fail-fast disabled: run all matrix jobs to completion

Selective fail-fast: cancel only dependent or identical-environment jobs


Trade-offs:

Fail-fast reduces compute usage and speeds up feedback termination

Full execution provides complete environment failure visibility

Selective strategies require more orchestration complexity but balance both goals


When to choose each option:

Fail-fast enabled: large matrices where each job is expensive

Fail-fast disabled: debugging environment-specific issues

Selective fail-fast: mature CI systems with well-defined matrix semantics


Common failure modes:

Cancelling matrix runs too early and missing secondary environment-specific failures

Running all jobs unnecessarily, wasting compute on already-invalid configurations

Misinterpreting partial matrix results as full validation

Hidden platform-specific failures only discovered in production


Example: GitHub Actions supports a fail-fast configuration for matrix jobs. When enabled, a failure in one OS/runtime combination can cancel remaining jobs, prioritizing speed of feedback over completeness of environment diagnostics.


---

Real Systems and Standard Patterns

What it is: The standard implementation of matrix builds in modern CI systems.

Why it exists: To provide scalable environment coverage without duplicating pipeline definitions.

Examples:

GitHub Actions matrix syntax defining OS × runtime × dependency axes

GitLab CI parallel matrix jobs with environment variables

Standard library testing matrices (e.g., Python, Node.js, Rust projects)


Trade-offs:

Enables compact specification of large environment spaces

Requires careful control of dimensions to avoid exponential cost growth

Can obscure cost scaling if not explicitly tracked


When to use:

Cross-platform libraries

Runtime-dependent frameworks

Systems with explicit multi-version compatibility guarantees


Common failure modes:

Uncontrolled growth of matrix dimensions over time

Misaligned CI configuration and actual deployment environments

Developers unaware of total job count scaling behavior


Example: Most open-source libraries use a minimal matrix such as:

Linux + macOS

Current LTS runtime version + previous LTS


This reflects actual user environments rather than theoretical completeness.


---

Why Smart Engineers Disagree

The disagreement is about what matrix builds represent.

One view:

Matrix builds are a correctness requirement across all plausible environments

More dimensions improve confidence and reduce unknowns

Compute cost is secondary to correctness guarantees


Another view:

Matrix builds should reflect explicit support commitments, not hypothetical environments

Each additional dimension must justify itself via real deployment risk

Over-testing creates false confidence and operational overhead


Both are valid under different constraints:

In ecosystem libraries, broader matrices are necessary because downstream environments are unknown

In application systems, deployment environments are controlled and known, making broad matrices unnecessary


The real axis of disagreement is whether the system is a platform or a product.


---

Position

Matrix builds are a mapping from supported environments to validation coverage.

The correct default is:

Define matrix dimensions strictly according to real support commitments

Prefer minimal but meaningful environment coverage over exhaustive combinatorial expansion

Treat each additional matrix axis as a cost multiplier, not a free increment

Use fail-fast selectively to balance compute cost and diagnostic completeness


A matrix that does not reflect actual deployment environments is not increasing confidence — it is simulating it.
