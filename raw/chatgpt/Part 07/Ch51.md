Chapter 51 — Commit Message Conventions


---

1. Purpose

This chapter defines the commit message as a durable, machine-consumable artifact of system history. A commit is not just a diff checkpoint; it is the atomic unit that powers git blame, git bisect, release tooling, and long-term system understanding.

Once a commit is merged, its message becomes more stable than the code it describes. Code gets refactored. History does not. That asymmetry is the reason commit discipline exists.


---

2. Prerequisites

Assumed concepts:

Branches represent integration topology, not semantic meaning (Ch 50)

PRs are integration vehicles, not historical truth

Issue-to-change linkage exists but is external metadata (Part VI, Ch 43)

Code comments are not a substitute for system history (Part IV, Ch 30)



---

3. Key Concepts

Commit history serves three distinct consumers:

Humans reading linear history (git log)

Humans debugging regressions (git bisect, git blame)

Machines generating release artifacts (changelogs, versioning, dependency graphs)


A commit message that serves only one of these is incomplete in a production system.


---

Commit Message as a Historical Contract

What it is: A commit message is a stable description of intent at a single point in time, tied to a diff that will likely be refactored, moved, or partially undone later.

Why it exists: Because diffs are transient explanations of what changed, while commit messages are persistent explanations of why this change existed at all.

Options:

Free-form descriptive message

Structured human-readable format (subject + body discipline)

Machine-structured format (Conventional Commits, metadata tags)


Trade-offs:

Free-form messages maximize flexibility but degrade long-term tooling and readability

Structured human-readable messages improve maintainability but require discipline

Machine-structured formats enable automation but constrain expressive freedom


When to choose each option:

Free-form: prototypes, throwaway branches, local-only work (rare in production)

Human-structured: most general-purpose software systems

Machine-structured: systems with automated release pipelines, dependency management, or changelog generation


Common failure modes:

Commit messages duplicating the diff (“fixed bug”, “updated code”) — no additional signal beyond git itself

Loss of causal information (“why this change exists”) leading to incorrect future refactors

Overloading commits with multiple unrelated intents, making history non-decomposable

Inconsistent formatting that breaks downstream tooling assumptions


Example: Linux kernel development enforces strong commit discipline at scale. Each commit includes a meaningful subject line, a body explaining rationale, and often subsystem prefixes. This is not stylistic — it is required for maintainers to safely navigate massive distributed history using tools like git bisect.


---

Chris Beams’ Commit Discipline Rules (Operational Baseline)

What it is: A widely adopted set of rules (“How to Write a Git Commit Message”) that standardizes human-readable commit structure.

Why it exists: To make commit history navigable under long time horizons and multiple contributors.

Options:

Subject-only commits

Subject + body separation

Full structured message with imperative mood and rationale


Trade-offs:

Subject-only commits are fast to write but lose reasoning context

Structured commits require discipline but remain interpretable after refactors

Imperative mood standardization improves readability in log aggregation tools


When to choose each option:

Subject-only: trivial or experimental changes

Full structured format: production systems with long-lived codebases

Mixed approach: teams with varying maturity, migrating toward stricter discipline


Common failure modes:

“Noise commits” that restate diffs without explaining intent

Missing rationale leading to incorrect future changes

Commit logs that become unreadable chronological noise rather than navigable history


Example: A well-formed commit message in this style:

Subject: “Fix null pointer in session initialization”

Body: explains that race condition occurs when session cache is cold and initialization path assumes pre-warmed state


The diff shows what changed. The message explains why the system was incorrect in the first place.


---

Structured Commit Formats (Conventional Commits)

What it is: A specification that encodes semantic meaning into commit messages using prefixes like:

feat: new user-visible functionality

fix: bug correction

chore: maintenance changes

refactor: internal restructuring without behavior change


Why it exists: To allow machines to derive meaning from commits without parsing natural language.

Options:

No structure (free text)

Lightweight semantic prefixing (Conventional Commits)

Fully structured metadata commits (rare, specialized systems)


Trade-offs:

Free text maximizes expressiveness but disables automation

Semantic prefixes enable deterministic tooling (version bumps, changelogs)

Heavy metadata formats increase overhead and reduce developer compliance


When to choose each option:

Free text: internal tools, low automation requirements

Conventional Commits: CI/CD pipelines with automated release management

Full structured metadata: dependency-managed ecosystems requiring strict traceability


Common failure modes:

Inconsistent prefix usage (fix, bugfix, hotfix used interchangeably)

Semantic drift (marking breaking changes as fix or chore)

Tooling dependence without enforcement (automation assumes discipline that doesn’t exist)


Example: Semantic-release systems consume Conventional Commit history to automatically determine version increments and generate changelogs. This directly connects commit discipline to release engineering (expanded in Ch 61).


---

Commit Messages vs Architectural Decision Records (ADR)

What it is: A commit message describes a single bounded change. An ADR describes a durable architectural decision that persists beyond any individual diff.

Why it exists: Because some decisions outlive their implementation history and must remain interpretable even after code churn.

Options:

Encode decisions in commit messages

Encode decisions in ADRs

Duplicate reasoning in both


Trade-offs:

Commit-only reasoning gets lost during refactors or squash merges

ADR-only reasoning loses proximity to implementation context

Duplication increases maintenance cost but improves discoverability


When to choose each option:

Commit-only: small systems, short-lived code

ADR-only: large architectural shifts requiring long-term reference stability

Both: production systems where implementation and architecture evolve independently


Common failure modes:

Architectural reasoning buried in commit history and effectively lost after rebases or squashes

ADRs becoming stale and detached from actual implementation reality

Developers using commits as pseudo-ADRs, producing inconsistent historical signal


Example: A commit may state “switch connection pool timeout to 30s due to observed PostgreSQL latency under load,” while an ADR defines the broader decision to adopt aggressive timeout tuning as a system-wide resilience strategy.


---

Why Smart Engineers Disagree

The disagreement is about where “truth” should live.

One camp argues:

The code + commit history is sufficient truth

Additional structure is overhead unless tooling absolutely requires it


Another camp argues:

Commit history is a data source, not documentation

Without structure, historical data is non-computable at scale


Both are correct under different constraints:

In small repositories with few contributors, unstructured commits are often adequate

In large distributed systems, commit history becomes a dataset, not a narrative — and datasets require schema


The tension is between expressive freedom and long-term machine usability.


---

Position

Commit messages are not optional metadata.

They are part of the system’s interface to its own past.

The baseline for production systems is:

Human-readable structure (subject + rationale)

Consistent semantic classification where automation depends on it

Clear separation between commit intent and architectural decisions (ADRs)


Free-form commit history is acceptable only in environments where history is never expected to be queried as a system of record.
