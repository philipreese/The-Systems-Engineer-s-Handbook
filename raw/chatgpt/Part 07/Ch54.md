Chapter 54 — Force Push: When It’s Acceptable


---

1. Purpose

This chapter defines when rewriting remote Git history via force push is acceptable, and when it is a production hazard. The operation itself is simple; the system-level consequences are not.

This is a direct application of the reversibility and blast radius model from Part I, Ch 09 to a single Git action: overwriting remote history.


---

2. Prerequisites

Assumed concepts:

Changes must be evaluated by reversibility × blast radius (Ch 09)

Branches are either private execution space or shared integration state (Ch 50–53)

Commit history may be rewritten locally before integration (Ch 51)

Merge policy is separate from history cleanup mechanics (Ch 52)



---

3. Key Concepts

Force push is not a single operation in practice. It is two distinct behaviors:

Safe context: rewriting your own unpublished or private branch

Unsafe context: rewriting shared history that other developers may depend on


The distinction is not technical — it is coordination scope.


---

Force Push on Private Branches

What it is: Overwriting the remote representation of a branch that only you control, typically after rewriting history locally (e.g., interactive rebase cleanup before PR).

Why it exists: To clean up noisy development history before it becomes shared system history.

Options:

No force push (append-only history, even locally messy)

Force push after local rebase or squash

Rebase locally without ever pushing intermediate states


Trade-offs:

Enables clean commit history before review (improves Ch 51 signal quality)

Prevents exposure of intermediate “work-in-progress” states

Introduces risk only if branch ownership assumptions are violated


When to choose each option:

Force push: personal feature branches prior to review

No force push: workflows requiring immutable published history even pre-merge (rare)

Local-only rebasing without push: strict hygiene workflows where remote branch is only a final snapshot


Common failure modes:

Accidentally treating a shared branch as private, overwriting others’ work

Confusion about branch ownership in loosely coordinated teams

Loss of local commits due to repeated rebasing without understanding remote divergence

Developers avoiding cleanup entirely to “avoid risk,” resulting in noisy PRs


Example: A developer rebases a feature branch to remove WIP commits and improve commit structure before opening a pull request. A force push updates the remote branch to reflect the cleaned history. No one else depends on that branch, so the operation is effectively safe.


---

Force Push on Shared Branches (High Blast Radius)

What it is: Overwriting history on a branch that other developers have already fetched, based work on, or integrated into their own local branches.

Why it exists (almost always incorrectly):

Misunderstanding of branch ownership boundaries

Attempt to “fix” history on a shared integration branch

Emergency correction of mistaken merges without proper rollback strategy


Options:

Force push shared branch (almost never acceptable)

Revert commits instead of rewriting history

Reset and coordinate recovery explicitly with the team


Trade-offs:

Force push creates silent divergence between local clones

Developers lose unpushed work without explicit notification

Git’s distributed model provides no automatic recovery coordination

History becomes non-monotonic and difficult to reason about causally


When to choose each option:

Force push shared branches: effectively never in mature workflows

Revert-based recovery: standard approach for correcting mistakes

Coordinated history rewrite: only in extreme cases with explicit team-wide synchronization


Common failure modes:

“My commits disappeared” after a branch was rewritten upstream

Divergent clones requiring manual reconciliation across multiple developers

CI systems acting on different histories than developer machines

Loss of trust in branch stability as a shared coordination mechanism


Example: A force push to a shared develop branch overwrites commits already pulled by other developers. Their local branches now contain commits that no longer exist upstream, forcing manual reconciliation or data loss if those commits were never pushed elsewhere. This is a well-known class of incident in distributed teams and is difficult to fully recover from without coordination overhead.


---

--force vs --force-with-lease

What it is: Two variants of force push with very different safety properties.

Why it exists: To distinguish between unconditional history overwrite and conditional overwrite based on remote state.

Options:

--force: unconditionally overwrite remote branch

--force-with-lease: only overwrite if remote branch is unchanged since last fetch


Trade-offs:

--force maximizes power but eliminates safety guarantees

--force-with-lease preserves intent while preventing accidental overwrites

Lease mechanism introduces a lightweight consistency check without requiring locking


When to choose each option:

--force: almost never justified outside scripted recovery in controlled environments

--force-with-lease: default for any legitimate force push scenario

No force push at all: preferred when history rewrite is unnecessary


Common failure modes:

Using --force out of habit, increasing risk of overwriting unseen remote changes

Failing to fetch before force pushing, causing unexpected rejections with lease

Assuming lease protects against all conflicts (it only protects remote divergence, not logical conflicts)


Example: --force-with-lease prevents overwriting a remote branch if another developer has pushed changes since your last fetch. This avoids silent destruction of newer history, making it the safe default for history rewriting workflows.


---

Branch Protection as Mechanical Enforcement

What it is: Repository-level rules that prevent force pushes to specific branches (e.g., main, release/*).

Why it exists: To eliminate reliance on human discipline for high-blast-radius operations.

Options:

Allow force push on all branches

Restrict force push on protected branches only

Fully disable force push in repository


Trade-offs:

Full allowance maximizes flexibility but increases systemic risk

Partial protection isolates high-risk branches while preserving workflow flexibility

Full disablement enforces immutability but may block legitimate recovery scenarios


When to choose each option:

Full protection: production systems with strict release control

Partial protection: most SaaS systems (protect main, allow feature branch flexibility)

No protection: experimental repositories or individual workflows


Common failure modes:

Accidentally enabling force push on shared branches in CI/CD workflows

Misconfigured protection rules excluding critical release branches

Developers bypassing protection via alternative branches and later merging unsafe history

Over-reliance on protection rules without understanding underlying risk model


Example: GitHub and GitLab both support branch protection rules that disallow force pushes on protected branches. This is the correct implementation of Principle 8: safety-critical operations should be enforced mechanically, not socially.


---

Why Smart Engineers Disagree

The disagreement is about where responsibility for history integrity should live.

One view:

Developers should be trusted to manage history carefully

Force push is a necessary tool for correcting mistakes

Over-restriction slows down legitimate workflows


Another view:

Shared history is infrastructure, not personal workspace

Mistakes in history rewrite are disproportionately expensive

System-level guarantees should prevent entire classes of human error


Both are valid under different assumptions:

In small teams with high coordination, force push is a controlled tool

In large distributed systems, the cost of accidental overwrite dominates flexibility benefits


The dividing line is not technical capability — it is coordination cost and blast radius.


---

Position

Force push is acceptable only when the branch is private in both intent and reality.

The safe default is:

Use --force-with-lease for all legitimate history rewrites

Allow force push only on branches where no other developer is expected to base work

Enforce restrictions on shared branches via repository-level protection


A system that relies on “remembering not to force push shared branches” is already incorrectly designed.
