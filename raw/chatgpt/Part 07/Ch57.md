Chapter 57 — What Belongs in CI (and What Doesn’t)


---

1. Purpose

This chapter defines the boundary of responsibility for Continuous Integration: what an automated pipeline should validate on every change versus what should be deferred to slower, non-blocking systems.

This directly resolves the scope question left open in Part VI, Ch 47: CI is not “everything you can test,” it is “everything that is cheap, fast, and deterministically enforceable per integration.”

If CI becomes the place where slow or ambiguous checks accumulate, it stops being integration infrastructure and becomes a queueing system.


---

2. Prerequisites

Assumed concepts:

Human review is not a substitute for mechanical validation (Part VI, Ch 47)

Testing is layered by feedback cost and scope (Part V, Ch 34–35)

Feedback loop latency determines system usability (Part V, Ch 34)

CI is a gate on integration, not a general-purpose analysis engine



---

3. Key Concepts

CI scope is defined by three constraints:

Cost per run (CPU time, wall time, infrastructure load)

Determinism (same input → same output)

Signal latency (time until a developer can act on result)


Anything that violates these constraints does not belong in the blocking CI path.


---

CI as Fast Deterministic Validation Layer

What it is: The minimal automated system that answers: “Does this change break known, cheaply testable invariants?”

Why it exists: To prevent broken code from entering shared integration points while maintaining rapid feedback loops.

Options:

Minimal CI (lint, type-check, unit tests)

Moderate CI (includes integration tests on small scope)

Heavy CI (includes broad system tests and extended validation)


Trade-offs:

Minimal CI is fast but may miss integration issues

Moderate CI balances coverage and speed

Heavy CI increases confidence but risks feedback latency collapse


When to choose each option:

Minimal CI: high-frequency deployment systems with strong local testing discipline

Moderate CI: most production SaaS systems (default balance)

Heavy CI: systems where deployment cost is high and failure tolerance is low


Common failure modes:

CI pipelines expanding until they become multi-hour jobs

Flaky tests degrading trust in CI output

Developers bypassing CI due to slow feedback

CI becoming a staging environment rather than a validation gate


Example: A typical modern CI pipeline runs linting, static type checks, and unit tests on every pull request. These checks are fast enough (< minutes) to provide actionable feedback before context switching costs become dominant.


---

What Belongs in CI

What it is: Checks that are cheap, deterministic, and directly tied to code correctness at the unit or small integration level.

Includes:

Linting and formatting validation

Static type checking (where applicable)

Unit tests (Part V, Ch 34–35)

Small-scope integration tests with controlled dependencies

Build verification (compilation, packaging sanity checks)


Why it exists: These checks catch high-frequency, low-cost-to-detect failures immediately at the integration boundary.

Options:

Strict gating (all must pass to merge)

Advisory mode (non-blocking but visible)

Tiered gating (critical checks block, others inform)


Trade-offs:

Strict gating enforces system integrity but increases friction when CI is noisy

Advisory mode improves flow but risks accumulating unnoticed regressions

Tiered gating balances enforcement and flexibility but requires careful classification of signals


When to choose each option:

Strict gating: mature CI systems with low flakiness

Advisory mode: early-stage systems or experimental repositories

Tiered gating: large systems with heterogeneous code quality and tooling


Common failure modes:

Treating slow or flaky tests as CI-grade checks

Allowing inconsistent enforcement across repositories

CI becoming a “best effort” signal rather than a gate

Overloading CI with duplicate checks already performed locally


Example: Google’s large-scale CI systems emphasize fast deterministic checks on every change, relying on strong local tooling and incremental validation to keep CI fast enough to remain continuously useful.


---

What Does NOT Belong in CI

What it is: Checks that are expensive, probabilistic, slow, or better evaluated in aggregate rather than per-change.

Includes:

Full end-to-end system test suites

Large-scale performance benchmarks

Comprehensive security scans (SAST/DAST at full depth)

Dependency vulnerability audits across full transitive graphs

Multi-service environment simulations


Why it exists: These checks provide value, but not at per-commit frequency. Their cost violates CI’s latency constraint.

Options:

Nightly or scheduled pipelines

Pre-release validation stages

Manual or semi-automated audit processes

Sampling-based execution strategies


Trade-offs:

Moving expensive checks out of CI preserves fast feedback loops

Deferring checks increases window of undetected regressions

Batch execution reduces cost per result but increases detection latency


When to choose each option:

Nightly pipelines: continuous monitoring of system-wide health trends

Pre-release pipelines: validation before deployment to production

Manual audits: compliance-heavy or high-risk domains requiring human oversight


Common failure modes:

CI pipelines blocked by slow E2E tests

Developers waiting hours for feedback on trivial changes

Flaky integration tests destabilizing merge confidence

Attempting to “fix CI” by scaling infrastructure instead of reducing scope


Example: Large systems often run full integration and performance test suites nightly rather than on every pull request. This preserves CI speed while still providing periodic system-level validation.


---

Martin Fowler’s Definition of CI

What it is: Martin Fowler’s 2006 definition of Continuous Integration describes CI as the practice of integrating code frequently with automated verification to ensure fast detection of integration problems.

Why it exists: To emphasize that CI is fundamentally about feedback speed, not test completeness.

Options:

CI as full validation system

CI as fast integration feedback loop

CI as hybrid with layered validation cadence


Trade-offs:

Full validation CI collapses under latency constraints

Fast feedback CI optimizes developer productivity and system stability

Hybrid systems require careful boundary management but scale best in practice


When to choose each option:

Fast feedback loop: default interpretation aligned with Fowler’s definition

Hybrid layering: large systems requiring both speed and depth

Full validation CI: generally a misapplication of the concept


Common failure modes:

Expanding CI until it no longer provides “continuous” feedback

Confusing CI with release validation pipelines

Treating CI as a correctness oracle rather than a fast signal generator


Example: Fowler explicitly frames CI as a discipline centered on frequent integration and rapid automated verification, not exhaustive system validation. Systems that take hours to return feedback violate the core intent of CI regardless of how thorough they are.


---

Why Smart Engineers Disagree

The disagreement is about whether CI is a gate or a safety net.

One view:

CI should be minimal and fast

Everything slow belongs elsewhere

Developer velocity depends on tight feedback loops


Another view:

CI should provide maximum confidence per merge

More checks reduce downstream risk

Slower CI is acceptable if it prevents production incidents


Both are valid under different system constraints:

High-frequency deployment systems optimize for fast feedback and accept layered validation

High-risk deployment systems optimize for pre-merge confidence and tolerate slower pipelines


The trade-off is between local responsiveness and global certainty.


---

Position

CI is not the place for completeness. It is the place for immediacy.

The correct default is:

CI contains only fast, deterministic, per-change checks

Expensive validation is moved to scheduled or pre-release systems

CI failure means “this change violates a known invariant,” not “this system might be unsafe in aggregate”


Once CI exceeds the latency threshold where developers stop waiting for it, it has already ceased to function as continuous integration.
