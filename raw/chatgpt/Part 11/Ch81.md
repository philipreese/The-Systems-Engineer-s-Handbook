Chapter 81 — Input Validation

Purpose

Every trust boundary identified in Chapter 79 and protected through the layered architecture of Chapter 80 ultimately receives data from a less-trusted source.

This chapter covers the first and most universal security control applied at those boundaries: validating that incoming data is actually what the system expects before acting on it.

The goal of input validation is not to determine whether a caller is authorized—that belongs to Chapter 82. It is to ensure that untrusted data remains data and cannot alter the behavior of the system interpreting it.

---

Every Trust Boundary Requires Validation

Data becomes dangerous only after software assigns meaning to it.

An HTTP request is just bytes.

A SQL engine interprets SQL.

A shell interprets commands.

A template engine interprets expressions.

A browser interprets HTML and JavaScript.

Security failures occur when untrusted input reaches an interpreter capable of acting on it without first being constrained to the forms the system actually expects.

The specific interpreter changes.

The architectural mistake does not.


---

Decision / Principle — Validate at Every Trust Boundary

What it is

Input validation verifies that data entering a more-trusted component conforms to the expectations of that component before any meaningful processing occurs.

Validation should occur whenever data crosses a trust boundary, not only at the outermost edge of the system.


---

Why it exists

Defense in depth assumes individual controls will eventually fail.

If one layer performs validation but downstream services assume all incoming data is permanently trustworthy, a compromised intermediary can bypass the system's only validation step.

Each trust boundary should independently verify the assumptions it depends upon.

Validation therefore becomes the code-level application of Chapter 80's layered-defense principle.


---

Options

Validate only at the system perimeter

Validate at every trust boundary

Trust upstream services completely



---

Trade-offs

Advantages of validation at every boundary:

Limits blast radius

Prevents trust from propagating unnecessarily

Makes services independently robust

Supports zero-trust architectures


Costs:

Additional implementation effort

Some duplicated validation logic

Slight runtime overhead


The redundancy is intentional.

Independent validation is more valuable than centralized validation that becomes a single point of failure.


---

When to choose each option

Validate at every boundary where assumptions change.

Internal APIs should validate their own inputs even when requests originate from another trusted service.


---

Common failure modes

Assuming internal traffic is inherently safe

Validating only HTTP requests while trusting message queues completely

Skipping validation for administrative interfaces

Trusting data solely because another component previously accepted it



---

Example

A microservice receiving requests from an API gateway should still validate required fields, value ranges, and data formats rather than assuming the gateway already rejected every malformed request.


---

Decision / Principle — Treat Untrusted Input as Data, Never Code

What it is

Many well-known security vulnerabilities arise because software allows untrusted input to become part of a command or expression interpreted by another system.

Instead of remaining data, the input becomes executable instructions.


---

Why it exists

Different technologies use different interpreters.

Databases interpret SQL.

Operating systems interpret shell commands.

Template engines interpret expressions.

Browsers interpret HTML and JavaScript.

Directory services interpret LDAP queries.

The underlying architectural mistake remains identical.

Untrusted input is combined with executable syntax before reaching an interpreter.


---

Options

Common interpreter targets include:

SQL

Shell commands

HTML

Template engines

Expression languages

LDAP

XPath



---

Trade-offs

Keeping data separate from executable syntax:

Advantages

Eliminates entire categories of injection attacks

Produces predictable interpreter behavior

Simplifies reasoning


Costs

Requires using appropriate APIs

Sometimes limits highly dynamic query construction


Those costs are generally insignificant compared to the security benefit.


---

When to choose each option

Whenever input reaches any interpreter, use mechanisms that preserve the distinction between executable instructions and user-supplied data.

Never construct executable strings by concatenating untrusted input.


---

Common failure modes

SQL injection

Command injection

Template injection

Expression-language injection

Cross-site scripting


Although these vulnerabilities appear different, they all share the same underlying architectural failure.


---

Example

Constructing SQL with string concatenation:

SELECT * FROM users WHERE username = '" + username + "'";

allows the database to interpret attacker-controlled input as SQL.

Using a parameterized query keeps the supplied value as data regardless of its contents.


---

Decision / Principle — Prefer Parameterization Over Filtering

What it is

Parameterization separates executable instructions from externally supplied values.

The interpreter receives both independently and never treats parameter values as executable syntax.

Prepared statements in SQL are the canonical example.


---

Why it exists

Filtering attempts to recognize malicious input.

Parameterization removes the interpreter's ability to execute the input as code in the first place.

One approach attempts to recognize attacks.

The other eliminates the relevant attack surface structurally.

This follows Principle 8: prefer mechanical enforcement over human discipline.


---

Options

String concatenation

Escaping

Pattern matching

Blocklisting

Parameterization



---

Trade-offs

Parameterization:

Advantages

Structural protection

Independent of attack syntax

Easier to review

Supported by mature libraries


Costs

Requires appropriate APIs

Dynamic query generation sometimes becomes less straightforward


Filtering:

Advantages

Simple to implement initially


Costs

Constant maintenance

Easily bypassed

Depends on anticipating attacker behavior



---

When to choose each option

Use parameterized APIs whenever they exist.

Treat manual escaping and string filtering as compatibility techniques rather than primary security mechanisms.


---

Common failure modes

Building SQL with string concatenation

Escaping only known attack sequences

Assuming quotes are the only dangerous characters

Believing sanitization replaces parameterization



---

Example

Virtually every mature database driver supports prepared statements and parameterized queries.

These APIs prevent user input from being interpreted as SQL regardless of whether the supplied value contains quotation marks, comments, or SQL keywords.


---

Decision / Principle — Prefer Allowlists Over Denylists

What it is

Validation may define either:

what is allowed, or

what is forbidden.


An allowlist accepts only inputs matching known-good expectations.

A denylist rejects only patterns already identified as dangerous.


---

Why it exists

The set of valid inputs is usually much smaller and more stable than the set of invalid ones.

Engineers generally know exactly what a postal code, UUID, email address, or account number should look like.

They cannot enumerate every malicious variation attackers may invent.


---

Options

Allowlist validation

Denylist validation

Hybrid validation



---

Trade-offs

Allowlists:

Advantages

Smaller rule set

Easier reasoning

Resistant to novel attack patterns

Stable over time


Costs

Must evolve when legitimate formats change

Can reject unexpected but valid inputs if designed poorly


Denylists:

Advantages

Easy to begin

Useful for detecting known abuse


Costs

Never complete

Continuous maintenance

Vulnerable to bypass techniques



---

When to choose each option

Use allowlists whenever legitimate input has a well-defined structure:

expected formats,

bounded lengths,

valid ranges,

known character sets,

enumerated values.


Reserve denylists for supplemental detection rather than primary validation.


---

Common failure modes

Blocking only a handful of SQL keywords

Rejecting obvious script tags while allowing equivalent encodings

Maintaining enormous regular-expression blocklists

Treating denylist matches as comprehensive protection



---

Example

Accepting only uppercase hexadecimal characters for a SHA-256 digest is substantially more robust than attempting to reject every possible malicious character sequence.


---

Case Study — Log4Shell

The Log4Shell vulnerability (CVE-2021-44228) is frequently described as a logging vulnerability.

Architecturally, it was an input-validation failure.

Applications commonly logged attacker-controlled values such as HTTP headers.

Those values eventually reached Log4j's JNDI lookup mechanism, which interpreted specially crafted strings as expressions to evaluate.

The attacker-controlled input did not remain data.

It reached an interpreter capable of performing external lookups, ultimately enabling remote code execution.

The underlying lesson extends far beyond logging.

The same architectural mistake appears whenever untrusted input reaches any interpreter without first being constrained to the forms the application actually intends to accept.

Log4Shell therefore belongs in the broader family of injection vulnerabilities alongside SQL injection, command injection, template injection, and cross-site scripting.

The specific interpreter differed.

The failure pattern did not.


---

Why Smart Engineers Disagree

Experienced engineers rarely disagree that validation is necessary.

They disagree about where it belongs.

One philosophy centralizes validation at system entry points to avoid duplication.

Another validates independently at every trust boundary, accepting repeated checks in exchange for stronger containment.

This handbook adopts the latter position.

Repeated validation is a deliberate consequence of defense in depth.

Trust should not automatically propagate simply because another component previously accepted the data.


---

Relationship to Error Handling

Validation failures should be reported using the error contracts established in Chapter 21.

The security objective of this chapter is determining whether input should be accepted.

The structure and semantics of the resulting error response have already been established and should remain consistent across the system.


---

Looking Ahead

Input validation answers one question:

"Is this data structurally safe to process?"

It does not answer:

"Should this caller be allowed to perform this operation?"

That decision belongs to authentication and authorization, the subject of the next chapter.
