What a Dependency Actually Is (Threat Model Baseline)

A dependency is:

Code the team did not write

Usually not fully reviewed

Pulled from a public or semi-public registry

Executed as part of the build or runtime process

Updated automatically or semi-automatically (Chapter 63)


The key security distinction:

Outdated dependency risk: known vulnerable code

Supply chain risk: trusted name, untrusted behavior


A dependency can be “up to date” and still malicious.


---

Decision / Principle — Treat Dependencies as External Code with Full Execution Privilege

What it is

Dependencies are not “inputs.” They are executable extensions of the system.

Once imported, they run with the same privileges as application code unless explicitly constrained.


---

Why it exists

Most modern systems implicitly trust dependency ecosystems:

npm

PyPI

Maven Central

crates.io


These ecosystems prioritize ease of publishing over strict identity validation.

As a result, dependency trust is often based on:

package name

version number

maintainer reputation


None of these are strong security guarantees.


---

Options

Treat dependencies as trusted code

Treat dependencies as untrusted executable input

Treat dependencies as verified artifacts with enforced provenance



---

Trade-offs

Implicit trust model

Advantages:

Simple development workflow

Fast adoption of libraries

Low friction ecosystem participation


Costs:

High supply chain exposure

Silent compromise risk

Hard-to-audit transitive dependency graphs


Explicit untrusted model

Advantages:

Strong isolation

Reduced blast radius

More predictable builds


Costs:

Engineering overhead

More restrictive build pipelines

Compatibility friction



---

When to choose each option

Implicit trust only holds in low-risk environments (local development, prototypes).

Production systems should assume dependencies are attack surface unless verified.


---

Common failure modes

Blindly trusting transitive dependencies

Assuming “popular package = safe package”

Installing packages by name without provenance checks

Ignoring build-time execution of dependency scripts



---

Example

A widely used npm or Python package can execute arbitrary code during installation or import, meaning compromise is not limited to runtime behavior.


---

Attack Patterns in the Supply Chain

Maintainer Compromise

A legitimate maintainer account is taken over or a legitimate package repository is compromised.

Attack pattern:

attacker publishes a new version under a trusted name

downstream systems auto-update or install new version

malicious code executes with full trust of dependency system


This is the most dangerous class because it preserves the expected identity of the package.


---

Typosquatting

An attacker publishes a package with a name similar to a legitimate one.

Example pattern:

reqeusts instead of requests


The attack relies on human error in package selection.


---

Dependency Confusion

An internal package is referenced by name in a private registry.

An attacker:

publishes a public package with the same name

build system resolves public registry first

malicious package is installed instead of internal one


This is a failure of namespace isolation between trust domains.


---

Decision / Principle — Enforce Lockfiles as the Minimum Supply Chain Control

What it is

A lockfile records the exact resolved dependency graph:

direct dependencies

transitive dependencies

exact versions

often cryptographic hashes of resolved artifacts


It ensures builds are reproducible and verifiable.


---

Why it exists

Without a lockfile:

dependency resolution is non-deterministic

builds may fetch different code over time

attackers can influence version selection indirectly


With a lockfile:

dependency graph becomes fixed

builds become repeatable

unexpected upgrades are prevented unless explicitly updated


This converts “trust the registry state at build time” into “verify exact expected artifact.”


---

Options

No lockfile (floating dependencies)

Version-constrained dependencies only

Fully pinned lockfile with hashes



---

Trade-offs

No lockfile

Advantages:

maximum flexibility

simplest initial setup


Costs:

non-reproducible builds

silent dependency drift

high supply chain exposure


Pinned lockfile

Advantages:

reproducibility

controlled updates

reduced attack surface variability


Costs:

update overhead

merge conflicts in lock changes



---

When to choose each option

Pinned lockfiles are the baseline for any production system.

Unpinned dependency graphs are only acceptable where reproducibility is not required (experiments, prototypes).


---

Common failure modes

Lockfiles not committed to source control

Partial pinning (only top-level dependencies pinned)

Ignoring lockfile integrity in CI

Blind regeneration of lockfiles without review



---

Example

package-lock.json, Cargo.lock, poetry.lock, and equivalent mechanisms ensure that the same dependency graph is installed across environments.

Decision / Principle — Require Artifact Provenance and Package Signing

What it is

Artifact signing extends the concept from Chapter 56 (signed release tags) to package ecosystems:

packages are signed by trusted build systems

signatures verify origin and integrity

provenance attestation confirms how the artifact was produced


This ensures:

not just “what version is this”

but “was this built by the expected pipeline from the expected source”



---

Why it exists

A correct version number does not guarantee correctness of origin.

An attacker can:

publish a malicious artifact under a legitimate version

inject code into build pipelines

compromise registry storage


Signing shifts trust from “registry contents” to “cryptographically verified build origin.”


---

Options

Unsigned packages (registry trust only)

Signature verification of packages

Full provenance attestation (build system + source + artifact linkage)



---

Trade-offs

Unsigned packages

Advantages:

simplest ecosystem compatibility

minimal tooling overhead


Costs:

registry compromise risk

indistinguishable malicious artifacts

no origin verification


Signed artifacts

Advantages:

integrity verification

origin validation

resistance to registry tampering


Costs:

tooling complexity

key management requirements

ecosystem adoption gaps



---

When to choose each option

Unsigned packages should not be assumed safe in high-trust environments.

Signed artifacts become increasingly important as build pipelines and registries become higher-value targets.


---

Common failure modes

verifying version but not provenance

trusting registry metadata without signature checks

incomplete signature validation in CI pipelines



---

Example

Modern package ecosystems increasingly adopt:

npm provenance

PyPI trusted publishing

Sigstore-based signing (in various ecosystems)


These systems tie published artifacts to a verifiable build identity rather than a mutable registry account alone.


---

Decision / Principle — Maintain a Software Bill of Materials (SBOM)

What it is

An SBOM is a complete, machine-readable inventory of:

direct dependencies

transitive dependencies

versions

identifiers and metadata


It answers a critical question:

> “What is actually inside this build?”




---

Why it exists

When a vulnerability is disclosed:

systems must identify exposure across entire dependency graphs

manual inspection is infeasible at scale


Without an SBOM:

organizations do not know where a vulnerable component is used

remediation becomes reactive and incomplete


With an SBOM:

impact analysis becomes deterministic

patching can be scoped precisely



---

Options

No formal inventory

Partial dependency tracking

Full SBOM generation and enforcement



---

Trade-offs

No SBOM

Advantages:

zero overhead


Costs:

no visibility into transitive risk

slow incident response

incomplete remediation


SBOM-enabled systems

Advantages:

complete dependency visibility

faster vulnerability response

structured compliance


Costs:

tooling integration

build-time overhead

maintenance of metadata accuracy



---

When to choose each option

SBOMs become necessary as soon as:

dependency graphs exceed trivial size

regulatory or security requirements exist

multiple teams share a codebase



---

Common failure modes

stale SBOMs not updated with builds

missing transitive dependencies

treating SBOM as documentation instead of an enforced artifact



---

Example

When a vulnerability is discovered in a widely used transitive library, SBOM data allows systems to immediately identify every affected service rather than manually tracing dependency trees.


---

Real Incident Pattern (Supply Chain Compromise)

A canonical class of incidents across npm, PyPI, and other ecosystems follows a consistent pattern:

1. A legitimate, widely used package is maintained by a small set of contributors.


2. Maintainer credentials are compromised or ownership is transferred.


3. A new version is published under the trusted package name.


4. The malicious version includes targeted behavior affecting specific downstream systems.


5. Automatic update mechanisms or loose version constraints pull the compromised version into production systems.



The key property:

Nothing about the package name or version indicates compromise.

The trust relationship is what was exploited.


---

Why Smart Engineers Disagree

The central disagreement is how far verification should extend beyond “known package name and version.”

One view assumes:

ecosystem reputation is sufficient

malware in major registries is rare enough to accept


Another view assumes:

any externally sourced code must be treated as hostile until proven otherwise

provenance, signing, and reproducibility are mandatory controls


In practice, most production systems evolve toward stronger guarantees as their exposure and dependency footprint grows.
