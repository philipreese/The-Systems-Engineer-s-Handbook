Chapter 70 — Metrics vs. Logs vs. Traces

Purpose

Observability is often described as having three primary signal types: logs, metrics, and traces. Engineers frequently know the names, but confusion arises because the same operational problem is often approached with the wrong signal. Teams attempt to answer a tracing question with metrics, a metrics question with logs, or a logging question with traces, resulting in unnecessary cost and incomplete answers.

This chapter establishes the observability taxonomy for the remainder of Part IX. Like Chapter 65's documentation taxonomy, it distinguishes related tools by the question each is designed to answer. The goal is not to declare one signal superior, but to identify the cheapest signal that can answer a particular operational question.

Chapter 69 established what belongs in a log. Chapter 72 will explain distributed tracing in depth. This chapter positions all three signal types relative to one another.

Key Concepts

The central ideas introduced here are:

Every observability signal answers a different class of question.

Logs preserve detailed historical events.

Metrics summarize system behavior over time.

Traces reconstruct the execution path of individual requests.

Metrics remain inexpensive because they intentionally limit cardinality.

The first step in instrumentation is identifying the question, not choosing the technology.



---

The Three Signals Answer Different Questions

What it is

Observability consists of three complementary signal types:

Logs

Metrics

Traces


They differ primarily in the shape of the information they preserve.

Why it exists

A common misconception is that the three signals contain the same information in different formats.

They do not.

Each intentionally sacrifices some kinds of information to make other kinds inexpensive to collect, store, and query.

Choosing the wrong signal often means paying more while learning less.

Options

Use:

logs for discrete historical events

metrics for aggregated measurements over time

traces for the execution history of individual requests


Trade-offs

No single signal can efficiently answer every operational question.

Logs preserve detail but become expensive at scale.

Metrics sacrifice detail to make large-scale aggregation inexpensive.

Traces preserve causality but only for individual executions.

When to choose each option

Begin with the operational question.

Do not begin with the instrumentation technology.

Common failure modes

Teams frequently standardize on whichever observability system they deployed first.

Organizations with mature logging systems often attempt to answer every question through log searches.

Organizations invested heavily in metrics sometimes attempt to encode detailed request information as metrics.

Neither scales well.

Example

The widely used "three pillars of observability" terminology exists precisely because production systems require multiple complementary signal types rather than a single universal one.


---

Logs Preserve Individual Events

What it is

A log is a timestamped record describing a discrete event.

Each event can contain arbitrary structured information.

Unlike metrics, logs preserve the full details of each occurrence.

Why it exists

Logs answer questions about specific historical events.

Typical questions include:

What happened?

What error occurred?

What configuration changed?

What did this request do?


Chapter 69 covered what information belongs in those records.

Options

Log:

request failures

lifecycle events

configuration changes

state transitions


Avoid using logs to summarize system health over long periods.

Trade-offs

Pros

Rich diagnostic context.

Arbitrary structure.

Excellent for investigation.


Cons

High storage cost.

Expensive indexing.

Slower queries across large datasets.


When to choose each option

Use logs when the investigation centers on a specific event or small collection of events.

Common failure modes

Teams expect log searches to behave like metric dashboards.

Instead of plotting one aggregated time series, the system must search millions of independent records.

Example

Searching the logs associated with a correlation ID from Chapter 21 reconstructs the events surrounding a failed request, even if no distributed trace exists.


---

Metrics Summarize System Behavior

What it is

A metric is a numeric time series representing an aggregated measurement.

Rather than storing every event individually, metrics continuously summarize system behavior.

Examples include:

requests per second

CPU utilization

memory usage

error rate

queue depth

consumer lag


Why it exists

Many operational questions concern trends rather than individual requests.

Examples include:

Is the system healthy?

Is traffic increasing?

Is latency rising?

Is capacity being exhausted?


Storing every request individually would be unnecessary for answering these questions.

Options

Metrics typically measure:

counters

gauges

histograms

summaries


The implementation details vary, but they all reduce large numbers of observations into compact numerical series.

Trade-offs

Pros

Extremely efficient storage.

Fast queries.

Long retention.

Excellent dashboards.

Efficient aggregation across machines.


Cons

Little individual request detail.

Cannot explain specific failures.

Cannot reconstruct execution history.


When to choose each option

Use metrics whenever the question asks:

how much?

how often?

how fast?

how many?


Common failure modes

A dashboard may show that error rate doubled at 2:17 PM.

The metric correctly reports that a problem exists.

It cannot explain why.

Another signal is required.

Example

Prometheus popularized the pull-based, low-cardinality time-series model that now dominates open-source metrics collection. Its data model intentionally favors aggregated measurements that remain efficient to scrape, store, and query over long periods.


---

Traces Preserve Causality

What it is

A trace records the execution path of a single request as it moves through a distributed system.

Unlike logs, which record isolated events, traces preserve the causal relationships between operations.

Why it exists

Distributed systems rarely fail inside a single process.

One request may pass through:

API gateway

authentication service

payment service

inventory service

database

message broker


Understanding where time was spent requires reconstructing that entire chain.

Options

A request may have:

no trace

a sampled trace

a complete trace


Chapter 72 discusses those implementation choices.

Trade-offs

Pros

Complete request timeline.

Cross-service visibility.

Latency attribution.

Causal relationships.


Cons

More expensive than metrics.

Higher per-request overhead.

Less suitable for broad statistical summaries.


When to choose each option

Use traces when investigating:

one failing request

unexpected latency

service interaction problems

bottlenecks spanning multiple services


Common failure modes

Teams expect traces to answer fleet-wide questions.

A trace explains one execution.

It does not replace statistical measurements across millions of requests.

Example

A customer reports that a single checkout request required twelve seconds.

A distributed trace can identify that nine of those seconds were spent waiting on an inventory service.

A metric cannot.


---

Cardinality Determines Metric Cost

What it is

Cardinality is the number of distinct label values associated with a metric.

A metric remains inexpensive because the number of unique time series remains intentionally small.

Why it exists

Metrics achieve efficiency through aggregation.

Aggregation only works when observations can be grouped into relatively few categories.

Options

Low-cardinality labels include:

HTTP method

endpoint

status code

region

availability zone


High-cardinality labels include:

user ID

request ID

email address

session ID

transaction ID


Trade-offs

Low-cardinality metrics aggregate efficiently across millions of observations.

High-cardinality labels create separate time series for nearly every request.

Storage, memory usage, and query cost increase dramatically.

Eventually the metric behaves less like a metric and more like a collection of log entries.

When to choose each option

Reserve high-cardinality identifiers for logs and traces.

Metrics should describe populations, not individuals.

Common failure modes

The most common instrumentation mistake is attaching request identifiers or user identifiers as metric labels.

For example:

http_requests_total{
    endpoint="/checkout",
    user_id="4819372"
}

This creates a new time series for every user.

Instead of a small collection of aggregated measurements, the monitoring system must manage millions of distinct series.

The metric loses the economic advantages that justified using a metric in the first place.

Example

Prometheus strongly encourages low-cardinality instrumentation because excessive label cardinality rapidly increases memory consumption, storage requirements, and query complexity.


---

Choosing the Right Signal

What it is

Instrumentation should begin with the operational question.

The signal follows from the question.

Why it exists

Engineers often ask:

> "Should we log this?"



The better question is:

> "What are we trying to learn later?"



Options

Operational Question	Preferred Signal	Why

Is the service healthy right now?	Metric	Aggregated system state
Is traffic increasing?	Metric	Time-series trend
How many requests failed today?	Metric	Efficient aggregation
What happened three hours ago?	Log	Historical event reconstruction
Why did this customer request fail?	Trace (or logs via correlation ID)	Individual execution history
Where did this request spend its time?	Trace	Causal latency analysis
Which configuration changed before the outage?	Log	Discrete historical event


Trade-offs

Using the correct signal minimizes both operational cost and investigation time.

Using the wrong signal usually increases both.

When to choose each option

Start by identifying whether the investigation concerns:

the system as a whole,

a historical event,

or one request.


That determines the appropriate signal.

Common failure modes

Some organizations duplicate information across all three signals.

Others rely entirely on one.

Both approaches waste resources.

The three signals are complementary precisely because each optimizes a different balance between detail and efficiency.

Example

An API begins returning elevated latency.

A metric dashboard shows average latency increasing.

A trace identifies that database calls are responsible.

Logs reveal that a database configuration change occurred shortly beforehand.

No individual signal provides the complete picture.

Together they do.

Kafka Consumer Lag: A Metric, Not a Log

What it is

Kafka consumer lag measures the difference between the latest available message offset and the latest offset processed by a consumer.

It is an aggregate measurement of backlog.

Why it exists

Chapter 17 deferred Kafka's operational visibility to this Part.

Consumer lag is an example of why metrics exist.

Operators rarely need to inspect every message in the queue.

They need to know whether consumers are keeping up with producers.

Options

Observe:

current lag

lag trend

maximum lag

lag by consumer group


Do not emit one log entry for every outstanding message simply to estimate backlog.

Trade-offs

A single lag metric immediately communicates processing health.

Equivalent information derived from logs would require searching and aggregating enormous numbers of events.

When to choose each option

Use metrics to monitor queue health.

Use logs when investigating why a particular message failed.

Use traces when following a request whose processing spans multiple services.

Common failure modes

Teams sometimes confuse queue depth with message history.

Backlog size is a metric.

The contents of an individual failed message belong in logs or traces.

Example

A Kafka dashboard showing steadily increasing consumer lag immediately indicates that consumers cannot keep pace with incoming traffic, even before any individual message failures become apparent.


---

Why Smart Engineers Disagree

Experienced engineers generally agree on the purpose of each signal but disagree about where the boundaries should be.

One philosophy favors rich logging and sparse metrics, arguing that detailed historical evidence can always be aggregated later if necessary.

The opposing philosophy emphasizes metrics wherever possible, arguing that operational dashboards, alerting, and capacity planning require efficient aggregation, while detailed investigations can rely on selective logs and traces.

Both approaches reflect real operational constraints.

Organizations operating thousands of services often optimize aggressively for storage and query efficiency, pushing information toward metrics whenever aggregation is sufficient. Teams building smaller systems may reasonably accept higher logging costs in exchange for richer investigative detail.

The consistent principle is not to maximize any particular signal, but to choose the least expensive one that can answer the question with the required fidelity.


---

Summary

Logs, metrics, and traces are complementary observability signals, each optimized for a different class of operational question. Logs preserve detailed historical events, metrics summarize system behavior as low-cardinality time series, and traces reconstruct the execution path of individual requests across service boundaries.

The key design decision is to identify the question before selecting the signal. Questions about system health, throughput, latency, and capacity belong to metrics. Questions about what happened during a particular event belong to logs. Questions about where time was spent or why a single request failed belong to traces or, for simpler systems, to logs connected by the correlation ID introduced in Chapter 21.

Metrics derive their efficiency from aggregation, making cardinality discipline essential. Introducing high-cardinality labels such as user IDs or request IDs causes metrics to lose the very properties that distinguish them from logs. Each signal therefore occupies a distinct place in an observability system: none replaces the others, and each is the least expensive tool capable of answering its intended question.

The next chapter builds on this taxonomy by addressing a different decision entirely: not which signal to collect, but which signals justify interrupting a human responder. Alerting turns observability into operational action, determining when an observation is important enough to page someone rather than simply record it.
