Chapter 72 — Distributed Tracing

Purpose

Logs explain what happened inside one component. Metrics summarize the behavior of an entire system. Neither can reliably answer one of the hardest questions in distributed systems:

"Where did this request spend its time?"

A modern request rarely executes inside a single process. It may traverse an API gateway, multiple services, several databases, caches, and asynchronous messaging systems before completing. No individual component observes the entire journey.

Distributed tracing exists to reconstruct that journey.

This chapter explains the trace data model, the context propagation that makes it possible, and the cost trade-offs involved in collecting traces at production scale. It builds on the taxonomy established in Chapter 70 and fulfills the forward references from Chapters 07 and 08, showing how engineers observe system-wide behavior rather than isolated service behavior.

Key Concepts

The central ideas introduced here are:

A trace represents one request's complete execution.

A span represents one unit of work within that request.

Parent-child relationships reconstruct causality across services.

Trace context must be propagated across every service boundary.

Context propagation is an organizational discipline, not merely a programming technique.

Sampling is the primary mechanism for controlling tracing cost.



---

A Trace Reconstructs One Request

What it is

A distributed trace records the complete path taken by a single request as it moves through a distributed system.

Rather than recording isolated events, it preserves the causal relationships between operations.

One incoming request produces one trace.

Why it exists

Distributed systems fragment execution across many independently deployed services.

Each service observes only its own work.

No individual log contains the complete story.

A trace reconstructs that story.

Options

Observe a request using:

logs only

logs connected by correlation IDs

a complete distributed trace


Trade-offs

Logs only

Pros

Simple.

Universally available.


Cons

Manual reconstruction.

Difficult to understand timing.

Poor visibility across many services.



---

Correlation IDs

Pros

Connect related logs.

Low implementation cost.


Cons

Still requires manual investigation.

Timing relationships remain difficult.



---

Distributed tracing

Pros

Complete request timeline.

End-to-end latency visibility.

Cross-service causality.


Cons

Higher collection cost.

More infrastructure.

Requires consistent instrumentation.


When to choose each option

Tracing is appropriate whenever requests routinely cross service boundaries and understanding end-to-end execution matters.

Small monolithic systems often obtain sufficient visibility through logs alone.

Common failure modes

Teams attempt to reconstruct distributed requests entirely from timestamps.

Clock skew, concurrent requests, retries, and asynchronous execution make this increasingly unreliable as systems grow.

Example

A customer checkout request may traverse:

API Gateway

Authentication Service

Cart Service

Inventory Service

Payment Service

Database


A distributed trace reconstructs that entire execution as one coherent timeline rather than six unrelated logs.


---

Spans Form the Building Blocks of a Trace

What it is

A trace consists of spans.

Each span represents one logical unit of work.

Examples include:

processing an HTTP request

executing a database query

calling another service

publishing a message

reading from a cache


Together, the spans form a tree describing the request's execution.

Why it exists

Requests naturally decompose into smaller operations.

Understanding overall latency requires understanding where each operation occurred.

Options

Spans typically represent:

server request handling

client requests

database operations

messaging operations

internal processing


Trade-offs

Large spans provide simple visualizations but hide detail.

Very small spans expose precise execution but increase storage and visualization complexity.

When to choose each option

Instrument meaningful units of work.

Individual language statements or helper functions rarely justify their own spans.

External operations and significant internal phases usually do.

Common failure modes

Over-instrumentation produces traces containing thousands of insignificant spans.

Investigations become harder because important operations disappear within excessive detail.

Example

A payment service might contain:

HTTP request span

fraud check span

payment gateway span

database update span


Each contributes to the overall request latency while remaining part of the same trace.


---

Parent-Child Relationships Capture Causality

What it is

Every span belongs to a trace.

Most spans also have a parent span.

These parent-child relationships form a causal tree describing how work was initiated.

Why it exists

Execution order alone does not explain why work occurred.

The tracing model preserves both timing and causality.

A child span exists because its parent initiated it.

Options

Relationships include:

root span

parent span

child span


The resulting structure is a tree rather than merely a sequence of timestamps.

Trade-offs

Capturing relationships adds metadata to every span.

In return, engineers gain an accurate execution hierarchy instead of inferring relationships from timestamps.

When to choose each option

Every operation initiated by another traced operation should become a child span.

Independent background work generally begins a new trace.

Common failure modes

Treating spans as unrelated timing measurements eliminates the causal information that distinguishes tracing from ordinary logging.

Example

A frontend request initiates an authentication call.

Authentication initiates a database lookup.

The database span is not merely adjacent to the authentication span.

It is its child, preserving the execution hierarchy.


---

Trace Context Must Cross Every Service Boundary

What it is

Tracing depends on propagating a shared context between services.

The correlation ID introduced in Chapter 21 becomes a richer trace context containing at least:

trace ID

span ID

parent span ID


Each service reads this context from incoming requests, creates its own span, and forwards the updated context on every outbound request.

Why it exists

Without shared context, independent services cannot determine that they participated in the same request.

Propagation is what transforms isolated spans into a distributed trace.

Options

A service may:

ignore incoming context

generate new identifiers

propagate existing trace context correctly


Trade-offs

Correct propagation introduces modest implementation complexity.

Ignoring propagation permanently loses end-to-end visibility.

When to choose each option

Every service participating in a traced request should propagate trace context across all supported communication mechanisms.

This includes:

HTTP

gRPC

messaging systems

asynchronous job dispatch


Common failure modes

One service generates a completely new trace identifier instead of continuing the existing trace.

Everything downstream appears unrelated.

The trace is fragmented into multiple independent executions even though the request itself never split.

Another common failure is propagating context only for synchronous requests while forgetting asynchronous messaging.

Example

A request enters an API gateway carrying an existing trace context.

Every downstream service extends that trace by creating new child spans while preserving the original trace identifier.


---

Trace Propagation Is a Cross-Team Discipline

What it is

Distributed tracing succeeds only when every participating service follows the same propagation rules.

One service cannot compensate for another that fails to propagate context.

Why it exists

A distributed trace spans organizational boundaries as much as technical ones.

Every participating team contributes one section of the complete execution path.

Options

Organizations may rely on:

voluntary instrumentation

shared instrumentation libraries

organization-wide standards


Trade-offs

Standardization requires coordination across teams.

Without standardization, traces become inconsistent and fragmented.

When to choose each option

Organizations operating multiple services should establish common propagation standards early.

Treat trace propagation as an interoperability requirement rather than an implementation preference.

Common failure modes

A newly deployed service neglects to forward trace context.

Every downstream dependency disappears from trace visualizations.

The service has not merely reduced its own visibility.

It has broken observability for every service beyond it.

Example

Vendor-neutral instrumentation standards such as OpenTelemetry exist largely to solve this interoperability problem by defining common APIs and context propagation conventions across languages and frameworks.


---

Sampling Controls Tracing Cost

What it is

Tracing every request is often prohibitively expensive.

Sampling determines which traces are retained.

Why it exists

Every collected trace consumes:

CPU

network bandwidth

storage

indexing

query resources


High-throughput systems may process millions of requests per minute.

Retaining every trace is rarely economical.

Options

Common strategies include:

fixed-rate sampling

tail-based sampling

error-biased sampling


Trade-offs

Fixed-rate sampling

Pros

Simple.

Predictable cost.

Easy implementation.


Cons

May discard the request under investigation.

Slow failures can be missed by chance.



---

Tail-based sampling

Pros

Preserves slow requests.

Preserves failed requests.

Higher diagnostic value.


Cons

Requires buffering requests.

More complex infrastructure.

Greater processing overhead.



---

Error-biased sampling

Pros

Captures failures at high probability.

Controls storage costs.


Cons

Successful but unusually slow requests may still be discarded.

More sophisticated policy management.


When to choose each option

Fixed-rate sampling is appropriate when operational cost dominates.

Tail-based or error-focused sampling becomes attractive when rare failures are significantly more valuable than ordinary successful requests.

Common failure modes

Teams adopt extremely low sampling rates without considering investigation needs.

The probability of capturing an unusual production failure becomes so small that tracing infrastructure provides little practical value during incidents.

Example

Many production tracing systems retain a high percentage of failed requests while aggressively sampling successful traffic, balancing investigative usefulness against storage cost.


---

Observing Global Behavior

What it is

Distributed tracing reveals behavior that no individual service can observe independently.

Why it exists

Chapter 08 argued that optimizing one component does not necessarily optimize the entire system.

Tracing provides the observational evidence.

Options

Without tracing, engineers optimize:

individual services

individual databases

individual hosts


With tracing, they optimize complete request paths.

Trade-offs

Local measurements remain simpler and less expensive.

Global measurements reveal interactions that local instrumentation cannot detect.

When to choose each option

When latency or failures emerge from interactions between services rather than behavior within a single component, tracing becomes the appropriate tool.

Common failure modes

Engineers optimize CPU utilization inside one service while requests continue experiencing unacceptable latency because the real bottleneck is queueing in a downstream dependency.

The local optimization succeeds.

The user experience does not improve.

Example

A trace may show:

15 ms in the frontend

20 ms in authentication

40 ms in payment processing

1,800 ms waiting for a downstream inventory service


CPU profiling of the frontend alone would never reveal the actual source of latency.


---

Why Smart Engineers Disagree

Experienced engineers generally agree on the value of distributed tracing. They disagree on how much tracing infrastructure is justified.

One philosophy favors aggressive tracing with rich instrumentation and sophisticated sampling policies. Its advocates argue that difficult production incidents justify the additional infrastructure because traces reveal interactions that logs and metrics cannot.

The opposing philosophy emphasizes simplicity. Organizations operating relatively small systems may conclude that structured logs, correlation IDs, and targeted metrics solve nearly every operational problem at substantially lower cost.

The disagreement is usually proportional to system complexity rather than engineering philosophy. As requests cross more services owned by more teams, the value of reconstructing end-to-end execution increases rapidly. In systems where requests rarely leave a single process, the additional machinery of distributed tracing often provides little benefit.

The consistent principle is that tracing exists to observe system behavior that cannot be reconstructed economically through logs or metrics alone.


---

Summary

Distributed tracing provides end-to-end visibility into individual requests as they traverse distributed systems. A trace represents the complete execution of one request, while spans divide that execution into meaningful units of work connected by parent-child relationships that preserve causality rather than merely chronology.

This model depends on propagating trace context across every service boundary. Extending the correlation ID introduced in Chapter 21 into a complete trace context allows independently deployed services to contribute spans to a single coherent execution history. Because any service that fails to propagate context fragments the trace for every downstream dependency, propagation is an organization-wide interoperability discipline rather than an isolated implementation detail.

Tracing also introduces significant operational cost, making sampling a fundamental design decision. Fixed-rate sampling offers predictable resource consumption, while tail-based and error-biased strategies retain the requests most likely to provide diagnostic value at the expense of additional complexity.

The final chapter of this Part steps back from the mechanics of observation to the operational objective they ultimately serve. Logs, metrics, traces, and alerts all exist to protect measurable service objectives, introducing the concepts of service-level indicators, objectives, and error budgets that define what "healthy" means for a production system.
