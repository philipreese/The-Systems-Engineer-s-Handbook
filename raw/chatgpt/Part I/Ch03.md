Chapter 03 — Coupling and Cohesion

1. Purpose



Most system design arguments collapse because engineers use “coupling” and “cohesion” as vibes instead of measurable properties.

This chapter makes them precise enough to reason about:

coupling: how changes propagate across boundaries

cohesion: how strongly related the responsibilities inside a component actually are


The goal is not to optimize either. It’s to make failure predictable when either one is wrong.

2. Prerequisites



From earlier chapters:

complexity comes from state, control flow, and code volume

systems optimize multiple competing objectives

complexity accumulates through change over time


Assumes familiarity with:

services and modules

APIs (REST/gRPC conceptually)

basic Unix tools and pipelines

3. Key Concepts



Coupling is about dependency relationships between components

Cohesion is about internal consistency within a component

High cohesion reduces local complexity; low coupling reduces global complexity

They are independent axes, not opposites

Most real systems fail due to hidden coupling, not lack of cohesion



---

4. Coupling (Formal Definition)



What it is: Coupling is the degree to which one component’s behavior depends on another component’s state, structure, or timing.

Why it exists: No system is truly isolated. Any useful system interacts with others: databases, services, files, APIs.

But the form of dependency determines how fragile the system becomes.

Options:

Tight coupling (shared state, shared schemas, synchronous dependencies)

Loose coupling (message passing, stable interfaces, isolated state)

Implicit coupling (undocumented contracts, shared assumptions)


Trade-offs:

Tight coupling improves performance and simplicity in small systems but breaks under independent evolution

Loose coupling improves scalability and change tolerance but increases system boundary complexity

Implicit coupling is the worst case: it behaves like tight coupling but looks like loose coupling


When to choose each option:

Tight: in-process modules, SQLite internal components, kernel subsystems

Loose: microservices, Unix pipelines, event-driven architectures

Implicit: usually accidental; only acceptable temporarily during system evolution


Common failure modes:

Microservices sharing PostgreSQL schema tables directly (breaking service boundaries)

“Independent” services coordinated via hidden timing assumptions

Refactoring one service breaks others without any interface change


Example: Unix pipes are loosely coupled because they communicate through a minimal contract (byte stream). Each tool (grep, awk, sort) can evolve independently as long as it preserves that contract.

5. Afferent vs Efferent Coupling



What it is:

Afferent coupling: number of components that depend on this component

Efferent coupling: number of components this component depends on


Why it exists: It distinguishes between “impact surface” (what breaks if I change this) and “dependency burden” (what I break if something else changes).

Options:

High afferent / low efferent (stable core systems)

Low afferent / high efferent (application glue code)

High / high (critical but fragile hubs)

Low / low (isolated utilities)


Trade-offs:

High afferent systems are stable but hard to change safely

High efferent systems are flexible but fragile

Low/low systems are easy to reason about but often underused or redundant

High/high systems become systemic risk multipliers


When to choose each option:

High afferent: databases like PostgreSQL internals, core libraries (libc)

High efferent: orchestration layers, application services

Low/low: CLI utilities, pure functions, stateless tools

High/high: often accidental architecture smells, not intentional design


Common failure modes:

Core service changes breaking many downstream consumers unexpectedly

Application services depending on too many external APIs (snowball failure risk)

“Orchestration services” becoming dependency hubs no one can safely modify


Example: PostgreSQL has extremely high afferent coupling (many systems depend on it) but carefully controlled efferent coupling, which is why it functions as a stable system core.

6. Connascence (Strength of Coupling)



What it is: Connascence is a taxonomy for describing how strongly coupled two components are based on the nature of their dependency.

Lower connascence = easier to change independently.

Key forms (ordered roughly from weak to strong):

Name: components depend on shared identifiers (e.g., function names, API routes)

Type: components depend on shared data types (e.g., JSON schema, protobuf contract)

Value: components depend on specific values (magic numbers, sentinel strings)

Execution order: components depend on sequencing (A must happen before B)

Timing: components depend on temporal behavior (timeouts, race conditions, async ordering)


Why it exists: Not all coupling is equal. Two systems can both “depend on each other,” but one breaks on rename, while another breaks only under race conditions.

Options:

Weak connascence (name/type)

Moderate connascence (value/execution order)

Strong connascence (timing/stateful coupling)


Trade-offs:

Weak connascence is easier to refactor but requires stricter interface discipline

Strong connascence improves performance or expressiveness but is fragile

Mixed connascence is inevitable in real systems but must be bounded


When to choose each option:

Name/type: API design, service boundaries, database schemas

Value: configuration systems, feature flags, protocol constants

Execution order: pipelines, transaction systems, build systems

Timing: distributed systems, concurrent systems, Kubernetes controllers


Common failure modes:

Hardcoded status values leaking across service boundaries

Race conditions in distributed job processing (Kafka consumers, Kubernetes controllers)

Hidden ordering assumptions in async workflows


Example: In Kubernetes, controllers rely heavily on execution-order and timing connascence: desired state reconciliation depends on eventual consistency and timing of control loops. This is why debugging “stuck pods” often involves timing rather than logic errors.

7. Measuring Coupling in Practice



What it is: Coupling is not just conceptual — it can be approximated by observing dependency graphs, change propagation, and runtime failure correlation.

Why it exists: Without measurement, coupling becomes subjective (“this feels tightly coupled”). In real systems, this leads to false refactors or ignored architecture debt.

Options:

Static analysis (dependency graphs, import graphs, build graphs)

Change-based analysis (how often changes co-occur across modules)

Runtime correlation (failure blast radius analysis)

Informal reasoning (architect intuition)


Trade-offs:

Static analysis is cheap but misses runtime coupling

Change-based analysis reflects reality but requires history

Runtime analysis is accurate but reactive (post-incident)

Informal reasoning is fast but unreliable at scale


When to choose each option:

Static: early-stage systems, code review workflows

Change-based: mature systems with git history (e.g., large monorepos)

Runtime: production systems with observability stacks (Kubernetes, distributed services)

Informal: small systems or initial design phases


Common failure modes:

Modules that look decoupled in code but always change together in practice

“Independent services” failing simultaneously due to shared database schema changes

Overestimating decoupling because coupling is mediated through configuration or timing


Example: Large Git-based systems (like Linux kernel or Chromium) often reveal coupling through co-change frequency: files that frequently change together are effectively coupled even if no explicit dependency exists.

8. Cohesion (Internal Consistency)



What it is: Cohesion measures how closely related the responsibilities inside a single component are.

Why it exists: Low cohesion means a component is doing multiple unrelated things, increasing internal state complexity and reducing predictability.

Options:

High cohesion (single responsibility, tightly aligned internal logic)

Medium cohesion (related but distinct responsibilities)

Low cohesion (unrelated responsibilities grouped together)


Trade-offs:

High cohesion improves understandability and change isolation

Medium cohesion is practical for real systems with shared state or domain overlap

Low cohesion increases accidental coupling inside the component itself


When to choose each option:

High: libraries, core domain modules, SQLite subsystems

Medium: application services with related business logic

Low: usually accidental, legacy “god modules”


Common failure modes:

“Utility” modules accumulating unrelated functions over time

Services that mix authentication, business logic, and persistence logic

Kubernetes manifests or CI pipelines that encode unrelated responsibilities


Example: SQLite maintains high cohesion by tightly integrating storage, indexing, and query execution into a single coherent system boundary. It avoids splitting responsibilities across separate processes or services.


---

9. Microservices Coupling Failure (Real System Pattern)



What it is: Microservices are often designed for loose coupling but become tightly coupled through shared databases or implicit contracts.

Why it exists: Distributed systems still need shared truth. Developers often fall back to:

shared PostgreSQL schemas

undocumented REST response structures

implicit event ordering assumptions


Options:

Shared database (tight coupling, fast iteration early)

Strict service boundaries (loose coupling, harder coordination)

Event-driven contracts (moderate coupling via messaging systems)


Trade-offs:

Shared DB is simple but destroys independence

Strict boundaries preserve autonomy but increase coordination overhead

Event-driven systems reduce direct coupling but introduce timing and ordering complexity


When to choose each option:

Shared DB: early-stage systems, internal tools

Strict boundaries: large-scale systems with independent teams

Event-driven: systems requiring scalability and decoupled evolution (Kafka-based architectures)


Common failure modes:

Schema changes breaking multiple services simultaneously

Event consumers depending on undocumented event ordering

“Independent services” coordinated through shared Redis keys or cron jobs


Example: A common failure pattern in microservices architectures is a “decoupled” service architecture where all services still read/write the same PostgreSQL database. Coupling has not been removed — only relocated to the schema layer.


---

10. Why Smart Engineers Disagree



Some engineers optimize for minimizing coupling aggressively (favoring strict boundaries, even at cost of performance and complexity).

Others prioritize cohesion first (favoring larger, more integrated modules that reduce internal fragmentation).

Others accept coupling but focus on making it explicit and observable rather than eliminating it.

The disagreement usually comes down to:

whether system boundaries should reflect organizational boundaries or runtime dependencies

whether change isolation is more important than performance

whether complexity should be reduced or centralized


Unix represents one extreme: extremely loose coupling via pipes. Monolithic systems like SQLite represent the other: tight internal coupling but extremely high cohesion.

Microservices often sit incorrectly in between, aiming for loose coupling while accidentally retaining hidden shared-state coupling.
