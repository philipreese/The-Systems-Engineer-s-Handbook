Chapter 06 — Cost Models and Mechanical Sympathy

1. Purpose



Most architecture decisions are made as if computation is abstract: “fast,” “slow,” “scales,” “doesn’t scale.”

That’s not enough. Systems behave the way they do because hardware costs are extremely uneven and often non-intuitive.

This chapter grounds design decisions in actual cost structures of computation so that choices like “add a cache,” “batch requests,” or “use a database index” are not aesthetic decisions but physical ones.

The goal is not to turn engineers into hardware experts. It’s to make them stop accidentally designing against physics.

3. Key Concepts



Computation cost is dominated by memory hierarchy, not CPU instructions

Latency is non-uniform and spans multiple orders of magnitude

Most “algorithmic” inefficiencies are irrelevant compared to cache misses or network calls

Mechanical sympathy is understanding how software maps onto hardware behavior

System design is often latency budgeting, not code optimization

4. The Memory and Latency Hierarchy (What Actually Costs Time)



These are the numbers that shape system architecture more than any algorithmic complexity analysis:

L1 cache: ~1 ns

L2 cache: ~4 ns

L3 cache: ~40 ns

Main memory (RAM): ~100 ns

SSD random read: ~100 μs

Network round trip (same datacenter): ~500 μs – 1 ms

Cross-region network call: ~100 ms


Why this matters: Each step is not a small slowdown — it is an order-of-magnitude jump. A cache miss is not “a bit slower.” It is often 100x–100,000x slower depending on where you fall.

Options:

Cache-resident computation (L1/L2 bound)

Memory-resident computation (RAM bound)

Disk-bound computation (SSD/HDD bound)

Network-bound computation (distributed systems)


Trade-offs:

Cache-resident systems are extremely fast but constrained by memory layout and working set size

Memory-resident systems are flexible but slower and sensitive to allocation patterns

Disk-bound systems scale beyond memory but introduce high latency variance

Network-bound systems scale horizontally but are dominated by latency and coordination costs


When to choose each option:

Cache-resident: hot loops, real-time systems, performance-critical components

Memory-resident: most application services, PostgreSQL working sets

Disk-bound: databases (PostgreSQL, Kafka logs, object storage)

Network-bound: microservices, distributed systems, cloud-native architectures


Common failure modes:

Treating SSD access as “fast enough” without considering random vs sequential patterns

Ignoring network round-trip cost in microservice design

Writing cache-unfriendly data structures that destroy CPU locality

Assuming CPU optimization matters before memory layout is correct


Example: A single L3 cache miss (~40 ns) is already 40x slower than L1 access. A full SSD read (~100 μs) is ~2,500x slower again. A remote network call (~1 ms) is ~10,000x slower than memory access. This is why “just add a service call” is not a neutral architectural decision.

5. Mechanical Sympathy (Martin Thompson)



What it is: Mechanical sympathy is the idea that software performs better when it respects how hardware actually executes instructions, moves data, and manages memory.

Why it exists: CPUs are not abstract processors. They are deeply optimized for:

sequential memory access

cache locality

branch prediction

pipeline execution


Violating these assumptions creates orders-of-magnitude slowdowns that dwarf algorithmic complexity concerns.

Options:

High sympathy design (cache-aware, memory-local, sequential access patterns)

Neutral design (standard application code, moderate awareness)

Low sympathy design (random memory access, heavy indirection, uncontrolled allocations)


Trade-offs:

High sympathy yields extreme performance but increases code complexity and reduces abstraction freedom

Neutral design is easiest to maintain but leaves performance on the table

Low sympathy design is often “clean-looking” but catastrophically inefficient at scale


When to choose each option:

High sympathy: databases (PostgreSQL internals), Kafka, game engines, trading systems

Neutral: most backend services, web applications

Low sympathy: prototypes, low-scale systems where performance is irrelevant


Common failure modes:

Object-oriented designs causing pointer chasing and cache misses

Excessive heap allocations in hot paths

Random memory access patterns in large datasets

Assuming CPU speed dominates system behavior


Example: Redis is fast largely because it avoids almost all mechanical inefficiency: it operates in-memory, uses simple data structures, and keeps access patterns cache-friendly. Its performance is not “algorithmic superiority” — it is hardware alignment.

6. Why Cache Misses Dominate Algorithmic Complexity



What it is: A theoretically efficient algorithm can be slower than a theoretically worse one if it causes poor cache behavior or memory access patterns.

Why it exists: CPU speed improvements have far outpaced memory latency improvements. The gap between computation and memory access is now the dominant bottleneck.

Options:

Cache-friendly algorithm (linear scans, contiguous memory)

Cache-ignorant algorithm (pointer-heavy structures, random access)

Hybrid approaches (tuned data structures, batching, prefetching)


Trade-offs:

Cache-friendly designs often require restructuring data rather than logic

Cache-ignorant designs are easier to express but perform poorly at scale

Hybrid systems require careful profiling and understanding of workloads


When to choose each option:

Cache-friendly: databases, search systems, real-time processing

Cache-ignorant: small datasets, prototype code, non-critical systems

Hybrid: most production systems after performance issues emerge


Common failure modes:

Linked-list-heavy designs that degrade under large datasets

Hash table overuse leading to unpredictable memory access

Ignoring CPU cache alignment in high-frequency operations


Example: A simple O(n) array scan often outperforms a more “efficient” O(log n) tree traversal when the array is cache-contiguous and the tree causes multiple cache misses per lookup.

7. Redis: Fast Because It Avoids the Slow Path Entirely



What it is: Redis is an in-memory key-value store designed to avoid disk and network latency on read paths as much as possible.

Why it works: It keeps working sets in memory, uses simple data structures, and avoids unnecessary abstraction layers in its core execution path.

What it avoids:

disk I/O on hot paths

complex query planning

cross-node coordination for basic reads


Trade-offs:

Extremely fast reads/writes for in-memory data

Limited by RAM capacity

Persistence is secondary and introduces complexity (AOF/RDB trade-offs)

Risk of data loss if not configured properly


Failure modes:

Treating Redis as a primary database without understanding persistence guarantees

Memory exhaustion under unbounded key growth

Latency spikes when persistence or eviction mechanisms activate


Why it matters: Redis demonstrates that performance is often about removing costs entirely, not optimizing them.


---

8. Kafka: Fast Through Sequential I/O, Not Avoiding Disk



What it is: Kafka is a distributed log system that achieves high throughput by relying on sequential disk writes rather than random access.

Why it works: Modern disks (especially SSDs) are dramatically faster at sequential writes than random reads/writes. Kafka exploits this by appending to logs instead of updating in place.

What it avoids:

random disk seeks

in-place mutation of stored data

complex read-modify-write cycles


Trade-offs:

Very high throughput

Strong durability guarantees

Increased storage usage due to append-only logs

Need for compaction and retention management


Failure modes:

Underestimating disk space growth due to log retention

Misunderstanding consumer lag as system failure

Assuming low latency guarantees (Kafka optimizes throughput, not latency)


Why it matters: Kafka is fast because it aligns with disk physics, not because it avoids disk usage.

9. PostgreSQL MVCC (Why It Trades Write Complexity for Read Isolation)



What it is: PostgreSQL uses Multi-Version Concurrency Control (MVCC) to allow readers and writers to operate without blocking each other.

Why it exists: Lock-based concurrency control creates contention under high write/read concurrency. MVCC avoids this by maintaining multiple row versions.

What it avoids:

reader-writer locking contention

blocking reads during writes

global locks on tables


What it introduces:

storage overhead (multiple versions of rows)

vacuuming complexity (cleanup of old versions)

write amplification under heavy update workloads


Trade-offs:

High read concurrency performance

Predictable transaction isolation

Increased storage and maintenance overhead

Complexity in understanding visibility rules


Failure modes:

table bloat due to insufficient vacuuming

performance degradation under high update workloads

confusion about why “deleted rows” still exist physically


Why it matters: PostgreSQL chooses a design that optimizes for concurrency at the cost of background complexity. This is a mechanical sympathy decision at the database level.

10. Why Smart Engineers Disagree



Some engineers prioritize theoretical algorithmic efficiency and clean abstractions, expecting the hardware to be a secondary concern.

Others design explicitly around memory and IO behavior, treating hardware constraints as primary design inputs.

The disagreement comes down to:

whether performance is primarily a software problem or a hardware alignment problem

whether abstractions should hide cost differences or expose them

whether correctness or locality is the dominant constraint


Redis, Kafka, and PostgreSQL represent three different points in this space:

Redis: eliminates cost paths entirely (memory-first)

Kafka: exploits hardware behavior (sequential disk I/O)

PostgreSQL: balances correctness and concurrency with hardware-aware trade-offs


None of them are “purely optimized.” They are aligned with different physical constraints.
