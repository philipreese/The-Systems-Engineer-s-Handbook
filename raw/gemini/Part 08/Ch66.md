Chapter 66 — Keeping Documentation Honest
## 1. Purpose
This chapter resolves the explicit forward reference from Chapter 49: the execution and management of the documentation maintenance burden. It scales the inline-comment rot argument established in Chapter 30 to the scope of an entire engineering documentation corpus.
While a stale code comment is a localized failure sitting directly adjacent to the logic it misrepresents, a stale README or architecture portal can live several directory layers or web clicks removed from production reality. Without a structural mechanism binding the documentation to the execution layer, there is nothing forcing an engineer changing the system's behavior to notice that the prose has diverged. This chapter applies strictly to documentation designed to track current reality; Chapter 65 has already isolated epoch snapshots (specs and ADRs) from this operational failure mode.
## 2. Prerequisites
 * **Chapter 30 (Inline Comments):** The mechanics of localized comment decay and code-level documentation boundaries.
 * **Chapter 49 (Engineering Process Audits):** The structural foundations of audit discipline and administrative pruning.
 * **Chapter 65 (README vs. Spec vs. ADR vs. Inline Comment):** The categorization of documents into living trackers versus frozen points in time.
## 3. Key Concepts
 * **Spatial Drift Decay:** The structural reality that the physical or logical distance between source code and its descriptive documentation is directly proportional to the speed at which that documentation rots.
 * **Executable Documentation:** A class of documentation artifacts that embed verifiable code blocks or syntax constraints that are actively compiled and verified by a continuous integration pipeline.
 * **The Write-Only Tracker:** A failure state where a documentation corpus expands unchecked due to low insertion costs and high deletion anxieties, resulting in an untrusted ecosystem that engineers entirely stop consulting.
## [Principle: The Proximity Co-Location Imperative (Docs-as-Code)]
### What it is:
The mechanical mandate to store living reality documentation within the identical repository, file format, and pull-request review pipeline as the functional source code it covers.
### Why it exists:
Decoupling living documentation from the codebase ensures its failure. When documentation is exiled to an external system (such as an isolated collaborative wiki), it escapes the developer’s immediate execution context. By treating documentation exactly like code—subject to branches, code reviews, and versioning—a behavioral change and its corresponding documentation change are granted a structural opportunity to land within the exact same atomic commit.
### Options:
#### 1. In-Repository Reviewed Text (Docs-as-Code)
Maintaining evergreen documentation exclusively as Markdown or AsciiDoc files directly within the application's source repository, processed through standard version-control review gates.
#### 2. External Collaborative Portals (Wikis)
Utilizing centralized, rich-text platforms completely separate from the source tree to compile and modify systems documentation.
### Trade-offs:
#### In-Repository Reviewed Text
 * **Pros:** Eradicates cross-system visibility failures. Changes to code pathways can be structurally blocked or flagged during peer review if the adjacent markdown documentation is left unmodified. All prose inherits the branching and versioning timeline of the system itself.
 * **Cons:** Increases friction for non-engineering stakeholders who lack Git proficiency or repository access. Lacks dynamic visual editors out-of-the-box, forcing engineers to manage text via basic text editors.
#### External Collaborative Portals
 * **Pros:** Provides zero-friction editing for product managers, technical writers, and business operations. Supports immediate web-based modifications without forcing a full code-review deployment loop.
 * **Cons:** Guarantees severe spatial drift. Because the editing loop is completely abstracted from the repository's development lifecycle, documentation changes rely entirely on human memory, leading to immediate synchronization failures during rapid development.
### When to choose each option:
 * **Choose In-Repository Reviewed Text** universally for all technical entrance vectors, configuration maps, local orchestration guides, and on-call runbooks (Chapter 68). If a document describes the current operational reality of the code, it must live inside the repo.
 * **Choose External Collaborative Portals** strictly for non-executable business contexts, compliance policies, high-level project management timelines, and product-org hiring charts that carry no relationship to application execution boundaries.
### Common failure modes:
 * **The Ghost PR:** An engineering team transitions their system layout from an in-repo folder structure to an external infrastructure provider. They meticulously update their code and configurations within an atomic pull request. However, because the repository's README.md is not verified by an automated check, they forget to touch the onboarding steps. The PR lands, and the next developer to clone the repo wastes their entire first morning executing setup steps that match an architecture that no longer exists.
### Example:
Both GitHub and GitLab maintain their extensive engineering, platform, and public product documentation using in-repository Markdown files. Changes to infrastructure or software capabilities require parallel edits to the core markdown directories within the same pull request, ensuring that the documentation's version history perfectly mirrors the software's operational state.
## [Decision: Mechanical Verification vs. Human Audit Pruning]
### What it is:
The architectural division between what can be mechanically validated via continuous integration and what must be verified through structured, human-driven lifecycle auditing.
### Why it exists:
Human discipline does not scale across massive engineering systems. Relying on an engineer's goodwill to verify that every sentence in a document remains factual is a design flaw. Systems engineers must offload everything that can be evaluated as a binary true/false to the CI pipeline, freeing up finite human cognitive bandwidth to focus strictly on conceptual truth during explicit, scheduled audit gates.
### Options:
#### 1. Mechanically Enforced Constraints
Deploying automated tooling within the CI pipeline—such as link checkers, docstring linters, and compiled code-block executors—to instantly fail builds when documentation errors are discovered.
#### 2. Contextual Auditing and Pruning
Establishing a rolling human process (leveraging the audit frameworks established in Chapter 49) to read documentation for conceptual accuracy and aggressively delete obsolete pages.
### Trade-offs:
#### Mechanically Enforced Constraints
 * **Pros:** Completely eliminates a specific class of rot—such as dead URLs, broken internal file references, or invalid code samples—without consuming human engineering cycles.
 * **Cons:** Structurally blind to conceptual accuracy. An automated tool can verify that a code snippet inside a markdown block compiles perfectly, but it cannot detect if the surrounding text describes a deployment philosophy that the organization abandoned six months ago.
#### Contextual Auditing and Pruning
 * **Pros:** Ensures that the documentation remains highly accurate, focused on the current recommended design patterns, and clear to the target audience.
 * **Cons:** Imposes a constant administrative tax on senior staff. Requires hard process discipline to execute regularly, and depends heavily on an engineering culture that rewards document maintenance and pruning.
### When to choose each option:
 * **Choose Mechanically Enforced Constraints** for any documentation containing concrete code syntax, command-line usage examples, internal repository hyperlinks, or public API wire-style schemas (Chapter 67).
 * **Choose Contextual Auditing and Pruning** as the explicit governance mechanism for all abstract engineering guides, design patterns, and system architectures. A document that cannot be mechanically tested must be given a strict expiration date, forcing a human review or immediate deletion.
### Common failure modes:
 * **The Write-Only Tracker (The Confluence Graveyard):** An engineering group treats documentation purely as an additive asset. Because writing new text is cheap and deleting old text feels risky, the corpus grows exponentially. Over time, outdated setup guides, deprecated system configurations, and active production runbooks sit side-by-side with no clear distinction. Because developers cannot easily tell which pages are accurate without checking the code directly, the entire corpus loses systemic trust, causing engineers to abandon the documentation entirely.
```
[Low Insertion Cost] ──> [Continuous Document Additions] 
                                    │
[Corpus Completely Untrusted] <── [Engineers Stop Consulting] <── [Prose Contradicts Code]

```
### Example:
The Rust ecosystem leverages **Executable Documentation** natively via rustdoc. When an engineer writes a code sample inside a documentation block within a Rust file, running cargo test explicitly extracts that code sample, compiles it, and executes it as a standard integration test. If an engineer changes an underlying library function signature but forgets to update the documentation example, the build breaks immediately in CI, transforming documentation drift into a hard block.
## Why Smart Engineers Disagree: The Threshold of Deletion
The primary friction regarding documentation maintenance centers on how to handle text that has fallen out of date but contains historical value:
### The Scorched-Earth Faction
This group argues that a page containing even a small amount of stale or inaccurate operational data must be instantly deleted from the living corpus. They operate on the principle that the presence of *any* incorrect data destroys the credibility of the entire platform. If a team lacks the immediate bandwidth to update a runbook or an architecture overview to match current reality, the file must be deleted rather than allowed to linger. They believe that no documentation is safer than wrong documentation, as no documentation forces a developer to safely read the source code directly.
### The Preservationist Faction
This group argues that deleting a document wholesale because a fraction of it has drifted is a catastrophic loss of institutional knowledge. They assert that even an outdated architecture document provides invaluable context regarding the system's structural shape, original bottlenecks, and domain constraints. Instead of deleting the page, they advocate for marking it with a prominent visual warning banner ([STALE / UNVERIFIED]) or moving it to an /archive directory, preserving the archeological context for future engineers.
### The Reality
This disagreement stems from a failure to separate the taxonomy tiers established in Chapter 65.
 * The **Scorched-Earth Faction** is correct when applied to **Living Reality Trackers** (onboarding instructions, production runbooks, and current API maps). An incorrect runbook leads directly to extended outages during an incident; archiving an inaccurate deployment step helps no one. It must be fixed or deleted immediately.
 * The **Preservationist Faction** is correct when applied to **Epoch Snapshots** (design specs and ADRs). These documents were never intended to track living reality, and deleting them destroys the exact trail needed to understand how the system evolved.
The systems engineer enforces this boundary by executing immediate deletion for stale reality trackers, while rigidly locking and preserving epoch snapshots.
