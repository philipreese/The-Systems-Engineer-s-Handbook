Chapter 64 — What to Document vs. What to Leave to the Code
## 1. Purpose
This chapter opens Part VIII by extending the WHY-vs-WHAT test introduced in Chapter 30 from the level of individual source files to the architecture of the entire system. While Chapter 30 established the boundaries for inline comments, this chapter addresses the prior question: whether external documentation is warranted at all before deciding which artifact type (the focus of Chapter 65) it should occupy.
Documentation is not a free good. The core complexity argument from Chapter 02 and the process constraint of Principle 6 apply directly here: every page of technical prose written is a page that must be parsed, trusted, maintained, and continuously corrected against production reality. This chapter provides the structural filters senior engineers use to determine when an information gap should be resolved by refactoring the code itself versus when it demands external documentation.
## 2. Prerequisites
 * **Chapter 02 (Complexity Is the Enemy):** Understanding the maintenance and cognitive liabilities of accidental complexity.
 * **Principle 6:** The operational rule that a process or artifact must only be introduced when it provides more systemic value than its cost of maintenance.
 * **Chapter 27 & 28 (Code Organization & Naming):** Mastery of structural and naming conventions required to maximize self-documenting code.
 * **Chapter 30 (Inline Comments):** The foundational WHY-vs-WHAT boundary test for inline documentation.
## 3. Key Concepts
 * **The Documentation Tax:** The ongoing engineering overhead of ownership. Documentation that is unmaintained becomes worse than missing; it becomes actively misleading, creating a split-brain state between prose and production reality.
 * **The Inexpressibility Filter:** The structural test used to verify if a system attribute can be realized directly in code syntax, type systems, or architecture before defaulting to human language.
 * **The Proxy-Metric Illusion:** The failure mode wherein tracking documentation volume or completion percentages creates an illusion of system clarity while driving up noise and stale debt.
## [Principle: The Inexpressibility Filter]
### What it is:
The strict architectural heuristic of choosing whether to fix an information gap by modifying the codebase (refactoring structures, types, or names) versus generating external prose.
### Why it exists:
Engineers frequently default to writing external documentation when a system becomes opaque. This shifts structural debt from the compiler to uncompiled text files. Because code executes and documentation drifts, external prose must be reserved exclusively for context that the code is mathematically or structurally incapable of expressing on its own.
### Options:
#### 1. Code-Level Realization
Resolving clarity gaps within the system design itself using explicit module boundaries (Chapter 27), intentional naming (Chapter 28), strong typing, or localized "WHY" inline comments (Chapter 30).
#### 2. External Documentation
Writing an external text artifact (such as a README overview, an Architecture Decision Record, or an operational runbook) because the context spans multiple systems, organizations, or distinct points in time.
### Trade-offs:
#### Code-Level Realization
 * **Pros:** Keeps context perfectly localized to execution logic. Changes to the code automatically drag the developer's eyes across the local context, minimizing drift.
 * **Cons:** Incapable of capturing cross-cutting, systemic realities. It cannot express the historical context of rejected options, macro organizational onboarding routes, or high-stress operational mitigations required during an infrastructure outage.
#### External Documentation
 * **Pros:** Successfully captures non-local information boundaries—including multi-service orchestration graphs, temporal business constraints, and cross-team dependencies.
 * **Cons:** Introduces an immediate maintenance liability. It lives outside the continuous integration pipeline, cannot be verified by compilers or automated test suites, and fails silently when production behavior changes.
### When to choose each option:
 * **Choose Code-Level Realization** when the confusion is contained within a specific subsystem, when an engineering team can clear up logic by renaming methods or variables, or when a localized exception can be cleanly explained by a localized inline docstring.
 * **Choose External Documentation** strictly when the information has nowhere else to live because it covers boundaries code cannot express:
   * *Onboarding Context:* High-level conceptual maps a newcomer requires before they can productively read the codebase (the README’s role, expanded in Chapter 65).
   * *Temporal Architecture Decisions:* Permanent records of a macro design trade-off and its rejected alternatives (the ADR’s role, Chapter 45).
   * *Operational Runbooks:* Cross-cutting mitigation workflows that only matter when a system is actively failing in production (Chapter 68).
### Common failure modes:
 * **The Textual Patch:** An engineer encounters a convoluted, highly coupled module that violates the boundary rules of Chapter 03. Instead of refactoring the accidental complexity away, they write a sprawling external wiki page detailing how to navigate the brittle architecture. The external document masks the underlying design flaw and goes stale within months, leaving future engineers with both broken code and incorrect documentation.
### Example:
Google's internal engineering practices documentation enforces an explicit boundary on this front. They mandate that code implementations must remain as self-documenting as possible. Code comments and external documents are explicitly restricted from repeating what the code *does*. Instead, they are reserved for details the code *cannot say*—such as security edge cases, specific hardware constraints (Chapter 06), or historical design trade-offs that explain why an obviously superior algorithm was rejected during code review.
## [Decision: Rejecting Proxy-Metric Governance]
### What it is:
Deciding how to govern and evaluate documentation quality within an organization—specifically choosing whether to enforce automated coverage metrics versus treating documentation as a high-scrutiny, low-volume engineering asset.
### Why it exists:
Engineering organizations often attempt to manage technical debt by introducing automated linting rules that mandate documentation artifacts. This violates Principle 9 by treating a proxy metric (lines of text written) as a substitute for systemic clarity.
```
[Mandated Coverage Rule] ──> [Automated CI Gate] ──> [Tautological Prose/Noise]
                                                             │
[Systemic Clarity Collapses] <── [Drift & Cognitive Tax] <───┘

```
### Options:
#### 1. Metric-Driven Coverage Enforcements
Utilizing CI linting tools to require public docstrings, comments, or standardized markdown templates for a strict percentage of files or methods.
#### 2. High-Scrutiny Filtering (Docs-as-Code Quality Bars)
Treating documentation additions with the same peer-review rigor as production code, forcing authors to justify the existence of every external page against its long-term maintenance cost.
### Trade-offs:
#### Metric-Driven Coverage Enforcements
 * **Pros:** Guarantees that documentation artifacts technically exist across the repository. Satisfies compliance checks, external regulatory audits, and automated documentation generation tooling easily.
 * **Cons:** Drives cargo-cult compliance. Engineers write low-signal, repetitive phrases to bypass the CI blocker, which increases the repository's noise-to-signal ratio and forces future maintainers to read text that transfers zero actual insight.
#### High-Scrutiny Filtering
 * **Pros:** Minimizes documentation volume and external noise. Ensures that the prose that does survive is high-signal, high-trust, and focused on essential system constraints.
 * **Cons:** Provides no automated indicator of system health or understandability to external stakeholders. It relies entirely on the discipline and experience of senior engineers during the code review process.
### When to choose each option:
 * **Choose Metric-Driven Coverage Enforcements** exclusively when shipping public, multi-tenant SDKs or external web APIs (Chapter 67) where automated documentation generation frameworks require placeholder docstrings to render reference manuals or provide IDE auto-complete tooltips.
 * **Choose High-Scrutiny Filtering** for all internal platforms, distributed microservices, and infrastructure codebases where developer velocity is bound by the reliability—rather than the sheer volume—of technical documentation.
### Common failure modes:
 * **The Tautological Linter Stub:** To satisfy a strict CI rule mandating docstrings for every public function, an engineer generates descriptions that add no information beyond the function's name and signature.
| Code Signature | Tautological Docstring Stub (High Noise) |
|---|---|
| func FetchUserByID(id string) (*User, error) | // FetchUserByID fetches a user by their ID. |
| func IsCacheValid(ts int64) bool | // IsCacheValid checks if the cache is valid based on timestamp. |
This satisfies the proxy metric but wastes the reader's cognitive cycles, consumes lines of code, and introduces text that can drift if the function is later refactored to fetch from a different identifier.
### Example:
The "Docs as Code" movement (formalized by communities like *Write the Docs*) treats documentation files with the exact same workflow as production logic. Rather than automating compliance through unthinking linting rules, documentation additions are routed through pull requests where reviewers evaluate them against the maintenance burden. If a markdown file or inline docstring duplicates information that can be readily extracted by looking at a clean type definition or a well-named module, the reviewer explicitly rejects the documentation change, forcing the author to keep the asset footprint minimal.
## Why Smart Engineers Disagree: The Illusion of "Self-Documenting Code"
The structural boundary between what belongs in code versus external prose is a point of continuous friction between senior architects, typically dividing along two philosophical lines:
### The Code Purists
This cohort argues that almost all external documentation outside of bare-minimum architecture overviews is an architectural smell. They believe that if a system requires a document to explain how it works, the design has failed Chapter 02's complexity test. If a piece of logic is confusing, refactor the names (Chapter 28), decouple the modules (Chapter 03), or add a precise, one-line inline comment (Chapter 30). They view prose as an un-compiled, un-tested liability that begins rotting the millisecond a pull request is merged.
### The Context Pragmatists
This cohort argues that the ideal of entirely "self-documenting code" is a myth that ignores human cognitive constraints. They assert that even perfectly clean code can only show *what* the system is doing and *how* it is doing it. Code is structurally blind to what is *absent*. It cannot show:
 * Why an obvious architectural approach was discarded.
 * What real-world hardware latency constraints (Chapter 06) forced a counter-intuitive optimization.
 * How a business pivot or team restructuring three years ago left an imprint on the database schema.
They advocate for comprehensive external prose to minimize the day-one cognitive friction for engineers entering the system.
### The Reality
This disagreement is an optimization mismatch based on organizational scale and the system's rate of evolution.
 * The **Code Purists** are optimizing for a compact, high-velocity codebase maintained by a stable team with deep, continuous contextual alignment. In this environment, the maintenance tax of external prose exceeds its value.
 * The **Context Pragmatists** are optimizing for large engineering organizations with high developer turnover, distributed service ownership, or legacy lifespans. Here, the time spent diving into code to reconstruct historical design intent represents a massive bottleneck to cross-team execution.
The pragmatic systems engineer balances this tension by drawing a hard line: **Implementation details belong exclusively in the code; system intent, historical trade-offs, and boundary constraints belong in external documentation.**
