Chapter 65 — README vs. Spec vs. ADR vs. Inline Comment
## 1. Purpose
This chapter resolves the technical documentation taxonomy forward-referenced across this handbook. Chapter 30 deferred the precise inline-comment boundary to this section; Chapters 45 and 46 similarly deferred the structural taxonomy of design proposals and architectural logs.
By evaluating our four core documentation artifacts side by side, this chapter establishes a strict selection test based on two clear axes: **who is the intended reader**, and **whether the artifact must track living reality or remain an immutable snapshot of a single point in time**.

## 3. Key Concepts
 * **The Onboarding Gateway:** The specialized, high-trust documentation tier designed exclusively to orient a newcomer entering a repository for the first time.
 * **The Living Reality Constraint:** The property of an artifact that requires continuous synchronization with production code. If a living reality tracker falls out of sync, its value drops to zero and it becomes an active systemic liability.
 * **The Epoch Snapshot:** The property of an artifact that acts as a historical record of a past moment. Epoch snapshots are intentionally allowed to diverge from modern production reality without being considered "wrong."
## [Principle: The Temporal Lifespan Taxonomy]
### What it is:
The classification of technical documentation into distinct categories based on mutability constraints: **Living Realities** (which must continuously mirror the active codebase) and **Epoch Snapshots** (which preserve a fixed historical state).
### Why it exists:
Conflating living reference documentation with historical snapshots is the single most common taxonomy failure in systems engineering. When an engineer treats an old design specification as an authoritative runtime map, they construct new features on obsolete assumptions. Conversely, if an engineer attempts to continuously update a historical decision record to reflect present-day schema modifications, they corrupt the immutable audit log required to understand *why* the original architecture was chosen.
### Options:
#### 1. Living Reality Trackers
Artifacts such as the repository README.md or live API schemas (expanded in Chapter 67) that map the current operational state of the codebase.
#### 2. Epoch Snapshot Records
Artifacts such as a design Specification (spec) or an Architecture Decision Record (ADR) that record a fixed moment in the system's chronological evolution.
### Trade-offs:
#### Living Reality Trackers
 * **Pros:** Provides an authoritative blueprint for active operations, local compilations, and continuous integrations.
 * **Cons:** Imposes an infinite maintenance tax. Every localized structural change or dependency update forces a synchronized documentation rewrite, risking immediate systemic drift if missed.
#### Epoch Snapshot Records
 * **Pros:** Zero long-term maintenance tax. Once written and locked, they require no further engineering updates, preserving pristine historical context about why specific paths were chosen and why alternative architectures were discarded.
 * **Cons:** Incapable of serving as an active system guide. A developer cannot use them to safely find the current configuration parameters or execution pathways of a running deployment.
### When to choose each option:
 * **Choose a Living Reality Tracker** when the text answers questions about the immediate, active state of the code—such as how to boot a local development environment, execute the test suite, or integrate with a public network interface.
 * **Choose an Epoch Snapshot Record** when the text details a specific proposal for future work yet to be built (a Spec) or documents the immutable rationale behind a decision that has already been executed (an ADR).
### Common failure modes:
 * **The Historical Rewrite (The Retrospective ADR):** An engineering team alters a service's persistence layer from PostgreSQL to Redis. Instead of opening a new ADR to log the change, they edit the original ADR written three years prior, rewriting its text to state that Redis was selected. This destroys the historical record of the startup's seed phase constraints, rendering the archival log useless for archeological analysis.
### Example:
Standard engineering repositories enforce this by separating Nygard-style ADR directories from the root documentation. The root README.md lives as a fluid document, updated continuously to reflect the active step-by-step onboarding flow. In contrast, the /docs/adr/ directory is managed as an append-only sequence of immutable files, where past records are never edited; they are simply marked as Deprecated or Superceded by a newer, separate record.
## [Decision: The Artifact Selection Framework]
### What it is:
The evaluation framework used by a systems engineer to select exactly one destination out of the four core structural options—README, Spec, ADR, or Inline Comment—for a given piece of technical knowledge.
### Why it exists:
Without a rigorous selection test, engineering organizations default to arbitrary documentation placement. Critical architectural context winds up buried inside isolated inline code comments, while ephemeral, line-level code anomalies are erroneously documented in macro-level system wikis.
### Options:
| Artifact | Intended Reader | Lifespan Strategy | Target Scope |
|---|---|---|---|
| **README** | Repository Newcomer / Contributor | Living Reality (Evergreen) | System/Service Entrance |
| **Specification (Spec)** | Active Team / Code Reviewers | Epoch Snapshot (Archived Proposal) | Feature/Component Initiative |
| **Architecture Decision Record (ADR)** | Future Architect / System Archeologist | Epoch Snapshot (Immutable Choice) | Macro-System Architecture |
| **Inline Comment** | Local Line Maintainer | Localized Reality (Moves/Dies with Code) | Source File/Statement |
### Trade-offs:
#### README
 * **Pros:** Highly discoverable; serves as the absolute entry gateway for developers.
 * **Cons:** Quickly becomes a dense, unreadable wall of text if stuffed with granular file mechanics, design histories, or algorithmic debates.
#### Specification (Spec)
 * **Pros:** Establishes consensus across teams *before* engineering capital is spent on implementation.
 * **Cons:** Becomes instantly stale the moment code diverges during real-world execution. If not explicitly labeled as an archived snapshot, it misleads downstream teams.
#### Architecture Decision Record (ADR)
 * **Pros:** Prevents perpetual re-debating of established architectural baselines by codifying why rejected alternatives were unfit.
 * **Cons:** Offers no help to an engineer trying to figure out why an individual line of code contains a specific syntactic optimization.
#### Inline Comment
 * **Pros:** Guarantees absolute spatial proximity to the source logic, minimizing the cognitive load of switching contexts.
 * **Cons:** Suffers from localized code rot (Chapter 30). It cannot express system-wide constraints or multi-service relationships cleanly.
### When to choose each option:
Apply the **Selection Test Questions** to identify the correct destination:
 * **Question 1:** *"What do I need to run this repo right now?"* * **Selection:** **README**. It must contain only what the system is, how to build/run it, and where to find adjacent logs. If it is stale, it breaks a newcomer's first, most critical hour of trust.
 * **Question 2:** *"What are we proposing to build next quarter, and how will we validate it?"*
   * **Selection:** **Spec**. It serves as a blueprint for work not yet done. Once the feature ships, the spec is treated as an inactive archive, not a live reference.
 * **Question 3:** *"Why did we choose an eventual consistency model over Raft consensus for our analytics pipeline last year?"*
   * **Selection:** **ADR**. It locks down past design decisions and details why rejected alternatives were structurally unviable under historical constraints.
 * **Question 4:** *"Why are we subtracting exactly 32 bytes from this specific network memory buffer before allocating it to the Linux kernel?"*
   * **Selection:** **Inline Comment**. It handles line-level, counter-intuitive anomalies that are tightly bound to immediate statements and must die or move with the code itself.
### Common failure modes:
 * **The Commented Architecture:** An engineer uncovers a critical, multi-service race condition across a distributed system. Rather than creating an ADR or detailing the constraint in the system overviews, they write a massive 40-line inline comment inside a single local file. Because engineers working on adjacent microservices never look inside this file, they inadvertently violate the system-wide constraint, triggering a cascading database lockup in production.
### Example:
The open-source development of Kubernetes demonstrates this taxonomy at scale. The repository's root features a living README.md focused strictly on the developer onboarding loop: cloning, compiling, and spinning up a local cluster.
For feature design and proposal phases, the ecosystem leverages **Kubernetes Enhancement Proposals (KEPs)**. These serve strictly as **Specs**—they establish consensus on API design or structural changes before code implementation begins. Once implemented, these proposals are frozen and archived as historical milestones; they are explicitly decoupled from the live API reference documentation (which lives as an independent, evergreen publishing engine covered in Chapter 67).
## Why Smart Engineers Disagree: The Cost of Archival Maintenance
The primary friction regarding documentation taxonomy manifests around whether Epoch Snapshots (Specs and ADRs) should be modified once code enters production.
### The Evergreening Faction
This school of thought argues that an out-of-date document is an inherently defective document. They believe that if a system’s production state mutates away from the original design spec or ADR, the original file should be dynamically updated, refactored, or rewritten to match modern truth. They optimize for a single, unified source of truth, attempting to reduce the cognitive load of future developers who might read an old document and mistake it for current architecture.
### The Structural Immutability Faction
This school argues that attempting to keep historical proposals and decision records aligned with living code is a fool's errand that violates Principle 6. They view specs and ADRs as **legal courtroom records** of engineering history. If you alter an ADR written in 2024 to fit the system layout of 2026, you erase the context that made that choice correct in 2024. They demand that historical records remain completely frozen; changes in system state must be handled by appending a *new* document that explicitly overrides the old one, leaving the original trail intact.
### The Reality
The disagreement is a function of organizational scale and tool maturity.
 * In small teams or fast-moving startups with a low total volume of documents, **Evergreening** keeps a small team aligned without forcing them to parse through chronological layers of old text.
 * In enterprise architectures, distributed open-source systems, or regulated environments, **Structural Immutability** is the only defensible model. At scale, the maintenance burden of retroactively updating old design specs collapses under its own weight.
The pragmatic systems engineer enforces structural immutability for specs and ADRs, ensuring that the historical record remains an anchor of engineering intent while leaving the tracking of active reality exclusively to the living README and auto-generated reference schemas.
