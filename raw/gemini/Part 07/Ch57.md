Chapter 57 — What Belongs in CI (and What Doesn't)
## 1. Purpose
This chapter opens the second half of Part VII—the continuous integration and continuous delivery (CI/CD) delivery pipeline—by establishing the strict operational boundaries of the automated verification loop. Continuous Integration (CI) is frequently misconstrued as an unrestricted staging ground for every test, audit, and security check an organization owns.
This chapter explicitly defines what types of verification belong in the blocking, on-every-commit pre-merge pipeline versus what must be relegated to out-of-band asynchronous cadences. It resolves the forward reference from Part VI (Ch 47) by transforming the philosophical mandate of automating human oversight into a concrete execution blueprint.
## 2. Prerequisites
This chapter assumes conceptual mastery of the following architectural and process frameworks:
 * **Part V (Testing Strategy):** The structural allocation of the testing pyramid, specifically the operational differences between fast unit-test tiers and heavy integrated test boundaries (Ch 34, Ch 35). This chapter references those tiers directly and does not re-derive them.
 * **Part V (Testing Strategy):** The systemic impact of feedback loop latency (Ch 34), treating pipeline execution speed as a hard operational constraint.
 * **Part VI (Engineering Process):** The code review paradigm and the mandate to automate machine-verifiable rules entirely out of human peer review (Ch 47), codified as Principle 8.
This chapter governs the *scope* of CI pipeline content. It does not cover the *ordering* of pipeline stages or the programmatic failure behaviors of conditional steps, which are covered in Ch 58.
## 3. Key Concepts
 * **Mechanical Verifiability:** The property of a codebase property (such as syntax formatting, type correctness, or baseline deterministic behavior) that can be verified by an algorithmic tool without human interpretation or execution context.
 * **Pipeline Latency Threshold:** The maximum elapsed time a developer can wait for a pipeline run before cognitive context-switching occurs, degrading team-wide development velocity.
 * **Deterministic Isolation:** The operational requirement that a pipeline's execution outcome must depend strictly on the code delta introduced, free from external state fluctuations, network dependencies, or multi-tenant test resource contention.
 * **Asynchronous Cadence Partitioning:** The architectural practice of separating verification tasks into a high-frequency, blocking pre-merge track and a low-frequency, non-blocking periodic track based on execution cost and resource utilization.
## [Pipeline Scope Partitioning (Pre-Merge Gates vs. Asynchronous Cadence)]
### What it is
The explicit architectural segmentation of the automated verification suite into two distinct execution paths: a blocking path that runs synchronously on every pull request commit, and a non-blocking path that executes asynchronously on a periodic cron cadence (e.g., nightly, weekly) or pre-release milestone.
### Why it exists
To manage the conflict between absolute system verification and developer feedback loop velocity. If every security scan, fuzzing array, and multi-hour end-to-end simulation runs on every single patch, the pipeline latency threshold is exceeded. This discourages small change units (Part VI, Ch 43) and leads to large, risky, grouped PRs that increase integration friction.
### Options
#### 1. Monolithic Gate Verification
The pipeline runs the entire testing portfolio—including deep static analysis, comprehensive container security scans, and full end-to-end integration flows—on every commit before allowing a merge to the main branch.
#### 2. Partitioned Cadence Verification
The blocking pre-merge pipeline is tightly constrained to run only tasks that are cheap, fast, and mechanically verifiable. Expensive, slow, or non-deterministic suites are offloaded to an asynchronous, out-of-band pipeline executed on a periodic schedule.
### Trade-offs
#### Partitioned Cadence Verification
 * **Pros:** Minimizes pipeline latency, keeping feedback loop times within the ideal sub-10-minute window. This supports continuous integration by encouraging developers to merge small, frequent commits into the main branch (Ch 50).
 * **Pros:** Enhances pipeline stability. Removing flakiness-prone outer adapters and end-to-end integration environments from the blocking path prevents transient network or infrastructure failures from halting the engineering organization.
 * **Cons:** Introduces a regression window. Shifting heavy tests out-of-band means a breaking change could pass pre-merge checks and land on the main branch, requiring an automated rollback strategy when the nightly build eventually fails.
#### Monolithic Gate Verification
 * **Pros:** Guarantees absolute catch rate metrics prior to any trunk mutation. It provides strong insurance against breaking changes landing on the primary trunk.
 * **Cons:** Distorts developer behavior due to high pipeline latency. When a merge takes two hours to clear CI, engineers batch unrelated features into massive pull requests to avoid paying the pipeline time tax repeatedly, maximizing batch risk.
 * **Cons:** High execution cost. Running heavy resource-intensive workloads on every single commit significantly increases cloud infrastructure spend.
### When to choose each option
 * **Choose Partitioned Cadence Verification** as the operational architecture for modern continuous delivery teams. Pre-merge pipelines must be reserved for light validation steps that complete within minutes.
 * **Choose Monolithic Gate Verification** only for small, low-velocity codebases where the entire test suite completes well under the 10-minute feedback loop latency threshold, or for mission-critical software systems (such as medical devices or aviation firmware) where zero trunk regressions are tolerated and development velocity is deliberately throttled.
### Common failure modes
 * **The Congested PR Chokepoint:** An engineering organization includes an extensive, three-hour Selenium end-to-end browser suite directly into the blocking pre-merge pipeline of a main web platform repository. A developer submits a one-line documentation or CSS color fix. The entire multi-tenant pipeline runner cluster fills with long-running browser instances. Pull requests back up in a multi-hour queue, code review velocity drops to zero, and developers begin bypassing CI checks entirely under administrative emergency flags to ship hotfixes.
### Example
The development pipeline for the Kubernetes project splits its testing vectors carefully. Pre-merge pull request pipelines execute focused unit tests, syntax linters, and light integration mocks that complete quickly. Heavy, multi-hour, multi-cloud conformance suites and comprehensive fuzzing setups are partitioned into distinct, periodic asynchronous pipelines that run on dedicated hourly or nightly schedules, reporting regressions to a release-blocking dashboard without blocking daily engineering output.
## [Mechanical Verifiability Enclosure (Automating Machine Tasks Out of Human Review)]
### What it is
The policy and mechanical practice of identifying every codebase attribute that can be programmatically verified by an abstract syntax tree (AST), compiler, linter, or type-checker, and forcing its evaluation into the automated CI pipeline.
### Why it exists
To enforce Part VI, Ch 47's Principle 8: *Anything a machine can check must be automated out of human review entirely.* Human code reviews are highly expensive cognitive events. Allowing human peer reviews to be wasted on mechanical issues like syntax formatting, indentation, import ordering, or basic type alignment introduces major process overhead and fuels subjective team arguments.
### Options
#### 1. Human-Regulated Style and Compliance
Coding styles, lint rules, and architectural dependency boundaries are codified in prose documentation (wikis, style guides). Peer reviewers are responsible for inspecting pull request diffs to manually identify and call out formatting variations or structural boundary violations.
#### 2. Programmatic Pipeline Enforcement
Mechanical compliance is managed entirely by automated binaries embedded directly within the blocking pre-merge CI runner (e.g., eslint, black, cargo fmt, golangci-lint). The pipeline immediately rejects any commit that does not comply with the established rules, aborting further test execution.
### Trade-offs
| Dimension | Human-Regulated Compliance | Programmatic Pipeline Enforcement |
|---|---|---|
| **Review Signal-to-Noise Ratio** | Low; pull request comments are dominated by formatting noise. | High; peer reviewers focus exclusively on logic and design architecture. |
| **Rule Consistency** | Variable; depends heavily on individual reviewer attention. | Absolute; rules are parsed identically across all code variations. |
| **Developer Onboarding Cost** | High; engineers must study long prose formatting wikis. | Low; the automated feedback loop corrects mistakes immediately. |
| **Configuration Rigidity** | High flexibility; reviewers can grant ad-hoc exceptions easily. | Low flexibility; requires explicit configuration overrides in source code. |
### When to choose each option
 * **Choose Programmatic Pipeline Enforcement** universally across all professional software teams. Every mechanical property must be validated by code tools prior to human review. A pull request should be physically unreviewable by a human until the automated mechanical pipeline turns green.
 * **Never use Human-Regulated Style Compliance** in shared engineering environments. It represents a systematic process-overhead failure that wastes expensive engineering focus on tasks easily handled by machines.
### Common failure modes
 * **The Peer-Review Style War:** An organization lacks automated formatting and lint checks in its CI pipeline. A developer submits a large architectural refactoring PR. Rather than analyzing the security model or data flow constraints, three different reviewers post 45 minor comments debating brace placement, snake_case vs. camelCase preferences, and un-sorted import blocks. The developer becomes defensive, the pull request stalls for two weeks over cosmetic details, and critical logical flaws in the architecture pass into production completely uninspected.
### Example
In high-performance engineering projects like Redis or SQLite, mechanical formatting is treated as an objective, binary gate. For instance, teams combine an automated formatter with a CI linter rule that blocks compilation if the code departs from the target style guide.
By forcing the linter to execute as the very first step of the pre-merge pipeline, the system catches formatting mistakes within seconds, preventing unformatted or style-violating code from ever landing in front of a human reviewer.
## Why Smart Engineers Disagree: The Illusion of "Complete Integration Validation"
The boundary defining what belongs inside a blocking pre-merge CI pipeline exposes a core disagreement regarding the fundamental promise of continuous integration. This friction splits engineering organizations into **Trunk Purists** and **Velocity Optimizers**.
**Trunk Purists** define the promise of CI as a mathematical guarantee that the primary branch remains clean, stable, and completely free of regressions at all times. They argue that allowing any code to merge without running every single validation tool available—including heavy end-to-end integration flows and deep security analysis—is an explicit abandonment of engineering discipline. To this group, the cost of a slow pipeline is a reasonable trade-off to ensure that the primary trunk is always completely verified and ready for deployment. They accept long queues and high pipeline latency as necessary safeguards to prevent regressions from slipping into production.
**Velocity Optimizers**, drawing heavily from Martin Fowler’s foundational continuous integration models, view CI primarily as a rapid, high-signal feedback loop designed to support developer momentum. They point out that a pipeline taking hours completely destroys the practice of continuous integration. When developers must wait a long time to clear CI, they stop integrating frequently. They work in isolation on long-lived branches (Ch 50), which defer major integration risks and lead to severe branch drift.
Velocity Optimizers argue that a blocking pipeline must complete in under ten minutes. To achieve this speed, they are willing to shift expensive integration tests out-of-band into asynchronous nightly pipelines. They accept the risk of occasional trunk regressions because they prioritize maintaining a fast developer loop, relying on rapid automated rollbacks to clean up any failures that land on the main branch.
### The Resolution
The systems engineer resolves this ideological conflict by analyzing the system's resource constraints and the team's pull request dimensions. If your engineering organization enforces small change units and utilizes runtime decoupling guardrails like feature flags (Part VI, Ch 43), you should lean toward the Velocity Optimizer model.
Keep your blocking pre-merge CI pipeline highly constrained: run only fast unit tests, style checks, and type validations that provide high-signal feedback within minutes. Offload heavy integration suites and exhaustive security scans to asynchronous periodic pipelines, and build robust tooling to handle rollbacks if an out-of-band run fails. This approach protects both developer velocity and system stability by making integration a fast, continuous habit rather than a slow, painful event.
## 4. Summary Checklist for CI Scope Governance
To maintain an optimal and fast pipeline across team boundaries, enforce the following configuration constraints:
 * **Enforce a Strict Latency Cap:** Programmatically track your pre-merge pipeline duration. If the blocking gate exceeds 10 minutes, mandate an architectural audit to prune or offload slow integration tests into an asynchronous cadence.
 * **Fail Fast on Mechanical Checks:** Position linting, formatting verification, and syntax type-checking at the absolute front of the pipeline. Ensure the runner aborts immediately on style failures before wasting execution time on compiled test suites.
 * **Isolate External Dependencies:** Block any blocking pre-merge test step from making live network calls to third-party APIs or unmanaged cloud databases. Force the use of localized mocks or hermetic in-memory test doubles to eliminate transient pipeline flakiness.
