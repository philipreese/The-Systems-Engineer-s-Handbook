Chapter 75 — Locks: When to Use Them
## 1. Purpose
This chapter provides the mechanical framework for managing shared state safely when the message-passing model detailed in Chapter 74 is bypassed. It establishes how to reason about, implement, and evaluate mutual exclusion primitives. The lock is the primary tool for safeguarding data integrity across concurrent execution threads, but it introduces severe structural risks to system latency and throughput. This chapter outlines how to balance lock scopes and identifies precisely when traditional locking mechanisms must be abandoned in favor of lock-free or lock-avoiding architectures.
## 2. Prerequisites
 * **Chapter 01 (What Engineering Actually Optimizes):** The throughput-vs-reasoning-simplicity constraint optimization model.
 * **Chapter 06 (Cost Models and Mechanical Sympathy):** CPU cache lines, memory access latency, and the baseline database-level distinction between pessimistic locking and versioning.
 * **Chapter 74 (Shared State vs. Message Passing):** The core definition of shared-state concurrency and the structural genesis of data races.
## 3. Key Concepts
 * **Mutual Exclusion (Mutex):** A runtime synchronization primitive designed to serialize access to a critical section of code, ensuring that exactly one execution unit can read or write to a specified block of memory at any given instant.
 * **Lock Contention:** The operational state where one or more threads are blocked waiting to acquire a lock currently held by another executing thread, forcing CPU cores to idle or execute context switches.
 * **Lock Granularity:** The operational scale of the data protected by a structural lock, defining the boundary between a single coarse guard and multiple highly isolated fine guards.
 * **Critical Section:** The minimal block of execution instructions that reads or mutates a shared resource and must execute atomically to prevent undefined behavior.
## Principles and Decisions
[Lock Granularity Selection: Coarse-Grained vs. Fine-Grained]
What it is:
The determination of how much data a single lock should protect—ranging from a single monolithic lock guarding an entire subsystem (coarse-grained) to multiple individual locks guarding micro-segments of data or array indices (fine-grained).
Why it exists:
Locks introduce a strict serialization bottleneck. While fine-grained locking minimizes lock contention and unlocks high parallel throughput across multiple CPU cores, it explodes the system's state space, making code exceptionally difficult to reason about and dramatically increasing the statistical probability of runtime structural failures.
Options:
 1. Coarse-Grained Locking: A single lock protecting an entire collection, object tree, or execution module.
 2. Fine-Grained Locking: Dissecting data structures into discrete sub-components, each managed by its own independent lock primitive.
Trade-offs:
 * Coarse-Grained Locking: Maximizes structural reasoning simplicity and minimizes human implementation error. It incurs zero multi-lock interaction hazards. However, it severely bottlenecks multi-core execution throughput under load; threads spend substantial cycles stalled in lock queues, collapsing system efficiency.
 * Fine-Grained Locking: Minimizes lock contention and maximizes parallel execution throughput across high-core-count hardware. However, it introduces an exponential surge in code complexity. Reasoning about concurrent interleavings becomes extraordinarily difficult, and it creates a highly acute risk of cascading deadlocks (expanded in Chapter 77).
When to choose each option:
 * Default explicitly to Coarse-Grained Locking. In alignment with this handbook's core focus on avoiding premature optimization, preserve coarse boundaries until rigorous production profiling (detailed in Part XII, Chapter 86) mathematically proves that lock contention on that specific boundary is the primary systemic bottleneck.
 * Transition to Fine-Grained Locking strictly when profiling reveals high contention metrics and when the underlying data can be cleanly decoupled into independent mathematical segments with zero overlapping transactional requirements.
Common failure modes:
 * Premature fine-grained structural fragmentation: An engineer splits a database cache into hundreds of bucket-level locks to optimize theoretical concurrency. The application code must now manage multiple lock acquisitions to execute multi-bucket movements, introducing a silent, non-deterministic deadlock that escapes static integration testing and surfaces only under non-linear production spikes.
Example:
Java's collections ecosystem perfectly illustrates this graduation. Early thread-safe structures like java.util.Vector or Hashtable utilized coarse-grained, exclusive synchronization on every method call, which instantly paralyzed performance on multi-threaded web servers. The platform evolved to introduce ConcurrentHashMap, which replaced the single monolithic table lock with a fine-grained array of discrete lock buckets (segments). This allows completely parallel writes to different regions of the hash map, dramatically elevating multi-core throughput at the cost of significantly more complex internal structural code.
[Lock Exclusivity Strategy: Mutual Exclusion vs. Shared Read-Write Locks]
What it is:
The decision of whether a critical section requires absolute, unconditional exclusivity for every thread, or whether the system can grant concurrent, shared access to readers while isolating writers.
Why it exists:
Many concurrent application payloads exhibit highly asymmetric read-to-write ratios. Forcing thousands of continuous read operations to queue sequentially behind an exclusive mutual exclusion lock degrades mechanical sympathy and artificially drives up latency.
Options:
 1. Exclusive Mutex: Standard locking where access is strictly binary; exactly one thread can enter the critical section regardless of whether it is reading or writing data.
 2. Read-Write Lock (RWMutex): A dual-mode lock allowing infinite concurrent threads to read data simultaneously, but requiring absolute exclusive access for any thread executing a write operation.
Trade-offs:
 * Exclusive Mutex: Imposes low internal bookkeeping overhead and exhibits faster raw lock acquisition and release speeds in low-contention environments. However, it causes severe, artificial thread starvation in high-volume, read-heavy workloads.
 * Read-Write Lock: Drastically mitigates reader-side latency and maximizes cache line shared usage across high-volume read paths. However, it carries significantly heavier internal state tracking and algorithmic overhead. Mismanaged prioritization rules can easily lead to severe writer starvation, where a continuous stream of incoming readers completely blocks pending writes from ever executing.
When to choose each option:
 * Choose Exclusive Mutex for balanced read/write profiles, short critical sections, or when write operations occur frequently (e.g., greater than 10-15% of total operations).
 * Choose Read-Write Locks strictly for workloads where reads outnumber writes by orders of magnitude (e.g., configuration maps, service discovery routing tables, routing engines) and where the time spent inside the read-side critical section is long enough to amortize the RWMutex's internal bookkeeping tax.
Common failure modes:
 * RWMutex Writer Starvation: An infrastructure team switches a highly contested local cache lock from a standard Mutex to a Read-Write Lock. The service handles a continuous torrent of 50,000 read requests per second. Because new readers are allowed to join the shared lock while existing readers are active, a background write thread attempting to flush updated data is blocked indefinitely, causing the application's configuration state to rot.
Example:
Go provides sync.Mutex and sync.RWMutex inside its core library. Senior engineers writing internal tools utilize standard sync.Mutex by default. They migrate to sync.RWMutex only when building read-dominated lookup subsystems—such as an in-memory geo-IP routing dictionary—where thousands of concurrent HTTP worker goroutines must parse incoming IP addresses against a static memory table that updates only once per day via a single background writer thread.
[Concurrency Control Abstraction: Pessimistic Memory Locking vs. Optimistic Lock Avoidance]
What it is:
The selection between physically blocking threads from entering a critical section (pessimistic locking) and allowing threads to execute operations unhindered, verifying state validity and retrying upon conflict after the fact (Optimistic Concurrency Control / Lock Avoidance).
Why it exists:
Blocking a thread using an OS-level mutex is expensive; it triggers context switches, kernel-space transitions, and CPU pipeline flushes. If conflicts between concurrent threads are statistically rare, paying the heavy operational cost of pessimistic mutual exclusion for every transaction degrades execution velocity.
Options:
 1. Pessimistic Mutual Exclusion: Physically blocking concurrent execution units via standard locks or kernel sleep states until the thread holding the critical section exits.
 2. Optimistic Concurrency Control (OCC): Executing updates into local memory variables or local buffers, checking if another thread mutated the source data during execution via hardware-level primitives (like Compare-And-Swap), and re-executing the entire transaction loop if a conflict is detected.
Trade-offs:
 * Pessimistic Mutual Exclusion: Guarantees a flat, predictable latency upper bound under extreme, saturated contention. Threads wait in orderly queues, preventing CPU starvation. However, it scales poorly in low-conflict environments due to continuous, unnecessary kernel scheduling and locking overhead.
 * Optimistic Concurrency Control: Delivers extreme, near-native execution performance and zero thread blocking when data contention is low. It keeps execution purely in user-space. However, under high, concentrated write contention, performance collapses catastrophically; threads enter infinite retry loops ("livelock"), burning 100% of available CPU cycles merely re-executing aborted work.
When to choose each option:
 * Choose Pessimistic Mutual Exclusion when write contention is mathematically high, workloads are write-heavy, or when the system operates under tight latency ceilings where retry loops are unacceptable.
 * Choose Optimistic Concurrency Control / Lock Avoidance when conflicts are statistically anomalous or when operations are highly distributed, ensuring that threads spend the vast majority of their time executing successful zero-block modifications.
Common failure modes:
 * OCC Retry Storm: An engineer implements a high-performance, lock-free counter using an optimistic atomic loop. During a promotional marketing blast, thousands of parallel network workers attempt to increment the exact same memory address simultaneously. Exactly one thread succeeds per iteration; the remaining 999 threads fail, loop around, and try again, driving machine CPU utilization to 100% and causing end-to-end service latency to spike exponentially.
Example:
The Linux kernel uses graduated locking and lock-avoidance models explicitly optimized for different operational contexts rather than enforcing a single strategy across the OS. For long-duration critical sections that interact with slow I/O, the kernel utilizes standard sleep-based mutexes. For ultra-short execution paths where blocking is prohibited, it switches to spinlocks. For read-heavy, performance-critical kernel subsystems—such as the network routing table or file descriptor lookups—Linux completely avoids traditional locks by leveraging Read-Copy-Update (RCU). RCU allows readers to traverse memory structures without acquiring locks or executing atomic instructions, while writers create an entirely new copy of the structure and safely swap pointers via atomic memory barriers, mimicking the database-level MVCC patterns implemented by platforms like PostgreSQL (Chapter 06).
## Why Smart Engineers Disagree: The Boundary of Lock-Free Implementation
The most persistent architectural debate between systems engineers in the shared-state domain is whether to replace lock-based critical sections with lock-free data structures using raw atomic hardware primitives.
Engineers advocating for pure lock-free code point to the mechanical reality of the operating system: traditional locks are an admission of failure. They argue that thread blocking destroys real-time characteristics, introduces unquantifiable latency jitter, and scales horribly as CPU core counts climb into the hundreds. They advocate for building data structures out of atomic primitives (Compare-And-Swap, Load-Linked/Store-Conditional), ensuring that the system as a whole always makes forward progress without kernel intervention.
Engineers prioritizing maintainability and system safety view lock-free data structures with extreme skepticism. They emphasize that lock-free programming is not merely hard; it is a specialized discipline where code readability goes to die. They point out that verifying the correctness of lock-free loops requires meticulous mapping of memory visibility orderings, CPU compiler reordering flags, and memory fence semantics. A single out-of-order instruction creates silent memory corruption that can take months to trace. To them, a well-placed, coarse-grained mutex is an elegant piece of engineering because its safety boundaries are explicit, transparent, and completely intelligible to any engineer on the team.
The Reality: The disagreement is a direct conflict between the limits of human comprehension and the limits of the underlying hardware architecture. Lock-free data structures should be heavily restricted. Unless you are writing a core execution scheduler, a low-level memory allocator, or a specialized ring-buffer queue for an extreme high-throughput telemetry engine, the performance gain of a lock-free structure rarely justifies the massive risk of introducing untraceable, non-deterministic memory alignment bugs. For 95% of systems infrastructure applications, a highly scannable, pessimistic locking strategy remains the correct engineering choice.
