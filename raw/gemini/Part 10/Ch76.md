Chapter 76 — Async vs. Threads vs. Processes
## 1. Purpose
This chapter separates a critical architectural axis from the choices detailed in Chapter 74 and Chapter 75: not how concurrent execution units coordinate access to shared state, but which physical or virtual execution vehicle runs concurrent work in the first place. Conflating the coordination model ("locks vs. channels") with the execution model ("async vs. threads") is a foundational engineering error. This chapter explicitly resolves that conflation, providing the structural criteria required to map workload characteristics to the correct operating system or runtime container.

## 3. Key Concepts
 * **Concurrency:** The structural decomposition of a program into discrete, independently executing units of work that can be processed during overlapping time windows. Concurrency is a property of code architecture and does not require multi-core hardware.
 * **Parallelism:** The literal, simultaneous execution of multiple computational tasks at the exact same physical instant. Parallelism is a property of execution throughput and strictly requires multiple physical CPU cores or hardware execution pipelines.
 * **OS Process:** The primary operating system container for execution, providing a fully isolated virtual address space, separate file descriptor tables, and strict kernel-enforced security boundaries.
 * **OS Thread:** A lightweight unit of execution scheduled preemptively by the operating system kernel within a process container. All threads of a single process share its virtual address space.
 * **Async / Coroutine:** A lightweight, software-managed unit of execution that yields control cooperatively back to a runtime event loop rather than relying on kernel-level preemption.
## Principles and Decisions
[Workload-Driven Execution Unit Selection: Threads/Processes vs. Async]
What it is:
Selecting the primary vehicle for task execution based on whether the performance bottleneck of the application is bounded by silicon computation or network/disk I/O operations.
Why it exists:
Hardware boundaries dictate that waiting for an external I/O packet costs hundreds of thousands of nanoseconds (~500μs–1ms for local network calls), while a CPU instruction executes in less than a nanosecond. If an application utilizes heavy OS threads merely to stall and wait for slow I/O, it wastes massive amounts of memory and kernel scheduling cycles. Conversely, if it uses a single-threaded async event loop for heavy computations, a single task will monopolize the execution thread, paralyzing the entire service.
Options:
 1. Kernel-Scheduled Parallelism (OS Threads or Processes): Assigning each concurrent task its own operating system thread or process.
 2. Cooperative Event-Driven Concurrency (Async / Coroutines): Mapping thousands of concurrent tasks onto a single or few-threaded execution loop that yields control at non-blocking I/O boundaries.
Trade-offs:
 * Kernel-Scheduled Parallelism: Unlocks true physical parallelism across multiple CPU cores, maximizing the execution speed of computational logic. However, it incurs severe memory overhead (typically 1MB to 8MB per thread stack size) and major CPU context-switching costs when thousands of threads block on network calls.
 * Cooperative Event-Driven Concurrency: Highly efficient resource consumption, permitting millions of concurrent tasks to reside in memory simultaneously due to minuscule per-task overhead (measured in kilobytes). However, it lacks CPU parallelism unless explicitly distributed across an underlying thread pool, and it is entirely vulnerable to non-cooperative, blocking execution paths.
When to choose each option:
 * Choose Kernel-Scheduled Parallelism (Threads/Processes) for CPU-bound workloads (e.g., cryptography operations, image processing pipelines, heavy JSON or protocol buffer serialization, mathematical matrix calculations).
 * Choose Cooperative Event-Driven Concurrency (Async) for I/O-bound workloads (e.g., highly concurrent HTTP/gRPC proxies, chat application web sockets, API gateways orchestrating multiple slow downstream service requests).
Common failure modes:
 * Thread Exhaustion Under I/O Satiation: An engineer builds a reverse proxy using a thread-per-request model. Under a sudden surge of slow network traffic, thousands of clients connect. The system spawns thousands of OS threads, exhausting host RAM on thread stacks and dropping global throughput to zero as the kernel spends all its cycles context-switching rather than processing data packets (thrashing).
 * Event-Loop Starvation: An engineer uses a single-threaded async runtime to build an API that processes incoming webhooks and applies a heavy CPU-intensive cryptographic signature to each payload. Under load, a single webhook execution blocks the shared event loop for 150ms. Every other pending request in the async queue stalls completely, breaching the service's end-to-end latency SLO for unrelated routes.
Example:
Node.js is a canonical async-first runtime explicitly built for I/O-bound workloads. It relies on a single-threaded JavaScript event loop powered by libuv underneath. If an application strictly performs database queries and network fetches, Node.js manages thousands of concurrent connections with minimal memory overhead. However, if a developer writes a long-running, blocking CPU loop directly in a request handler, the entire Node.js server stops accepting any incoming network packets until that specific calculation completes.
[Task Boundary Isolation: Processes vs. Threads]
What it is:
The choice of whether execution units should run within a shared virtual address space (OS Threads) or be segregated into entirely separate memory spaces (OS Processes).
Why it exists:
Security boundaries and fatal crash isolation differ论 sharply between threads and processes. If a single execution thread encounters a fatal memory error or segmentation fault, the operating system terminates the entire parent process, killing all other sibling threads instantly. Processes provide rigid blast-radius protection at the cost of hardware resource overhead.
Options:
 1. Multithreaded Shared Memory Architecture: Running all concurrent tasks as distinct threads within a single shared process space.
 2. Multi-Process Isolated Architecture: Spawning distinct operating system processes for every independent unit or pipeline of work.
Trade-offs:
 * Multithreaded Shared Memory Architecture: Extremely fast inter-task communication and zero pointer-passing overhead since all memory is shared natively. However, it completely lacks fault isolation; a single unhandled panicking thread or unsafe pointer violation kills the entire application container.
 * Multi-Process Isolated Architecture: Guarantees absolute fault isolation and robust security containment. If one process crashes or exhibits a memory leak, the remaining execution nodes continue operating unimpeded. However, inter-process communication (IPC) requires crossing kernel boundaries via domain sockets, shared memory mapping, or pipes, introducing serialization latency and heavy CPU usage.
When to choose each option:
 * Choose Multithreaded Shared Memory for high-performance application tiers where tasks are deeply cooperative, data sets are massive, and the latency tax of IPC serialization violates the execution SLO.
 * Choose Multi-Process Isolated Architecture when executing untrusted third-party code, processing independent multi-tenant payloads with strict security boundaries, or when building high-assurance master-worker control loops where the crash of a worker must never terminate the master engine.
Common failure modes:
 * Cascading Shared-Process Collapse: A large web framework uses a single multi-threaded process to handle thousands of requests. One edge-case route executes an unmanaged FFI call that triggers a segmentation fault. The OS kernel immediately terminates the entire process, violently disconnecting thousands of completely unrelated, healthy client connections and spiking the system's MTTR.
Example:
PostgreSQL implements a strict Multi-Process Isolated Architecture. Every incoming client connection causes the Postgres master daemon to fork a completely separate backend process. If a specific client connection executes a highly complex query that triggers an internal bug or memory corruption fault, only that individual backend process terminates. The core database cluster remains online and unbothered, protecting global availability. This stands in direct contrast to multithreaded relational databases, where an internal thread crash can compromise the entire server instance's uptime.
[Scheduling Strategy: Preemptive vs. Cooperative Execution Control]
What it is:
Determining whether the system relies on the operating system kernel to forcibly interrupt and switch between tasks (Preemptive) or leaves it to the individual execution tasks to explicitly yield processing time back to a runtime manager (Cooperative).
Why it exists:
Fairness vs. Throughput. Preemptive scheduling guarantees that no single rogue loop can monopolize a CPU core, but it forces frequent, non-negotiable scheduling checks. Cooperative scheduling minimizes runtime overhead, but relies entirely on code behaving perfectly to maintain systemic equilibrium.
Options:
 1. Preemptive Kernel Scheduling: Utilizing standard OS-level primitives (threads/processes) where the kernel scheduler automatically rotates tasks based on hardware timer interrupts.
 2. Cooperative Runtime Scheduling: Utilizing user-space async/coroutines where the code explicitly yields execution control during network waits or via keyword markers (e.g., await).
Trade-offs:
 * Preemptive Kernel Scheduling: Provides absolute fairness. An infinite while(true) loop in one task will not stall sibling tasks, as the kernel will forcibly suspend it. However, context switches occur non-deterministically, destroying CPU cache locality and forcing frequent registers-saving operations.
 * Cooperative Runtime Scheduling: Eliminates the kernel context-switching tax entirely, keeping execution within user-space. Tasks yield only when they hit natural pauses (like waiting for a network socket), maximizing raw instruction execution throughput. However, fairness is completely sacrificed; a single task that fails to yield will permanently seize the execution core, freezing all concurrent tasks assigned to that loop.
When to choose each option:
 * Choose Preemptive Kernel Scheduling when writing multi-tenant platforms, execution runtimes, or applications where workloads are unpredictable, diverse, and cannot be trusted to self-regulate execution durations.
 * Choose Cooperative Runtime Scheduling when designing high-concurrency network servers where every task follows a uniform pattern of short processing bursts interrupted by explicit, predictable I/O waits.
Common failure modes:
 * Rogue Cooperative Monopolization: An engineer uses an async framework to process incoming network streams. An execution path encounters an edge-case bug that triggers an infinite compute loop containing no I/O operations or yield checkpoints. The coroutine never yields control, permanently locking that execution thread. The runtime cannot step in, causing a massive, silent freeze of all other coroutines queued on that specific worker thread.
Example:
Python's reference implementation (CPython) features a unique concurrency constraint: the Global Interpreter Lock (GIL). Because of the GIL, even when an engineer spawns multiple OS threads, the Python runtime internally restricts bytecode execution to exactly one CPU core at a time, using a preemptive scheduling mechanism to rotate thread ownership of the lock. Therefore, spawning Python OS threads does not buy true CPU parallelism for numerical calculations. To bypass this, performance-critical Python tools must drop down across the FFI boundary into C to release the GIL, or utilize multi-processing instead, demonstrating how runtime constraints dictate the actual utility of scheduling models.
## Why Smart Engineers Disagree: The M:N Hybrid Runtime Paradigm
The sharpest architectural divide in modern systems design centers on whether runtimes should obscure the execution-unit distinction by implementing an M:N hybrid scheduling model.
Engineers championing explicit M:N hybrid runtimes (most notably exemplified by Go's goroutine scheduler) argue that forcing developers to manually select between explicit async/await keywords or raw OS threads is an unnecessary cognitive tax that introduces systemic complexity. Go handles this by mapping M green threads (goroutines) onto N OS threads dynamically via its runtime scheduler. If a goroutine makes a blocking network call, the runtime intercepts it, swaps it out in user-space, and continues running other work on that same OS thread. If it executes a blocking syscall, the runtime detaches the thread and assigns a new one to keep the remaining goroutines moving. To these engineers, this represents the ultimate synthesis: the performance and low overhead of async concurrency paired with the linear, blocking reasoning simplicity of threads.
Engineers favoring explicit, distinct models (such as those in the Rust, C++, or Node.js ecosystems) view M:N hybrid schedulers as an architectural mistake. They argue that hiding the execution vehicle introduces massive, invisible accidental complexity. A hybrid scheduler must carry a permanent, non-trivial runtime engine packed into every compiled binary, destroying its viability for low-level embedded devices or lightweight FFI plugins. Furthermore, they point out that when an M:N system encounters performance degradation, debugging the internal, opaque heuristics of a dynamic runtime scheduler requires deep specialist knowledge, whereas explicit async models or direct OS thread architectures keep the execution boundaries fully transparent and controllable by the systems engineer.
The Reality: The disagreement stems from a fundamental divergence in optimization targets. The hybrid model optimizes for global developer throughput and uniform execution design across a corporate ecosystem, happily accepting the baseline runtime fatness and loss of micro-granular control. The explicit model optimizes for absolute resource predictability and zero-cost abstractions, ensuring that if an execution boundary is crossed, the cost is visible directly in the code syntax rather than hidden behind runtime magic.
