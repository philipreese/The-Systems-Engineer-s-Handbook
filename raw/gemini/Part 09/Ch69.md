# Chapter 69 — Logging: What to Log and at What Level

## 1. Purpose

This chapter opens Part IX by extending this handbook's core information-cost argument from written artifacts to runtime telemetry. The foundational complexity arguments from Chapter 02 and the systemic process boundaries of Principle 6 apply to a runtime log line exactly as they apply to a line of source code documentation. Every log line emitted by an application in production is an engineering liability: it must be generated, serialized, transported over networks, indexed by central aggregators, and stored securely at a non-zero financial and operational cost.

An unread, low-signal log line that merely states an obvious execution flow introduces pure accidental complexity to the observability pipeline. This chapter establishes the rigorous technical filters senior engineers use to differentiate high-signal telemetry from operational noise, defines how to utilize log levels as strict triage vectors rather than expressions of developer emotion, and dictates how to balance historical debugging completeness against global infrastructure and security costs.

## 3. Key Concepts

* **The Telemetry Tax:** The ongoing financial, computing, and search-latency overhead of data ingestion. Excess volume degrades query execution speeds across log backends when an on-call responder is fighting an active incident.
* **The Actionability Filter:** The primary test for log inclusion. Telemetry must exist solely to inform an explicit engineering or operational decision after the fact, rather than serving as an unstructured journal of successful execution.
* **Contextual Severity Triage:** The design paradigm that a log level is a dynamic classification based on the target operational audience and response expectations, not an static property of an individual code statement.
* **The Correlation Vector:** Carrying a unique request identifier natively through every process step within a service boundary to enable single-request reconstruction prior to full distributed tracing integration.

## 4. Operational Principles and Decisions

### [Principle: The Operational Actionability Test]

**What it is:** The architectural filter that mandates a runtime log entry must be written exclusively to facilitate a concrete operational decision or debug mitigation step by a specific human consumer after the event occurs.

**Why it exists:** Engineers frequently treat application logs as a stream-of-consciousness diary of execution, logging events simply because a code path was reached. This pattern shifts the cognitive burden of data filtering from the developer at write-time to the on-call responder at midnight. In a high-throughput system, unstructured and non-actionable logs create a thick layer of static noise that obscures actual failure signatures, directly inflating Mean Time to Recovery (MTTR) during an outage.

**Options:**

* **Option 1: Severity-Driven Logging (Emotional Baselines)** Emitting log lines based on how significant or interesting an execution state feels to the individual engineer during initial feature development (e.g., logging INFO for every successful database record insertion).
* **Option 2: Action-Driven Logging (Decision Baselines)** Restricting log lines to state transitions, invariant violations, boundary crossings, or data anomalies that require a specific response, downstream investigation, or structural post-mortem analysis.

**Trade-offs:**

* **Severity-Driven Logging:**
  * *Pros:* Requires minimal upfront design or code-review friction. Provides highly detailed, verbose step-by-step audit trails during initial local software prototyping.
  * *Cons:* Generates massive accidental complexity and noise in production systems. High log-to-signal ratios mask true architectural failures and create chronic alerting fatigue if hooked to log scanners.
* **Action-Driven Logging:**
  * *Pros:* Minimizes the production log footprint, keeping search and aggregation engines fast. Ensures that when an on-call engineer queries the logs during an incident, every returned line carries high diagnostic density.
  * *Cons:* Requires deliberate planning during the design and code-review phases. Risks omitting granular execution breadcrumbs if an entirely unpredicted, novel failure mode occurs outside the anticipated boundaries.

**When to choose each option:**

* Choose **Action-Driven Logging** universally for all production deployments, cloud microservices, and stable long-term infrastructure.
* Choose **Severity-Driven Logging** strictly within ephemeral local development branches, standalone test suites, or alpha prototypes that have a lifespan measured in weeks.

**Common failure modes:**

* *The Warning Flood:* An application logs a WARN level message every time an external user submits an invalid form parameter or an expired authentication token. Because these events occur constantly due to normal web traffic and malicious scanners, the log repository fills with millions of warnings. The operational team grows entirely blind to the WARN level, causing them to miss a legitimate warning signal—such as an intermittent connection drop to a core database replica—until a catastrophic outage manifests.

**Example:** The classic Linux syslog protocol establishes a standardized severity-level taxonomy (from LOG_EMERG down to LOG_DEBUG). Experienced systems engineers do not use these levels to express personal frustration at a code bug. They align them strictly to operational postures: an ERROR level implies a localized failure that requires immediate engineering intervention or post-mortem attention; a WARN indicates a non-fatal anomaly that requires automated tracking but no immediate wake-up call; an INFO captures macro-level lifecycle changes (such as a service startup or configuration reload), while a DEBUG is filtered out entirely at the infrastructure boundary during standard operations.

### [Decision: Structured JSON Fields vs. Free-Text Streams]

**What it is:** The structural choice between formatting log outputs as human-readable, unconstrained text strings or as machine-parseable, strongly typed key-value pairs (typically structured as serialized JSON objects).

**Why it exists:** Free-text logs (e.g., `log.info("User " + userId + " connected from IP " + ip)`) are simple for a human to interpret when tailing a single log file on a local terminal. However, at scale—where a distributed system processes tens of thousands of requests per second across hundreds of parallel container arrays—free-text logs require complex, fragile regular expressions (Regex) to parse, index, and query inside a centralized log engine. A minor typographical change to a log string made by a developer during a routine patch will silently break downstream regex parsers, destroying log visibility and breaking metric extraction pipelines.

```
Free-Text (Fragile Parsing):
[INFO] 2026-07-02 09:15:00 - User 98321 connected from IP 192.168.1.50

Structured JSON (Machine-Parseable Schema):
{
  "timestamp": "2026-07-02T09:15:00Z",
  "level": "INFO",
  "event": "user_connected",
  "user_id": "98321",
  "client_ip": "192.168.1.50",
  "correlation_id": "req-01j1p5x7"
}
```

**Options:**

* **Option 1: Free-Text Serialization** Writing unconstrained string messages directly to standard output (stdout), appending variable parameters natively into the prose.
* **Option 2: Structured JSON Schemas** Enforcing a rigorous, uniform JSON schema across the entire application ecosystem, where every contextual variable is explicitly isolated into its own strongly typed field.

**Trade-offs:**

* **Free-Text Serialization:**
  * *Pros:* Zero runtime serialization overhead; extremely low cognitive friction for product engineers during feature writing.
  * *Cons:* Impossible to query efficiently at scale. Filtering logs based on specific variable thresholds (e.g., finding all failures where `user_id` matches a specific tenant) requires executing heavy full-text scans or computing expensive runtime regex extractions that saturate log cluster CPU.
* **Structured JSON Schemas:**
  * *Pros:* Turns logs into a queryable relational dataset. Centralized log aggregators (such as an OpenSearch or Elasticsearch/ELK pipeline) can immediately index specific keys, allowing engineers to filter billions of log records in milliseconds.
  * *Cons:* Introduces a non-zero CPU and memory allocation tax due to continuous string serialization and JSON map construction on the application execution path. Requires disciplined enforcement of field naming rules to prevent data type collisions.

**When to choose each option:**

* Choose **Structured JSON Schemas** universally for any production environment utilizing centralized log aggregation, multi-service architectures, or high-throughput transaction paths.
* Choose **Free-Text Serialization** exclusively for low-level, isolated command-line interface (CLI) utilities or standalone scripts where execution outputs are intended purely for immediate local human display on a terminal screen.

**Common failure modes:**

* *The Ingestion Schema Mapping Collision:* An application utilizes an Elasticsearch/ELK cluster for log aggregation. Team A deploys a service that logs a field named `context` as a plain string payload (`"context": "database_retry"`). Team B concurrently deploys a service that logs `context` as a nested JSON object (`"context": {"error_code": 500, "retry_count": 3}`). When the centralized log indexer encounters the conflicting structures for the identical key, it rejects the log payloads from Team B entirely, causing a silent drop of critical failure telemetry during a production incident.

**Example:** Every structured log entry generated within a microservice ecosystem must carry the unified **Correlation ID** introduced in Chapter 21. Even before full distributed tracing networks (Chapter 72) are deployed, embedding a consistent `correlation_id` key inside every structured JSON log block allows an on-call engineer to execute a single indexed search query against an ELK cluster. This query instantly reassembles the linear chronological history of a request across distinct thread boundaries and internal service calls, completely bypassing the confusion of concurrent text logs.

### [Decision: Intentional Ingestion Filtering vs. "Just-in-Case" Logging]

**What it is:** The architectural policy governing the volume of context recorded during ordinary operations—specifically choosing whether to explicitly drop verbose local context to minimize cost and security risk, versus capturing massive diagnostic volumes to guard against unpredicted edge-case failures.

**Why it exists:** Engineering teams often fall into the overcorrection trap of logging everything "just in case" an outage occurs. This mirrors the proxy-metric failures observed in documentation coverage targets: volume is falsely equated with system understanding. In high-volume systems, an unrestrained logging policy results in log ingestion costs that can scale exponentially relative to production traffic growth, often rivaling the financial cost of primary compute infrastructure. Furthermore, un-scrubbed verbose logs frequently leak protected user data, access keys, or personal identifiable information (PII) into secondary, less-audited storage backends, creating severe security and regulatory exposure.

**Options:**

* **Option 1: Comprehensive "Just-in-Case" Ingestion** Logging nearly all execution steps, trace variables, and peripheral payload contexts across all services, relying on massive storage layers to retain the data for future troubleshooting.
* **Option 2: Intentional Ingestion Filtering (Contextual Guardrails)** Restricting logging to highly structured boundary state metrics during standard operations, relying on dynamic runtime log-level configuration changes or targeted, explicit error-context wrapping to capture deep details only when an actual error occurs.

**Trade-offs:**

* **Comprehensive "Just-in-Case" Ingestion:**
  * *Pros:* Guarantees that if an obscure, non-reproducible race condition or memory state violation occurs, the historical forensic evidence is already captured in the central log engine.
  * *Cons:* Imposes an enormous financial and operational tax on infrastructure. Degrades query performance across the entire observability stack and significantly expands the system's security attack surface by recording transient memory states.
* **Intentional Ingestion Filtering:**
  * *Pros:* Minimizes data ingestion volumes and infrastructure costs. Ensures high-trust, high-velocity log queries by keeping the indexing footprint compact. Minimizes security leakage risks by applying tight code review filters over what keys are permitted to serialize.
  * *Cons:* Requires advanced observability tooling—such as dynamic runtime log-level manipulation or local ring-buffered log extraction—to pull granular data if a subtle, silent calculation bug slips through standard telemetry guardrails.

**When to choose each option:**

* Choose **Intentional Ingestion Filtering** for mature, high-traffic systems, multi-tenant enterprise platforms, and heavily regulated financial or medical backends where data security and cost predictability are hard constraints.
* Choose **Comprehensive Ingestion** strictly during early-stage alpha testing, greenfield system boot-ups, or deeply isolated, low-volume mission-critical pipelines where computing costs are trivial relative to immediate engineering discovery speed.

**Common failure modes:**

* *The Data Poisoning Cluster Wipeout:* A high-throughput service experiences an un-throttled, external denial-of-service (DoS) attack. Because the system follows a "Just-in-Case" logging policy, it records the complete request body of every incoming malformed payload to the log system. The ingestion volume spikes by 10,000%, saturating the disk IOPS of the underlying Elasticsearch indexer nodes. The central log cluster crashes under disk exhaustion, leaving the engineering team completely blind to the state of *all* services across the enterprise during the active attack.

**Example:** The catastrophic Log4Shell vulnerability (CVE-2021-44228) highlighted that logging libraries are active execution surfaces, not passive text utilities. Applications running the Apache Log4j library that blindly logged raw user input strings (such as an HTTP User-Agent string containing a JNDI injection payload) allowed attackers to execute remote code execution (RCE) vectors directly inside the application process context. This historical failure demonstrated that logging everything without rigorous validation, scrubbing, and ingestion filtering introduces immense architectural risk. The technical mechanics of handling credentials and sanitizing data structures at system boundaries are isolated within Part XI (Chapter 83's secrets management framework), but the logging layer must enforce the physical perimeter by refusing to process raw, unvalidated payload context.

## 5. Why Smart Engineers Disagree: The Continuous Production Debug Dispute

The structural boundary governing production log density exposes a fundamental alignment friction between software engineers optimizing for local diagnostic speed and Site Reliability Engineers (SREs) optimizing for infrastructure reliability, performance, and cost containment.

### The Verbose Diagnostics Faction

This cohort argues that disabling deep DEBUG level logging in production environments represents an anti-pattern born of artificial caution. They assert that modern microservice interactions fail in emergent, non-deterministic ways that can never be replicated in local testing environments or staging clusters. To them, attempting to diagnose an intermittent multi-service distributed deadlock without step-by-step state logs forces engineers to guess blindly, directly extending MTTR. They demand that DEBUG logs remain permanently active or be easily toggled on globally, viewing the accompanying infrastructure costs and storage bills as a necessary, justifiable cost of engineering clarity.

### The Cost and Performance Containment Faction

This cohort argues that continuous, high-volume production debugging is an architectural failure that violates foundational systems engineering discipline. They point out that at high scale—such as a system executing hundreds of thousands of operations per second—the mechanical tax of string allocation, concatenation, and synchronous or asynchronous disk and network I/O required to emit DEBUG telemetry visibly throttles application throughput, consumes localized CPU cycles, and inflates cloud ingestion metrics. They advocate for strict INFO or WARN default limits in production, insisting that if an engineering team requires a continuous trail of text statements to verify whether an application is functioning correctly, the system has failed the fundamental code design and modular encapsulation tests outlined in Part IV.

### The Reality

The disagreement is resolved by evaluating the structural placement of the service and its throughput realities.

* In low-volume, high-complexity orchestration engines (such as a multi-step enterprise billing settlement workflow), the financial and performance tax of deep context logging is negligible relative to the catastrophic business cost of an incorrect state calculation. Here, verbose logging is correct.
* Conversely, in a high-throughput, latency-critical data ingestion pipeline (such as a Kafka message ingest layer or a network routing gateway), emitting a text log line for every processed unit of data will immediately cripple performance and cause localized buffer exhaustion.

The pragmatic systems engineer manages this tension by decoupling log recording from log extraction. They maintain tight, production-filtered ingestion profiles by default, but design the architecture to support local, in-memory ring buffers that store transient DEBUG traces entirely in RAM. If, and only if, an explicit transaction error is encountered, the service flushes that local memory buffer to the central log aggregator, capturing pristine forensic context for that specific failure block while incurring zero ingestion cost for the millions of surrounding successful operations.
