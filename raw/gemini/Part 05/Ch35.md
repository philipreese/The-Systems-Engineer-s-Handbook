Chapter 35 — What Belongs at Each Layer
## 1. Purpose
This chapter transitions the automated testing framework from an abstract structural concept to an actionable architectural practice. While Chapter 34 established the geometric constraints of the testing pyramid, this chapter resolves the daily mechanical question faced by engineering teams: *For any given file, module, or behavioral flow, at which layer of the pyramid does its verification naturally belong?* Distributing assertions incorrectly carries severe structural penalties: testing pure logic at the integration layer unnecessarily inflates feedback loop latency, while testing infrastructure adapters using pure unit tests creates dangerous architectural blind spots. This chapter defines the exact engineering boundaries separating unit, integration, and end-to-end (E2E) test targets. It frames testability not as an afterthought or a compliance metric, but as a direct real-time diagnostic tool for measuring the structural health and decoupling efficiency of a system's code organization.
## 2. Prerequisites
This chapter assumes a comprehensive mastery of the structural and systemic mental models established in previous sections of this handbook. Readers must thoroughly understand:
 * **Complexity Classification (Chapter 02):** The strict differentiation between essential problem domain logic and accidental solution complexity.
 * **Information Hiding Mechanics (Chapter 04):** Parnas's core directive to insulate volatile engineering decisions behind stable boundaries.
 * **The Structural Taxonomy of Coupling (Chapter 03 & Chapter 13):** The operational hazards of afferent/efferent vectors and database-level connascence.
 * **The Hexagonal Architectural Pattern (Chapter 11):** The explicit physical isolation of a business domain ("Core") from delivery mechanisms and persistent storage engines via interfaces ("Ports") and implementations ("Adapters").
## 3. Key Concepts
 * **The Determinism Rule:** The architectural requirement that a unit test must yield an identical result across infinite consecutive runs on any computing environment, isolated from system state, current time, or local file layout.
 * **The Architectural Seam:** A design-level boundary—most frequently manifested as an abstract interface or a port—that acts as a clean dividing line where unit test suites stop and integration test suites begin.
 * **The Testability Diagnostic:** The direct correlation between a module’s internal design quality and the difficulty of its testing setup. If a function is hard to unit-test, it is mathematically coupled to volatile dependencies.
 * **Package-by-Feature Co-location:** An organizational philosophy dictating that testing code must be managed as an immediate neighbor to the production code it exercises, preventing structural drift between feature evolution and verification assets.
## Decision / Principle Format
### [The Unit Layer Boundary (Pure Logic and Determinism)]
**What it is:**
The strict confinement of low-level unit testing to stateless, I/O-free components including pure data transformations, core domain entities, validation routines, and algorithmic business rules.
**Why it exists:**
To protect the development feedback loop by guaranteeing sub-millisecond assertion execution and absolute localization precision. By ensuring the base of the pyramid contains zero dependencies on external state, compilers and local runners can execute thousands of checks instantly, immediately pointing an engineer to the exact file and line number causing a logical regression.
**Options:**
 1. *Pure-Domain Isolation:* Confining unit tests strictly to plain language primitives and stateless domain models that reject any framework, network, or persistent storage dependencies.
 2. *Framework-Entangled Unit Isolation:* Writing tests that assert business rules while relying on localized framework test runners, embedded context engines, or base controller configurations to manage object lifetimes.
**Trade-offs:**
 * *Pure-Domain Isolation:* Delivers unmatched execution speed and eliminates vendor lock-in. The domain core can survive major framework upgrades (e.g., migrating from Spring Boot to Quarkus, or rewriting a Django engine) entirely without rewriting the test suite. However, it demands rigorous architectural discipline upfront to separate business logic from structural framework conveniences.
 * *Framework-Entangled Unit Isolation:* Accelerates day-one development velocity by allowing engineers to write validation rules directly within framework-provided models or database entities. However, it introduces significant bootstrapping latency taxes into every local execution loop and turns the unit test suite into a fragile dependency vulnerable to framework configuration changes.
**When to choose each option:**
 * Choose *Pure-Domain Isolation* universally for all core business systems whose domain rules are complex, long-lived, and define the primary asset value of the enterprise (e.g., ledger calculation engines, pricing matrices, parsing libraries).
 * Choose *Framework-Entangled Unit Isolation* strictly for trivial CRUD utilities, single-purpose data-entry forms, or short-lived experimental prototypes where the business logic is entirely secondary to rapid transport delivery.
**Common failure modes:**
 * *The Hidden I/O Leak:* An engineer attempts to unit-test an validation utility, but that function internally invokes a method that reads from an environment variable file, queries a local timezone database, or checks a local directory. The test runs perfectly on the author's machine but breaks unpredictably when executed within an un-provisioned container in a CI runner or when run in a different time zone.
**Example:**
In a Hexagonal architecture, a business requirement dictates that an international billing invoice must compute compounded regional VAT dynamically. The code responsible for this calculation lives exclusively within a pure domain module. Its corresponding unit tests pass raw floats and custom domain values into the method and assert the output. The test file imports zero network, HTTP, or SQL framework elements, guaranteeing execution remains bound entirely to process memory space.
### [The Integration Layer Boundary (I/O, Adapters, and Contracts)]
**What it is:**
The deliberate application of testing sweeps across a process or database engine boundary to verify the correctness of data persistence mapping, raw SQL queries, cross-component collaboration paths, and infrastructure adapter behaviors.
**Why it exists:**
Unit tests are structurally incapable of verifying that an advanced SQL index behaves correctly under real workloads, that a database schema migration matches the code’s ORM definitions, or that a network client respects a specific third-party HTTP header payload contract. The integration layer exists to validate the *seams* and *translations* of the architecture where code directly encounters physical hardware or external systems.
**Options:**
 1. *Real Infrastructure Integration:* Executing integration assertions directly against localized, production-identical instances of the actual dependency (e.g., testing database repositories against a transient PostgreSQL or Redis instance running via container virtualization).
 2. *In-Memory Emulation:* Running integration tests against an in-memory language stub or simplified embedded driver (e.g., using an in-memory H2 database to emulate an external PostgreSQL engine).
**Trade-offs:**
 * *Real Infrastructure Integration:* Provides maximum fidelity and prevents silent production crashes. It accurately surfaces vendor-specific runtime anomalies, edge cases, and connection failures. However, it levies a noticeable latency tax on the test suite, requires a local Docker or orchestration daemon, and complicates parallel test execution due to shared database state constraints.
 * *In-Memory Emulation:* Executes significantly faster than fully containerized infrastructure and scales horizontally across local development boxes without external configuration dependencies. However, it introduces severe architectural risk due to dialect divergence. An SQL query containing precise locking behaviors or optimized vendor functions may execute cleanly inside an in-memory mock engine but trigger a syntax error or a deadlock when deployed to a real production database.
**When to choose each option:**
 * Choose *Real Infrastructure Integration* for all production database adapters, transactional repositories, messaging brokers (like Apache Kafka), and any operation relying on specific vendor performance primitives (like PostgreSQL JSONB structures).
 * Choose *In-Memory Emulation* exclusively for structural data caching strategies where the data access model is mathematically uniform and possesses near-zero structural divergence from standard language collection memory types.
**Common failure modes:**
 * *The Mocked Database Illusion:* A development team tests its primary database adapter layer by mocking out the database driver client. The tests run blazingly fast and pass perfectly in CI. Upon deployment to production, the application instantly crashes with severe SQL syntax exceptions because a database column was renamed in a migration script but the mocked test object continued to accept the obsolete schema field without complaining.
**Example:**
An application's billing module features a outbound database adapter that writes financial records to a table via a Port contract. The integration test suite exercises the concrete PostgresBillingAdapter by spinning up a production-identical PostgreSQL engine via Testcontainers. The test executes a live insertion query, reads the on-disk state directly, and verifies that relational foreign keys and database constraints are properly enforced, bypassing mocks entirely.
### [The End-to-End (E2E) Layer Boundary (System Orchestration)]
**What it is:**
The minimal deployment of automated verification scripts that exercise the complete full-stack topology of the system—from the external client interface down to persistent state engines—to validate critical, non-negotiable user journeys.
**Why it exists:**
To provide a macro-level smoke safety net over the entire operational architecture. While lower tiers confirm that code logic and adapters match their local specs, they cannot detect when the system breaks due to a bad CDN caching rule, an invalid API gateway route configuration, a broken load balancer SSL handshake, or a missing deployment environment variable.
**Options:**
 1. *Critical Journey Guardrails:* Limiting the E2E automation sweep strictly to a small, high-value selection of transactional paths that define the core business capability (e.g., user signup, successful check-out, core data sync).
 2. *Comprehensive Boundary Regression:* Attempting to duplicate all business logic verification, field input validations, error message variations, and secondary button states at the browser or API gateway layer.
**Trade-offs:**
 * *Critical Journey Guardrails:* Preserves deployment velocity, maintains a reliable and trustworthy CI signal, and keeps test maintenance costs low by remaining insulated from superficial frontend aesthetic changes. However, it relies on lower-level unit and integration suites to capture minor localized feature regressions before code hits the main environment.
 * *Comprehensive Boundary Regression:* Theoretically provides total verification that the visible interface behaves correctly under every conceivable user input combination. Practically, it creates an unmaintainable, exceptionally slow, non-deterministic test pipeline that rapidly degrades into the "Ice Cream Cone" anti-pattern, forcing engineers to waste hours debugging brittle selector failures.
**When to choose each option:**
 * Choose *Critical Journey Guardrails* as the absolute maximum boundary constraint for all public-facing SaaS applications, enterprise distributed web topologies, and single-page client applications.
 * Never choose *Comprehensive Boundary Regression* at the E2E tier. It is an expensive architectural misallocation of testing resources that shifts local logic checks to the most complex operational environment possible.
**Common failure modes:**
 * *The Brittle DOM Selector Blockade:* An engineer writes an E2E test to verify that a descriptive error message appears when an invalid coupon code is entered. A frontend developer later moves the error component to a side modal or updates a CSS class selector name for a styling layout change. The underlying API, database, and business validation code continue to function perfectly, but the E2E test crashes in CI, halting a critical deployment pipeline for hours to fix a superficial locator string.
**Example:**
A large distributed e-commerce architecture restricts its E2E test suite to exactly three targeted user scripts executed by a headless browser engine against a fully staged cluster environment: verifyUserCanRegister(), verifyUserCanSearchAndCart(), and verifyUserCanCompleteCheckout(). These scripts assert only broad success states (e.g., reaching a /receipt route). Every edge case variation—such as expired coupons, alphanumeric string length limits, or decimal rounding rules—is intentionally pushed down to unit tests in the domain layer.
### [Test File Location Topology (Co-location vs. Parallel Trees)]
**What it is:**
The structural organization of testing assets within a source code repository—either placing the test file as an immediate neighbor inside the same folder as its target production file (Co-location) or isolating all test files within a mirrored, distinct directory hierarchy (Parallel Trees).
**Why it exists:**
File layout directly governs a developer’s daily friction and cognitive overhead. Proximity shapes behavior: package structure enforces team alignment with modular boundaries, affects information hiding visibility, and directly dictates whether engineers actively maintain corresponding verification assets during feature refactoring.
**Options:**
 1. *Structural Co-location:* Organizing files so that testing code sits immediately next to the production component it asserts within the same package directory (e.g., user.go and user_test.go live in /internal/user).
 2. *Parallel Test Trees:* Segregating all automated test files into a dedicated top-level folder tree completely removed from production modules (e.g., production lives in /src and all test code lives in /tests).
**Trade-offs:**
 * *Structural Co-location:* Enhances discoverability, visibility, and supports package-by-feature cohesion. It encourages engineers to treat test code as an intrinsic element of the feature rather than an optional secondary artifact. In certain language environments, it allows unit tests to evaluate package-private functions directly without forcing the public exposure of internal details to the rest of the application. However, it mixes distinct testing assets directly with operational code inside the file navigator and can complicate simple packaging configurations if the toolchain cannot naturally exclude test files during compilation.
 * *Parallel Test Trees:* Retains an entirely clean production directory structure, simplifying basic compilation script paths and strictly treating the application as a black-box consumer. However, it injects constant cognitive friction into daily workflows; engineers must continuously traverse separate, deep folder branches to keep code and assertions synchronized. It also restricts unit tests to asserting functionality through the module's public surface area, which frequently tempts developers to break information hiding principles by exposing private module mechanics simply to make them accessible to the remote test tree.
**When to choose each option:**
 * Choose *Structural Co-location* as the default baseline standard for software systems built in compiled languages with native engineering toolchains that explicitly filter test assets (such as Go, Rust, or modern modular TypeScript setups).
 * Choose *Parallel Test Trees* strictly when required by the structural defaults of legacy dynamic frameworks or tools (such as native Python pytest discovery patterns) that expect a centralized testing root directory to manage external environment overrides.
**Common failure modes:**
 * *The Ghost Module Regression:* An engineering team utilizing a Parallel Test Tree pattern executes an extensive structural refactoring, moving or deleting an old production module. Because the test folder lives entirely across a separate directory branch, the developer fails to notice or clean up the corresponding mirrored test file. The obsolete tests are left orphaned in the repository, either rotting silently or introducing unpredictable compilation and CI breakages weeks later during unrelated changes.
**Example:**
In a Go application designed around package-by-feature principles, a developer navigates to the /internal/shipping directory. The folder contains shipping.go and its immediate neighbor shipping_test.go.
To assert strict black-box abstraction, the engineer declares package shipping_test inside the test file, ensuring the test runner cannot inspect un-exported package structures. If they must verify a volatile, internal multi-stage parsing algorithm without leaking it to the public API surface, they switch the file to package shipping, keeping the localized decision cleanly hidden within the module boundary.
## Testability as an Architectural Design Diagnostic
A fundamental tenet of systems engineering is that code testability is an immediate indicator of design quality. When a piece of business logic is exceptionally difficult to unit test—requiring dozens of mocked objects, deep nesting of framework configurations, or elaborate setup phases simply to evaluate a single validation branch—it is a clear diagnostic signal that the production architecture has decayed.
This friction reveals a failure of information hiding and modular cohesion. The code is mixing volatile infrastructural concerns (like formatting an SQL transaction or parsing an incoming HTTP header) directly into the execution path of core business calculations.
The Hexagonal architecture resolves this problem structurally by drawing a strict physical line through the code. The domain logic resides securely at the center of the architecture, entirely insulated from external frameworks; it can be fully verified using fast, lightweight unit tests. The external database adapters and network clients are pushed to the outermost perimeter, where they are exercised using targeted integration tests against real infrastructure container instances.
The interfaces (Ports) serve as the clear architectural seams where these two test domains meet, ensuring that the structure of your automated test suite mirrors the decoupled design of your application.
## Why Smart Engineers Disagree: The Public Surface Area vs. Structural Isolation Debate
One of the most persistent tactical disagreements among senior systems architects centers on whether automated unit tests should interact exclusively with a component’s public API surface or target internal, un-exported modules directly.
Engineers advocating for *Strict Public Surface Testing* (often refactoring classicists) argue that unit tests must treat a module as a complete black box. They demand that assertions only invoke the outermost public entry points of a feature package. They argue that testing internal helper methods or private functions couples the test suite to the transient implementation details of the code.
