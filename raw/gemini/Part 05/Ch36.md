Chapter 36 — When to Mock vs. Use Real Dependencies
## 1. Purpose
This chapter addresses the most polarized tactical split in automated testing: whether to isolate a code unit by replacing its collaborators with synthetic substitutes or to execute it alongside real, production-grade dependencies. This choice is not an aesthetic or stylistic preference; it is a fundamental system optimization choice that directly governs test suite determinism, execution latency, and resistance to refactoring rot.
Choosing the wrong isolation strategy introduces heavy accidental complexity. Over-mocking transforms a test suite into a brittle mirror of current implementation details, destroying its utility during structural refactorings. Conversely, an absolute rejection of test doubles introduces non-determinism, network taxes, and brittle environment configurations into the local engineering feedback loop. This chapter establishes a rigorous framework for identifying when to deploy a test double and when to enforce the use of real dependencies.

## 3. Key Concepts
 * **The Test Double:** The formal overarching taxonomy for any synthetic object, function, or service injected into a system under test to take the place of a real production collaborator.
 * **Refactoring Fragility:** A pathological state where an automated test suite breaks during internal code modifications (such as splitting a class or renaming a private method) despite the system's external behavioral contract remaining perfectly intact.
 * **Interaction Verification:** A testing paradigm that assets correctness by confirming *how* a module invokes its neighbors (methods called, arguments passed, execution sequence).
 * **State Verification:** A testing paradigm that asserts correctness by inspecting the final *value*, memory state, or persistent output of a system after an operation concludes.
 * **The Cost Threshold of Realism:** The evolving economic and operational calculation determining whether spinning up a real piece of infrastructure (like a database or cache) is cheaper than maintaining a simulated model of its behavior.
## Decision / Principle Format
### [The Test Double Taxonomy]
**What it is:**
The precise structural classification of test dependencies as defined by Martin Fowler in his canonical 2004 reference *"Mocks Aren't Stubs."* It separates synthetic substitutes into five distinct mechanical types based on how they process inputs and record internal state: Dummies, Stubs, Spies, Mocks, and Fakes.
**Why it exists:**
Engineering teams routinely introduce high cognitive overhead and communication failures by using the word "mock" as a loose catch-all for any test substitute. Because different doubles enforce fundamentally different verification behaviors (state versus interaction), using them interchangeably obscures the true intent of a test file and leads to poor test design.
**Options:**
 1. *Dummy:* An object passed into a method signature purely to satisfy compiler type requirements (e.g., an empty configuration struct or an unused logger reference). It is never invoked and holds no logic.
 2. *Stub:* A component that provides pre-programmed, canned answers to specific method invocations during execution. It contains no internal state tracking and does not assert how or if it was called.
 3. *Spy:* An enhanced stub that records the chronological history of its internal interactions—such as tracking the number of times a method was executed or capturing the specific argument structures passed to it—for later evaluation.
 4. *Mock:* A specialized object pre-programmed with strict behavioral expectations that form a mandatory specification of the calls it must receive. If the code deviates from the mock's expected call sequence or method signatures, the mock automatically fails the test.
 5. *Fake:* A lightweight, fully operational implementation of a port interface that utilizes an alternative shortcut mechanism unsuitable for production concurrency or durability (e.g., an in-memory repository or a local SMTP server).
**Trade-offs:**
 * *Stubs and Fakes (State-Oriented):* Support clean state verification. They allow the system under test to execute naturally, ensuring tests remain completely insulated from minor implementation variations. However, fakes require manual design and maintenance upkeep to guarantee their logic does not drift from production reality.
 * *Mocks and Spies (Interaction-Oriented):* Enable precise verification of side effects and outbound communications that yield no direct return values (e.g., verifying that an asynchronous notification event was dispatched). However, they permanently lock the test suite into the structural layout of the source code, maximizing refactoring fragility.
**When to choose each option:**
 * Choose *Dummies and Stubs* when a component requires read-only configuration values or static data structures from a collaborator to execute its immediate calculation path.
 * Choose *Fakes* for persistent interfaces (such as storage repositories) within unit testing layers to sustain rapid, repeatable state validations without infrastructure overhead.
 * Choose *Mocks and Spies* strictly when validating outbound side effects across separate architectural boundaries where interaction tracking is the specific behavior under review.
**Common failure modes:**
 * *The Brittle Expectation Lock:* An engineer builds a test using a mock framework and configures strict expectations requiring method fetchData() to be called exactly twice with specific parameter strings. A later optimization combines these calls into a single batch query. The output of the component remains identical, but the mock framework throws a fatal assertion failure, forcing the engineer to rewrite the test code to accommodate a harmless implementation shift.
**Example:**
In a billing engine designed around a Hexagonal structure, a test must verify an invoice processor. Instead of initializing a live database connection to fetch customer records, the developer uses a *Stub* that returns a fixed PremiumCustomer payload whenever an ID is requested. To verify that an email notification is issued upon successful invoicing, they inject a *Spy* that captures the outgoing payload metadata for state assertion at the conclusion of the test loop.
### [Isolation Methodology (Detroit vs. London Schools)]
**What it is:**
The structural architectural choice between evaluating an application module alongside its real, live collaborators (the Classicist or Detroit school of thought) or isolating the system under test completely by replacing all surrounding dependencies with mocks (the Mockist or London school of thought).
**Why it exists:**
This division represents a fundamental divergence in how engineers define a "unit" within unit testing. The London school treats a unit as a single class or source file, utilizing mocks to decouple it from all neighbors. The Detroit school treats a unit as an isolated behavioral capability, happily executing multiple real files and classes simultaneously as long as they run entirely within a single memory space.
**Options:**
 1. *The Classicist (Detroit) Strategy:* Prioritizing the use of real objects and components for all internal system collaborators. Test doubles are strictly reserved for resources that cross volatile macro boundaries external to the immediate application context (such as third-party APIs, real hardware clocks, or network sockets).
 2. *The Mockist (London) Strategy:* Systematically replacing every downstream collaborator class or module with a mock framework interface. This focuses unit testing exclusively on the local method mechanics and interaction protocols of the single file under review.
**Trade-offs:**
 * *The Classicist Strategy:* Verifies real production outcomes and provides deep architectural confidence. Because tests focus on behavior rather than file boundaries, internal code modules can be completely restructured, merged, or renamed without breaking a single assertion. However, if a low-level utility contains a hidden bug, it will cause cascading failures across dozens of unrelated tests, reducing localized failure precision.
 * *The Mockist Strategy:* Delivers absolute failure isolation. If a test fails, the defect resides definitively within the single file under test. It also forces engineers to think deeply about interface design and call interactions during initial development. However, it creates an incredibly high refactoring tax; the test suite acts as an rigid carbon copy of the internal code architecture, heavily penalizing ongoing maintenance velocity.
**When to choose each option:**
 * Choose the *Classicist Strategy* as the default operational model for core business systems, domain modules, and complex algorithmic paths where internal code layouts are expected to evolve dynamically over time.
 * Choose the *Mockist Strategy* strictly when designing high-level workflow orchestrators, routing matrices, or microservice interaction boundaries where the *sequence of coordination* itself constitutes the essential value of the component.
**Common failure modes:**
 * *The Green Mirage:* A mock-heavy test suite features thousands of isolated specifications that pass flawlessly in under three seconds on every CI execution run. However, the system consistently crashes upon production deployment. Because every collaborator was replaced by a mock, the tests were validating assumptions about how components interacted rather than how they physically operated. The mocks were configured to return values that the real production dependencies were structurally incapable of producing.
**Example:**
Consider a user registration service that invokes a password hasher, a database repository, and a messaging publisher. A *Mockist* test replaces all three modules with mocks, verifying only that hash() and save() are triggered in sequential order. A *Classicist* test initializes the real password hashing algorithm and utilizes an in-memory repository fake, confirming that a valid, securely hashed record is physically committed to the working state store.
### [The Realism Cost Threshold (Virtualization and Fakes)]
**What it is:**
The strategic selection framework governing whether an infrastructure dependency (such as a persistent database, caching engine, or message queue) should be simulated via an in-memory fake or run natively using ephemeral container virtualization tools like Testcontainers.
**Why it exists:**
Historically, the financial cost, initialization latency, and environment volatility of standing up heavy infrastructure instances forced engineers to rely heavily on database mocking. The rapid evolution of lightweight container orchestration (Docker, Testcontainers) in modern environments (Go, Java, Python) has significantly lowered the cost threshold of realism, shifting the optimal balance point between synthetic fakes and native engines.
**Options:**
 1. *The In-Memory Fake Architecture:* Developing and maintaining a lightweight, stateful language substitute that implements the same interface boundary as the production adapter (e.g., a map-backed repository struct).
 2. *Containerized Infrastructure Integration (Testcontainers):* Leveraging automated lifecycle containers to spin up clean, production-identical instances of the actual dependency (like PostgreSQL or Redis) directly inside the execution sweep of the test runner.
**Trade-offs:**
 * *The In-Memory Fake:* Runs entirely in-process in microseconds, requires zero external container runtimes, and remains completely deterministic across all local development environments. However, it can leak abstractions by hiding real vendor realities. It fails to catch syntax errors in raw SQL strings, structural errors in database migration assets, or locking deadlocks.
 * *Containerized Infrastructure:* Guarantees 100% production fidelity. It catches real configuration flaws, unique constraint violations, and specialized dialect features (such as PostgreSQL JSONB or Redis transactions) before code hits staging. However, it injects a noticeable initialization latency tax (seconds per container lifecycle), demands a local container daemon, and requires defensive isolation to prevent parallel test threads from clobbering shared table states.
**When to choose each option:**
 * Choose *In-Memory Fakes* within low-level domain unit testing layers where the interface boundary is stable, and the objective is to thoroughly evaluate high-velocity business rule variations without triggering I/O latency.
 * Choose *Containerized Infrastructure* for adapter boundaries, repository classes, migration assets, and integration testing sweeps where the structural correctness of the query or infrastructure configuration is the explicit risk under validation.
**Common failure modes:**
 * *Dialect Drift:* A development team relies exclusively on an in-memory SQLite configuration to test an application whose production engine is PostgreSQL. An engineer writes a specialized analytical query utilizing advanced Postgres window functions. The tests pass cleanly on local development boxes against SQLite, but the system immediately encounters fatal runtime syntax execution crashes the moment it is deployed to production.
**Example:**
An engineering team utilizes a two-pronged strategy for an order fulfillment system. For rapid local unit sweeps of the core domain state machine, they pass an in-memory repository fake that stores records inside a thread-safe map structure. For the persistence adapter tier, they reject abstract mocking frameworks completely and utilize Testcontainers to launch a real, transient PostgreSQL instance in CI, ensuring every database trigger and index optimization is validated natively.
## The Handbook Position on Dependency Realism
This handbook establishes a definitive position on the realism axis: **Prefer real or near-real dependencies wherever the physical and operational cost is acceptable.** Automated tests derive their primary economic value from their ability to provide an authentic safety net during code modification.
Every test double injected into a codebase represents an unverified assumption—a synthetic contract written by a human that can silently drift from production reality. Consequently, you must systematically push the boundaries of your test execution to incorporate real dependencies, leveraging the low latency costs provided by container virtualization tools like Testcontainers.
```
       [ HIGH REALISM CORE ]
     - In-Memory Fakes for Storage
     - Ephemeral Virtualized Containers (PostgreSQL, Redis)
     - Real Inter-Module Collaborators
               |
               |  <-- The Cost Threshold Boundary
               V
       [ TEST DOUBLES ONLY FOR: ]
     - Non-Deterministic Inputs (System Clock, Crypto Random)
     - Prohibitively Expensive Assets (Mainframes, Paid SaaS APIs)
     - Uncontrollable Volatility (Third-Party Network Ingress)

```
You must explicitly restrict the deployment of test doubles to three clear systemic scenarios where using real dependencies becomes physically or economically impractical:
 1. *External Control Boundaries:* Components managed by third-party vendor systems outside your organization's physical network (such as Stripe payment processing, Twilio SMS gateways, or external cloud provider services).
 2. *Uncontrollable Non-Determinism:* Internal language or hardware operations that introduce unpredictable state paths into a test execution loop, specifically including the system clock, random number generation utilities, or ambient network latency.
 3. *Prohibitive Resource Economics:* Enterprise infrastructure assets whose operational footprint makes them impossible to initialize locally or whose licensing models impose unacceptable financial constraints (such as legacy corporate mainframes, proprietary data lakes, or heavily rate-limited external enterprise systems).
## Why Smart Engineers Disagree: Interaction vs. Outcome Verification
The fiercest architectural debates regarding test dependency management center on whether a test suite's primary role is to serve as an active design tool for internal interfaces or as an unbiased validator of system outcomes.
Engineers aligning with the London (Mockist) school of thought argue that outcome-based testing allows software design to degenerate. They view the widespread deployment of mocks not as a maintenance chore, but as a crucial diagnostic tool. If a component requires ten independent mocks to be configured before a test can run, the mock framework is providing immediate, loud feedback that the component violates the Single Responsibility Principle and suffers from excessive efferent coupling. They argue that classicist testing masks poor design choices by allowing highly entangled, poorly abstract modules to pass verification as long as the global outcome happens to be correct. To them, the test suite is an essential blueprinting engine for system architecture.
Engineers aligning with the Detroit (Classicist) school view this design feedback loop as an incredibly expensive illusion. They argue that an automated test suite has exactly one primary objective: to facilitate fearless, high-velocity refactoring. If a test suite is built upon interaction verification, it permanently couples the assertions to the precise structural syntax of the current code layout. The moment an engineer modifies an internal interface, extracts a helper method, or optimizes an execution loop, dozens of mock expectations collapse, despite the outward-facing business behavior remaining perfectly stable. They contend that a test suite that requires simultaneous updates every time code is refactored is an architectural failure that actively slows down engineering velocity and encourages developers to tolerate technical debt rather than fight it.
**The Reality:**
This conflict represents an optimization mismatch between the *rate of architectural design iteration* and the *long-term maintenance lifespan* of the software system.
During the initial design phase of a highly uncertain, deeply coordinated system boundary, adopting a mockist strategy can accelerate the discovery of optimal component protocols and interface boundaries.
However, once an application boundary stabilizes and enters its long-term operational lifecycle, maintaining an interaction-heavy suite imposes an unsustainable accidental complexity tax. The pragmatic systems engineer manages this continuum by prioritizing outcome-driven, classicist verification with real or fake dependencies as the baseline standard for the application, ensuring the team buys long-term refactoring autonomy rather than short-term design conformity.
