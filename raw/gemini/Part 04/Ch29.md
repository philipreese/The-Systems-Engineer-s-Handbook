Chapter 29 — When to Split Files vs. Keep Together
 1. Purpose
   This chapter governs the physical granularity of code within a defined module. While Chapter 27 established the macro-level organization of packages and directories, this chapter addresses the micro-level decision: should a specific class, type, or concept live in its own isolated file, or alongside closely related code? This is an application of cohesion at the smallest practical scale. It dictates that file splitting should be driven by concrete evidence of divergent concerns, not preemptive organizational aesthetics.
 2. Prerequisites
   Assumes mastery of the foundational constraints from Part I. Specifically, readers must understand cohesion (Part I, Ch 03) and the Rule of Three governing premature abstraction (Part I, Ch 04). This chapter assumes a module boundary already exists (Part IV, Ch 27). It does not cover naming conventions (expanded in Part IV, Ch 28) or whether splitting code constitutes a new systemic abstraction (expanded in Part IV, Ch 31). Test strategies mentioned here are expanded upon fully in Part V (Testing Strategy).
 3. Key Concepts
 * The File as a Cognitive Unit: A file is not just a compiler construct; it is the physical boundary of a developer's working memory. Code that must be understood together should ideally be read together.
 * Premature Fragmentation: Splitting a file based on an arbitrary line-count limit before distinct, separable concerns have actually emerged.
 * The Rule of Three (Granular Application): Do not extract a concept into its own file preemptively on a guess. Wait until the code naturally bifurcates into distinct, unrelated concerns before introducing the indirection of a new file.
[Granularity Paradigm: Singular Entity vs. Conceptual Co-location]
What it is:
The baseline convention for how source code is distributed across files within a module.
Why it exists:
To establish predictable navigation. Engineers need a reliable mental model for locating the implementation of a given identifier. The paradigm dictates whether the file system acts as an exact index of classes, or as an index of holistic concepts.
Options:
 1. One Class per File: Every class, struct, or interface is strictly isolated into its own dedicated file, named exactly after the entity it contains.
 2. Conceptual Co-location (File as a Unit of Related Concepts): A single file contains multiple distinct types, functions, and interfaces that collectively implement one cohesive business requirement or mechanism.
Trade-offs:
 * **One Class per File:** Makes finding a specific entity trivial via file-tree navigation. However, it destroys reading flow. For a small utility class that is exclusively consumed by a parent class, isolating it forces the developer to open multiple tabs to understand a single, simple operation.
 * **Conceptual Co-location:** Preserves high cohesion and reading flow. A developer can understand the entire lifecycle of a domain concept by scrolling top-to-bottom. However, it risks degrading into a monolithic dump if engineers lack the discipline to stop adding tangentially related logic to the file.
When to choose each option:
 * Choose **One Class per File** when enforced by the language ecosystem (e.g., Java's public class mandate) or when dealing with massive, standalone domain aggregates that have no strict mutual dependency on sibling types.
 * Choose **Conceptual Co-location** universally in modern languages (Go, Python, JavaScript/TypeScript) for small, tightly coupled types, helper functions, and data transfer objects that have no distinct identity outside their immediate context.
Common failure modes:
 * **The "God File":** The canonical failure case of conceptual co-location. A file named user_processing.py begins with cohesive user-creation logic but gradually accumulates unrelated password-hashing utilities, email formatting, and database connection pooling over several years. It becomes an unnavigable 5,000-line monolith of low-cohesion code.
Example:
Idiomatic Go explicitly champions Conceptual Co-location. It is standard practice to declare an interface, the struct that implements it, and the small helper types it relies on within the same domain.go file. Conversely, Java historically enforced the One Class per File rule, which often resulted in directories cluttered with dozens of tiny, single-method classes (e.g., UserActionFactory.java, UserActionBuilder.java), massively increasing the indirection tax for simple workflows.
[Signals for Splitting vs. Keeping Together]
What it is:
The operational heuristic used to decide whether an existing file has accumulated enough independent concepts to warrant a physical split.
Why it exists:
As software evolves, files grow. Applying the Rule of Three to file splitting prevents the introduction of unnecessary navigational indirection until the evidence of low cohesion is undeniable.
Options:
 1. Keep Together: Retaining growing logic within a single file.
 2. Split by Concern: Extracting a subset of the logic into a new, sibling file within the same module.
Trade-offs:
 * **Keep Together:** Avoids the cognitive penalty of context-switching between tabs. However, as the file grows, visual noise increases. Modifying the file forces the developer to scroll past hundreds of lines of irrelevant logic to find the target function.
 * **Split by Concern:** Physically isolates distinct concerns, restoring scan-ability. However, if executed prematurely, it obscures the holistic relationships between mutually dependent types, forcing the developer to mentally reconstruct the execution path across multiple files.
When to choose each option:
 * Choose **Keep Together** when dealing with a tightly coupled state machine whose transitions lose all contextual meaning if separated, or when types are strictly mutually dependent and only make sense when read side-by-side.
 * Choose **Split by Concern** when modifying one feature within the file consistently requires ignoring the rest of the file. If you have to use a text-search tool to jump between two related functions because they are separated by 800 lines of unrelated parsing logic, the file lacks cohesion and must be split.
Common failure modes:
 * **The Fragmented State Machine:** An engineer dogmatically splits code to keep files under 100 lines. A cohesive 300-line state machine is shattered across six files. Understanding a single state transition now requires opening six tabs and tracing imports, artificially introducing high afferent and efferent coupling at the file level.
Example:
If an auth_middleware.go file contains the logic for intercepting an HTTP request, checking a JWT, and returning a 401 Unauthorized, that code must be kept together. If that same file grows to include a custom base64 decoding algorithm and an RSA key-rotation cron job, those concerns must be split into jwt_decoder.go and key_rotation.go. They are separate conceptual axes of variation.
[Test Co-Location]
What it is:
The decision of where to physically store the automated tests that verify a specific source file's behavior.
Why it exists:
Tests are the primary consumer of a file's contract. The physical proximity of the test file directly influences the likelihood of the test being maintained and understood alongside the source logic.
Options:
 1. Adjacent Co-location: The test file lives in the exact same directory, immediately next to the source file (e.g., billing.js and billing.test.js).
 2. Parallel Test Tree: Tests live in a completely separate root directory that mirrors the structure of the source code (e.g., src/domain/billing.java and test/domain/billingTest.java).
Trade-offs:
 * **Adjacent Co-location:** Maximizes physical cohesion. The test acts as immediate, adjacent documentation for the source file. You cannot modify the source without staring at the test file. However, it requires modern build tooling to successfully strip the test artifacts out of the final production binary.
 * **Parallel Test Tree:** Keeps the production source directories pristine and guarantees that test code never ships to production. However, it introduces physical distance. A developer modifying a deeply nested source file must manually navigate to a completely different directory tree to update the corresponding tests.
When to choose each option:
 * Choose **Adjacent Co-location** universally for unit tests in modern frontend ecosystems, Node.js, Go, and Rust. It is a deliberate cohesion-first choice. (Deep testing strategy is deferred to Part V).
 * Choose **Parallel Test Tree** only when dictated by legacy build systems (e.g., Maven/Gradle in Java) or when writing large-scale integration/E2E tests that span dozens of source files and have no single "adjacent" owner.
Common failure modes:
 * **The Orphaned Test:** In a parallel test tree, a developer refactors the src/ directory, moving and renaming several files. They forget to execute the exact same structural changes in the test/ directory. The tests become disjointed, orphaned scripts that no longer clearly map to the production architecture.
Example:
The React and Vue frontend ecosystems heavily mandate Adjacent Co-location. A specific UI concept (a component) is stored in a single folder containing Button.tsx (logic/markup), Button.css (styles), and Button.test.tsx (verification). This is a deliberate rejection of the older, layered convention of placing all markup in one folder, all CSS in another, and all tests in a third. It prioritizes the cohesion of the concept over the technical type of the files.
Why Smart Engineers Disagree: The Cost of Scrolling vs. The Cost of Tabs
The most frequent argument regarding file granularity pits engineers who hate scrolling against engineers who hate context switching. Engineers optimizing for visual brevity will argue that any file over 200 lines is a failure. They actively champion hyper-fragmentation, insisting that small files force developers to adhere to the Single Responsibility Principle. To them, scrolling is a sign of an overgrown monolith.
Engineers optimizing for cognitive continuity view file fragmentation as an unmitigated disaster. They argue that the human brain operates like a CPU stack; every time an engineer has to switch to a new tab to find the definition of a helper function, they risk a cache miss. They will happily accept a 2,000-line file if it means the entire narrative of the operation is contained in one linear scroll.
The Reality: The systems engineer recognizes that scrolling and tab-switching are both taxes, but they scale differently. The cognitive cost of scrolling is linear; the cognitive cost of tab-switching is exponential because it shatters the developer's execution context.
The deciding factor is never the raw line count—it is the density of cohesion. If a 1,500-line file is highly cohesive (e.g., a massive switch statement for a protocol parser), splitting it into ten files of 150 lines introduces catastrophic accidental complexity. However, if scrolling feels like searching through a junk drawer, the file lacks cohesion. You split files to isolate volatile, unrelated concerns, never simply because you are tired of scrolling.
