Chapter 28 — Naming Conventions and When They Matter
 1. Purpose
   Naming is the cheapest form of information hiding available to a systems engineer. A well-chosen name communicates intent and establishes boundaries without introducing any mechanical indirection, latency, or layers of abstraction. This chapter separates naming decisions that carry actual architectural weight from naming conventions that are purely stylistic. It establishes that a name is a contract with the reader, and picking the right one is fundamental to code organization.
 2. Prerequisites
   Assumes mastery of the Ubiquitous Language concept and bounded contexts (Part II, Ch 13) and an understanding of physical file and module boundaries (Part IV, Ch 27).
This chapter does not cover file and module naming (expanded in Part IV, Ch 27), the granular rules of abstraction design (expanded in Part IV, Ch 31), or the use of comments as an alternative communication mechanism (expanded in Part IV, Ch 30).
 3. Key Concepts
 * Naming as Information Hiding: A name can either expose the mechanical implementation of a variable, or it can hide it behind the business intent.
 * Signal-to-Noise Ratio: The amount of descriptive text in a name should correlate directly to the size of its lexical scope.
 * Style vs. Substance: Distinguishing between conventions that prevent runtime or cognitive errors versus conventions that exist solely for aesthetic uniformity.
[Intention-Revealing Names vs. Implementation-Revealing Names]
What it is:
The choice of whether an identifier's name describes the business domain concept it represents or the underlying data structure and mechanics it utilizes.
Why it exists:
Code is written once but read thousands of times. When reading code, the engineer's cognitive limit is constrained by how much they must translate. If names mirror the implementation, the engineer must mentally reverse-engineer the business intent every time they read the file. A name should match the domain's own vocabulary (the Ubiquitous Language).
Options:
 1. Implementation-Revealing Names: Naming variables after their structural type (e.g., userListArray, fetchFromPostgres(), redisCacheMap).
 2. Intention-Revealing Names: Naming variables after their systemic purpose (e.g., activeUsers, authenticateUser(), sessionState).
Trade-offs:
 * **Implementation-Revealing Names:** Provides immediate mechanical context to the developer. You know exactly what methods are available on userListArray. However, it completely destroys information hiding. It couples the caller to the specific data structure.
 * **Intention-Revealing Names:** Perfectly preserves abstraction and aligns with the business domain. The underlying mechanics can be completely rewritten without renaming the caller's variables. However, it requires the developer to rely on the compiler or IDE to understand the memory footprint or available methods of the object.
When to choose each option:
 * Choose **Intention-Revealing Names** universally for 99% of application code, business logic, and API boundaries.
 * Choose **Implementation-Revealing Names** strictly at the absolute outer edges of the architecture—inside the infrastructure adapters where the implementation *is* the domain (e.g., a specific postgresDriver variable inside a database connection pooler).
Common failure modes:
 * **The Refactoring Ripple:** An engineer names a variable customerListArray. Six months later, performance dictates that the array must become a HashSet to guarantee uniqueness. Because the data structure was hardcoded into the name, changing the type requires a cascading rename operation across 50 dependent files, turning a localized performance optimization into a massive, risky pull request.
Example:
The Extreme Programming (XP) and Clean Code traditions rigorously mandate intention-revealing names. An XP codebase will define find_eligible_customers() rather than iterate_customer_records(). The former hides the search mechanism; the latter leaks that a linear scan is occurring.
[Scope-Proportional Naming]
What it is:
The convention determining the length and descriptiveness of a variable name relative to how long that variable exists in the code.
Why it exists:
Names carry cognitive weight. A 30-character variable name is highly descriptive, but if it is used six times within a tight three-line mathematical loop, the visual noise obscures the logic of the algorithm.
Options:
 1. Uniform Verbosity: Every identifier, regardless of scope, is given a fully descriptive, multi-word name (e.g., loopIndex, databaseConnectionError).
 2. Scope-Proportional Naming: Identifiers in very small, localized scopes use single letters or abbreviations (i, err, db), while identifiers at the package or global scope use fully descriptive names.
Trade-offs:
 * **Uniform Verbosity:** Ensures absolute clarity when a snippet of code is read entirely out of context (e.g., in a diff tool). However, it clutters tight algorithmic blocks, making the code read like a legal contract rather than a mathematical formula.
 * **Scope-Proportional Naming:** Drastically improves the scan-ability of local logic. The visual focus remains on the operators and control flow. However, it requires the reader to hold the local context in their head; if the scope grows too large (e.g., a 200-line function), the reader forgets what c stands for.
When to choose each option:
 * Choose **Scope-Proportional Naming** universally. Short names are for short scopes; long names are for long scopes.
 * Never choose **Uniform Verbosity** for transient loop variables or error-checking blocks.
Common failure modes:
 * **The Global Abbreviation:** An engineer falls in love with short names and defines a package-level variable var c *Client. Because it is globally scoped, every other function in the package uses it. When a new engineer tries to introduce a Cache object, catastrophic collision and confusion occur.
Example:
Idiomatic Go heavily enforces Scope-Proportional Naming. It is a deliberate, considered convention by Go's maintainers, not laziness. In Go, it is standard practice to use err for local error checking and i for loop counters. This contrasts sharply with Java's cultural tendency toward extreme verbosity at every scope (e.g., ArrayIndexOutOfBoundsException exception), which often forces Java developers to rely heavily on IDE auto-completion just to write a simple loop.
[Semantic Encoding vs. Type System Reliance]
What it is:
The decision of whether to encode metadata about a variable (its type or state) directly into its prefix or suffix.
Why it exists:
Historically, compilers were dumb. In weakly typed languages, a developer needed to know if a memory address held a string or an integer, so they encoded that information into the name itself.
Options:
 1. Systems Hungarian Notation: Prefixing the mechanical data type to the variable name (e.g., strName for a string, intAge for an integer, pUser for a pointer).
 2. Semantic Predicate Naming: Prefixing booleans with verbs that ask a question (e.g., isActive, hasPermission, canExecute).
 3. Type-Agnostic Naming: Providing the pure domain noun and relying entirely on the compiler (e.g., name, age).
Trade-offs:
 * **Systems Hungarian Notation:** Provides immediate, raw type visibility without an IDE. However, it is an egregious violation of information hiding. It duplicates the work of the type system and creates massive churn when types change.
 * **Semantic Predicate Naming:** Carries real disambiguating value. It instantly clarifies that a variable represents a binary state (True/False) rather than an object or a string, without tying it to a specific memory layout.
 * **Type-Agnostic Naming:** Clean, purely domain-focused, and highly resilient to refactoring. However, it completely relies on the language having a strong static or gradual type system to prevent developers from doing math on a string.
When to choose each option:
 * Choose **Semantic Predicate Naming** universally for all boolean variables and functions that return booleans. The is/has/can/should prefixes are the few naming conventions that carry active correctness value.
 * Choose **Type-Agnostic Naming** for everything else in a modern, typed language.
 * Never choose **Systems Hungarian Notation** unless you are writing unsafe C code without a modern IDE.
Common failure modes:
 * **The Liar Variable:** An engineer using Hungarian notation in a loosely typed language names a variable intAge. Years later, a requirement changes to support fractional ages, and the variable is updated to hold a floating-point number. The name intAge is never updated. The convention has degraded from being redundant to being actively malicious.
Example:
Hungarian notation originated in the C-era Windows API (e.g., lpszName for a long pointer to a null-terminated string) out of absolute physical necessity. Today, it is redundant cargo-culting. If you use TypeScript, Rust, Go, or Java, the compiler already tracks the type. Adding IUser for an interface or strEmail is an anti-pattern. However, using is_active remains a universally accepted convention across all languages because it communicates domain logic, not memory layout.
Why Smart Engineers Disagree: Arbitrary Style vs. Correctness
Engineers will spend hours debating whether a codebase should use camelCase, snake_case, or PascalCase. They will write long architectural documents arguing that snake_case is objectively more readable, or that camelCase is more modern.
The Reality: The systems engineer recognizes that these are pure style choices. They carry zero correctness value, they do not hide information, and they do not prevent bugs. The human brain's pattern recognition adapts to any consistent formatting in roughly three days.
The only failure mode regarding casing is inconsistency. A codebase that mixes camelCase and snake_case arbitrarily forces the developer to memorize the casing of every individual function, increasing cognitive load. The pragmatic engineer enforces a standard via an automated linter on day one, and then explicitly bans any further debate on the topic.
This strict separation of style from substance ties directly into the broader concept of code documentation (expanded in Part IV, Ch 30). If a block of code requires a sprawling comment to explain what it is doing, the naming has failed entirely. Extracting that block into a single function named isEligibleForDiscount() eliminates the need for the comment. Good naming makes the code self-documenting; arbitrary style arguments only distract from the architecture.
