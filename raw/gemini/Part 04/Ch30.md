Chapter 30 — Comments: What to Comment, What Not To
 1. Purpose
   A comment is a maintenance liability. It is the only artifact in a codebase that must be kept in sync with the logic it describes, but structurally lacks any compiler enforcement to guarantee that it is. This chapter establishes the strict conditions under which that ongoing maintenance cost is worth paying. It defines the boundary between comments that act as critical systemic context and comments that are merely band-aids for poorly structured code.
 2. Prerequisites
   Assumes an understanding of naming as the primary vehicle for information hiding (Part IV, Ch 28). This chapter explicitly differentiates localized in-code commentary from large-scale architectural documentation.
This chapter does not cover repository-level Architectural Decision Records (ADRs) used for historical "why" decisions (expanded in Part VI, Ch 45), nor does it cover the deep mechanics of generating API documentation websites (expanded in Part VIII, Ch 64–68).
 3. Key Concepts
 * **The Compiler Blindspot:** Because comments are ignored by the parser, they are uniquely vulnerable to silent degradation over time.
 * **Comment Rot:** A comment that no longer matches the underlying code. It is strictly worse than no comment at all, as it actively misleads the reader.
 * **The Invisible Context:** Code defines the mechanical execution of a system; it cannot easily express the external, physical, or historical constraints that forced the code to be written that way.
[The WHY vs. WHAT Distinction]
What it is:
The central heuristic for determining if an inline comment is justified. It categorizes comments into those that restate the code's mechanics ("What") and those that explain the invisible constraints surrounding the code ("Why").
Why it exists:
Reading code is cognitively taxing. If a comment merely translates a programming language into English, it adds zero new information while doubling the maintenance burden. Code tells you *what* happened; only a comment can tell you *why* it had to happen that way.
Options:
 1. Restating Mechanics (The "What"): Explaining the operational steps of the code (e.g., // increment index by 1).
 2. Capturing Context (The "Why"): Explaining the non-obvious invariant, hardware quirk, or external bug that the code is working around (e.g., // delay 50ms to allow the external controller time to reset before querying).
Trade-offs:
 * **Restating Mechanics:** Incurs infinite maintenance cost with absolutely no cognitive benefit to a competent engineer.
 * **Capturing Context:** Preserves critical, invisible systemic knowledge that prevents future regressions. However, it requires discipline to write accurately and must be ruthlessly deleted if the external constraint ever changes.
When to choose each option:
 * Choose **Capturing Context ("Why")** whenever the code relies on a non-obvious invariant, handles a lock-ordering requirement, circumvents a known bug in a third-party API, or utilizes an unintuitive algorithmic optimization.
 * Never choose **Restating Mechanics ("What")**. If the mechanics of the code are so complex that they require an English translation to be understood, the code itself must be refactored.
Common failure modes:
 * **The Stale Lie:** An engineer writes a comment explaining exactly *what* a complex block of code does. Six months later, a bug fix alters the code's behavior. The engineer updates the code but ignores the comment. The next maintainer reads the comment, trusts it, and wastes three days debugging a physical reality that contradicts the English text.
Example:
The Linux kernel coding style documentation enforces this discipline aggressively. Kernel developers are explicitly instructed never to explain *how* or *what* the code does in inline comments. Comments in the kernel are reserved exclusively for genuinely non-obvious systemic realities, such as documenting memory barrier requirements or explaining physical hardware timing quirks that a reader of the C code could not possibly deduce on their own.
[Extraction vs. Explanation]
What it is:
The structural choice between writing a comment to summarize a dense block of procedural code, versus extracting that exact block into a well-named function.
Why it exists:
Engineers instinctually write comments acting as headers or summaries for logical chunks of operations within a larger function. This is a missed opportunity for actual structural organization. As established in Chapter 28, naming is a native abstraction; commenting is not.
Options:
 1. Comment the Block: Leaving the procedural code inline and adding a summary comment above it (e.g., // 1. Verify user is eligible for discount).
 2. Extract to Function: Moving the block into a new function whose name is the exact text of the intended comment (e.g., isEligibleForDiscount()).
Trade-offs:
 * **Comment the Block:** Fast to write and requires no physical file modification or new stack frames. However, it masks the underlying structural failure: a function that is doing too many things. It also fails to create a testable boundary for that specific logic.
 * **Extract to Function:** Eliminates the maintenance burden of the comment entirely, as the compiler now enforces the name. It isolates the logic into a purely testable unit. However, it slightly increases indirection and the total function count of the module.
When to choose each option:
 * Choose **Extract to Function** almost universally when you feel the urge to write a comment explaining what a specific block of logic accomplishes.
 * Choose **Comment the Block** only when dealing with a hyper-optimized, contiguous mathematical algorithm that would suffer catastrophic performance penalties or lose its conceptual cohesion if physically split into separate function calls.
Common failure modes:
 * **The Header Comment Monolith:** A 300-line function is visually divided into sections using comments like // --- FETCH DATA ---, // --- PARSE DATA ---, and // --- SAVE DATA ---. The comments are desperately trying to provide the visual organization that physical function extraction was designed to solve.
Example:
In the Clean Code tradition, a comment explaining a block of code is explicitly treated as a code smell indicating a missed extraction. If an engineer writes if (employee.flags & HOURLY_FLAG && employee.age > 65), they are forced to comment it. By extracting it to if (employee.isEligibleForPension()), the comment becomes obsolete, and the code becomes self-documenting through its structure.
[Implementation Comments vs. Contractual Docstrings]
What it is:
The architectural distinction between comments aimed at the internal maintainer of a function (inline comments) and comments aimed at the external consumer of an interface (docstrings/API comments).
Why it exists:
An API consumer should never be forced to read the implementation source code to understand how to interact with a system boundary. Contractual comments define the public surface area; implementation comments explain the private mechanics. Conflating the two destroys information hiding.
Options:
 1. Implementation Commentary (Inline): Standard comments living inside the function body, ignored by external tooling.
 2. Contractual Documentation (Docstrings): Specially formatted comments attached to the declaration of a module, class, or function, often parsed by tools to generate IDE tooltips or static sites.
Trade-offs:
 * **Implementation Commentary:** Highly localized and contextual. However, it provides zero value to the consumer of the API, who cannot see it without opening the source file.
 * **Contractual Documentation:** Creates a strict, visible boundary contract (inputs, outputs, side effects, exceptions). It powers the developer experience (DX) via IDE integrations. However, it requires rigid adherence to formatting rules and adds visual bulk to the interface definition.
When to choose each option:
 * Choose **Contractual Documentation** for every public interface, exported function, and shared data structure. You must define the parameters, the expected state changes, and the exact failure modes the caller must handle.
 * Choose **Implementation Commentary** strictly for the localized, private "why" inside the function body (as defined in the first principle).
Common failure modes:
 * **The Leaky Docstring:** A developer documents a public interface, but populates the docstring with internal implementation details (e.g., """ Fetches the user by executing a linear scan over the Redis cache array """). This couples the consumer's mental model to a private detail. When the cache is swapped from Redis to Memcached, the public contract changes purely because the docstring was poorly scoped.
Example:
Python explicitly codifies this distinction via PEP 257. In Python, an inline comment uses the # symbol and is discarded by the parser. A docstring uses triple quotes ("""...""") directly below a declaration. Crucially, the docstring is not discarded; it is attached to the object at runtime as the __doc__ attribute. Python structurally acknowledges that the public contract of a function is fundamentally different from the internal commentary of its maintainer.
Why Smart Engineers Disagree: The "Self-Documenting Code" Fallacy
A persistent dogma in modern software engineering is the absolute mandate for "self-documenting code." Proponents of this view argue that any comment is a failure of expressiveness. If your code needs a comment, they argue, your naming is bad, your functions are too long, or your abstractions are wrong. Consequently, they attempt to ban all comments from the repository.
Engineers optimizing for systemic maintainability view this zero-tolerance policy as naive. They argue that while perfect naming eliminates the need to explain the *what* and the *how*, it can never explain the *why*. Code executes in a hostile physical universe, interfacing with undocumented third-party APIs, idiosyncratic hardware, and legacy databases.
The Reality: The systems engineer knows that a bug in an external vendor's API is not "self-documenting." A temporary workaround for a race condition in a deprecated network switch cannot be expressed purely through clever variable names. The pragmatic engineer writes code that is structurally clear enough to render all mechanical comments obsolete, fiercely reserving the right to use comments to document the invisible constraints of reality that the compiler cannot see. Banning comments entirely does not make the code cleaner; it merely erases the system's history.
