Chapter 27 — File and Module Structure
 1. Purpose
   This chapter dictates how architectural theory translates into physical files and directories. Part I and Part II established the conceptual rules of coupling, cohesion, and dependency inversion. However, those principles are useless if they are not mapped to the file system. This chapter answers the question that previous chapters explicitly deferred: how do you physically organize the interfaces and implementations produced by dependency inversion and hexagonal architecture? It establishes that directory structure is not a matter of aesthetic preference; it is the physical enforcement mechanism for architectural boundaries.
 2. Prerequisites
   Assumes mastery of the foundational constraints from Part I and Part II. Specifically, readers must understand accidental versus essential complexity (Part I, Ch 02), information hiding versus simple encapsulation (Part I, Ch 04), and the mechanics of afferent and efferent coupling (Part I, Ch 03). Readers must also possess a deep understanding of Hexagonal Architecture / Ports and Adapters (Part II, Ch 11) and the Dependency Inversion Principle (Part II, Ch 12).
This chapter does not cover naming conventions for files and identifiers (expanded in Part IV, Ch 28), the granular rules for splitting a single concept across multiple files (expanded in Part IV, Ch 29), or the practice of writing unsafe/low-level systems code across FFI boundaries (expanded in Part IV, Ch 33).
 3. Key Concepts
 * **Physical Boundaries vs. Logical Boundaries:** A logical boundary is an interface in code; a physical boundary is a directory or package. If the physical boundary does not enforce the logical boundary, the architecture will inevitably rot.
 * **Package-by-Layer:** Organizing a codebase by the technical role of the files (e.g., all controllers in one folder, all database queries in another).
 * **Package-by-Feature:** Organizing a codebase by the business domain concept (e.g., all billing logic, billing controllers, and billing queries in one folder).
 * **The Dependency Rule as a File Constraint:** The architectural mandate that source code dependencies must point inward can be physically enforced by restricting which directories are allowed to import other directories.
[Package-by-Layer vs. Package-by-Feature]
What it is:
The primary organizational decision for an application's root directory structure. It dictates whether files are grouped by what they *are* technically, or by what they *do* for the business.
Why it exists:
Code organization optimizes for the engineer's cognitive load during a change. When a business requirement changes, the engineer must modify code. The directory structure determines whether those modifications are localized to a single folder or scattered across the entire repository.
Options:
 1. Package-by-Layer: Directories are named after technical frameworks or architectural layers (e.g., src/controllers/, src/services/, src/repositories/).
 2. Package-by-Feature: Directories are named after bounded contexts or domain aggregates (e.g., src/orders/, src/billing/, src/inventory/).
Trade-offs:
 * **Package-by-Layer:** Maximizes technical cohesion (all HTTP parsing logic is in one folder). However, it completely destroys domain cohesion. Adding a single new "User Profile" feature requires touching files in controllers/, services/, and repositories/, forcing the developer to mentally juggle the entire repository layout to complete one logical task. It practically guarantees that distinct domain concepts will bleed into each other, as they share the same technical directories.
 * **Package-by-Feature:** Maximizes domain cohesion. All logic required to execute a "Billing" use case is physically co-located in the billing/ directory. Refactoring a domain concept is safe and localized. However, it can result in minor duplication of technical boilerplate (e.g., HTTP error formatting) across multiple feature packages.
When to choose each option:
 * Choose **Package-by-Feature** universally for 99% of business applications, microservices, and product code. It has won broad industry consensus because it aligns the physical codebase with the axes of business change.
 * Choose **Package-by-Layer** strictly for isolated framework development, generic platform code, or low-level network drivers where the "domain" genuinely is the technical role (e.g., building a custom ORM where parsing, executing, and mapping *are* the features).
Common failure modes:
 * **The Controller God-Folder:** In a Package-by-Layer structure, the controllers/ directory grows to contain 150 files spanning 20 different, unrelated business domains. Developers lose the ability to reason about the system's overall capabilities because the domain is completely obscured by technical grouping.
Example:
Idiomatic Go communities fiercely reject Package-by-Layer. A Go repository structured with models/, controllers/, and utils/ is considered a canonical anti-pattern by the language's maintainers. Because Go utilizes package-level visibility rules, grouping unrelated models in a models/ package destroys information hiding and virtually guarantees circular dependency compiler errors when a User model inevitably needs to reference an Order model, and vice-versa.
[Physical Placement of Ports and Adapters]
What it is:
The literal directory mapping for the Hexagonal Architecture (Ports and Adapters) concepts established in Chapter 11. It defines where interfaces live and where the infrastructure that implements those interfaces lives on the disk.
Why it exists:
Dependency Inversion dictates that high-level policy must not depend on low-level details. If the Core domain package and the PostgreSQL adapter are kept in the exact same directory, a developer will eventually import a PostgreSQL-specific struct directly into the business logic. Physical directory segregation allows the compiler to enforce the dependency rule.
Options:
 1. Flat Hexagon: The core logic, the ports (interfaces), and the adapters (implementations) are all files sitting alongside each other in a single feature directory.
 2. Segregated Core and Adapters: The feature directory contains a strictly isolated core/ package containing business logic and ports. Adapters live in separate sibling directories (e.g., postgres/, http/) that explicitly import the core/.
Trade-offs:
 * **Flat Hexagon:** Highly pragmatic for small services. Minimizes directory nesting and import boilerplate. However, it relies entirely on human discipline during code reviews to ensure the core business logic does not accidentally import infrastructure files.
 * **Segregated Core and Adapters:** Mathematically enforces the Dependency Rule. Because postgres/ imports core/, the compiler will physically prevent core/ from importing postgres/ (as this would trigger a circular import). However, it adds significant file nesting and structural boilerplate for simple CRUD operations.
When to choose each option:
 * Choose **Segregated Core and Adapters** universally for long-lived systems, critical financial engines, and any codebase touched by more than a handful of engineers. The compiler is a better enforcer of boundaries than a code review checklist.
 * Choose **Flat Hexagon** only for throwaway prototypes or hyper-focused microservices under 1,000 lines of code.
Common failure modes:
 * **The Bleeding Adapter:** A team adopts Hexagonal Architecture logically, but puts everything in one package. An engineer imports a third-party AWS SDK exception directly into the core PaymentService.java file. Because there is no physical package boundary to stop them, the architecture silently degenerates into a tightly coupled Big Ball of Mud.
Example:
Rust enforces this pattern aggressively at the language level via its module system. A Rust project will define a core module containing public traits (Ports). It will define a separate infrastructure module containing the database adapters. Using explicit pub use re-exports, the infrastructure module can hide the concrete adapter implementations completely, ensuring the rest of the application can only interact with the core traits. This is information hiding enforced by the physical module tree.
[Circular Dependency Avoidance]
What it is:
The structural reality of two physical packages attempting to import each other (Package A imports Package B, and Package B imports Package A).
Why it exists:
Circular dependencies are rarely a compiler quirk; they are almost universally a symptom of a failed architectural boundary. If two packages must mutually depend on each other, they are highly connascent. A concept was split across a physical boundary that it should never have crossed, or a shared foundational primitive was left entangled in a higher-level feature.
Options:
 1. Package Merging: Collapse the two mutually dependent packages into a single package.
 2. Primitive Extraction: Identify the shared state or interface causing the cycle and extract it downward into a third, independent package that both A and B can safely import.
Trade-offs:
 * **Package Merging:** The fastest, most direct solution. It correctly acknowledges that the two concepts are actually highly cohesive and belong together. However, it increases the total volume of the resulting package, potentially violating the single-responsibility focus of the original design.
 * **Primitive Extraction:** Preserves the independent identities of Package A and Package B while eliminating the cycle. However, it increases the total number of packages in the system, adding indirection and cognitive load.
When to choose each option:
 * Choose **Package Merging** when the cycle is caused by overlapping domain concepts (e.g., the orders package imports the invoices package to calculate a total, and invoices imports orders to read the line items). They are fundamentally the same lifecycle boundary and belong in one feature package.
 * Choose **Primitive Extraction** when the cycle is caused by a shared mechanical interface or foundational type (e.g., both packages need to pass a generic RequestContext struct back and forth).
Common failure modes:
 * **The Lazy Init Hack:** In languages that permit runtime circular imports (like Python or Node.js), engineers avoid fixing the structural boundary and instead move the import statement inside the function body to delay resolution. This masks the architectural failure and frequently results in unpredictable runtime deadlocks or uninitialized state crashes in production.
Example:
The Go compiler famously refuses to compile code containing circular dependencies. While developers often complain about this restriction, it forces them to confront poor physical boundaries immediately. If package users imports package billing, and package billing imports package users, the Go compiler physically halts the build, forcing the systems engineer to either merge them into a single domain package or extract the shared interfaces into a third, independent package.
Why Smart Engineers Disagree: The "Utils" God Package
The most frequent disagreement regarding code organization is what to do with cross-cutting, shared logic. Engineers optimizing for rapid deduplication (often adherents to dogmatic DRY principles) will create a common/, shared/, or utils/ directory. They argue this is a highly pragmatic place to put date formatters, string manipulators, and generic math functions so that all feature packages can consume them instantly without duplicating code.
Engineers optimizing for architectural cohesion view the utils/ directory as a catastrophic failure of domain modeling. They argue that utils/ is a "god package" — a garbage dump of unrelated functions that guarantees high afferent coupling (everything depends on it) and low internal cohesion (the date formatter has nothing to do with the string parser).
The Reality: The systems engineer understands that utils is a symptom of lazy naming, not a valid architectural layer. If code is genuinely shared across boundaries, it deserves an explicit, purpose-built package that accurately describes its function. You do not create a utils/ package; you create a currency/ package, a timeparse/ package, or a metrics/ package. By giving the shared code a concrete physical boundary and an explicit name, you prevent it from accumulating unrelated garbage and protect the physical integrity of the repository.
