Chapter 11 — Layered, Hexagonal, and Ports-and-Adapters Architecture

**1. Purpose**
This chapter covers the primary structural patterns for organizing code within a single service or application boundary. While Chapter 10 addressed when to split a system into distributed components, this chapter addresses how to structure the internal codebase of whichever component you are building. It explains the mechanical problem each internal architectural pattern solves, establishing that the structural organization of code is fundamentally an exercise in managing the direction of dependencies.

**2. Prerequisites**
Assumes mastery of Part I. Specifically, readers must understand the mechanics of afferent and efferent coupling, and the distinction between mere encapsulation and true information hiding as defined in Chapter 04. This chapter does not cover the mechanics of dependency inversion (expanded in Part II, Ch 12), API surface design (expanded in Part II, Ch 15), or module file structure within a repository (expanded in Part IV, Ch 27).

**3. Key Concepts**

* **Internal Architecture:** The structural rules governing how code modules interact within a single memory space and deployment unit.
* **Dependency Direction:** The single most critical vector in software architecture. If Module A calls Module B, Module A is coupled to Module B's volatility.
* **Domain Isolation:** The concept that core business logic should have zero awareness of the delivery mechanism (HTTP, gRPC, CLI) or the storage mechanism (SQL, NoSQL, memory).
* **Ports and Adapters:** The vocabulary for isolating the domain. Ports are the interfaces defined by the business logic; adapters are the infrastructure implementations that fulfill them.

### [The Layered (N-Tier) Architecture]

**What it is:** The classic, sequential organization of an application into horizontal tiers, typically Presentation (UI/API) $\rightarrow$ Business Logic $\rightarrow$ Data Access.

**Why it exists:** It is intuitively simple and directly maps to the chronological flow of a standard web request. A request enters the controller, the controller calls a service, the service calls a database repository, and the data flows back up.

**Options:**

1. **Strict Layering:** A layer can only call the layer immediately below it. The Presentation layer cannot bypass the Business Logic layer to query the database directly.
2. **Relaxed Layering:** Higher layers are allowed to bypass intermediate layers to call foundational layers directly, reducing boilerplate for simple CRUD operations.

**Trade-offs:**

* **Strict Layering:** Maintains strict operational cohesion. However, it forces engineers to write redundant "pass-through" methods in the business layer for simple reads, increasing code volume and accidental complexity.


* **Relaxed Layering:** Maximizes developer velocity for simple operations. However, it scatters business rules; if the Presentation layer queries the Data layer directly, authorization and validation logic meant to live in the Business layer are bypassed.

**When to choose each option:**

* Choose Layered Architecture for simple CRUD applications, tightly coupled internal pipelines, and rapid prototypes where the business rules are trivial.
* Default to Strict Layering to maintain a predictable control flow, dropping to Relaxed Layering only for read-only analytical queries.

**Common failure modes:**

* **Layer Leakage:** The most frequent failure of layered architectures. Because the dependencies point downwards, the Business layer structurally depends on the Data layer. Over time, infrastructure concepts bleed upward. SQL exceptions bubble into the API layer, and database ORM annotations pollute the core domain entities. The intended information hiding fails, exposing the underlying paradigms.



**Example:** Django’s MTV (Model-Template-View) is a highly successful, pragmatic realization of layered architecture. The View (controller) processes the HTTP request, invokes the Model (which handles both business logic and data access), and renders the Template. It is fiercely optimized for velocity, accepting the structural trade-off that business logic and database access are fundamentally intertwined.

### [The Hexagonal (Ports-and-Adapters) Architecture]

**What it is:** Introduced by Alistair Cockburn in 2005, this pattern places the business logic (the "Core") at the absolute center of the architecture. The Core defines interfaces ("Ports") for anything it needs to do. External infrastructure (databases, UI, message queues) wrap the Core by implementing those interfaces ("Adapters").

**Why it exists:** To completely isolate the business logic from infrastructure volatility. This is Information Hiding applied architecturally. The Core knows nothing about HTTP, JSON, SQL, or Kafka. It only knows about its own domain rules.

**Options:**

1. **Driving (Primary) Adapters:** Infrastructure that initiates the action (e.g., a REST API Controller, a CLI command, an event listener). They map external input into a format the Core's Port accepts.
2. **Driven (Secondary) Adapters:** Infrastructure that the Core utilizes to perform side effects (e.g., a PostgreSQL repository, a third-party payment gateway). They implement the Ports the Core requires.

**Trade-offs:**

* **Hexagonal Isolation:** Mathematically forces domain logic to be framework-agnostic. Allows massive internal refactoring without breaking consumers. Crucially, it allows the entire business domain to be unit-tested in milliseconds using fast, in-memory mock adapters.


* **The Indirection Tax:** Extricating business logic from infrastructure is expensive. It requires defining explicit interface contracts (Ports) for every external interaction and maintaining complex dependency injection graphs. It introduces massive upfront cognitive overhead for simple applications.

**When to choose each option:**

* Choose Hexagonal Architecture when building complex, core business systems whose domain rules will outlive the current database technology or delivery framework.
* Do not choose Hexagonal Architecture for simple data-entry applications or ephemeral microservices where the cost of the indirection tax outweighs the domain complexity.



**Common failure modes:**

* **Empty Core:** Engineers implement the heavy interface boilerplate of Hexagonal Architecture for an application that has zero actual business logic (e.g., a service that simply saves JSON payloads to a database). The architecture becomes purely accidental complexity, providing no isolation value while destroying readability.

**Example:** A complex enterprise REST API transitioned from a Layered structure to a Hexagonal structure. Instead of the `BillingService` calling the `StripeClient` directly (Layered), the `BillingService` now calls a `PaymentProcessor` interface (the Port). The `StripeClient` (the Adapter) implements that interface. The business logic is insulated; if the company migrates to Adyen, the Core remains untouched. Spring Boot allows for both: it defaults to a standard Layered configuration, but experienced teams utilize Spring's dependency injection to strictly wire a Hexagonal "Clean Architecture."

### [Shared Models vs. Strict Boundary Mapping]

**What it is:** The mechanical strategy for moving data structures between architectural layers or across Port/Adapter boundaries.

**Why it exists:** Data must travel from the database, through the business logic, and out to the network. Engineers must decide whether a single data structure traverses these boundaries, or if the data is explicitly translated into a new structure at every boundary crossing.

**Options:**

1. **Shared Model:** A single class or struct represents the data end-to-end. The same object queried from the database is passed through the business logic and serialized into the HTTP response.
2. **Strict Boundary Mapping:** Each layer defines its own distinct data shape. The database uses an ORM Entity, the business logic uses a Domain Model, and the presentation layer uses an API Data Transfer Object (DTO). Data is mapped between them at the boundaries.

**Trade-offs:**

* **Shared Model:** Highly efficient to write. Eliminates hundreds of lines of mapping boilerplate. However, it creates catastrophic coupling via Connascence of Type and Value. A change to a database column instantly alters the JSON API payload, risking downstream breakages.


* **Strict Boundary Mapping:** Establishes rigid interfaces that protect downstream consumers. The API contract is mathematically decoupled from the database schema. However, it requires writing tedious mapper functions and allocates significantly more memory as objects are constantly duplicated.



**When to choose each option:**

* Choose Shared Models in Layered architectures where rapid delivery is prioritized and the API is only consumed by a tightly coupled, internal frontend.
* Choose Strict Boundary Mapping in Hexagonal architectures and any service exposing a public API, where an accidental contract change would cause cascading systemic failure.



**Common failure modes:**

* **The God Model:** An engineer uses a Shared Model in a complex system. Over time, the `User` class accumulates database ORM annotations, JSON serialization tags, GraphQL schema directives, and core validation logic. It becomes a massive, fragile bottleneck where a change to support a new UI feature breaks the database write path.



**Example:** In a Hexagonal REST API, a user creation request enters as a `CreateUserRequest` JSON DTO. The driving adapter maps it to a plain `User` Domain Model and passes it through the Port to the Core. The Core applies business rules, then passes the `User` Domain Model to the driven repository adapter. The repository maps it to a `UserEntity` to execute the PostgreSQL insert. The boundaries are mathematically enforced through explicit mapping.

### Why Smart Engineers Disagree: Indirection vs. Simplicity

The most persistent architectural debate within a service boundary is the friction between isolation purists and pragmatic minimalists.

Engineers optimizing for system longevity and testability (often proponents of "Clean Architecture" or Hexagonal) view the framework and the database as hostile dependencies. They argue that infrastructure is inherently volatile. They gladly pay the heavy tax of mapping layers, DTOs, and interface indirection to ensure their business rules are utterly pure and can be unit-tested without spinning up a Docker container.

Engineers optimizing for delivery and simplicity argue that "swapping out the database" is a myth. They point out that in the real world, a company almost never migrates a mature system from PostgreSQL to MongoDB. Therefore, writing dozens of interfaces to protect against a migration that will never happen is the literal definition of speculative future-proofing and unnecessary accidental complexity. They prefer the Layered approach, happily coupling their logic to their SQL framework to ship faster.

**The Reality:** The disagreement is resolved by evaluating the essential complexity of the business logic.

If the application is a basic data-entry tool where the primary challenge is moving forms to tables, the Hexagonal architecture is a massive, unwarranted tax. The database *is* the domain. Conversely, if the application is a tax-calculation engine with hundreds of compounding, overlapping rules, testing that logic against a live database will make the test suite impossibly slow and brittle. The systems engineer understands that the true value of Hexagonal architecture is not the theoretical ability to change databases; it is the concrete, daily ability to execute thousands of complex business-rule tests in sub-millisecond isolation.
