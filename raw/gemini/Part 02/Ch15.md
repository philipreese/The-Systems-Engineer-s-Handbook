Chapter 15 — API Surface Design: What to Expose, What to Hide

**1. Purpose**
This chapter applies the concept of information hiding directly to the network boundary. Every field, parameter, and operation exposed in an API is a permanent liability—a structural commitment you must honor in perpetuity or break at a massive coordination cost. This chapter establishes the architectural discipline of minimal surface area: designing APIs not by asking what data is available, but by deciding what data you are willing to maintain forever.

**2. Prerequisites**
Assumes mastery of Part I. Specifically, readers must understand the distinction between information hiding and encapsulation (Chapter 04), the mechanics of afferent coupling (Chapter 03), and the reality that the wrong abstraction is worse than no abstraction. This chapter does not cover versioning strategies for evolving an existing API (expanded in Part II, Ch 16), the REST vs. RPC vs. event-driven transport choice (expanded in Part III, Ch 19), authentication and authorization at the boundary (expanded in Part III, Ch 24), or the internal module file structure of the API implementation (expanded in Part IV, Ch 27).

**3. Key Concepts**

* **The API as a Liability:** Every element of an API surface is cheap to add but exponentially expensive to remove once consumers exist.
* **Minimal Surface Area:** The practice of exposing only what the caller strictly requires, deliberately hiding internal database schemas and state fields to prevent connascence of value.
* **Progressive Disclosure:** Designing the API such that the most common use cases require the simplest possible payloads, while advanced capabilities remain hidden behind optional parameters.
* **The Proximity of the Consumer:** The stability guarantees of an API are dictated entirely by whether the consumer is internal (coordinated deployment is possible) or external (coordination is impossible).

### [The Principle of Minimal Surface Area]

**What it is:** The strict architectural mandate to hide all internal state and capabilities by default, exposing only the specific fields a consumer explicitly needs to fulfill a business requirement.

**Why it exists:** APIs are the structural manifestation of afferent coupling. If you expose a field, a downstream system will inevitably build a dependency on it. If you expose an internal database identifier merely because the ORM generated it automatically, you are now permanently prevented from changing your database schema without breaking the consumer.

**Options:**

1. **Internal State Passthrough:** The API directly serializes and returns the internal domain model or database entity.
2. **Explicit Boundary Mapping:** The API utilizes a dedicated Data Transfer Object (DTO) that contains a strictly reduced, intentionally designed subset of the internal data.

**Trade-offs:**

* **Internal State Passthrough:** Extremely fast to write. Eliminates mapping boilerplate and accelerates day-one delivery. However, it completely destroys information hiding. It couples the consumer directly to the provider’s internal persistence strategy.
* **Explicit Boundary Mapping:** Mathematically protects the internal implementation, allowing the database to be radically refactored without altering the API contract. However, it incurs the indirection tax of maintaining translation layers and duplicated data structures.

**When to choose each option:**

* Choose Internal State Passthrough only for tightly coupled internal administrative scripts or rapid prototypes where the consumer and provider are deployed simultaneously by the same developer.
* Choose Explicit Boundary Mapping universally for any production API with distinct consumers.

**Common failure modes:**

* **The Leaky Database Column:** A developer adds a `deleted_at` timestamp column to the database for internal auditing. Because the API uses State Passthrough, this field accidentally leaks into the public JSON response. A third-party client begins using the presence of `deleted_at` to filter records. The provider can now never change their soft-delete database strategy without breaking the external client.

**Example:** The POSIX file descriptor API (`open`, `read`, `write`, `close`) is a masterclass in minimal surface area. It has remained stable for 50 years precisely because it exposes almost nothing. It forces the consumer to deal with simple byte arrays, completely hiding the massive underlying complexity of file systems, block storage, and hardware interrupts.

### [Progressive Disclosure of Capabilities]

**What it is:** An API design strategy where the default, baseline request is radically simple, but complex capabilities are available via optional, layered parameters for advanced users.

**Why it exists:** Systems must serve both the 90% of consumers who need a simple default and the 10% of power users who need granular control. If you force the 90% to understand the parameters required by the 10%, you create massive cognitive overload and guarantee integration errors.

**Options:**

1. **The God Resource (Flat API):** A single endpoint that accepts dozens of parameters and always returns a massive, deeply nested payload containing every possible related entity.
2. **Progressive Disclosure (Layered API):** An endpoint that requires minimal input and returns a flat, concise response by default, relying on query parameters (e.g., `?expand=`) or distinct sub-routes for advanced relationships.

**Trade-offs:**

* **The God Resource:** Minimizes the total number of network round-trips (reducing the latency tax). However, it maximizes parsing overhead, wastes massive amounts of bandwidth transmitting unused data, and makes the API documentation nearly impossible to read.
* **Progressive Disclosure:** Provides an exceptional developer experience and keeps the baseline response blazing fast. However, if a client actually needs the complex data, they are forced to either make multiple $N+1$ network calls or learn a complex expansion syntax.

**When to choose each option:**

* Choose The God Resource only for heavy, asynchronous data-science extraction pipelines where the goal is raw data transfer, not human integration.
* Choose Progressive Disclosure universally for public REST APIs and SDKs.

**Common failure modes:**

* **Configuration Bankruptcy:** An endpoint begins simple, but over years accumulates 15 different optional boolean flags (`include_history`, `skip_validation`, `dry_run`). Because these flags interact in undocumented, non-deterministic ways, the endpoint becomes a brittle nightmare that is impossible to safely test.

**Example:** Stripe's API heavily utilizes progressive disclosure. When you query a `Charge` object, the default response is minimal and flat. The `customer` field simply returns a string ID (e.g., `cus_123`). This keeps the API incredibly fast. However, if an advanced client needs the full customer object, they do not hit a different "God" endpoint; they simply append `?expand[]=customer`, and Stripe layers the complex nested object into the response.

### [Internal vs. External API Boundaries]

**What it is:** The strategic distinction in how an API is designed and evolved based entirely on the organizational proximity of its consumers.

**Why it exists:** An API is a contract. If both parties of the contract work in the same building, the contract can be renegotiated and deployed synchronously. If one party is an unknown third-party enterprise on another continent, the contract is immutable.

**Options:**

1. **Aggressive Evolution (Internal API):** The provider deprecates fields, changes payloads, and forces downstream consumers to update their code to match the new reality.
2. **Strict Immutability (External API):** The provider never removes a field or alters an existing behavior. All new features are strictly additive.

**Trade-offs:**

* **Aggressive Evolution:** Prevents the accumulation of tech debt. The codebase remains clean, and the provider does not have to maintain legacy adapter layers. However, it weaponizes Conway’s Law, forcing constant, synchronous cross-team coordination to manage breaking changes.
* **Strict Immutability:** Builds immense trust with consumers. Integrations written a decade ago still function flawlessly. However, the provider absorbs all the accidental complexity. The backend codebase becomes a museum of legacy translation layers and backward-compatibility hacks.

**When to choose each option:**

* Choose Aggressive Evolution for internal microservice-to-microservice communication where you maintain full visibility into every downstream caller.
* Choose Strict Immutability for public-facing SaaS APIs, mobile application backends (where users refuse to update the app), and third-party enterprise integrations.

**Common failure modes:**

* **The Unannounced Deprecation:** A team treats a public API like an internal one. They remove a "rarely used" field from a JSON payload to clean up their database. Thousands of third-party integrations instantly crash, permanently destroying the company's engineering reputation.

**Example:** GraphQL was designed by Facebook to handle internal API boundaries. It allows the consumer to dynamically select exactly which fields they want. This is a brilliant internal optimization to prevent over-fetching. However, when exposed as a public External API, GraphQL forces the provider to support an infinite number of query permutations, making it notoriously difficult to optimize database execution plans or guarantee strict rate limiting against third parties.

### Why Smart Engineers Disagree: Consumer-Driven Flexibility vs. Provider-Driven Control

The most polarizing debate in API surface design is whether the consumer or the provider should dictate the shape of the data.

Engineers optimizing for frontend velocity (often proponents of GraphQL) argue for consumer-driven flexibility. They argue that strict REST APIs are rigid bottlenecks. Every time the UI needs a new field, the frontend team must open a Jira ticket and wait two weeks for the backend team to add it to the API surface. By exposing a flexible graph, the frontend can query whatever it needs instantly, eliminating organizational friction.

Engineers optimizing for backend stability and performance (often proponents of strict REST or gRPC) argue for provider-driven control. They understand that a database cannot efficiently answer arbitrary, infinitely nested queries. They argue that consumer-driven APIs are leaky abstractions that inevitably result in $N+1$ query disasters and database CPU exhaustion, because the backend loses the ability to perfectly tune the execution path for a known, rigid request shape.

**The Reality:** The disagreement is a classic application of the Theory of Constraints. If the organizational bottleneck is UI iteration speed, consumer-driven flexibility (GraphQL) is the correct architectural choice. If the system bottleneck is database IOPS or strict data security, provider-driven control (REST) is mandatory. The systems engineer understands that exposing flexibility to the consumer mathematically requires absorbing complexity in the provider. You must choose where the complexity is cheaper to maintain.
