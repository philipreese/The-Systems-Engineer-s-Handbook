Chapter 20 — Resource Modeling
 1. Purpose
   This chapter dictates how to structurally model business entities once a resource-oriented transport (REST or resource-shaped RPC) has been selected. While Chapter 19 established the wire-level mechanics of the transport choice, this chapter addresses the structural shape of the payloads and URIs themselves. A resource is not a remote procedure call masquerading behind a URL; it is a stable, addressable noun with defined identity and state. This chapter covers how to map volatile business domains into stable resource hierarchies, how to handle state transitions without resorting to RPC verbs, and how to model relationships without destroying network throughput or cache viability.
 2. Prerequisites
   Assumes mastery of the foundational constraints from Part I and Part II. Specifically, readers must understand the mechanics of information hiding and the distinction between encapsulating data and concealing volatile design decisions. Readers must also understand the principle of minimal API surface area and how progressive disclosure prevents payload bloat, as well as the structural dangers of Connascence of Value across network boundaries.
This chapter does not cover the initial transport decision (expanded in Part III, Ch 19), pagination strategies for collections (expanded in Part III, Ch 23), error response structures (expanded in Part III, Ch 21), or idempotency guarantees for state mutations (expanded in Part III, Ch 22).
 3. Key Concepts
 * The Resource Noun: An entity defined by its business value to the consumer, not by its structural layout in the provider's database.
 * Domain-Driven Routing: Designing URI paths that reflect the consumer's mental model of the business workflow, isolating the API contract from internal infrastructure volatility.
 * The Action Problem: The structural limitation of standard CRUD verbs (GET, POST, PUT, PATCH, DELETE) when modeling complex, multi-step business state transitions.
 * Sub-Resource Actions: The deliberate, pragmatic use of POST requests appended to resource URIs to execute explicit state machine transitions without breaking the noun-based architecture.
[Domain Nouns vs. Implementation Schemas]
What it is: The architectural mandate to define URIs and resource payloads based on the external business domain, intentionally obscuring the underlying relational database schema.
Why it exists: To protect the consumer. If an API simply serializes internal database tables over HTTP, the API contract is permanently coupled to the provider's storage engine. If the provider normalizes a table, the API contract breaks.
Options:
 1. Schema-Driven Resources: URIs map 1:1 with internal database tables (e.g., exposing /users, /user_roles, and /user_preferences as distinct endpoints).
 2. Domain-Driven Resources: URIs map to aggregated business concepts (e.g., exposing a single /accounts resource that internally aggregates data from multiple highly normalized tables).
   Trade-offs:
 * Schema-Driven Resources: Maximizes initial developer velocity, as ORM entities can be serialized directly to JSON with zero mapping boilerplate. However, it completely destroys information hiding. It forces the external consumer to understand the provider's internal relational algebra and execute multiple network requests to rebuild the business entity.
 * Domain-Driven Resources: Mathematically isolates the external contract from internal storage volatility, allowing the provider to radically refactor their database without breaking clients. However, it requires maintaining explicit translation layers (Data Transfer Objects) and incurs CPU overhead to aggregate and map data at runtime.
   When to choose each option:
 * Choose Domain-Driven Resources universally for any external, public-facing API or cross-boundary microservice contract.
 * Choose Schema-Driven Resources strictly for rapid, internal administrative tooling where the consumer and the database are maintained by the exact same engineering team.
   Common failure modes:
 * The Normalization Leak: A database engineer normalizes a shipping_address out of the orders table to optimize storage. Because the API uses Schema-Driven Resources, the API suddenly stops returning the address in the /orders response, breaking millions of mobile clients that expected the unified domain noun.
   Example: Stripe's Charge object is a triumph of Domain-Driven Resources. When a client fetches a Charge, they receive a single, cohesive JSON noun. Internally, Stripe must query legacy banking gateways, fraud-detection models, and dozens of heavily sharded relational tables to construct that noun. The client remains blissfully ignorant of this internal chaos.
[Hierarchical Nesting Limits]
What it is: The structural constraint applied to URI path depth when modeling resources that logically belong to other resources (e.g., an Item within an Order within a User).
Why it exists: To prevent URIs from devolving into rigid, brittle traversal paths. A resource's identity should stand on its own. Deep nesting permanently couples the identity of a resource to a single specific access pattern.
Options:
 1. Deep Nesting: URIs reflect the entire relational ownership chain (e.g., /users/{user_id}/orders/{order_id}/items/{item_id}).
 2. Shallow Nesting (Max ~2 Levels): URIs are nested only to indicate collection scoping, but individual resources are accessed via flat, globally unique identifiers (e.g., /users/{user_id}/orders to list, but strictly /orders/{order_id} and /items/{item_id} to retrieve).
   Trade-offs:
 * Deep Nesting: Provides absolute contextual clarity in the URL and mathematically guarantees that a client cannot fetch an item without also proving they know the user ID. However, it creates brittle access patterns. If an Item is eventually allowed to belong to multiple Orders, the URI structure completely collapses.
 * Shallow Nesting: Maximizes routing flexibility and provides stable, persistent identifiers that survive business logic changes. However, it requires the underlying database to rely on globally unique identifiers (UUIDs) rather than composite primary keys, and forces authorization logic to be handled explicitly in the business tier rather than implicitly via the URL path.
   When to choose each option:
 * Choose Shallow Nesting universally as the default standard for REST API design. Stop nesting after one level of dependency.
 * Choose Deep Nesting only if a child resource mathematically cannot exist without the parent and strictly relies on a sequential integer ID scoped to that parent (e.g., /orders/123/lines/1).
   Common failure modes:
 * The Traversal Trap: A client application receives an alert containing an item_id. Because the API enforces deep nesting, the client cannot simply fetch the item. It must execute expensive, brute-force search queries against the parent /orders endpoints just to construct the valid URL required to fetch the child resource.
   Example: GitHub's API relies heavily on valid, shallow nesting. Because an "issue" is fundamentally scoped to a repository, the URI /repos/{owner}/{repo}/issues is structurally correct. However, GitHub stops there. They do not force you to fetch a specific comment via /repos/{owner}/{repo}/issues/{issue_number}/comments/{comment_id}. Once the entity has global identity, it is accessed directly or via minimally scoped paths.
[The Action Problem (State Machine Transitions)]
What it is: The design pattern for handling complex, multi-step business mutations—such as "canceling an order," "refunding a payment," or "publishing a draft"—that do not map cleanly to standard HTTP PUT (full replace) or PATCH (partial update) verbs.
Why it exists: HTTP verbs are infrastructural; business processes are domain-specific. Forcing complex state transitions into generic CRUD verbs obscures the business intent and makes executing strict side-effects dangerously unreliable.
Options:
 1. Generic State Patching: The client modifies the resource by sending a generic mutation (e.g., PATCH /orders/123 { "status": "canceled" }).
 2. Sub-Resource Action Endpoints: The provider exposes explicit verbs appended to the resource noun (e.g., POST /orders/123/cancel).
 3. Shapeless Command Endpoints: The provider abandons the resource routing entirely and exposes global RPC-style endpoints (e.g., POST /cancelOrder).
   Trade-offs:
 * Generic State Patching: Architecturally pure according to strict REST semantics. However, it forces the server to diff the incoming payload against the database to infer the business intent. The explicit "cancel" command is lost, risking that critical side-effects (like refunding the credit card) are bypassed if the state machine logic is flawed.
 * Sub-Resource Action Endpoints: Preserves the resource-oriented URL structure while providing an explicit, unambiguous command trigger for complex state machines. However, it litters the REST API with non-noun path segments, offending architectural purists.
 * Shapeless Command Endpoints: Maximizes operational clarity for the specific command. However, it quietly turns the entire API into an RPC system with extra steps, destroying the predictability and cacheability of the resource model.
   When to choose each option:
 * Choose Sub-Resource Action Endpoints (POST as the verb of last resort) universally for state machine transitions, complex domain actions, and any mutation that triggers asynchronous side effects.
 * Choose Generic State Patching strictly for basic, literal data-entry updates (e.g., changing a user's display name).
   Common failure modes:
 * The Implicit Side-Effect: An engineer relies on state patching. A client sends PATCH /invoices/1 { "status": "paid" } to update the UI. The backend database updates successfully, but because the explicit intent was obscured, the backend fails to trigger the integration with the external Payment Gateway. The invoice is marked paid, but the capital is never actually collected.
   Example: Stripe's PaymentIntent API models the complex lifecycle of a transaction as a strict state machine. You do not PATCH a payment to force it through security checks. You explicitly call POST /v1/payment_intents/{id}/capture or POST /v1/payment_intents/{id}/cancel. The resource is the noun, but the state transitions are explicit sub-resource verbs.
[Resource Relationships (Embed vs. Reference)]
What it is: The structural choice of how a resource represents data that belongs to a separate, distinct domain entity (e.g., how an Order JSON payload represents the User who placed it).
Why it exists: To balance the mechanical cost of network round-trips against payload bloat and cache invalidation complexity.
Options:
 1. Referencing: The parent resource includes only the unique identifier or full URI of the related resource (e.g., "user_id": "usr_123").
 2. Embedding: The parent resource serializes the entire JSON object of the related resource directly into its own payload.
   Trade-offs:
 * Referencing: Keeps payloads minimal, ensuring high throughput and preserving strict cache invalidation boundaries (if the User changes, the Order cache remains perfectly valid). However, it forces the client to execute subsequent, synchronous API calls (the N+1 network problem), introducing massive cumulative latency.
 * Embedding: Provides an exceptional developer experience by delivering all required data in a single network round-trip. However, it results in catastrophic payload bloat, wastes massive amounts of bandwidth transmitting redundant data, and makes cache invalidation mathematically impossible without centralized distributed locking.
   When to choose each option:
 * Choose Referencing as the absolute default for all cross-domain entities and public APIs.
 * Choose Embedding only when the child resource is structurally bounded by the parent and has no standalone lifecycle (e.g., embedding line_items inside an Invoice), or when the client explicitly requests the embedded data via Progressive Disclosure parameters (e.g., ?expand=user).
   Common failure modes:
 * The Accidental God Object: An engineer optimizes for the UI by embedding relations by default. Fetching a single Comment embeds the Author, which embeds the author's Company, which embeds the company's BillingHistory. A request for a 10-word text string silently generates a 4MB JSON payload, exhausting the database's connection pool and crippling network bandwidth.
   Example: GitHub’s API strictly relies on Referencing to protect its infrastructure. A request to fetch a pull request returns flat IDs and explicit hypermedia links ("commits_url": "https://api.github.com/..."). The client must explicitly traverse the reference to fetch the related data, protecting GitHub's databases from accidental infinite-depth serialization loops.
Why Smart Engineers Disagree: Purity vs. Pragmatism in Action Verbs
The most persistent argument in resource modeling pits REST purists against pragmatic systems engineers regarding the "Action Problem."
Engineers optimizing for architectural purity will argue that *everything* can and must be modeled as a noun. If a business requires a user to cancel an order, the purist will refuse to implement POST /orders/123/cancel. Instead, they will mandate the creation of a new, abstract entity: POST /cancellations { "order_id": 123 }. They argue this preserves the strict noun-verb uniformity of HTTP, perfectly aligns with REST constraints, and allows the "cancellation" itself to be tracked as an independent resource.
Engineers optimizing for consumer velocity and operational clarity view this as architectural gymnastics. They argue that a cancellation is an action, not an entity. Forcing a verb to masquerade as an abstract noun ("a cancellation") creates opaque, highly confusing developer experiences. It forces clients to manage the lifecycles of manufactured entities that only exist to satisfy architectural dogma.
The Reality: The systems engineer optimizes for the consumer's mental model and systemic safety. While creating a Cancellation noun is technically pure, it is usually accidental complexity. A sub-resource action endpoint (POST /orders/123/cancel) is semantically imperfect, but it is operationally superior. It retains the primary resource (Order) as the focal point, provides a mathematically unambiguous trigger for internal side-effects, and drastically reduces the cognitive load on the consuming developer. You optimize the boundary for the human reading the code, not for the philosophical purity of the transport protocol.
