Chapter 22 — Idempotency
 1. Purpose
   Part II Chapter 17 established that across heterogeneous distributed boundaries, at-least-once delivery is the only physically realistic messaging default. Exactly-once delivery is a localized illusion. This chapter dictates how to survive that reality. Idempotency is the mechanical pattern that makes at-least-once delivery harmless. It does not prevent the network from sending a request twice; it guarantees the system only applies the business side effects once. This chapter covers how to construct natural idempotency, how to design idempotency keys for unsafe mutations, and how to safely cache and expire those keys at scale.
 2. Prerequisites
   Assumes mastery of the foundational constraints from Part I and Part II. Specifically, readers must understand the reality of partial failure in distributed environments, the at-least-once delivery semantics of asynchronous message brokers, and the fundamental temporal coupling decisions established in Chapter 17. Readers must also understand the error handling contracts defined in Chapter 21.
This chapter does not re-derive the impossibility of exactly-once delivery across network boundaries (assume it is impossible), it does not cover the Saga pattern or compensating actions for distributed rollbacks (expanded in Part II, Ch 17), and it does not cover the structural format of error responses (expanded in Part III, Ch 21).
 3. Key Concepts
 * **Idempotency:** The mathematical property of an operation whereby applying it multiple times yields the exact same state as applying it once (f(f(x)) = f(x)).
 * **Natural Idempotency:** Operations that are structurally idempotent by their very definition, allowing clients to safely retry them without coordination.
 * **Designed Idempotency:** The introduction of synthetic identifiers (Idempotency Keys) to force non-idempotent operations to behave idempotently.
 * **The Retention Window:** The explicit operational boundary defining how long a server is obligated to remember a synthetic identifier before a duplicate request is treated as a new request.
[Natural vs. Designed Idempotency]
What it is:
The architectural distinction between operations that are inherently safe to retry based on their standard HTTP verb semantics, and operations that require explicit, engineered protection to prevent duplicate execution.
Why it exists:
Networks drop packets. When a client sends a request, and the network drops the server's acknowledgment, the client has no way of knowing if the server processed the request. The client *must* retry. If the API is not designed with idempotency in mind, that default network retry will corrupt the system state.
Options:
 1. Natural Idempotency (PUT, DELETE, GET): The operation targets a specific, existing identity and replaces or removes its state. Repeating PUT /orders/123 { "status": "canceled" } fifty times results in the exact same database state as doing it once.
 2. Designed Idempotency (POST): The operation instructs the server to create a new entity or execute a side effect without prior identity. Repeating POST /orders { "item": "laptop" } fifty times results in fifty laptops being shipped unless an explicit Idempotency Key is provided.
Trade-offs:
 * **Natural Idempotency:** Requires zero backend tracking, zero storage overhead, and allows clients to implement "dumb," aggressive retry loops safely. However, it severely restricts API design; the client must already know the ID of the resource (often requiring an extra network round trip to fetch or generate it) before they can mutate it.
 * **Designed Idempotency:** Provides an exceptional developer experience, allowing the client to safely execute complex, multi-step creations in a single POST request. However, it forces the provider to build and maintain complex, highly concurrent state-tracking mechanisms to catch the duplicates.
When to choose each option:
 * Choose Natural Idempotency universally for reads, replacements, and deletions. Standard HTTP clients and service meshes will automatically retry GET, PUT, and DELETE requests precisely because the protocol guarantees they are safe.
 * Choose Designed Idempotency for any POST request that initiates a state machine transition, charges capital, or triggers asynchronous side effects.
Common failure modes:
 * **The Unsafe Default Retry:** A mobile client library defaults to retrying all HTTP timeouts. The backend implements checkout via a standard, non-idempotent POST /checkout. The user enters a tunnel, the connection drops while waiting for the response, the library retries, and the user is charged twice for the same cart.
Example:
In standard web architecture, DELETE /users/456 is naturally idempotent. If the user is deleted on the first request, the second request might return a 404 Not Found instead of a 200 OK, but the *state of the system* (the user is gone) remains identical. The retry is harmless.
[The Idempotency Key Implementation]
What it is:
The mechanical pattern of the client generating a globally unique identifier (e.g., a UUID v4) for a logical operation, sending it in an HTTP header, and the server utilizing a strict database constraint to prevent duplicate execution while returning the original response.
Why it exists:
To provide the exact safety of exactly-once delivery semantics over an inherently at-least-once network. By shifting the generation of the unique ID to the client, the server can definitively distinguish between "the user clicked checkout twice" and "the network retried the first checkout request."
Options:
 1. Application-Level Check (Read-Before-Write): The application code executes a SELECT to see if the key exists. If not, it executes the business logic and INSERTs the key.
 2. Database-Level Constraint (Unique Index): The application relies entirely on the ACID guarantees of the relational database. It attempts an INSERT into an idempotency_keys table with a UNIQUE constraint. If it violates the constraint, the database rejects it atomically.
Trade-offs:
 * **Application-Level Check:** Easy to write in the framework layer and avoids touching the database schema. However, it is structurally flawed. It introduces a massive race condition (Time-of-Check to Time-of-Use). Two concurrent retries will both SELECT, both see no key, and both execute the charge.
 * **Database-Level Constraint:** Mathematically guarantees safety against race conditions by offloading the serialization to the storage engine. However, it requires writing the idempotency key and the resulting response payload in the same transactional boundary, increasing database write volume and coupling the API response layer to the persistence layer.
When to choose each option:
 * Choose the Database-Level Constraint universally for financial transactions, state machine mutations, and any designed idempotency endpoint.
 * Never choose Application-Level Read-Before-Write checks in a distributed, concurrent system.
Common failure modes:
 * **The Lost Response:** An engineer correctly implements the UNIQUE database constraint, preventing the double charge. However, when the duplicate request hits the constraint, the server simply returns 400 Bad Request or 409 Conflict. The client, which retried because it *never saw the first response*, is now trapped. It doesn't know if the operation succeeded or failed.
Example:
Stripe's API is the textbook standard for this pattern. The client includes an Idempotency-Key: <UUID> header in a POST request. Stripe attempts to insert this key into a heavily sharded PostgreSQL database with a UNIQUE constraint. If the insert succeeds, Stripe processes the payment, saves the resulting JSON response alongside the key, and returns the response. If the client retries the same key due to a network timeout, the UNIQUE constraint violation triggers Stripe to skip the payment processing, fetch the cached JSON response from the first attempt, and return it. The client perceives a seamless success.
[The Retention Window]
What it is:
The explicit, documented time limit defining how long the server will store an idempotency key and its associated cached response before purging it from the database.
Why it exists:
Storage is finite. An API processing millions of mutations a day cannot permanently store every idempotency key and every JSON response payload forever. The server must eventually reclaim that disk space, which means it must eventually "forget" that an operation occurred.
Options:
 1. Infinite Retention: The idempotency key is permanently linked to the created resource in the primary operational table (e.g., adding an idempotency_key column to the orders table).
 2. Finite Retention Window (e.g., 24 Hours): Keys and responses are stored in a dedicated, high-throughput table (like a Redis cluster with TTLs, or a PostgreSQL table with a daily cleanup cron job) and are aggressively purged after a set window.
Trade-offs:
 * **Infinite Retention:** Guarantees absolute mathematical safety; a retry sent five years later will still be caught. However, it permanently bloats the primary domain tables with network-level transport metadata, violating separation of concerns. It also prevents returning the exact original HTTP response (headers, status codes), as only the domain entity is retained.
 * **Finite Retention Window:** Keeps the database lean, allows for specialized, ultra-fast storage of idempotency data, and perfectly caches the exact HTTP response. However, it introduces a terrifying edge case: if a client's retry loop is broken and it retries the exact same payload *after* the window closes, the server will treat it as a brand new request and execute it a second time.
When to choose each option:
 * Choose a Finite Retention Window (typically 24 to 48 hours) as the industry-standard default for API idempotency.
 * Choose Infinite Retention only if the idempotency key maps 1:1 with a core domain identifier (e.g., the client generates the order_id itself as a UUID, effectively merging the idempotency key and the natural identity).
Common failure modes:
 * **The Late Retry Double Charge:** A mobile app places an order with an idempotency key. The server succeeds, but the network drops the connection. The user's phone goes to sleep in an area with zero service. Three days later, the phone connects to WiFi and the background worker blindly retries the exact same HTTP request. Because the server's retention window is 24 hours, it has forgotten the key. The order is processed twice.
Example:
Modern APIs explicitly document their retention windows. Stripe promises to retain idempotency keys for exactly 24 hours. If a client attempts to retry a request after 24 hours with the same key, it is the client's fault, not the provider's, that the side-effect executes again. The contract dictates that network retries must be resolved within the defined operational window.
Why Smart Engineers Disagree: The HTTP Response to a Duplicate Key
The most intense friction in idempotency design occurs over what HTTP status code a server should return when it detects a duplicate request.
Engineers optimizing for architectural purity (strict REST purists) argue that if a client sends a POST request, and the resource has already been created by a previous request, the current request is fundamentally a conflict. They mandate returning 409 Conflict. They argue that returning a 200 OK or 201 Created for a request that did not actually create anything in that specific execution cycle is a lie that obscures the true state of the system from the client.
Engineers optimizing for operational resilience and consumer velocity (pragmatists) argue that returning 409 Conflict entirely defeats the purpose of the idempotency key. The client is only retrying because the network failed; they do not know the first request succeeded. If they receive a 409 Conflict, their automated retry loop breaks, and they must write complex error-handling logic to execute a GET request just to figure out what happened.
The Reality: The pragmatist is correct. The systems engineer understands that the Idempotency-Key header is a contract that says: "Make sure this happens exactly once, and give me the result." If the operation has already occurred, returning the cached success response (200 OK or 201 Created with the original JSON body) fulfills the contract flawlessly. You do not punish the client for a network partition by forcing them to parse a generic 409 Conflict error; you hide the network's failure behind the idempotency abstraction and let the client proceed as if the network were perfectly reliable.
