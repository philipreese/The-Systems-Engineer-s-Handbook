Chapter 21 — Error Handling Contracts
 1. Purpose
   Error responses are part of the API surface, and every parameter exposed is a permanent liability. They carry the exact same backward-compatibility obligations as success responses; if a consumer relies on a field, modifying it constitutes an outage. An undocumented or inconsistent error shape is a contract violation, even if nobody explicitly calls it "the contract." This chapter establishes how to structurally design machine-readable error responses, how to mathematically separate client faults from server faults to protect automated retry loops, and how to represent the reality of partial failure over the wire without destroying data integrity.
 2. Prerequisites
   Assumes mastery of the foundational constraints from Part I and Part II. Specifically, readers must understand the principle of minimal API surface area and how exposed fields become permanent liabilities, the strict categorization of backward-compatible vs. breaking changes, and the reality of partial failure in distributed environments.
This chapter does not cover idempotency and safe retry mechanics (expanded in Part III, Ch 22), the specific 401-vs-403 authentication and authorization boundaries (expanded in Part III, Ch 24), versioning the error shape itself (which strictly follows the rules of Part II, Ch 16), or general fail-fast reliability principles at the system level (expanded in Part I, Ch 07).
 3. Key Concepts
 * **The Error Contract:** The structural, immutable promise that failures will be reported in a predictable, parseable format.
 * **Actionable Fault Domains:** The categorization of an error based entirely on who is responsible for fixing it—the client (who sent bad data) or the server (which degraded).
 * **Traceability:** The mandate to include correlation identifiers in every failure response, bridging the gap between an external network failure and the provider's internal observability platform.
[Structured Error Responses]
What it is:
The architectural mandate that an API returns a strictly typed, predictably shaped payload detailing the failure, containing a machine-readable code, a human-readable message, and a correlation identifier.
Why it exists:
HTTP status codes are fundamentally imprecise. A 400 Bad Request tells the client they made a structural mistake, but it does not tell them *which* field was malformed, *why* it was malformed, or how to programmatically recover. A structured response bridges this gap, providing automated clients with the explicit metadata required to act.
Options:
 1. Flat String Messages: Returning a simple human-readable string in the body (e.g., "Invalid email format").
 2. Structured Error Objects: Returning a canonical, heavily documented JSON object containing a stable error code, a descriptive message, and a unique trace/correlation ID.
Trade-offs:
 * **Flat String Messages:** Minimizes payload size and is trivial to implement. However, it completely destroys machine actionability. If a client wants to execute specific logic (e.g., prompting the user for a new card only if the error is insufficient_funds), they must parse and regex the human-readable string, which breaks instantly when the provider updates their copywriting.
 * **Structured Error Objects:** Mathematically decouples the human-readable message from the machine-actionable state. However, it expands the API surface area; every specific code or param emitted becomes a permanent part of the contract that must be supported forever.
When to choose each option:
 * Choose Structured Error Objects universally for all machine-to-machine APIs, public SaaS REST APIs, and mobile backends.
 * Choose Flat String Messages only for internal administrative shell scripts or CLI tools consumed exclusively by humans.
Common failure modes:
 * **The "200 OK" Error:** An engineer catches an exception at the framework level but returns an HTTP 200 OK with a payload of { "error": true, "message": "Failed to save" }. This bypasses all standard HTTP-level monitoring, CDN caching rules, and API gateway metrics. The infrastructure reports 100% success while the clients are fundamentally broken.
Example:
Stripe's API error object is the industry standard for structured errors. It contains a type (a broad category like card_error), a stable machine code (insufficient_funds), a human-readable message, and the specific param that caused the failure. RFC 7807 ("Problem Details for HTTP APIs") standardizes this convergence across the industry using standard fields like type, title, status, and detail. Google's API design dictates a similar standard for gRPC, utilizing status, code, and an array of strongly-typed details passed via trailer metadata.
[The Fault Domain Boundary (4xx vs. 5xx)]
What it is:
The strict semantic segregation of network errors into Client Errors (4xx) and Server Errors (5xx).
Why it exists:
To dictate the mechanical behavior of automated client retry loops. If an error is in the client's domain (4xx), retrying the exact same payload will mathematically never succeed; the client must change the request. If an error is in the server's domain (5xx), retrying the exact same payload later might succeed because the provider's degradation may have resolved.
Options:
 1. Generic Error Bucketing: Returning a broad 400 for everything the client does wrong, and a broad 500 for everything else, heavily relying on the JSON payload for distinction.
 2. Semantic Fault Segregation: Strictly mapping the specific failure mode to the appropriate 4xx (e.g., 409 Conflict for state issues, 429 Too Many Requests) or 5xx (e.g., 502 Bad Gateway, 503 Service Unavailable).
Trade-offs:
 * **Generic Error Bucketing:** Reduces the cognitive load on the backend developer writing the exception handler. However, it forces the client to parse the JSON body just to figure out the nature of the HTTP transport failure, increasing client-side complexity and breaking out-of-the-box edge infrastructure routing.
 * **Semantic Fault Segregation:** Perfectly aligns with standard web infrastructure (proxies, load balancers, and service meshes can automatically retry 503s but immediately reject 400s). However, it requires mapping complex, internal domain exceptions to a rigid, decades-old HTTP standard, which often feels like a leaky abstraction.
When to choose each option:
 * Choose Semantic Fault Segregation universally for all RESTful boundaries and public APIs.
 * Never choose Generic Error Bucketing for a true resource-oriented API.
Common failure modes:
 * **The "500 Bad Request" amplification:** An engineer fails to catch a basic data validation exception (e.g., a missing required field). The web framework catches the unhandled exception at the top level and translates it into an HTTP 500 Internal Server Error. The client's automated SDK sees the 5xx, assumes the server is temporarily degraded, and retries the exact same malformed payload 10 times with exponential backoff. This amplifies a simple user error into a localized denial-of-service attack against the provider's infrastructure.
Example:
When a client hits rate limits on GitHub's API, they do not receive a generic 400 Bad Request or 500 Internal Server Error. They receive a mathematically precise 429 Too Many Requests status code. Because the fault domain is explicitly categorized, generic HTTP clients and service meshes know exactly how to handle it (backing off) without needing custom, GitHub-specific parsing logic.
[Partial Failure in Batch Operations]
What it is:
The wire-level representation of an API request that mutates multiple distinct resources simultaneously, where some items succeed and others fail.
Why it exists:
Because distributed systems suffer from partial failures, an operation like POST /users/batch with 100 users might successfully insert 99 but fail on 1 due to a duplicate email. A single HTTP status code cannot accurately represent this split state.
Options:
 1. All-or-Nothing (Atomic): The entire batch is wrapped in a strict database transaction. If one item fails, the entire batch is rolled back, and the API returns a strict 4xx or 5xx.
 2. Per-Item Result (Multi-Status): The API accepts the batch, processes what it can, and returns a 207 Multi-Status (or a 200 OK) with a structured payload explicitly listing the success or failure state of each individual item.
Trade-offs:
 * **All-or-Nothing:** Provides absolute systemic consistency and a trivial client experience (it either worked entirely, or it didn't). However, a single bad item poisons the entire batch. At scale, the probability of at least one item failing approaches 100%, making massive batch insertions mathematically impossible.
 * **Per-Item Result:** Maximizes throughput and isolates failures, guaranteeing that good data is processed. However, it pushes massive accidental complexity onto the client. The client must parse the response, determine exactly which items failed, and manually construct a new, smaller batch of retries for just those items.
When to choose each option:
 * Choose All-or-Nothing for small, tightly coupled domain mutations where the items are logically inseparable (e.g., creating an Invoice and its associated Line Items).
 * Choose Per-Item Result universally for bulk ingestion endpoints, asynchronous webhooks, and massive data synchronization tasks where throughput is the primary constraint.
Common failure modes:
 * **The Poisoned Retry Loop:** An API uses All-or-Nothing for a batch of 100 items. Item 45 fails validation. The API returns 400 Bad Request. The client's crude integration simply retries the entire batch of 100. If the provider's API lacks strict idempotency mechanics, items 1-44 are inserted a second time before item 45 fails the batch again. This continues until the database is flooded with duplicate records.
Example:
Elasticsearch's _bulk API explicitly relies on the Per-Item Result pattern. Because it is designed to ingest massive logs, an entire batch failing because one JSON document is malformed is unacceptable. It returns a 200 OK (the network request succeeded) and provides an errors: true flag in the payload, accompanied by an array detailing the exact HTTP status code and error string for each individual document.
Why Smart Engineers Disagree: HTTP Semantics vs. Payload Pragmatism
The most persistent argument regarding error contracts pits REST purists against RPC pragmatists.
Engineers optimizing for architectural purity (REST advocates) view the HTTP protocol as the application itself. They argue that HTTP status codes are sacred. A 404 Not Found is fundamentally different from a 409 Conflict, and edge infrastructure (proxies, load balancers, CDNs) relies on these distinctions to route, cache, and shed load efficiently. They argue that ignoring status codes neuters the fundamental machinery of the internet.
Engineers optimizing for operational simplicity and consumer velocity (RPC and GraphQL advocates) view this as outdated dogma. They argue that HTTP is merely a dumb transport tunnel. Forcing complex business state transitions into a limited set of status codes defined in 1999 is accidental complexity. They advocate for returning 200 OK for all successfully delivered network payloads, embedding the *actual* business error code inside the structured JSON or Protobuf response. They argue this provides a mathematically perfect, type-safe contract that is infinitely more expressive than an HTTP header.
The Reality: Both approaches are valid, but they cannot be mixed. The systems engineer understands that if you claim to build a public REST API, you are structurally bound to honor HTTP semantics; returning a 200 OK with an error body in a REST API violates the contract and breaks automated infrastructure. However, if you are explicitly building an internal RPC system over HTTP, pushing the error semantics entirely into the payload (like Google's gRPC error model) is a highly cohesive, robust optimization. The failure is not in choosing one or the other; the failure is refusing to commit to the paradigm and creating an API that unpredictably does both.
