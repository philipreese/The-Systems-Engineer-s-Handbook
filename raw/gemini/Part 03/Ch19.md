Chapter 19 — REST vs. RPC vs. Event-Driven
 1. Purpose
   This chapter addresses the structural format of network communication. Part II Chapter 17 established the architectural decision of temporal coupling—whether a system should communicate synchronously or asynchronously and outlined the failure modes of each. This chapter covers the subsequent decision: once you know whether a call is synchronous or asynchronous, what shape does the contract take on the wire? The choice between REST, RPC, and Event-Driven transport mechanics encodes fundamentally different mental models of what an API is built to achieve. They are not interchangeable defaults.
 2. Prerequisites
   Assumes mastery of the foundational constraints from Part I and Part II. Specifically, readers must understand the mechanics of information hiding and how to conceal volatile design decisions. Readers must also understand the principle of minimal API surface area and how exposed fields become permanent liabilities, progressive disclosure and layering API complexity, the distinction between internal and external API stability constraints and how proximity dictates evolution, backward compatibility and the sunset lifecycle, temporal coupling and how blocking network boundaries create fragility, the synchronous vs. asynchronous communication decision and the blast radius of network failures, and the Saga pattern for distributed transaction choreography.
This chapter does not cover resource modeling deep-dives (expanded in Part III, Ch 20), idempotency mechanics and utilizing unique database constraints (expanded in Part III, Ch 22), pagination strategies (expanded in Part III, Ch 23), or authentication and authorization at the boundary which handles security mechanisms (expanded in Part III, Ch 24).
 3. Key Concepts
 * The Wire-Level Paradigm: The semantics and assumptions encoded into the payload and transport protocol, dictating how the caller and callee reason about the action.
 * Resource-Oriented Architecture (ROA): Modeling a system as a collection of distinct nouns (resources) with state, manipulated by a standard, uniform set of verbs.
 * Action-Oriented Architecture: Modeling a system as a collection of verbs (procedures), prioritizing the execution of a business process over the state of a specific entity.
 * Fact-Oriented Payloads: A messaging paradigm where payloads declare an immutable historical truth ("X occurred"), explicitly containing no instructions on what downstream systems should do about it.
[REST (Resource-Oriented Architecture)]
What it is: Representational State Transfer models the business domain as interconnected nouns (resources) identified by URLs (e.g., /users/123/invoices). State mutations are requested using a strictly uniform interface of standard HTTP verbs (GET, POST, PUT, PATCH, DELETE).
Why it exists: To leverage the fundamental mechanics of the World Wide Web. By mapping domain operations to standard HTTP methods, REST perfectly aligns with standard web infrastructure, allowing CDNs, API Gateways, and reverse proxies to cache reads, rate-limit writes, and load-balance traffic natively without deep packet inspection.
Options:
 1. Pragmatic REST (JSON over HTTP): Resources are identified by URLs and manipulated via HTTP verbs, but clients manually construct URLs based on out-of-band documentation (like OpenAPI/Swagger).
 2. Strict HATEOAS (Hypermedia As The Engine Of Application State): The textbook ideal of REST. The server drives the state machine by embedding contextual hypermedia links in every response, telling the client exactly what actions are currently allowed without the client needing prior knowledge of the URL structure.
Trade-offs:
 * Pragmatic REST: Highly legible for human developers and universally supported by every language ecosystem. However, because it rigidly maps to nouns, it forces complex business transactions (e.g., "approve a loan and transfer the funds") into unnatural, manufactured nouns (e.g., POST /loan-approvals).
 * Strict HATEOAS: Theoretically perfect for decoupling clients from server routing logic, allowing the provider to radically change internal URIs without breaking external clients. Practically, it imposes massive cognitive overhead, dramatically bloats JSON payload sizes, and breaks static type generation.
When to choose each option:
 * Choose Pragmatic REST universally for external, public-facing APIs, mobile application backends, and third-party SaaS integrations where human discoverability and ecosystem trust are paramount.
 * Almost never choose Strict HATEOAS. The organizational and computational coordination tax far outweighs the theoretical decoupling benefits.
Common failure modes:
 * The HTTP Verb Abuse: An engineer maps a destructive, non-idempotent action to a GET request (e.g., GET /users/123/delete). Standard web browsers or edge caches pre-fetch the URL to optimize load times, accidentally executing mass data deletion across the architecture.
 * The "God Resource": Attempting to avoid multiple network round trips, an engineer builds a single REST endpoint that accepts massive payloads and returns entirely different data shapes depending on arbitrary query parameters, completely destroying the predictable resource model and creating un-cacheable traffic.
Example: Stripe and GitHub are the canonical triumphs of Pragmatic REST. They expose vast, complex business domains strictly through predictable noun-based URIs. They actively reject HATEOAS in favor of explicit, versioned, heavily documented API contracts.
[RPC (Remote Procedure Call)]
What it is: An action-oriented architecture that allows a client to invoke a function on a remote server as if it were a local function call. It abandons the noun-based limitations of REST in favor of exposing explicit, highly specific verbs (e.g., CalculateCompoundInterest()).
Why it exists: To maximize computational performance, enforce strict mathematical contracts, and optimize for code generation over human legibility.
Options:
 1. gRPC: Google's high-performance RPC framework utilizing Protobuf (Protocol Buffers) for binary serialization and HTTP/2 for multiplexed, streaming transport.
 2. Twirp: A pragmatic RPC framework (originating from Twitch) that utilizes Protobuf for strict schema definitions but transmits over standard HTTP/1.1, avoiding the operational complexity of HTTP/2 load balancing.
Trade-offs:
 * gRPC: Blazingly fast, heavily compresses network payloads, and mathematically enforces backward and forward compatibility at compile time. However, the binary format destroys naive human legibility; it makes basic curl debugging tedious and requires specialized layer 7 load balancers to route the multiplexed HTTP/2 streams correctly.
 * Twirp: Retains the compile-time safety and code generation of Protobuf while easily routing through legacy infrastructure and standard AWS Application Load Balancers. However, it lacks the bidirectional streaming capabilities of gRPC.
When to choose each option:
 * Choose gRPC universally for high-throughput, internal microservice-to-microservice communication where the system is bound by network serialization latency.
 * Choose Twirp when you require strict schema enforcement but your infrastructure team is not yet prepared to operate and debug a full HTTP/2 service mesh.
Common failure modes:
 * The Local Method Illusion: Because the RPC framework generates native client stubs in the developer's IDE (e.g., client.FetchUser(id)), the developer forgets they are crossing a physical network boundary. They place the remote call inside a tight loop, ignoring mechanical sympathy, and inadvertently collapse system throughput via the N+1 network latency tax.
Example: Google's internal Stubby (the predecessor to gRPC) proved that at hyperscale, parsing JSON strings is an unacceptable waste of CPU cycles. By enforcing binary Protobuf RPCs internally, Google ensured that services only communicate via strictly typed, compiled contracts.
[Event-Driven (Fact-Oriented Payloads)]
What it is: The structural shape of messages placed onto asynchronous message brokers. While RPC and REST are fundamentally command-oriented ("Do this action"), Event-Driven payloads are fact-oriented ("This action has already occurred").
Why it exists: To achieve absolute temporal decoupling and invert domain dependencies. The publishing service does not know, and does not care, who consumes the fact or what side effects the consumption triggers.
Options:
 1. Thin Events (Notification / Pointer Events): The payload contains only the identity of the resource and the event type (e.g., { "event": "OrderPlaced", "order_id": "123" }).
 2. Fat Events (Event-Carried State Transfer): The payload contains the entire state of the resource at the moment the event occurred (e.g., { "event": "OrderPlaced", "order_id": "123", "items": [...], "total": 100.00 }).
Trade-offs:
 * Thin Events: Extremely lightweight and guarantees the consumer will never process stale data, because the consumer must immediately make a synchronous API call back to the publisher to fetch the required state. However, this re-introduces massive temporal coupling; if the publisher's API is down, the consumer cannot process the event.
 * Fat Events: Mathematically eliminates temporal coupling. The consumer has everything it needs to execute its side effects without ever querying the publisher. However, it risks eventual consistency collapse; if the message broker is delayed, the downstream system processes a state that may have already been superseded by a newer mutation.
When to choose each option:
 * Choose Thin Events when the data payload is massive (e.g., video processing pipelines) or when strict data privacy/security regulations prevent broadcasting sensitive PII into a global event bus.
 * Choose Fat Events universally when building CQRS read projections or when executing autonomous Saga orchestrations where cross-domain data availability is the primary constraint.
Common failure modes:
 * Commands Disguised as Events: An engineer publishes an event named SendWelcomeEmail to a topic. This is an RPC command fundamentally disguised as an event. The publisher is now tightly coupled to the implementation details of the Email Service. The correct, fact-oriented event shape must be UserRegistered, allowing the Email Service to independently decide to react to it.
Example: Kafka event schemas heavily rely on Fat Events modeled in Avro or Protobuf. In highly decoupled architectures, a PaymentSecured event contains the full transactional metadata, allowing the Fulfillment Service, Analytics Service, and Ledger Service to operate completely autonomously from the Billing API's uptime constraints.
Why Smart Engineers Disagree: The Noun vs. Verb Friction in Complex Domains
The most intense architectural friction regarding API wire design occurs when modeling complex, multi-step business transactions.
Engineers optimizing for ecosystem consistency and public consumption (REST advocates) demand that everything be modeled as a noun. If a business process requires checking fraud, verifying inventory, and securing a payment, they will create an abstract noun—a CheckoutSession—and mutate its state via HTTP PATCH requests. They argue this preserves the predictability of the web architecture and leverages standard HTTP caching and idempotency headers perfectly.
Engineers optimizing for procedural clarity (RPC advocates) view this as architectural gymnastics and accidental complexity. They argue that a checkout process is fundamentally an action, not an entity. Forcing a verb to masquerade as a noun creates opaque, brittle state machines. They advocate for exposing an explicit ExecuteCheckout() RPC endpoint, arguing that it makes the system's execution path immediately obvious to the consuming developer.
The Reality: Both approaches are valid, but they belong on different sides of the network boundary. The systems engineer understands that public interfaces demand the stability and infrastructural compatibility of REST nouns. Internal microservices, however, are frequently tasked with coordinating complex procedural Sagas across distributed boundaries. Inside the perimeter, RPC verbs eliminate the cognitive overhead of mapping procedures to arbitrary nouns, strictly optimizing for performance and mechanical clarity.
