Chapter 24 — Authentication and Authorization Boundaries
 1. Purpose
   This chapter establishes the architectural placement of authentication and authorization boundaries within a distributed system. Part II outlined how to physically decompose a system; this chapter dictates how trust is managed across those new physical perimeters. It answers where identity is validated and where permissions are enforced, without delving into the underlying cryptography or threat modeling.
 2. Prerequisites
   Assumes mastery of the foundational constraints from Part I and Part II. Specifically, readers must understand the principle of minimal API surface area (expanded in Part II, Ch 15), the network latency tax of distributed calls (Part I, Ch 06), and the mechanical reality of coupling across service boundaries.
This chapter does not cover the cryptographic mechanics of protocols, deep threat modeling, or secrets management (expanded in Part XI, Security, Ch 79–84). It does not cover specific implementation details of Role-Based Access Control (RBAC) or Attribute-Based Access Control (ABAC) beyond their architectural placement, nor does it cover general API surface design (expanded in Part II, Ch 15).
 3. Key Concepts
 * **Authentication (Authn):** The verification of identity ("Who are you?"). At the boundary, this is the cryptographic proof that the caller is who they claim to be.
 * **Authorization (Authz):** The verification of permission ("What are you allowed to do?"). At the boundary, this is the domain-specific evaluation of whether the authenticated identity has the right to mutate a specific resource. (A full treatment of Authn vs. Authz policies is expanded in Part XI, Ch 82).
 * **The Trust Boundary:** The perimeter within which a system implicitly trusts the identity claims of incoming requests.
 * **Zero-Trust Architecture:** A structural security model that assumes the internal network is perpetually hostile and demands explicit identity verification at every discrete service boundary, regardless of network location.
[Enforcement Placement: Gateway vs. Service]
What it is:
The structural choice of where to execute the logic that verifies a caller's identity and evaluates their permissions.
Why it exists:
To balance centralized operational control and performance against localized security and defense-in-depth. In a distributed architecture, deciding whether to check identity at the front door or at the destination fundamentally alters the system's vulnerability to internal lateral movement.
Options:
 1. Centralized API Gateway: A single edge proxy handles all authentication, terminates the external request, and passes only trusted identity headers to internal services.
 2. Decentralized Service Enforcement: Every individual microservice natively evaluates the raw identity payload (e.g., verifying a token signature itself) before executing business logic.
Trade-offs:
 * Centralized Gateway: Maximizes developer velocity and operational consistency. Engineers writing microservices do not have to implement auth middleware, and identity validation only costs one CPU cycle at the edge. However, it creates a brittle, implicit trust boundary. If an attacker bypasses the gateway or compromises a single internal service, the entire backend is defenseless because services blindly trust internal network traffic.
 * Decentralized Service Enforcement: Maximizes security by enforcing verification at every hop (Zero-Trust). If one service is compromised, it cannot easily pivot to attack another. However, it incurs massive CPU overhead (verifying cryptographic signatures on every internal network hop) and risks inconsistent security implementations across different autonomous teams.
When to choose each option:
 * Choose Centralized Gateway for Authentication (Authn) in tightly coupled environments or modular monoliths where the internal network is genuinely isolated.
 * Choose Decentralized Service Enforcement universally for Authorization (Authz)—the gateway cannot possibly know if User A is allowed to mutate Resource B in the downstream domain—and for Authn in any modern, zero-trust microservice architecture.
Common failure modes:
 * The Blind Passthrough: An API gateway verifies an external token, strips it, and injects an X-User-Id: 123 HTTP header, passing it to the internal service. A malicious internal developer (or compromised sidecar container) simply makes a direct network request to http://billing-service/invoices with the header X-User-Id: 1 (the admin user). The internal service, blindly trusting the perimeter, executes the command, bypassing all security controls entirely.
Example:
Kong and Envoy are widely used as centralized API gateways to terminate complex external identity flows. However, modern implementations deploy Envoy as a sidecar proxy at the pod level within a Kubernetes cluster. This forces Decentralized Service Enforcement, verifying mutual TLS (mTLS) and token signatures at every single microservice boundary rather than just at the edge.
[The Token Propagation Pattern]
What it is:
The architectural mechanism for maintaining the caller's identity as a single logical request traverses multiple discrete microservices (e.g., Gateway \rightarrow Service A \rightarrow Service B).
Why it exists:
When Service A calls Service B to fulfill a user's request, Service B needs to know who the original user was to enforce strict authorization constraints. If identity is lost across the hop, downstream services are forced to grant god-level access to upstream services.
Options:
 1. Perimeter Trust (Service Identity): Service A authenticates with Service B using its own service-level credentials (e.g., a shared secret or mTLS). Service B trusts Service A completely and performs the requested action on behalf of the user.
 2. Delegated Token Forwarding (User Identity): Service A extracts the original user's signed token and forwards it intact in the Authorization header when calling Service B. Service B mathematically verifies the user's token independently.
Trade-offs:
 * Perimeter Trust: Simple to implement and requires no complex token lifecycle management across internal boundaries. However, it creates a massive security vulnerability: Service B has no idea who the actual user is, meaning it cannot enforce granular user-level authorization (RBAC/ABAC). It must assume Service A already authorized the action.
 * Delegated Token Forwarding: Preserves the strict user context across the entire distributed trace, allowing granular authorization at the deepest layers of the architecture. However, it couples internal service communication to the external token schema and risks token leakage if an internal service logs the payload or forwards it to an untrusted external third-party API.
When to choose each option:
 * Choose Perimeter Trust strictly for administrative, system-level background jobs where there is genuinely no interactive user context (e.g., a nightly billing cron job).
 * Choose Delegated Token Forwarding universally for any call chain initiated by an external user.
Common failure modes:
 * The Confused Deputy: A user asks the "Report Service" to generate a summary. The Report Service uses its own god-level Perimeter Trust credentials to query the downstream "User Service" for all records, intending to filter them locally. Due to a bug or parameter injection in the Report Service, it returns another company's data to the user. The User Service (the deputy) was "confused" into leaking data because it trusted the network perimeter of the Report Service rather than demanding the original user's token.
Example:
OAuth 2.0 and OpenID Connect (OIDC) are the dominant frameworks for delegated auth flows. A client receives a signed JWT, passes it to the gateway, the gateway passes it to Service A, and Service A propagates the exact same JWT to Service B. Service B independently verifies the OIDC signature to guarantee the user identity before returning data, structurally preventing the confused-deputy problem.
[The Identity Representation: Reference vs. Value Tokens]
What it is:
The structural format of the identity credential passed over the wire—whether it is an opaque pointer requiring a lookup, or a self-contained payload.
Why it exists:
To balance the architectural need for immediate, centralized revocation against the brutal latency cost of distributed database lookups on every API request.
Options:
 1. Reference Tokens (Opaque Strings): The token is a random, meaningless string (e.g., a session ID). Every service must make a synchronous network call to the Identity Provider (IdP) to validate it and fetch the user data.
 2. Value Tokens (JWTs): The token is a JSON payload containing the user's identity and permissions, cryptographically signed by the IdP. Services can verify the signature locally without calling the IdP.
Trade-offs:
 * Reference Tokens: Mathematically guarantees immediate revocation. If a user is banned, the next network call to the IdP fails instantly. However, it creates a massive global bottleneck and latency tax, as the central IdP must process a query for every single internal microservice hop.
 * Value Tokens (JWTs): Eliminates the IdP bottleneck and allows microservices to authenticate requests locally in microseconds. However, it introduces the Token Expiry Problem; if a user is banned, their JWT remains cryptographically valid across the internal network until its expiration time is reached, creating an unavoidable window of vulnerability.
When to choose each option:
 * Choose Reference Tokens for highly sensitive financial, healthcare, or administrative mutations where instant revocation is a hard regulatory or business constraint.
 * Choose Value Tokens universally for standard distributed REST APIs, high-throughput RPC systems, and microservice mesh propagation.
Common failure modes:
 * The Unverifiable JWT: An engineer decodes the base64 JSON payload of a JWT to read the user_id inside an internal service, but fails to execute the CPU-intensive step of cryptographically verifying the signature against the IdP's public key. An attacker simply modifies the user_id in their browser, base64 encodes it again, and gains full system access because the service trusts the unverified payload.
Example:
GitHub's Personal Access Tokens are opaque reference tokens; checking their validity requires a round trip to GitHub's servers, ensuring compromised tokens can be killed instantly. Conversely, Kubernetes Service Account Tokens and standard OIDC flows rely heavily on JWTs as value tokens, prioritizing decentralized, low-latency validation within the cluster.
Why Smart Engineers Disagree: Edge vs. Depth Authorization
The most intense architectural friction regarding boundaries occurs over where authorization should physically live.
Engineers optimizing for operational simplicity and perimeter security argue that the API Gateway should handle both Authentication AND Authorization. They want to define routes in Envoy or Kong configuration files like "Only Admins can POST to /billing". They argue this isolates security logic into a single, highly auditable location and prevents unauthorized traffic from ever entering the internal network, saving internal CPU cycles.
Engineers optimizing for domain cohesion and strict zero-trust argue that Authorization is fundamentally a domain concern. An edge gateway might know if a user has the "Admin" role, but it cannot possibly know if User A is the creator of Invoice B without querying the Billing database. They argue that placing authorization rules in the gateway creates massive temporal coupling; the API Gateway team must update their routing config every time the Billing team adds a new domain rule. Therefore, the gateway should only handle Authentication (identity), and Authorization MUST live deep within the business logic of the service.
The Reality: The pragmatic systems engineer splits the difference. The gateway handles coarse-grained Authentication (verifying the JWT signature mathematically) and superficial routing authorization (rejecting expired tokens or invalid global scopes). However, fine-grained domain authorization (can this specific user edit this specific resource?) is mathematically required to live deep inside the service boundary. Only the owning service possesses the domain data necessary to make that decision. You secure the perimeter with identity, but you secure the data with domain logic.
